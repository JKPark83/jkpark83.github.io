# 📚 주목 논문

### 1) GRACE — 지시문을 그래프로 관리하면 장기 신뢰성이 뛴다

- **저자:** Dan C. Hsu, Luke Lu
- **arXiv:** [2607.09175](https://arxiv.org/abs/2607.09175) · cs.AI · 2026-07-10
- **핵심 기여:** 에이전트의 "지속 시스템 지시문"을 평문이 아니라 **타입드 시맨틱 그래프**로 유지(GRACE). 모델·툴·하네스는 고정하고, 운영 경험으로 지시문만 진화시킨다.
- **방법(2–3줄):** 지시문 수정 제안을 해당 노드의 **국소 타입드 이웃** 안에서만 검증 → 승인된 그래프 변경을 배포용 텍스트 체크포인트의 증분 편집으로 재구성. "검증을 국소화하는 구조"와 "누적 지시문을 계속 쓸 수 있게 하는 통합"이 둘 다 필요하다는 결론.
- **결과(검증됨):** τ²-bench 파생 텔레콤 하네스·분포 이동 프로토콜·5회 반복에서 strict reliability(pass³)가 Gemini 2.5 Flash zero-shot **0.091 → 0.673±0.136**. 같은 held-out에서 Gemini 3.1 Pro zero-shot(0.242)도 상회, 평문 HCE baseline은 0.191±0.051에서 멈춤.
- 🛠 **에이전트 빌더 시사점:** 계속 자라는 `CLAUDE.md`/시스템 프롬프트를 통짜 텍스트로 굴리지 말고, **항목을 타입·의존성으로 구조화**하고 수정 시 영향받는 이웃만 검증하는 파이프라인을 만들 것.
- ⚠️ **한계:** 단일 텔레콤 도메인·특정 하네스에서의 결과. 그래프 스키마 설계·유지 비용은 별도.

### 2) Tool-Making — 반복 단계를 "런타임 코드 생성" 말고 "사전 컴파일된 툴"로

- **저자:** Kalle Kujanpää, Ning Liu, Shahnawaz Alam 외
- **arXiv:** [2607.08010](https://arxiv.org/abs/2607.08010) · cs.CL · 2026-07-09
- **핵심 기여:** 프로덕션 에이전트가 매 요청마다 같은 절차 코드를 재생성하며 지연·신뢰성을 낭비하는 문제를, **배포 전 SOP 단계를 검증된 버전드 툴로 컴파일**하는 파이프라인으로 대체.
- **방법:** 툴메이커가 라이브 환경에서 실행 트레이스 수집·백엔드 스키마 관찰 → 후보 툴 생성 → 라벨 케이스로 수리. 런타임엔 에이전트가 툴을 직접 호출하고 필요할 때만 코드 생성으로 폴백.
- **결과(검증됨):** Fulfillment Center 알람 트리아지(44-노드 SOP)에서 툴 호출로 **p50 지연 -42%**. 1,500건 과거 알람에서 반복 단계의 편차를 억제해 **end-to-end 오류율 최대 -53%**. 구조화 verdict 덕에 직접호출 아키텍처로 단순화 시 ablation에서 p50 추가 -62%.
- 🛠 **에이전트 빌더 시사점:** 워크플로에서 **반복되는 결정 단계를 프로파일링**해 상위 몇 개를 스크립트/툴로 굳혀라. 버전드 툴은 감사성도 올리고 스펙 갭·데이터 드리프트도 드러낸다.
- ⚠️ **한계:** 산업용 단일 시스템 사례. SOP가 자주 바뀌면 툴 재컴파일 비용 고려 필요.

![Tool-Making 파이프라인: 런타임 코드 재생성 대비 사전컴파일 버전드 툴, 결과 p50 -42%·오류 -53%·추가 -62%](/images/ai-news-2026-07-14/toolmaking-pipeline.svg)
*런타임 코드 재생성 vs 사전 컴파일 버전드 툴, 프로덕션 결과 (수치: arXiv 2607.08010, 생성 이미지)*

# 🛠 오픈소스 스포트라이트

- **GRACE류(2607.09175) 재현 포인트:** 지시문 저장소를 평문 대신 노드+타입+엣지로 모델링하고, 수정 diff에 대해 "영향받는 이웃 노드"만 재검증하는 eval 게이트를 CI에 붙여보기.
- **Tool-Making류(2607.08010) 재현 포인트:** 에이전트 트레이스에서 반복 툴콜 시퀀스를 집계 → 상위 N개를 결정적 함수로 승격 → 회귀 케이스로 잠그기.

# 🧑‍💻 Codex & Claude Code 실전 팁

### 🟩 Codex — 멀티에이전트 위임을 "필요할 때만" 켜기 (category: Subagent)

- **소스:** [Codex changelog](https://developers.openai.com/codex/changelog)
- 💡 **핵심:** 앱서버 클라이언트에서 멀티에이전트 위임을 `disabled` / `explicit-request-only` / `proactive`로, **스레드·턴 단위**로 제어할 수 있다.
- 📝 **How-to:** 비용·예측성이 중요한 파이프라인은 기본을 `explicit-request-only`로 두고, 명시 요청이 있을 때만 서브에이전트를 스폰하도록 한다.

```text
delegation: explicit-request-only   # 기본은 위임 끔
# "이 작업은 병렬 조사 필요" 같은 명시 신호에서만 subagent 확장
```

- 🎯 **언제:** 서브에이전트가 토큰·지연을 예측 불가하게 늘려 곤란할 때.
- ⚠️ **주의:** `proactive`는 탐색형 작업엔 좋지만 비용 급증. 프로덕션 배치엔 부적합.
- ⭐ **추천:** 기본 보수, 필요 시 턴 단위 승격.

### 🟦 Claude Code — 메모리 파일을 잘게 쪼개 `@import`로 조립하기 (category: Context)

- **소스:** [Claude Code Memory 문서](https://code.claude.com/docs/en/memory)
- 💡 **핵심:** GRACE의 교훈을 그대로 실무에: 거대한 `CLAUDE.md` 하나 대신 **주제별 파일로 분리**하고 `@path`로 임포트해 조립한다. 수정 시 영향 범위가 파일 단위로 국소화된다.
- 📝 **How-to:**

```markdown
# CLAUDE.md
@./.claude/rules/testing.md
@./.claude/rules/api-conventions.md
@./.claude/rules/deploy.md
```

- 🎯 **언제:** 규칙이 수십 개로 불어나 "어디를 고치면 뭐가 깨지는지" 감이 안 올 때.
- ⚠️ **주의:** 임포트 파일도 컨텍스트를 먹는다. 자주 안 쓰는 규칙은 서브에이전트·슬래시커맨드로 분리.
- ⭐ **추천:** 한 파일이 화면 한 페이지를 넘기면 쪼갤 신호.

### 🟪 공통 — 반복 툴콜은 코드 생성 말고 함수로 굳혀라 (category: Workflow)

- **소스:** [arXiv 2607.08010](https://arxiv.org/abs/2607.08010)
- 💡 **핵심:** 매 턴 같은 절차를 LLM이 다시 짜게 두면 지연·편차가 쌓인다. 반복 단계는 **검증된 결정적 함수**로 승격.
- 📝 **How-to:** ①트레이스에서 반복 시퀀스 집계 → ②상위 단계를 함수화(입출력 스키마 고정) → ③회귀 케이스로 잠금 → ④런타임은 함수 우선, 미지의 상황만 LLM 폴백.
- 🎯 **언제:** SOP·정형 워크플로가 있는 프로덕션 에이전트.
- ⚠️ **주의:** 스키마·데이터가 자주 바뀌면 재검증 파이프라인 필수(드리프트가 조용히 오답을 만든다).
- ⭐ **추천:** "3번 이상 반복되는 툴콜 시퀀스"부터 굳히기 시작.
