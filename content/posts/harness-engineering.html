한 분석에 따르면, Claude Code 같은 코딩 에이전트의 코드베이스에서 모델에게 무엇을 시킬지 정하는 'AI 로직'이 차지하는 비중은 아주 작고, 대부분은 권한 검사·컨텍스트 관리·도구 연결·복구 처리 같은 주변 인프라입니다.

— 즉, 에이전트의 실력은 모델이 아니라 그 모델을 감싼 하네스에서 갈린다는 뜻입니다.

요즘 "AI 에이전트를 만든다"는 말을 자주 듣습니다. 그런데 막상 좋은 모델(GPT, Claude, Gemini…)을 API로 불러 보면 당황하게 됩니다. 모델은 질문을 받으면 글을 써서 돌려줄 뿐, 파일을 읽지도, 명령어를 실행하지도, 어제 한 일을 기억하지도 못하니까요. "이걸로 어떻게 코딩을 대신 시키지?" 싶은 거죠.

그 간극을 메우는 것이 이 글의 주제, 하네스(harness)입니다. 비유하자면 모델은 엔진입니다. 엔진은 강력하지만, 엔진만으로는 도로를 달릴 수 없습니다. 섀시, 바퀴, 핸들, 계기판, 브레이크, 안전벨트 — 이 나머지 전부를 합쳐야 비로소 "자동차"가 되고 사람을 태우고 달립니다. 그 '나머지 전부'에 해당하는 것이 하네스이고, 그걸 잘 설계하는 일이 하네스 엔지니어링(harness engineering)입니다.

이 글은 그 분야를 기초부터 심화까지 한 바퀴 돌아봅니다. 에이전트가 아직 낯선 주니어 개발자도 따라올 수 있게, 비유와 짧은 코드부터 시작합니다.

1부 · 기초

1. 하네스란 무엇인가 — 엔진과 자동차의 차이

먼저 용어를 맞추겠습니다. 여기서 말하는 하네스는 전기 배선 뭉치나 테스트 하네스가 아니라, LLM을 감싸서 '실제로 일하는 에이전트'로 만들어 주는 런타임 시스템 전체를 가리킵니다. Anthropic은 이를 "복잡한 작업을 모델이 수행할 수 있게 해주는 구조적 틀(framework)"이라고 설명합니다.

왜 이게 필요한지는, 모델 혼자서 할 수 있는 일과 못 하는 일을 나눠 보면 분명해집니다.

모델은 입력(프롬프트)을 받아 출력(텍스트)을 내놓는 함수에 가깝습니다. 파일을 읽으려면 누군가 파일을 읽어다 텍스트로 넣어 줘야 하고, 모델이 "이 명령을 실행해 줘"라고 말해도 실제로 실행하고 결과를 다시 넣어 주는 누군가가 필요합니다. 그 '누군가'가 전부 하네스입니다. 구체적으로는 이런 부품들로 이뤄집니다.

  • 에이전트 루프 — "생각 → 행동 → 관찰"을 반복시키는 제어 구조 (3장)
  • 시스템 프롬프트 — 에이전트가 어떻게 행동할지 정한 운영 규칙 (4장)
  • 툴(tool) — 파일 읽기, 명령 실행, 검색 같은 에이전트의 손발 (5장)
  • 컨텍스트 관리 — 한정된 입력 공간을 어떻게 채우고 비울지 (6장)
  • 메모리 — 대화가 끝나도 남는 장기 기억 (7장)
  • 검증·권한·평가 — 결과를 믿어도 되는지, 위험한 행동을 막는 안전장치 (10~12장)
핵심 한 문장. 하네스 엔지니어링은 "더 똑똑한 모델을 만드는 일"이 아닙니다. 이미 똑똑한 모델의 잠재력을, 실제 업무에서 안정적으로 끌어내도록 그 주위를 설계하는 일입니다. 모델을 바꾸지 않고도 에이전트의 성공률을 크게 끌어올릴 수 있는 영역이죠.

2. 왜 지금 하네스 엔지니어링인가 — 프롬프트 → 컨텍스트 → 하네스

이 분야가 갑자기 주목받는 데는 이유가 있습니다. LLM을 다루는 기술의 무게중심이 단계적으로 바깥으로 이동해 왔기 때문입니다.

한때 가장 뜨거웠던 건 프롬프트 엔지니어링이었습니다. "이 마법의 주문(프롬프트)을 쓰면 모델이 더 잘 답한다"는 식이었죠. 하지만 모델이 똑똑해질수록, 한 줄짜리 주문보다 모델에게 무엇을 보여 줄지가 더 중요해졌습니다.

그래서 등장한 것이 컨텍스트 엔지니어링(context engineering)입니다. Anthropic은 2025년 9월, Claude Sonnet 4.5와 함께 낸 글에서 이렇게 정의합니다. 핵심 질문은 더 이상 "어떤 단어를 쓸까"가 아니라, "어떤 컨텍스트 구성이 우리가 원하는 모델 동작을 만들어 낼 가능성이 가장 높은가"라고요. 여기서 컨텍스트란 모델이 답을 생성할 때 입력으로 보는 토큰 전체를 말합니다. 가이드는 간단합니다 — "유익하되, 간결하게(informative, yet tight)."

그리고 지금, 무게중심은 한 걸음 더 바깥으로 나갔습니다. 컨텍스트를 잘 채우는 것만으로는 부족하고, 그 컨텍스트를 매 턴 어떻게 채우고 비울지, 도구를 어떻게 부르고, 결과를 어떻게 검증하고, 위험한 행동을 어떻게 막을지 — 시스템 전체를 설계해야 합니다. 그게 하네스 엔지니어링입니다.

오해 방지. 이 셋은 서로를 대체하지 않습니다. 좋은 하네스 안에도 좋은 프롬프트(시스템 프롬프트)와 좋은 컨텍스트 관리가 들어갑니다. 프롬프트·컨텍스트는 하네스의 부품이고, 하네스는 그 부품들을 묶어 돌리는 차체에 가깝습니다.

3. 모든 것의 심장: 에이전트 루프(ReAct)

하네스의 가장 안쪽에는 에이전트 루프가 있습니다. 이름은 거창하지만 구조는 의외로 단순합니다. 생각(reasoning) → 행동(action) → 관찰(observation)을 끝없이 반복하는 것뿐입니다. 이 패턴을 ReAct(Reason + Act)라고 부릅니다.

코드로 보면 더 분명합니다. 거의 모든 코딩 에이전트의 핵심은 이 while 루프 한 덩어리입니다.

# 에이전트 루프의 뼈대 — 과장 없이, 사실 이게 거의 전부입니다
context = [system_prompt, user_request]   # 모델에게 줄 맥락(토큰 묶음)

while True:
    response = llm(context)               # ① 생각: 다음 수를 모델이 정한다
    context.append(response)

    if response.is_final_answer:          #    끝났다고 판단하면 루프 종료
        return response.answer

    tool_name, tool_args = response.tool_call   # ② 행동: 어떤 툴을 부를지
    result = run_tool(tool_name, tool_args)     #    하네스가 실제로 실행
    context.append(result)                # ③ 관찰: 결과를 맥락에 덧붙인다
    # 다시 위로 — 모델이 그 결과를 보고 다음 수를 둔다

Anthropic이 만든 Claude Code 같은 도구도, 안을 들여다보면 본질은 이 ReAct 패턴의 while 루프입니다. 거창한 의사결정 그래프가 아니라, "맥락을 모은다 → 모델을 부른다 → 도구를 실행한다 → 반복한다"의 단순한 순환이죠. 하네스 엔지니어링의 많은 부분은 결국 이 루프의 각 단계를 어떻게 더 똑똑하고 안전하게 만드느냐의 문제입니다.

여기서 질문이 생깁니다. "매 턴 context에 결과를 계속 append하면, 맥락이 끝없이 부풀지 않나?" 정확합니다. 바로 그 문제를 다루는 게 6장(컨텍스트 관리)이고, 이 블로그의 루프 엔지니어링 글에서 루프 자체를 더 깊이 파고듭니다.

2부 · 핵심 구성요소

4. 시스템 프롬프트 — 에이전트의 운영 매뉴얼

루프가 매 턴 모델을 부를 때, 맨 앞에 항상 따라붙는 글이 있습니다. 시스템 프롬프트입니다. 사용자의 그때그때 질문이 아니라, "너는 어떤 존재이고, 어떻게 행동해야 하는가"를 정해 두는 상시 지침이죠. 신입에게 건네는 업무 매뉴얼이자 행동 강령에 가깝습니다.

여기에는 보통 이런 것들이 들어갑니다.

  • 역할과 목표 — "너는 코드를 작성·수정하는 엔지니어링 에이전트다"
  • 행동 규칙 — "큰 변경 전에는 계획을 먼저 보여라", "테스트를 통과시킨 뒤 끝냈다고 말하라"
  • 가진 도구와 사용법 — 어떤 툴이 있고 언제 쓰는지
  • 출력 형식·말투 — 결과를 어떤 형태로 돌려줄지
  • 안전 경계 — 하면 안 되는 일, 사람에게 물어봐야 하는 일

시스템 프롬프트는 길수록 좋은 게 아닙니다. 컨텍스트 엔지니어링의 원칙 그대로 "유익하되 간결하게"가 핵심입니다. 규칙이 너무 많거나 서로 모순되면 모델이 헷갈리고, 무엇보다 그 길이만큼 매 턴 토큰(=비용)과 주의력을 깎아 먹습니다. 좋은 시스템 프롬프트는 꼭 필요한 규칙만, 모델이 오해 없이 따를 수 있게 적은 것입니다.

주니어를 위한 팁. 시스템 프롬프트는 코드처럼 다루세요. 한 번 쓰고 끝이 아니라, 에이전트가 자주 저지르는 실수를 보면서 규칙을 한 줄씩 고쳐 나가는 반복 개선의 대상입니다. "왜 자꾸 테스트를 안 돌리지?" 싶으면, 모델을 탓하기 전에 시스템 프롬프트에 그 규칙이 분명히 적혀 있는지부터 확인하세요.

5. 툴(Tool) — 에이전트의 손발, 그리고 좋은 툴의 조건

모델이 "이 파일을 읽고 싶다"고 생각해도, 실제로 읽으려면 툴(tool)이 필요합니다. 툴은 에이전트의 손발입니다. 모델은 "어떤 툴을 어떤 인자로 부를지"를 글로 표현하고, 하네스가 그걸 받아 진짜로 실행한 뒤 결과를 돌려줍니다.

모델에게 툴은 이런 스키마(설명서) 형태로 주어집니다. 이름, 무엇을 하는지, 어떤 인자를 받는지를 적어 두면 모델이 그걸 보고 호출 방법을 정합니다.

{
  "name": "read_file",
  "description": "파일 내용을 읽어 텍스트로 반환한다. 경로는 프로젝트 루트 기준 상대 경로.",
  "input_schema": {
    "type": "object",
    "properties": {
      "path": { "type": "string", "description": "읽을 파일의 경로" }
    },
    "required": ["path"]
  }
}

흥미로운 사실이 있습니다. 서로 따로 만들어진 코딩 에이전트들을 비교해 보면, 도구의 개수는 제각각이어도 핵심 능력은 거의 같은 네 가지로 수렴한다는 점입니다. 바로 읽기(read) · 검색(search) · 편집(edit) · 실행(execute)입니다. 즉 좋은 에이전트를 만들기 위해 수십 개의 툴이 필요한 게 아니라, 이 최소 집합을 얼마나 신뢰성 있게 만드느냐가 관건이라는 뜻입니다.

"신뢰성 있는 툴"이 무슨 뜻인지, 파일 편집 툴을 예로 들어 보겠습니다. 모델이 코드를 고치게 하려면 "몇 번째 줄을 바꿔라"라고 시킬 수도 있고, "이 문자열을 저 문자열로 바꿔라"라고 시킬 수도 있습니다. 실무에서 여러 팀이 독립적으로 도달한 결론은 후자, 즉 '기존 문자열 → 새 문자열'을 정확히 매칭해 치환하는 방식이 더 안정적이라는 것이었습니다. 줄 번호는 파일이 조금만 바뀌어도 어긋나지만, 정확한 문자열 매칭은 모델이 "무엇을 바꾸려는지"를 그대로 표현하기 때문이죠.

이게 왜 중요할까요? 툴 설계는 사소해 보이지만, 에이전트의 실패 대부분이 "모델이 멍청해서"가 아니라 "툴이 애매하거나 깨지기 쉬워서" 일어납니다. 설명이 모호한 툴, 에러를 이상하게 돌려주는 툴, 한 번에 너무 많은 걸 하는 툴은 똑똑한 모델도 헛발질하게 만듭니다. 좋은 툴 = 좋은 에이전트라고 해도 과언이 아닙니다.

6. 컨텍스트 관리 — 유한한 작업 기억을 지키는 법

3장에서 예고한 문제로 돌아옵니다. 루프가 돌 때마다 도구 결과가 context에 쌓이는데, 모델이 한 번에 볼 수 있는 입력에는 상한(컨텍스트 창)이 있습니다. 게다가 창이 크다고 해서 마냥 좋은 것도 아닙니다.

책상 비유. 컨텍스트 창은 책상입니다. 책상이 넓다고 서류를 무한정 쌓아 두면, 정작 지금 필요한 한 장을 못 찾습니다. 입력이 길어질수록 모델이 중간 내용을 놓치는 현상(흔히 "컨텍스트 부패, context rot"라 부릅니다)도 잘 알려져 있죠. 그래서 책상을 넓히는 것보다, 지금 필요한 서류만 올려 두는 일이 더 중요합니다.

그래서 하네스는 매 턴 모델을 부르기 전에 컨텍스트를 손봅니다. 대표적인 두 가지 전략이 있습니다.

  • 잘라내기(pruning / observation masking) — 오래된 도구 결과처럼 더는 필요 없는 부분을 통째로 빼거나 자리표시자로 치환합니다. 가장 단순하고, 토큰을 가장 크게 줄입니다.
  • 요약(summarization / compaction) — 버리기 전에 그 내용을 한 줄 요약으로 압축해 끼워 넣습니다. "무엇을 했는지"라는 진행 맥락을 잃지 않으면서 분량을 줄이죠. Claude Code의 /compact가 이 방식입니다.

최근 연구들은 이 두 가지를 함께 쓸 때 가장 좋다고 입을 모읍니다. 잘라내기는 토큰을 줄이고 낡은 정보로 인한 혼란을 막지만, 너무 잘라내면 진행 상황을 잊어버립니다. 요약이 그 빈자리를 메워 주죠. 핵심 메시지는 하나입니다 — 긴 작업에서 컨텍스트는 "무한정 쌓아 두는 로그"가 아니라 "능동적으로 관리하는 대상"이라는 것입니다.

더 깊이 읽기. "맥락을 잘라냈더니 오히려 성능이 올랐다"는 실험을 구체적인 수치와 함께 다룬 글이 이 블로그에 있습니다 → 맥락은 적을수록 똑똑하다. 컨텍스트 관리만 따로 파고들고 싶다면 추천합니다.

7. 메모리 — 컨텍스트 너머를 기억하기

컨텍스트 창은 단기 기억입니다. 대화가 끝나거나 컴팩션으로 잘려 나가면 사라지죠. 그런데 "어제 짜다 만 기능을 오늘 이어서" 같은 긴 작업을 하려면, 세션이 끝나도 남는 장기 기억이 필요합니다. 사람으로 치면 머릿속 단기 기억과, 노트에 적어 둔 기록의 차이입니다.

Anthropic이 장기 실행 에이전트를 다룬 글에서 권하는 방식이 바로 이 파일 기반 메모리입니다. 구체적으로 세 가지 산출물을 둡니다.

  • 진행 로그(claude-progress.txt 같은 텍스트 파일) — 무엇을 했고 어떤 결정을 내렸는지 줄글로 기록
  • 기능 목록(JSON 파일) — 만들어야 할 기능들을 적고, 각각 완료/미완료를 표시
  • git 히스토리 — 의미 있는 커밋 메시지와 함께 변경을 남겨, 잘못되면 되돌릴 수 있는 체크포인트로 활용

핵심은 "기억을 머릿속(컨텍스트)에만 두지 말고, 밖(파일)에 적어 둔다"는 발상입니다. 그러면 컨텍스트가 잘려도, 다음 세션이 그 파일을 읽고 "아, 여기까지 했구나" 하며 이어서 일할 수 있습니다.

3부 · 심화

8. 장기 실행 에이전트 — 컴팩션만으론 부족하다

여기서부터가 진짜 어려운 영역입니다. 몇 분짜리 작업이 아니라 몇 시간, 여러 세션에 걸친 작업을 시키면 어떻게 될까요? "컨텍스트 관리(6장)랑 메모리(7장)만 잘하면 되는 것 아냐?"라고 생각할 수 있습니다. 그런데 Anthropic의 실험이 보여준 결론은 단호합니다 — 컴팩션만으로는 부족하다는 것입니다.

아무리 뛰어난 모델(예: Opus 4.5)에게도 "고수준 한 줄 지시"만 던지면 프로덕션급 결과를 못 냅니다. 에이전트가 흔히 보이는 실패가 두 가지 있습니다.

  • 한꺼번에 너무 많이 시도한다 — 기능 열 개를 동시에 건드리다 다 어중간하게 만든다.
  • 덜 됐는데 "다 됐다"고 선언한다 — 실제로는 절반만 구현해 놓고 끝났다고 한다.

그래서 효과적인 하네스는 모델에게 일을 시키는 방식 자체를 설계합니다. Anthropic이 제시한 패턴 하나가 2-에이전트 아키텍처입니다.

  • 초기화 에이전트(Initializer) — 딱 한 번 돌면서 환경을 세팅합니다. 작업용 스캐폴딩 파일, 기능 명세(JSON), 첫 git 커밋을 만들어 둡니다.
  • 코딩 에이전트(Coding) — 매 세션 돌면서 한 번에 한 기능씩 점진적으로 진행하고, 다음 세션을 위한 흔적(진행 로그·커밋)을 분명히 남깁니다.

재미있는 건, 이 둘이 시스템 프롬프트도, 도구도, 하네스도 똑같이 공유한다는 점입니다. 차이는 오직 맨 처음 받는 지시(프롬프트)뿐이죠. 같은 일꾼인데, 첫 지시가 "환경을 세팅하라"냐 "이어서 한 기능을 끝내라"냐만 다른 겁니다.

그리고 한 세션의 흐름도 정해 둡니다. ① 시작할 때 진행 산출물을 먼저 읽고 ② 기준이 되는 동작 테스트를 돌려 "지금 멀쩡한지" 확인한 뒤 ③ 한 기능만 골라 작업하고 ④ 끝나면 설명적인 커밋을 남긴다. 이 규율 덕분에, 세션이 수십 번 바뀌어도 작업이 무너지지 않고 쌓입니다.

핵심 통찰. 장기 실행 에이전트의 비결은 "모델이 모든 걸 기억하게 만드는 것"이 아니라, "기억하지 못해도 이어서 일할 수 있는 구조를 밖에 만들어 두는 것"입니다. 사람도 큰 프로젝트를 머릿속에만 담지 않고 이슈 트래커와 문서로 관리하잖아요. 똑같은 원리입니다.

9. 멀티에이전트 오케스트레이션 — 일을 나눠 맡기기

작업이 더 커지면, 에이전트 하나가 모든 걸 처리하는 대신 여러 에이전트에게 일을 나눠 맡기는 방식을 씁니다. 회사가 커지면 한 명이 다 하지 않고 팀을 나누는 것과 같습니다. 이걸 멀티에이전트 오케스트레이션이라고 합니다.

이 방식의 진짜 장점은 "여러 개라서 빠르다"가 아니라 컨텍스트 격리(context isolation)입니다. 하위 에이전트가 자료 100개를 뒤지는 지저분한 과정은 그 에이전트의 컨텍스트 안에서만 벌어지고, 메인 오케스트레이터에게는 깔끔하게 정리된 결론만 돌아옵니다. 메인의 책상이 깨끗하게 유지되는 거죠. 또 서로 독립적인 일은 병렬로 동시에 굴릴 수도 있습니다.

물론 공짜는 아닙니다. 에이전트가 많아질수록 토큰 비용이 곱절로 늘고, 누가 무엇을 했는지 조율·추적하기가 어려워집니다. 그래서 "무조건 멀티에이전트"가 아니라, 일이 자연스럽게 독립된 덩어리로 나뉘고 각 덩어리의 중간 과정이 길 때 쓰는 게 정석입니다.

10. 검증과 피드백 루프 — "다 됐어요"를 믿지 않기

8장에서 봤듯, 에이전트는 덜 됐는데 다 됐다고 말하는 버릇이 있습니다. 그래서 잘 만든 하네스는 모델의 "다 됐어요"를 곧이곧대로 믿지 않고, 스스로 확인하는 단계를 루프 안에 넣습니다. 이것이 검증과 피드백 루프입니다.

검증의 형태는 다양합니다. Anthropic의 권고는 특히 실전적입니다 — 단위 테스트만으로 끝내지 말고, 브라우저 자동화 같은 걸로 "사용자 입장에서 진짜 동작하는지" 끝까지(end-to-end) 확인하라는 것입니다. 모델이 "버튼을 추가했다"고 말하는 것과, 실제로 그 버튼을 눌렀을 때 동작하는 것은 다른 이야기니까요.

한 발 더 나아간 기법도 있습니다.

  • 자기 비판(self-critique) — 모델이 행동하기 전에 "이게 정말 맞나?"를 한 번 스스로 따져 보게 한다.
  • 어드버서리얼 검증(adversarial verification) — 결과를 그대로 믿는 대신, "이게 틀렸음을 증명해 보라"고 시키는 별도의 검증자를 둔다. 여러 검증자에게 투표를 시켜 다수가 반박하면 폐기하는 식이다.
  • 휴먼인더루프(human-in-the-loop) — 위험하거나 애매한 결정은 사람에게 확인을 받는다.
솔직한 메모. 사실 이 블로그 글 자체도, 자료를 모은 뒤 "각 주장을 반박해 보라"는 어드버서리얼 검증을 거쳤습니다. 그 과정에서 출처가 불분명한 정밀 수치 몇 개를 일부러 뺐습니다. 검증 루프가 없으면, 그럴듯하지만 틀린 내용이 그대로 살아남습니다 — 사람이 쓰든 에이전트가 쓰든 마찬가지죠.

11. 권한과 샌드박스 — 생각과 실행을 분리하라

에이전트에게 손발(툴)을 주는 순간, 위험도 함께 생깁니다. 모델이 실수로(혹은 악의적인 입력에 속아) rm -rf 같은 파괴적 명령을 내리면 어떻게 될까요? 그래서 하네스에는 반드시 안전장치가 들어갑니다.

핵심 설계 원칙은 "생각하는 부분과 실행을 허가하는 부분을 분리한다"는 것입니다. 모델은 "이 명령을 실행하고 싶다"고 제안할 수 있을 뿐, 실제로 실행할지 말지는 모델과 별개의 결정론적 코드(권한 검사기)가 정합니다. 이렇게 분리해 두면, 설령 모델이 이상한 행동을 하려 해도 권한 계층을 모델이 마음대로 뚫지 못합니다.

  • 권한 게이트(permission gate) — 위험한 툴(파일 삭제, 외부 전송 등)은 실행 전에 사람의 승인을 요구하거나, 허용 목록에 있는 것만 통과시킨다.
  • 샌드박스(sandbox) — 에이전트를 격리된 환경(컨테이너, 임시 작업 폴더 등)에서 돌려, 사고가 나도 피해가 그 안에 갇히게 한다.
  • 읽기 전용 모드 — 계획만 세우는 단계에서는 아예 쓰기 도구를 주지 않아, 실수의 여지를 원천 차단한다.
비유. 자동차의 엔진이 아무리 강해도, 브레이크와 안전벨트 없이는 도로에 내보내지 않습니다. 권한·샌드박스는 에이전트의 브레이크와 안전벨트입니다. 그리고 앞서 본 "코드베이스의 대부분이 AI 로직이 아니라 인프라"라는 사실의 큰 부분이 바로 이 안전장치들입니다.

12. 평가·신뢰성·비용 — 한 번 성공이 진짜 성공은 아니다

하네스를 잘 만들었는지 어떻게 알까요? "한 번 돌려 봤더니 잘되더라"로는 부족합니다. 에이전트는 같은 작업도 돌릴 때마다 조금씩 다르게 행동하기 때문입니다(모델 출력에 무작위성이 있습니다). 그래서 하네스 엔지니어링에는 평가(evaluation, 줄여서 eval)가 따라붙습니다.

여기서 주의할 함정이 있습니다. 많은 벤치마크가 "한 번 돌려 성공했나"만 측정하는데, 실제 프로덕션에서 중요한 건 "열 번 돌리면 몇 번 성공하나(신뢰성)"입니다. 입력을 살짝만 바꿔도 성공률이 눈에 띄게 떨어지는, 즉 에이전트가 의외로 깨지기 쉽다(fragile)는 연구 결과도 있습니다. 그래서 좋은 평가는 같은 작업을 여러 번 반복하고, 입력을 조금씩 비틀어 보며 안정성을 봅니다.

또 하나 빼놓을 수 없는 축이 비용입니다. 에이전트는 루프를 돌 때마다 토큰을 쓰고, 토큰은 곧 돈입니다. 그래서 같은 작업을 더 적은 토큰으로 끝내는 하네스가 좋은 하네스입니다. 6장의 컨텍스트 관리(잘라내기·요약)가 단지 "성능"만이 아니라 비용의 문제이기도 한 이유죠.

흔한 착각실제로 봐야 할 것
한 번 돌려서 성공 → 완성여러 번 돌렸을 때의 성공률(신뢰성)
벤치마크 점수만 높으면 됨입력을 비틀어도 버티는지(견고성)
성능만 좋으면 됨작업당 토큰·시간(비용)까지
모델을 바꾸면 다 해결같은 모델로도 하네스가 점수를 가른다
중요한 미묘함. 많은 벤치마크가 "모델의 실력"과 "하네스의 실력"을 뒤섞어 측정합니다. 그래서 점수가 올랐을 때 그게 더 좋은 모델 덕인지, 더 좋은 하네스 덕인지 떼어 보기 어렵습니다. 내 에이전트를 개선할 때는 한 번에 하나만 바꾸고(모델 고정 → 하네스만 손보기) 측정하는 습관이 중요합니다.

마치며

13. 주니어 개발자를 위한 학습 경로

한 번에 다 만들려 하면 막막합니다. 아래 순서대로 작게 만들고 하나씩 붙여 가는 걸 권합니다. 각 단계는 앞 단계 위에 자연스럽게 쌓입니다.

  1. 가장 작은 ReAct 루프부터. 3장의 while 루프를 그대로 따라 만들어 보세요. 툴은 get_current_time() 같은 장난감 하나면 충분합니다. "모델이 툴을 부르고, 내가 실행해서 결과를 돌려주는" 한 사이클을 직접 보는 게 첫걸음입니다.
  2. 진짜 툴 1~2개를 붙인다. 파일 읽기, 셸 명령 실행 같은 걸 추가해 보세요(샌드박스 안에서!). 이때부터 "툴 설명을 어떻게 써야 모델이 헷갈리지 않는지"가 피부로 느껴집니다(5장).
  3. 컨텍스트가 터지는 걸 경험한다. 루프를 길게 돌려 보면 입력이 부풀어 오르는 걸 직접 보게 됩니다. 그때 "최근 N개만 남기기"부터 적용해 보세요(6장).
  4. 파일 메모리를 더한다. 진행 상황을 progress.txt에 적게 하고, 다음 실행이 그걸 읽고 이어 가게 만들어 보세요(7장).
  5. 검증 루프를 넣는다. "테스트가 통과할 때까지 고쳐라"를 루프에 넣으면, 에이전트의 결과 품질이 확 올라갑니다(10장).
  6. 그다음에 멀티에이전트. 위가 안정된 뒤에야 일을 나눠 맡기는 단계로 갑니다(9장). 처음부터 멀티에이전트로 시작하지 마세요 — 디버깅 지옥입니다.

그리고 가장 좋은 교재는 잘 만든 하네스의 1차 출처입니다. 아래 Anthropic의 두 글은 이 분야의 사실상 표준 참고서이니 꼭 읽어 보세요.

14. 마무리 (TL;DR)

"에이전트의 실력은 모델이 아니라, 그 모델을 감싼 하네스에서 갈린다."

  • 하네스란 — LLM(엔진)을 '실제로 일하는 에이전트'로 만드는 주변 시스템 전체. 루프·프롬프트·툴·컨텍스트·메모리·검증·권한.
  • 왜 지금 — 엔지니어링의 무게중심이 프롬프트 → 컨텍스트 → 하네스로 옮겨 왔다. 모델이 똑똑해진 만큼 병목이 바깥으로 이동했다.
  • 심장 — 에이전트 루프(ReAct): 생각 → 행동 → 관찰의 단순한 while 반복. 하네스의 많은 일은 이 루프의 각 단계를 다듬는 것.
  • 핵심 부품 — 시스템 프롬프트(운영 매뉴얼), 신뢰성 있는 툴(읽기·검색·편집·실행), 컨텍스트 관리(잘라내기+요약), 파일 기반 메모리.
  • 심화 — 장기 작업엔 컴팩션만으론 부족 → 2-에이전트·세션 규율. 큰 작업엔 멀티에이전트(컨텍스트 격리). 결과는 검증 루프로 확인. 권한·샌드박스로 안전 확보. 평가는 한 번이 아니라 반복·비용까지.
  • 시작은 — 가장 작은 ReAct 루프부터. 툴 → 컨텍스트 → 메모리 → 검증 → 멀티에이전트 순으로 하나씩 쌓아라.

모델은 앞으로도 계속 똑똑해질 겁니다. 하지만 그 똑똑함을 실제 업무에서 안정적으로 꺼내 쓰는 일은 여전히 우리 몫입니다. 좋은 엔진을 좋은 자동차로 만드는 것 — 그게 하네스 엔지니어링이고, 지금 에이전트를 만드는 개발자가 가장 많은 차이를 만들어 낼 수 있는 자리입니다.
