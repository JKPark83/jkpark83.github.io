# 🔥 오늘의 핵심

오늘의 축은 둘로 갈린다. **모델**: Moonshot의 **Kimi K3**(2.8T MoE·1M 컨텍스트)가 오픈웨이트를 예고하며 프런티어를 추격 중이다. **에이전트**: 새 벤치마크 **OmniaBench**는 Claude-Sonnet-5·GPT-5.6-Sol조차 일반 에이전트 태스크에서 **Pass@1 58%대**에 머문다는 걸 실측했다. 즉 모델은 싸지고 강해지는데, 에이전트의 진짜 벽은 성능이 아니라 **계획·제약 유지·적응적 수정**이다. 오늘 할 일: 값싼 오픈 모델을 서브태스크에 라우팅하고, 에이전트엔 "제약 위반"을 잡는 최소 eval을 붙여라.

# 📚 주목 논문

## 1) OmniaBench — 일반 에이전트, 어디서 무너지나
- **저자**: Chengyu Shen, Yujie Fu, Gangtao Xin 외 (arXiv `2607.14989`, cs.CL, 2026-07-16 제출)
- **핵심**: 소비자·비즈니스·엔터프라이즈를 아우르는 **1,431개 태스크**(644개 챌린지 서브셋), 90개 L1·354개 L2 도메인, 10차원 역량 프레임 + 8개 난이도 인자로 "일반 에이전트"를 폭넓게 평가.
- **결과(검증됨)**: 프런티어도 고전한다. **Claude-Sonnet-5 Overall Pass@1 58.54, GPT-5.6-Sol 57.14**. 실패는 **계획 수립·제약 유지·적응적 수정**에 집중.
- **🛠 agent-builder 시사점**: "성능 좋은 모델 = 좋은 에이전트"가 아니다. 태스크를 잘게 쪼개고, 각 단계에서 제약(포맷·권한·예산)을 **명시적으로 재확인**하는 노드를 넣어라.
- **한계**: 합성 태스크 비중이 커, 실제 프로덕션 분포와의 격차는 별도 검증 필요.
- 링크: [arxiv.org/abs/2607.14989](https://arxiv.org/abs/2607.14989)

![OmniaBench Overall Pass@1 막대그래프: Claude-Sonnet-5 58.54, GPT-5.6-Sol 57.14, 완벽선 100](/images/ai-news-2026-07-19/omniabench.svg)
*프런티어 모델도 일반 에이전트 태스크에선 절반 남짓. 병목은 모델이 아니라 제약 유지. (자체 제작 SVG, 값 출처 arXiv 2607.14989)*

## 2) SearchOS-V1 — 검색 에이전트를 OS처럼 스케줄링
- **저자**: Yuyao Zhang, Junjie Gao, Zhengxian Wu 외 (arXiv `2607.15257`, cs.AI, 2026-07-16 제출)
- **핵심**: 취약한 **암묵적 검색 진행상태**를 "명시적·영속·공유 상태"로 바꾼다 — task frontier(미해결 목표), evidence graph(근거 그래프), coverage map(커버리지 지도).
- **방법(요약)**: Search-Oriented Context Management(SOCM)와 툴 미들웨어 하니스로 근거를 자동 기록하고, **파이프라인-병렬 스케줄링**으로 유휴 슬롯을 미해결 갭 태스크로 계속 채운다.
- **결과**: WideSearch·GISA 벤치에서 기존 단일/멀티 에이전트 대비 우위. 코드·280개 사전 정의 스킬 공개.
- **🛠 agent-builder 시사점**: 멀티스텝 에이전트의 "무엇을 이미 했나"를 프롬프트가 아니라 **외부 상태(그래프/맵)**에 두면, 컨텍스트가 날아가도 진행상태가 살고 중복 검색·토큰 낭비가 준다.
- **한계**: 정보탐색 도메인 특화 — 코드/툴 실행 에이전트로의 이전은 별도 설계가 필요.
- 링크: [arxiv.org/abs/2607.15257](https://arxiv.org/abs/2607.15257)

![SearchOS 명시적 공유 상태 패턴 다이어그램: 암묵적 진행상태 → task frontier·evidence graph·coverage map → 루프 제거·병렬 스케줄](/images/ai-news-2026-07-19/searchos-state.svg)

# 🛠 오픈소스 스포트라이트

오늘의 두 리포는 위 논문·모델과 바로 맞물린다.

- **[antins-labs/SearchOS](https://github.com/antins-labs/SearchOS)** (Python) — SearchOS-V1의 공식 코드. "검색 에이전트를 OS가 프로세스 스케줄하듯" 돌린다. 280개 사전 정의 스킬(전략·접근)과 툴 미들웨어 하니스 포함. **쓸 곳**: 근거 추적·커버리지 관리가 필요한 리서치/딥서치 에이전트의 참조 구현.
- **[MoonshotAI/kimi-code](https://github.com/MoonshotAI/kimi-code)** (TypeScript, MIT) — 터미널 코딩 에이전트 Kimi Code CLI. `/mcp-config`로 MCP 서버를 대화형으로 추가·인증하고, coder/explore/plan 서브에이전트를 격리 컨텍스트로 굴린다. ACP로 Zed·JetBrains 연동. **쓸 곳**: Kimi K3를 쓰는 CLI 에이전트 워크플로, 또는 MCP 설정을 손으로 JSON 편집하기 싫을 때.
- ⚠️ **투명성 노트**: 이번 세션은 GitHub API 접근이 현재 리포로 제한돼 **스타 수를 검증하지 못했다.** 수치를 지어내지 않기 위해 별점은 생략했으니, 채택 전 각 리포에서 직접 확인하길 권한다.

# 🧑‍💻 Codex & Claude Code 실전 팁

## 🟩 Codex — 서브에이전트 계층화 (Subagent)
- 출처: [developers.openai.com/codex/subagents](https://developers.openai.com/codex/subagents)
- 💡 **핵심**: 메인 에이전트는 요구사항·결정·최종 산출에 집중시키고, 탐색·테스트·로그 분석 같은 **읽기 중심** 작업만 서브에이전트로 병렬 위임한다. 서브에이전트는 원시 출력이 아니라 **요약**을 반환.
- 📝 **How-to**: 서브에이전트 프롬프트에 (1) 작업 분할 방법, (2) 전부 기다릴지 여부, (3) 반환할 요약 형식을 명시. `AGENTS.md`는 32KiB를 넘으면 잘리니, 상세 규칙은 skills/MCP로 밀어낸다.
- 🎯 **언제**: 코드베이스 탐색·테스트·트리아지처럼 읽기 위주 작업을 넓게 훑을 때.
- ⚠️ **주의**: 쓰기(write) 작업을 여러 서브에이전트로 병렬화하면 동시 편집 충돌이 난다 — 쓰기는 직렬 또는 워크트리 격리.
- ⭐ **추천도**: ★★★★☆

## 🟦 Claude Code — Kimi K3로 서브태스크 라우팅 (Workflow)
- 출처: [platform.kimi.ai/docs/guide/claude-code-kimi](https://platform.kimi.ai/docs/guide/claude-code-kimi)
- 💡 **핵심**: Claude Code는 Anthropic Messages API를 말하므로, `ANTHROPIC_BASE_URL`만 바꾸면 값싼 오픈 모델(Kimi K3)로 돌릴 수 있다. 대량·저위험 서브작업의 비용을 낮추는 라우팅.
- 📝 **How-to**:

```bash
export ANTHROPIC_BASE_URL=https://api.moonshot.ai/anthropic
export ANTHROPIC_AUTH_TOKEN=$MOONSHOT_API_KEY
export ANTHROPIC_MODEL=kimi-k3
claude   # /status 로 Base URL·Model 반영 확인
```

- 🎯 **언제**: 리팩터·대량 파일 요약·테스트 생성처럼 토큰은 많이 먹지만 프런티어급 추론이 필수는 아닌 작업.
- ⚠️ **주의**: 코드·프롬프트가 서드파티 엔드포인트로 나간다. 사내 정책·데이터 민감도를 먼저 확인하고, 아키텍처 결정 같은 고위험 작업은 프런티어 모델에 남겨라.
- ⭐ **추천도**: ★★★★☆

## 🟪 공통 — "제약 유지" 회귀 eval 붙이기 (Eval)
- 출처: [arxiv.org/abs/2607.14989](https://arxiv.org/abs/2607.14989)
- 💡 **핵심**: OmniaBench가 보여준 프런티어의 약점은 성공률이 아니라 **제약 유지**다. 성공/실패 이진 평가 대신, 각 단계의 제약(출력 스키마·허용 툴·예산 상한) 위반을 세는 작은 회귀 eval을 CI에 넣어라.
- 📝 **How-to**: 실패했던 궤적 5~10개를 골든셋으로 고정 → 매 배포마다 "제약 위반 수"를 리포트 → 성공률과 **별개 지표**로 추적. 위반이 오르면 성공률이 유지돼도 릴리스를 막는다.
- 🎯 **언제**: 툴 호출·포맷 계약이 많은 프로덕션 에이전트.
- ⚠️ **주의**: 골든셋이 작으면 과적합된다 — 새 실패가 나올 때마다 셋을 갱신.
- ⭐ **추천도**: ★★★★☆
