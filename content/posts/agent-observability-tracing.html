일반 서버는 200 아니면 500으로 성패를 알려 줍니다. 그런데 에이전트는 200을 반환하면서 틀립니다. 도구를 엉뚱하게 호출하고, 같은 검색을 세 번 반복하고, 빈 결과를 받고도 태연히 답을 지어냅니다. 로그엔 아무 에러도 안 남죠.

— 그래서 "왜 이렇게 답했지?"의 답은 로그가 아니라 트레이스에 있습니다.

에이전트를 만들어 데모까지 성공시킨 junior 개발자가 프로덕션에서 처음 겪는 좌절은 대개 이렇습니다. 사용자가 "답이 이상해요"라고 제보합니다. 로그를 열어 봅니다. 200 OK, 예외도 없습니다. 그런데 답은 분명히 틀렸습니다. 대체 에이전트가 무슨 생각으로 그렇게 답했는지, 어떤 도구를 어떤 인자로 불렀는지, 검색 결과가 실제로 뭐였는지 — 아무것도 보이지 않습니다.

이 깜깜함을 걷어 내는 게 관측성(observability)입니다. 한 문장으로 줄이면 이렇습니다. 시스템이 밖으로 내보내는 신호(로그·메트릭·트레이스)만 보고, 안에서 무슨 일이 일어났는지 되짚을 수 있는 성질. 에이전트에서 이 되짚기의 주인공은 트레이스(trace)입니다. 사용자의 요청 하나를 통째로 따라가며, 그 안에서 벌어진 LLM 호출·도구 실행·검색을 한 단계씩 스팬(span)으로 기록한 것이죠.

이 주제가 2026년 여름에 부쩍 화제입니다. "에이전트의 해"라던 흐름이 프로토타입에서 프로덕션으로 넘어오면서, 병목이 '똑똑하게 만들기'에서 '돌아가는 걸 지켜보기'로 옮겨 왔기 때문입니다. 최근 한 조사에서 조직의 89%가 이미 에이전트에 어떤 형태로든 관측성을 붙였고, 62%는 개별 스텝·도구 호출까지 들여다보는 상세 트레이싱을 갖췄다고 답했을 정도입니다. 이번 주만 해도 Braintrust의 '에이전트 관측성 완벽 가이드', Latitude의 '에이전틱 워크플로에서 도구 호출 오류 잡아내기' 같은 실무 글이 잇따라 돌았고, 표준인 OpenTelemetry GenAI 시맨틱 컨벤션도 계속 다듬어지는 중입니다.

이 글은 그 지형을 처음부터 짚습니다. 로그·메트릭·트레이스가 어떻게 다른지, 스팬과 트레이스가 정확히 뭔지, OpenTelemetry의 gen_ai.* 표준은 무엇을 정해 두었는지, 몇 줄로 계측하는 코드는 어떻게 생겼는지, 그리고 트레이스를 보고 실패를 읽어 내는 법까지 — 쉽고 자세하게 살펴봅니다.

1. 왜 에이전트엔 특별한 관측성이 필요한가

보통의 웹 서비스는 실패가 정직합니다. DB 연결이 끊기면 예외가 터지고, 잘못된 요청은 4xx를 받고, 서버가 죽으면 5xx가 뜹니다. 상태 코드와 스택 트레이스만 봐도 "어디서 터졌는지"가 대충 잡힙니다.

에이전트는 다릅니다. 실패가 성공의 얼굴을 하고 옵니다. LLM은 무슨 입력을 받아도 그럴듯한 문장을 뱉으니까요. 대표적인 실패 모드는 이렇습니다.

  • 형식은 멀쩡한데 내용이 틀림 — JSON도 잘 맞고 문장도 자연스러운데 사실이 틀렸습니다. 검증기(validator)는 통과하고 사용자만 압니다.
  • 도구를 잘못 부름 — 존재하는 도구를 엉뚱한 인자로 부르거나, 필요 없는데 부르거나, 필요한데 안 부릅니다.
  • 같은 일을 반복(loop) — 똑같은 검색을 세 번 하며 토큰과 시간을 태웁니다.
  • 빈 결과를 못 알아챔 — 도구가 []를 돌려줬는데도 "찾았습니다"라며 답을 지어냅니다(hallucination).

이 모든 게 200 OK 안에서 벌어집니다. 그래서 에이전트는 자기만의 관측 계층이 필요합니다. 한 요청 안에서 모델이 뭘 받아 뭘 뱉었고, 어떤 도구를 어떤 인자로 불렀고, 그 결과가 뭐였고, 그게 몇 번 반복됐는지를 통째로 남겨야, 비로소 "왜 이렇게 답했지?"에 답할 수 있습니다.

관측성 ≠ 모니터링. 모니터링은 "미리 정해 둔 질문"에 답합니다 — CPU 몇 %냐, 에러율 몇 %냐. 관측성은 "미리 몰랐던 질문"에 답할 수 있는 성질입니다 — "어제 오후 3시에 이 사용자 요청이 왜 3초나 걸렸고 왜 틀렸지?" 에이전트 디버깅은 대부분 후자입니다.

2. 배경 — 로그·메트릭·트레이스, 그리고 스팬

관측성은 흔히 세 기둥(three pillars)으로 설명합니다. 각자 답하는 질문이 다릅니다.

  • 로그(logs) — "무슨 일이 있었나"의 이벤트 기록. 한 줄 한 줄의 사건. 흩어져 있어서 하나의 요청을 처음부터 끝까지 잇기 어렵습니다.
  • 메트릭(metrics) — "얼마나"를 재는 숫자. 초당 요청 수, 평균 지연, 토큰 사용량. 집계엔 좋지만 개별 사건의 사연은 없습니다.
  • 트레이스(traces) — "하나의 요청이 시스템을 어떻게 통과했나"의 여정. 여러 단계를 인과와 시간 순서로 엮습니다. 에이전트 디버깅의 핵심입니다.

트레이스는 스팬(span)들로 이루어집니다. 스팬 하나는 "이름 붙은 작업 한 조각"입니다 — 시작 시각, 끝 시각(그래서 얼마나 걸렸는지), 성공/실패 상태, 그리고 부가 정보(속성, attributes)를 담습니다. 스팬들은 부모-자식 관계로 묶여 하나의 트리를 이루고, 그 트리 전체가 하나의 트레이스입니다.

이 개념은 원래 마이크로서비스의 분산 추적(distributed tracing)에서 왔습니다. 요청 하나가 여러 서비스를 건너뛰며 처리될 때 "어느 구간이 느렸나"를 보려고 만든 것이죠. 에이전트도 구조가 똑같습니다. 요청 하나가 LLM → 도구 → LLM → 도구를 건너뛰며 처리되니까요. 그래서 분산 추적의 도구와 표준을 거의 그대로 빌려 씁니다.

3. 에이전트 트레이스의 구조 — 스팬 트리

에이전트 트레이스가 실제로 어떻게 생겼는지 보겠습니다. 사용자가 "최근 결제 실패 건을 요약해 줘"라고 물으면, 에이전트는 대략 이렇게 움직입니다.

  1. 에이전트가 요청을 받아 판단을 시작합니다. → 이 전체가 루트 스팬(invoke_agent)
  2. LLM에게 "무슨 도구를 부를까?" 물어봅니다. → 모델 스팬(chat)
  3. LLM이 search_payments 도구를 호출하라고 답합니다. 그 도구를 실행합니다. → 도구 스팬(execute_tool)
  4. 도구 결과를 다시 LLM에 넣어 최종 답을 만듭니다. → 모델 스팬(chat)

이 네 단계가 부모-자식으로 엮여 아래 같은 트리가 됩니다. 각 막대의 가로 길이가 걸린 시간이라, 어디서 느렸는지가 한눈에 보입니다. 이렇게 시간축에 스팬을 눕힌 그림을 워터폴(waterfall)이라 부릅니다.

여기서 핵심은, 각 스팬이 단순한 타이머가 아니라는 점입니다. 모델 스팬엔 어떤 모델을 썼는지·입력/출력 토큰이 몇 개였는지, 도구 스팬엔 어떤 인자로 불렀고 결과가 뭐였는지가 속성(attribute)으로 붙습니다. 그래서 트레이스 하나만 펼치면 "무엇을·언제·왜·얼마나"가 전부 담깁니다.

4. 표준 — OpenTelemetry GenAI 시맨틱 컨벤션

스팬에 정보를 담는 건 좋은데, 이름을 제각각 붙이면 도구마다 따로 놀게 됩니다. 어떤 팀은 토큰 수를 tokens_in, 다른 팀은 prompt_tokens, 또 다른 팀은 n_input이라 부르면, 어떤 대시보드도 이걸 한꺼번에 이해할 수 없습니다.

그래서 OpenTelemetry(OTel)가 나섭니다. OTel은 로그·메트릭·트레이스를 어떤 언어·어떤 백엔드에서도 같은 방식으로 다루자는 벤더 중립 표준입니다. 그 안에 LLM/에이전트용 규약이 따로 있는데, 바로 GenAI 시맨틱 컨벤션(semantic conventions)입니다. "이런 상황엔 스팬 이름을 이렇게, 속성 키를 이렇게 붙여라"를 못 박아 둔 사전이죠. 핵심은 gen_ai.*로 시작하는 표준 속성입니다.

스팬의 종류는 gen_ai.operation.name이라는 속성으로 구분합니다. 대표적인 값은 이렇습니다.

gen_ai.operation.name무엇을 나타내나스팬 이름 규칙
chatLLM 대화형 추론 호출chat {gen_ai.request.model}
execute_tool도구(함수) 실행execute_tool {gen_ai.tool.name}
invoke_agent에이전트 한 번 돌리기(루트)invoke_agent {gen_ai.agent.name}
embeddings임베딩 생성embeddings {gen_ai.request.model}
retrieval벡터/검색 조회retrieval {gen_ai.data_source.id}

그 밖에 자주 쓰는 표준 속성은 다음과 같습니다. 이 키들을 그대로 붙여 두면, 어떤 관측성 백엔드에 보내도 알아서 "이건 토큰 수, 이건 모델 이름"으로 인식합니다.

  • gen_ai.provider.name — 제공자(예: openai, anthropic)
  • gen_ai.request.model / gen_ai.response.model — 요청/응답 모델 이름
  • gen_ai.usage.input_tokens / gen_ai.usage.output_tokens — 입력/출력 토큰 수(비용·병목 분석의 핵심)
  • gen_ai.conversation.id — 여러 요청을 한 대화로 묶는 ID
  • gen_ai.agent.name / gen_ai.agent.id — 에이전트 이름/ID
  • gen_ai.tool.name / gen_ai.tool.call.id — 도구 이름/호출 ID
프롬프트 내용은 기본적으로 안 담깁니다. 실제 프롬프트·응답 메시지는 gen_ai.input.messages·gen_ai.output.messages라는 속성에 담기지만, 컨벤션은 "계측 라이브러리는 기본값으로는 이걸 기록하지 말고, 켤 수 있는 옵션만 제공하라"고 규정합니다. 프롬프트엔 개인정보(PII)가 섞이기 쉬워서입니다. 디버깅하려면 이 옵션을 의도적으로 켜야 하고, 켤 땐 PII 마스킹을 함께 고민해야 합니다.
성숙도 주의. GenAI 시맨틱 컨벤션은 2026년 현재 아직 Development(개발 중) 단계라, 세부 속성 이름이 바뀔 수 있습니다. 그래도 방향은 정해졌으니, 직접 키를 지어내기보다 이 표준을 따르는 편이 나중에 갈아타기 훨씬 쉽습니다. 최근 GenAI 규약은 별도 저장소(semantic-conventions-genai)로 옮겨져 관리되고 있습니다.

5. 실전 — 몇 줄로 계측하고 직접 스팬 만들기

이론은 됐고, 코드로 봅시다. 좋은 소식은 대부분 자동이라는 겁니다. OTel 기반 자동 계측 라이브러리(예: Traceloop의 OpenLLMetry)를 붙이면, OpenAI·Anthropic·LangChain 같은 SDK 호출을 가로채 알아서 chat 스팬을 만들어 줍니다. 내 코드는 거의 안 건드립니다.

{{< highlight python >}} # pip install traceloop-sdk openai from traceloop.sdk import Traceloop from openai import OpenAI # 한 줄로 초기화 — 이후의 LLM 호출이 자동으로 트레이스에 잡힌다. # app_name은 대시보드에서 서비스를 구분하는 이름이 된다. Traceloop.init(app_name="payments-agent") client = OpenAI() resp = client.chat.completions.create( # 이 호출이 자동으로 'chat' 스팬이 된다 model="gpt-5.6", messages=[{"role": "user", "content": "최근 결제 실패 건을 요약해 줘"}], ) # 모델 이름·입력/출력 토큰 수가 gen_ai.* 속성으로 스팬에 자동으로 붙는다. print(resp.choices[0].message.content) {{< /highlight >}}

하지만 자동 계측이 모르는 부분이 있습니다. 내가 직접 만든 도구나 에이전트 루프의 경계는 내가 알려 줘야 합니다. 이때 스팬을 손으로 여는 법이 이렇습니다. 표준 gen_ai.* 속성을 직접 붙여 주는 게 요령입니다.

{{< highlight python >}} from opentelemetry import trace tracer = trace.get_tracer("payments-agent") def run_tool(name: str, args: dict) -> dict: # 도구 실행을 execute_tool 스팬으로 감싼다. # 스팬 이름은 컨벤션대로 "execute_tool {도구이름}". with tracer.start_as_current_span(f"execute_tool {name}") as span: span.set_attribute("gen_ai.operation.name", "execute_tool") span.set_attribute("gen_ai.tool.name", name) try: result = TOOLS[name](**args) # 실제 도구 호출 span.set_attribute("tool.result_count", len(result)) # 결과가 비었는지 추적 return result except Exception as e: span.record_exception(e) # 예외를 스팬에 붙인다 span.set_status(trace.StatusCode.ERROR) # 상태를 명시적으로 실패로 raise {{< /highlight >}}

핵심은 span.set_status(...ERROR)와 record_exception(e)입니다. 이걸 안 붙이면 도구가 터져도 스팬은 성공으로 남습니다 — 딱 우리가 피하고 싶은 "성공처럼 보이는 실패"죠. 그리고 tool.result_count처럼 내 도메인 특화 속성을 하나 붙여 두면, 나중에 "결과가 0인데도 답을 지어낸 트레이스"를 그 속성으로 골라낼 수 있습니다.

에이전트 루프 전체는 이렇게 루트 스팬으로 감싸면, 앞서 본 워터폴이 그대로 만들어집니다.

{{< highlight python >}} def invoke_agent(user_msg: str) -> str: with tracer.start_as_current_span("invoke_agent payments-agent") as root: root.set_attribute("gen_ai.operation.name", "invoke_agent") root.set_attribute("gen_ai.agent.name", "payments-agent") messages = [{"role": "user", "content": user_msg}] for step in range(MAX_STEPS): # 에이전트 루프(관측성이 loop를 드러낸다) reply = call_llm(messages) # 자동으로 chat 스팬 if reply.tool_calls: for tc in reply.tool_calls: out = run_tool(tc.name, tc.args) # execute_tool 스팬 messages.append(tool_result(tc, out)) else: return reply.content # 도구를 더 안 부르면 종료 root.set_attribute("agent.hit_max_steps", True) # 루프 상한에 걸렸다 = 위험 신호 return "(최대 스텝 도달)" {{< /highlight >}}

이렇게 계측하면 스팬들은 OTel 형식으로 만들어져, 원하는 백엔드로 보낼 수 있습니다. 자체 호스팅으로는 Langfuse(오픈소스, prompt·eval 중심), Arize Phoenix(OpenTelemetry·OpenInference 규약 기반)가 널리 쓰이고, SaaS로는 Braintrust, LangSmith, Laminar 등이 있습니다. 표준 gen_ai.*로 계측해 두면 백엔드를 갈아타도 코드는 그대로라는 게 큰 장점입니다.

6. 트레이스로 실패를 읽는 법

계측을 마쳤으면, 이제 트레이스를 진단 도구로 씁니다. 앞서 본 실패 모드는 각각 트레이스의 특정 모양으로 나타납니다. 이 매핑을 알면 대시보드를 여는 순간 원인이 보입니다.

실전에선 트레이스 하나만 보지 않습니다. 관측성 백엔드에 쌓인 수천 개의 트레이스를 속성으로 거를(filter) 수 있는 게 진짜 힘입니다. 예를 들어 이런 질문에 몇 초 만에 답합니다.

  • "tool.result_count = 0인데도 긴 답을 낸 트레이스" → 환각(hallucination) 후보를 모아 본다.
  • "gen_ai.usage.output_tokens 상위 1%" → 비용을 태우는 요청을 찾는다.
  • "같은 gen_ai.tool.name이 한 트레이스에서 3번 이상" → 반복 루프에 빠진 요청을 잡는다.
  • "status = ERROR인 execute_tool 스팬" → 어떤 도구가 자주 실패하는지 순위를 낸다.
관측성과 평가(evals)는 짝꿍입니다. 평가(evals)가 배포 전 시험이라면, 관측성은 배포 후 CCTV입니다. 둘은 이어집니다. 프로덕션 트레이스에서 발견한 실패 케이스를 골라 평가 데이터셋에 넣으면, 다음 배포 땐 그 실패를 자동으로 걸러 낼 수 있습니다. 트레이스 → 데이터셋 → 평가로 이어지는 이 고리가 에이전트를 꾸준히 개선하는 엔진입니다.

7. 흔한 실수 · 팁

1) print/로그로 버티기. 초기엔 print(messages)로 그럭저럭 됩니다. 하지만 요청이 여러 스텝·여러 도구·동시 사용자로 늘어나면 로그는 뒤엉켜 어느 줄이 어느 요청인지 알 수 없게 됩니다. 트레이스는 애초에 이 문제를 풀려고 만든 것입니다. 처음부터 트레이스로 시작하세요.
2) 도구 스팬의 상태를 안 챙기기. 도구가 예외를 던져도 set_status(ERROR)를 안 붙이면 스팬은 초록색(성공)으로 남습니다. 최악은 도구가 조용히 빈 값을 반환하는 경우 — 예외조차 없죠. result_count 같은 속성으로 "성공했지만 알맹이가 없는" 상태를 명시적으로 기록하세요.
3) 프롬프트를 통째로, 마스킹 없이 기록. 디버깅하려고 gen_ai.input.messages를 켜는 건 좋은데, 여기엔 사용자 개인정보가 그대로 담깁니다. 컨벤션이 이 기록을 기본 off로 둔 이유입니다. 켤 땐 이메일·주민번호·카드번호 같은 PII를 마스킹하고, 접근 권한을 제한하세요.
4) 표준을 무시하고 키를 지어내기. my_tokens, llm_name처럼 마음대로 붙이면 당장은 되지만, 관측성 백엔드의 자동 대시보드·비용 집계가 안 먹힙니다. gen_ai.usage.input_tokens처럼 표준 키를 쓰면 도구가 알아서 이해합니다. 나중에 백엔드를 갈아탈 때도 편합니다.
5) 샘플링을 안 걸어 비용 폭탄. 트래픽이 크면 모든 트레이스를 100% 저장·전송하는 건 비쌉니다. 다만 무작정 10%만 남기면 정작 실패한 요청을 놓칠 수 있습니다. 에러·느린 요청은 항상 남기고 정상 요청만 표본을 줄이는 tail 기반 샘플링을 고려하세요.
6) 트레이스 ID를 사용자에게 노출하라. 에러 응답이나 지원 화면에 트레이스 ID를 함께 보여 주면, 사용자가 "이 요청이 이상해요 (ID: abc123)"라고 제보할 때 정확히 그 트레이스를 바로 열 수 있습니다. 디버깅 시간이 극적으로 줄어드는 작은 습관입니다.

8. 마무리 (TL;DR)

  • 에이전트는 500 대신 "성공처럼 보이는 실패"를 한다 — 형식은 맞는데 틀리고, 도구를 잘못 부르고, 루프에 빠지고, 빈 결과를 무시한다. 로그로는 안 보인다.
  • 트레이스(trace)가 답이다. 요청 하나를 스팬(span) 트리로 기록하면 "무엇을·언제·왜·얼마나"가 워터폴로 펼쳐진다. 분산 추적을 그대로 빌려 온 개념이다.
  • 이름은 OpenTelemetry GenAI 시맨틱 컨벤션을 따르라 — gen_ai.operation.name(chat·execute_tool·invoke_agent), gen_ai.usage.input_tokens, gen_ai.tool.name 등. 표준을 쓰면 백엔드를 갈아타도 코드는 그대로다. (2026년 현재 Development 단계.)
  • 계측은 대부분 자동(OpenLLMetry 등)이고, 내 도구·에이전트 루프만 수동 스팬으로 감싼다. 도구 스팬엔 반드시 set_status(ERROR)와 결과 개수 같은 도메인 속성을 붙여라.
  • 트레이스를 속성으로 걸러 실패를 사냥하라 — result_count=0인데 긴 답, 같은 도구 3번 이상, 상위 1% 토큰. 프롬프트 내용은 PII 때문에 기본 off, 켤 땐 마스킹.
  • 관측성은 평가(evals)의 짝꿍이다 — 프로덕션 트레이스에서 건진 실패를 평가셋에 넣어 다음 배포에서 자동으로 막는 고리를 만들어라.
