같은 GPU 한 장, 같은 모델. HuggingFace model.generate()로 직접 돌리면 초당 몇 요청이 고작인데, vLLM에 그대로 올리면 수십 배가 나옵니다. 모델을 손댄 것도, 더 좋은 GPU를 산 것도 아닙니다.

— 달라진 건 딱 두 가지, 언제 어떤 요청을 GPU에 태울지와 KV 캐시 메모리를 어떻게 담을지입니다.

모델 하나를 API로 서비스해 본 junior 개발자라면 한 번쯤 당황합니다. 로컬에서 transformers로 잘 돌던 모델을, 사용자 여러 명이 동시에 때리는 서버로 올렸더니 처리량(throughput)이 형편없습니다. GPU 사용률을 찍어 보면 더 이상합니다. 비싼 GPU가 대부분의 시간 동안 놀고 있습니다. 분명 일은 밀려 있는데 말이죠.

그래서 다들 vLLM이나 SGLang 같은 서빙 엔진(serving engine)을 씁니다. 재미있는 건, 이 엔진들이 모델을 더 똑똑하게 만들어서 빠른 게 아니라는 점입니다. 똑같은 가중치를 똑같이 계산합니다. 대신 요청을 스케줄링하는 방식과 메모리를 관리하는 방식을 바꿔서, 놀고 있던 GPU를 꽉 채웁니다.

이 주제가 이번 주에 다시 화제였습니다. 2026년판 "vLLM vs SGLang" H100 벤치마크 글들이 여럿 돌면서, SGLang의 RadixAttention이 프리픽스가 겹치는 RAG·멀티턴 워크로드에서 vLLM을 앞선다는 수치가 실무자 사이에서 오갔거든요. 그런데 그 비교를 제대로 읽으려면, 먼저 서빙 엔진이 애초에 왜 빠른지를 알아야 합니다. 이 글은 그 밑바닥을 처음부터 짚습니다.

1. LLM 추론은 왜 느린가 — prefill과 decode, 그리고 KV 캐시

서빙 엔진이 뭘 최적화하는지 이해하려면, LLM이 텍스트를 만드는 과정을 두 단계로 나눠 봐야 합니다.

  • prefill(프리필) — 입력 프롬프트 전체를 한 번에 읽어 들이는 단계입니다. 토큰이 500개면 500개를 병렬로 쭉 처리해 각 토큰의 내부 상태를 계산합니다. 연산량이 많지만 한 방에 끝나서 GPU를 잘 씁니다.
  • decode(디코드) — 답을 한 토큰씩 만들어 내는 단계입니다. 다음 토큰 하나를 뽑고, 그걸 다시 입력에 붙여 또 하나를 뽑고… 이 자기회귀(autoregressive) 반복이 답 길이만큼 이어집니다. 매 스텝이 토큰 딱 하나짜리 연산이라, GPU 입장에선 일감이 너무 작아 대부분 놀게 됩니다.

여기서 핵심 조연이 KV 캐시(KV cache)입니다. 트랜스포머는 새 토큰을 만들 때마다 앞에 나온 모든 토큰의 정보를 참조합니다(어텐션). 그때 필요한 게 각 토큰의 Key·Value 벡터인데, 매 스텝마다 처음부터 다시 계산하면 낭비가 엄청납니다. 그래서 한 번 계산한 K·V를 메모리에 저장해 두고 재사용합니다. 이게 KV 캐시입니다.

문제는 이 캐시가 요청 하나마다, 토큰이 늘어날수록 계속 커진다는 점입니다. 예를 들어 13B 모델에서 시퀀스 하나의 KV 캐시가 수백 MB에 달할 수 있습니다. 동시에 처리하는 요청이 많아질수록 GPU 메모리에서 KV 캐시가 차지하는 몫이 폭발합니다. 그래서 "GPU에 요청을 몇 개나 동시에 올릴 수 있는가"는 사실상 "KV 캐시를 얼마나 알뜰하게 담느냐"로 결정됩니다. 이 문장을 기억해 두세요. 뒤에 나올 최적화 두 개가 전부 여기서 출발합니다.

한 줄 요약. prefill은 GPU를 잘 쓰지만 짧고, decode는 GPU를 놀리면서 길게 이어집니다. 그리고 decode 내내 요청마다 KV 캐시가 GPU 메모리를 잡아먹습니다. 서빙 최적화 = decode의 놀고 있는 GPU 채우기 + KV 캐시 낭비 없애기입니다.

2. 순진한 서빙의 두 가지 낭비

이제 "순진하게" 서빙하면 어떤 일이 벌어지는지 봅시다. 사용자가 동시에 들어오니, 당연히 요청을 묶어서(batch) 한 번에 GPU에 태우고 싶습니다. 여기서 첫 번째 함정이 나옵니다.

낭비 ①: 정적 배칭(static batching)의 빈 자리. 순진한 방식은 요청 8개를 한 배치로 묶어 GPU에 올리고, 그 8개가 전부 끝날 때까지 기다린 뒤 다음 배치를 받습니다. 그런데 답 길이는 제각각입니다. 어떤 요청은 20토큰 만에 끝나고, 어떤 요청은 500토큰을 뱉습니다. 짧은 요청이 먼저 끝나도 그 자리는 배치 전체가 끝날 때까지 빈 채로 놀게 됩니다. 새 요청이 대기줄에서 기다리고 있어도 끼워 넣지 못합니다. GPU 사용률이 바닥을 치는 주범이 이겁니다.

낭비 ②: KV 캐시 메모리의 과다 예약과 파편화. KV 캐시는 요청이 진행되며 점점 커지는데, 순진한 시스템은 이 캐시를 메모리에 하나의 연속된 큰 덩어리로 잡습니다. 얼마나 길어질지 미리 모르니, "최대 2048토큰까지 갈 수도 있으니 일단 2048토큰 분량을 통째로 예약"해 둡니다. 실제로는 30토큰만 쓰고 끝나도 나머지는 붙잡아 둔 채 낭비되죠. 게다가 요청들이 들쭉날쭉 들어오고 나가면서 메모리 사이사이에 못 쓰는 빈틈(파편화, fragmentation)이 생깁니다.

원조 vLLM 논문은 이 낭비를 측정했습니다. 순진한 할당 방식(초기 Orca 등)에서는 KV 캐시로 잡은 메모리의 20~40%만 실제 토큰에 쓰이고, 60~80%가 예약·파편화로 버려졌습니다. 절반 넘게 그냥 놀린 겁니다. 메모리가 곧 동시 처리 요청 수인데, 그 메모리의 절반 이상을 버리고 있었던 거죠.

이 두 낭비를 각각 겨냥한 게 바로 continuous batching(낭비 ①)과 PagedAttention(낭비 ②)입니다. 하나씩 봅시다.

3. 아이디어 1 — continuous batching (놀지 마라)

continuous batching(연속 배칭), 다른 말로 iteration-level scheduling(반복 단위 스케줄링)의 발상은 단순합니다. "배치 하나가 전부 끝날 때까지 기다리지 말고, 토큰 생성 스텝 한 번이 끝날 때마다 배치를 다시 짜자."

즉 매 스텝마다 스케줄러가 확인합니다. 방금 끝난 요청이 있으면 배치에서 빼내 곧바로 사용자에게 돌려주고, 비게 된 그 자리에 대기줄의 새 요청을 즉시 끼워 넣습니다. 요청 단위가 아니라 반복(iteration) 단위로 스케줄링한다고 해서 iteration-level scheduling이라 부릅니다. 이 아이디어는 2022년 Orca 논문(OSDI '22)이 처음 제시했고, vLLM이 이를 받아 표준으로 굳혔습니다.

효과는 극적입니다. Orca 논문은 iteration-level scheduling으로 기존 시스템 대비 최대 36.9배 높은 처리량을 보고했고, Anyscale의 실측 글은 continuous batching만으로 처리량이 최대 23배 올랐다고 정리합니다. 배치 크기를 아무리 키워도 정적 배칭에서는 "짧은 요청이 긴 요청을 기다리는" 낭비가 사라지지 않는데, continuous batching이 그 구조적 낭비를 없앤 겁니다.

왜 이렇게 효과가 클까. decode 단계는 원래 GPU를 놀립니다(토큰 하나짜리 연산이라). 그 노는 연산 자원에 다른 요청들을 동시에 얹으면 거의 공짜로 처리량이 붙습니다. continuous batching은 그 "얹을 요청"을 매 스텝 최대한 확보하는 장치입니다.

4. 아이디어 2 — PagedAttention (메모리를 페이지로)

continuous batching으로 "빈 자리에 새 요청을 넣자"고 했지만, 넣으려면 그 요청의 KV 캐시를 담을 메모리가 있어야 합니다. 그런데 2장에서 봤듯 순진한 방식은 KV 캐시를 큰 연속 덩어리로 과다 예약해서, 정작 메모리가 금방 동납니다. 여기서 vLLM의 간판 기술 PagedAttention이 등장합니다.

발상의 원천은 운영체제(OS)의 가상 메모리와 페이징입니다. OS는 프로그램에게 메모리를 하나의 연속된 공간처럼 보여 주지만, 실제로는 뒤에서 작은 페이지(page) 단위로 쪼개 물리 메모리 여기저기에 흩어 담습니다. PagedAttention은 이걸 KV 캐시에 그대로 적용합니다.

  • KV 캐시를 고정 크기 블록(block)(예: 토큰 16개 분량)으로 잘게 나눕니다.
  • 요청이 길어지면 필요할 때마다 블록을 하나씩 새로 할당합니다. 미리 2048토큰을 통째로 예약하지 않습니다.
  • 블록들은 물리 메모리에서 연속일 필요가 없습니다. 논리적 순서를 블록 테이블(page table 격)이 기억해 두니, 흩어져 있어도 됩니다.

결과는 놀랍습니다. vLLM 논문에 따르면 PagedAttention은 KV 캐시 낭비를 4% 아래로 떨어뜨렸습니다(순진한 방식은 60~80% 낭비). 낭비가 사라지니 같은 GPU에 훨씬 많은 요청을 동시에 올릴 수 있고, 그만큼 continuous batching이 채울 "빈 자리"도 넉넉해집니다. 두 기술이 맞물려 vLLM은 초기 HuggingFace Transformers 대비 최대 24배 처리량을 냈습니다. 이 조합이 오늘날 거의 모든 서빙 엔진의 표준이 됐습니다.

보너스 — 공유가 공짜가 된다. KV 캐시를 블록으로 관리하면, 여러 요청이 같은 앞부분을 쓸 때 그 블록을 그대로 공유할 수 있습니다(OS의 copy-on-write처럼). 병렬 샘플링이나 같은 시스템 프롬프트를 쓰는 요청들이 대표적이죠. 이 "공유" 아이디어를 극단까지 밀어붙인 게 다음 절의 RadixAttention입니다.

5. 아이디어 3 — RadixAttention (프리픽스를 재사용하라)

실무 워크로드를 보면, 요청들이 앞부분을 자주 공유합니다. 긴 시스템 프롬프트를 모든 요청에 똑같이 붙이거나, few-shot 예시가 매번 동일하거나, 멀티턴 대화에서 앞선 대화가 계속 반복되거나, RAG에서 같은 문서 조각이 여러 질문에 다시 등장하거나. 이 공유되는 앞부분(prefix)의 KV 캐시를 매번 다시 계산하는 건 명백한 낭비입니다.

SGLang의 RadixAttention은 이 프리픽스 재사용을 자동으로 해 줍니다. 핵심 자료구조는 이름 그대로 radix tree(래딕스 트리)입니다. 처리한 시퀀스들의 KV 캐시를 토큰 단위로 트리에 보관하되, 공유되는 앞부분은 트리에서 한 경로로 합쳐 둡니다. 새 요청이 오면 트리를 따라 내려가며 "어디까지 이미 계산돼 있나"를 찾아, 겹치는 프리픽스는 계산을 건너뛰고 캐시를 그대로 씁니다. 메모리가 부족하면 LRU(least recently used) 방식으로 오래된 가지를 evict 합니다.

SGLang 논문(2023)은 이 구조로 당시 최고 수준 시스템 대비 최대 6.4배 높은 처리량을 보고했습니다. 특히 agent 제어, 논리 추론, few-shot, JSON 디코딩, RAG처럼 프리픽스가 크게 겹치는 워크로드에서 이득이 컸습니다.

혼동 주의 — 서버 프리픽스 캐싱 vs API 프롬프트 캐싱. RadixAttention은 서빙 엔진이 GPU 메모리 안에서 자동으로 KV 캐시를 재사용하는 기술입니다. Anthropic·OpenAI API에서 cache_control로 명시하는 프롬프트 캐싱과는 층위가 다릅니다. 다만 원리는 같습니다 — "겹치는 앞부분을 다시 계산하지 마라". 참고로 vLLM도 --enable-prefix-caching으로 자동 프리픽스 캐싱을 지원합니다. RadixAttention은 그중에서도 토큰 단위 트리로 매칭이 촘촘한 구현이라고 보면 됩니다.

6. 실전 — vLLM으로 직접 서버 띄우기

이론은 됐고, 실제로 얼마나 쉬운지 봅시다. vLLM의 큰 장점 하나는 OpenAI 호환 API 서버를 명령어 한 줄로 띄운다는 점입니다. 기존에 OpenAI SDK로 짠 코드를 base_url만 바꿔 그대로 붙일 수 있습니다.

먼저 설치하고 서버를 실행합니다. 위에서 배운 최적화들은 기본으로 켜져 있습니다. continuous batching과 PagedAttention은 끄는 옵션을 찾는 게 더 어렵죠.

{{< highlight bash >}} pip install vllm # OpenAI 호환 서버 실행 (continuous batching·PagedAttention 기본 ON) vllm serve Qwen/Qwen3-8B \ --enable-prefix-caching \ # 자동 프리픽스 캐싱 켜기 (반복 시스템 프롬프트에 유리) --max-model-len 8192 \ # 최대 컨텍스트 길이 --gpu-memory-utilization 0.90 # GPU 메모리의 90%까지 KV 캐시에 사용 {{< /highlight >}}

이제 클라이언트에서 부릅니다. 진짜로 OpenAI SDK 그대로입니다.

{{< highlight python >}} from openai import OpenAI # base_url만 로컬 vLLM 서버로 돌린다. api_key는 아무 값이나 OK. client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY") resp = client.chat.completions.create( model="Qwen/Qwen3-8B", messages=[{"role": "user", "content": "continuous batching을 한 문장으로 설명해줘"}], ) print(resp.choices[0].message.content) {{< /highlight >}}

처리량을 체감하려면 요청을 동시에 많이 던져 봐야 합니다. 여기가 핵심인데, 순진한 서버라면 100개를 동시에 던지면 거의 순차 처리되지만, vLLM은 continuous batching으로 이들을 촘촘히 묶어 처리합니다. 아래는 감을 잡기 위한 간단한 부하 스크립트입니다.

{{< highlight python >}} import asyncio, time from openai import AsyncOpenAI client = AsyncOpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY") async def one(i): r = await client.chat.completions.create( model="Qwen/Qwen3-8B", messages=[{"role": "user", "content": f"{i}번 질문: 파이썬 리스트와 튜플의 차이는?"}], max_tokens=128, ) return r.usage.completion_tokens async def main(n=100): t = time.perf_counter() # 100개를 '동시에' 던진다 → 서버가 continuous batching으로 함께 처리 toks = await asyncio.gather(*[one(i) for i in range(n)]) dt = time.perf_counter() - t print(f"{n}개 요청 / {sum(toks)} 토큰 / {dt:.1f}s → {sum(toks)/dt:.0f} tok/s") asyncio.run(main()) {{< /highlight >}}

동시 요청 수(n)를 1 → 10 → 100으로 올려 보면, 순진한 서버는 처리량이 거의 평평한데 vLLM은 어느 지점까지 쭉 오르는 걸 보게 됩니다. 바로 그 차이가 이 글에서 설명한 두 아이디어의 힘입니다.

SGLang도 거의 똑같습니다. pip install "sglang[all]" 후 python -m sglang.launch_server --model-path Qwen/Qwen3-8B로 띄우면 역시 OpenAI 호환 API가 열립니다. RadixAttention은 기본으로 켜져 있어, 시스템 프롬프트가 반복되는 워크로드라면 별도 설정 없이 이득을 봅니다.

7. vLLM vs SGLang, 언제 무엇을 (2026)

이제 이번 주 화제였던 벤치마크 이야기로 돌아갑시다. 2026년 여러 비교 글을 종합하면 큰 그림은 이렇습니다. 아래 수치는 특정 블로그의 H100 벤치마크값이라 워크로드·설정에 따라 달라진다는 점을 꼭 감안하세요.

상황추천이유
프리픽스가 크게 겹침
(RAG · 멀티턴 · 공용 시스템 프롬프트)		SGLangRadixAttention의 자동 프리픽스 재사용이 prefill 비용을 크게 줄임. 한 H100 비교에서 프리픽스 공유 시 vLLM 대비 약 29% 높은 처리량(예: 16,200 vs 12,500 tok/s), 프리픽스 집약 워크로드에선 최대 수 배까지.
모델을 자주 바꿔 끼움 · 넓은 하드웨어(그리고 생태계) 필요vLLM지원 모델·백엔드(NVIDIA·AMD·TPU 등)와 기여자 층이 가장 두껍고, 기능 업데이트가 빠름. 별 약 70K로 사실상 표준.
고유(중복 없는) 프롬프트 위주둘 다 비슷프리픽스가 안 겹치면 RadixAttention 이득이 거의 없어 기본 성능 싸움이 됨.
단일 모델을 최대 throughput으로 고정 서빙TensorRT-LLM도 후보NVIDIA에 특화해 커널 최적화가 깊음(대신 유연성·이식성은 떨어짐).

정리하면, SGLang은 프리픽스 재사용이 큰 워크로드에서 빛나고, vLLM은 생태계·유연성·범용성의 기본값입니다. 둘 다 continuous batching + 페이지 방식 KV 관리라는 토대는 공유합니다. 차이는 주로 프리픽스 캐싱을 얼마나 공격적으로, 어떤 자료구조로 하느냐에서 갈립니다.

8. 흔한 실수 · 팁

  • "vLLM 올렸는데 왜 안 빨라지죠?" — 요청을 순차로 던지고 있진 않나요. continuous batching은 동시에 밀려드는 요청이 있어야 진가가 납니다. for 문으로 하나 보내고 응답 기다리고 또 보내면, 배치에 채울 게 없어 그냥 느립니다. 부하를 병렬로 넣으세요.
  • --gpu-memory-utilization을 너무 낮게 잡으면 KV 캐시 공간이 좁아집니다. 이 값이 KV 캐시로 쓸 메모리를 정하고, 그게 곧 동시 처리 가능 요청 수입니다. 여유가 있으면 0.9 안팎까지 올려 동시성을 확보하세요(단, 다른 프로세스와 GPU를 공유하면 OOM 주의).
  • 프리픽스 캐싱은 "앞부분이 실제로 같을 때"만 듣습니다. 시스템 프롬프트에 매 요청 타임스탬프·랜덤 ID를 맨 앞에 넣으면 프리픽스가 매번 달라져 캐시가 통째로 깨집니다. 변하는 값은 뒤로, 고정된 값은 앞으로 두세요.
  • throughput과 latency는 다릅니다. 배치를 키우면 처리량은 오르지만 개별 요청의 응답 시간(특히 첫 토큰까지)은 늘 수 있습니다. 사용자 대면 챗봇이면 지연을, 대량 배치 처리면 처리량을 우선하는 식으로 목표를 정하고 설정을 맞추세요.
  • continuous batching이 있어도 prefill이 길면 튈 수 있습니다. 아주 긴 프롬프트의 prefill이 한 스텝을 오래 잡으면 다른 요청들의 디코딩이 잠깐 밀립니다. chunked prefill(긴 prefill을 잘게 쪼개 디코딩과 섞는 기법) 옵션이 이를 완화합니다. 긴 컨텍스트를 다룬다면 켜 볼 만합니다.
  • 벤치마크 숫자를 그대로 믿지 마세요. "SGLang이 29% 빠르다" 같은 수치는 특정 모델·입력 길이·프리픽스 공유율에서 나온 값입니다. 여러분의 실제 트래픽 분포로 직접 재보는 것이 유일하게 믿을 수 있는 비교입니다.

9. 마무리 (TL;DR)

  • 서빙 엔진이 빠른 건 모델이 똑똑해져서가 아니라, GPU와 KV 캐시 메모리의 낭비를 없애서입니다. 같은 가중치, 같은 계산, 다른 스케줄링·메모리 관리.
  • LLM 추론은 prefill(입력 한 번에)과 decode(토큰 하나씩)로 나뉘고, decode는 GPU를 놀립니다. 그 내내 요청마다 KV 캐시가 메모리를 잡아먹습니다.
  • continuous batching(Orca → vLLM)은 매 토큰 스텝마다 배치를 다시 짜, 끝난 요청 자리에 새 요청을 곧바로 끼워 넣어 GPU를 채웁니다. 최대 20배대 처리량.
  • PagedAttention(vLLM)은 KV 캐시를 OS 페이징처럼 작은 블록으로 관리해, 낭비 60~80%를 4% 아래로 줄입니다. 그만큼 동시 요청이 늘어납니다.
  • RadixAttention(SGLang)은 겹치는 프리픽스를 radix tree로 자동 재사용해, 반복 시스템 프롬프트·RAG·멀티턴에서 prefill을 크게 아낍니다.
  • 실무 선택: 프리픽스가 많이 겹치면 SGLang, 생태계·유연성이 우선이면 vLLM. 둘 다 OpenAI 호환 API를 한 줄로 띄웁니다. 그리고 반드시 직접 벤치마크하세요.
