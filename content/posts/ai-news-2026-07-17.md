# 🔥 오늘의 핵심

D-day였던 **Gemini 3.5 Pro**는 오늘(7/17) 목표마저 놓쳤다. 세 번째 지연이고, 구글은 완성판 대신 **스톱갭 릴리스**를 저울질 중 — 여전히 API·가격·모델카드가 없다. 그러니 오늘 코드에 반영할 실무 신호는 모델이 아니라 **에이전트 논문 두 편**이다. ① 에이전틱 코딩의 진짜 병목은 모델 성능이 아니라 **"한 명이 리뷰/승인하는" 1인 감독 루프**였고(2,361개 repo 실측), ② 옵티마이저 이득은 **회귀 통제가 있어야만 복리**로 쌓인다(RELAI-VCL 평생 76.4%).

# 📚 주목 논문

## 1) 에이전틱 코딩, 실제로 얼마나 쓰나 — GitHub 실측

- 저자: Maliha Noushin Raida, Daqing Hou (Clarkson Univ.)
- arXiv: **2607.14037** (cs.SE, 2026-07-15)
- 핵심 기여: 인기 GitHub repo **2,361개**의 **에이전트 생성 PR 25,264건**을 분석해, PR 단위가 아니라 **프로젝트 단위 채택·운영 패턴**을 처음으로 실측.
- 결과(요지):
  - 중앙값 repo는 3개월간 **PR 1~2건**뿐 — 집중 채택은 소수에 쏠려 있다.
  - **소규모 프로젝트(1~5명)**가 오히려 참여율·평균 활동이 높다.
  - 인간-에이전트 협업은 **1인 감독(single-human oversight)**이 압도적. 여러 사람이 함께 다루는 패턴은 드물다.
  - 업계가 말하는 "참여자당 3개월 36 PR" 수준을 넘는 repo는 극소수.
- 🛠 **에이전트 빌더 시사점:** 처리량 상한은 모델 성능이 아니라 **한 명이 리뷰·승인할 수 있는 양**이다. PR을 작게·설명 가능하게 쪼개고, diff 크기·리스크 기준으로 1인이 소화 가능한 단위로 자동 분할하는 라우팅을 앞단에 둬라.
- 한계: 초기 스냅샷(공개 repo 3개월)이라 장기 추세로 일반화는 이르다.
- 링크: <https://arxiv.org/abs/2607.14037>

![1인 감독 루프 다이어그램: 에이전트가 PR을 병렬 생성해도 사람 1명의 리뷰가 병목](/images/ai-news-2026-07-17/oversight-loop.svg)
*에이전트는 PR을 무한히 뽑아도, 병목은 "사람 1명의 리뷰 처리량". (직접 생성, 출처: arXiv 2607.14037)*

## 2) 에이전트 옵티마이저는 복리되는가

- 저자: Wenxiao Wang, Priyatham Kattakinda, Soheil Feizi (Univ. of Maryland)
- arXiv: **2607.14004** (cs.AI, 2026-07-15)
- 핵심 기여: 대부분의 옵티마이저 성과는 **고정 벤치마크 1회 최적화**로 보고된다. 이 논문은 **새 실패·새 과제가 계속 들어오는 배포 환경**을 흉내낸 2단계 연속학습으로 "이득이 누적되는가"를 검증.
- 방법(3줄): Terminal-Bench 2.0의 어려운 과제로 2단계 구성. GEPA·Meta Harness·RELAI-VCL을 **동일 예산**에서 비교. 1단계 최적화 후 새 과제를 넣고 다시 최적화해 전이·재개선을 측정.
- 결과: 정적 단일 단계에선 셋 다 baseline을 넘지만, 새 과제가 들어오면 갈린다. **GEPA는 baseline 아래로 전이**, Meta Harness는 전이는 되나 2차 최적화에서 정체. **RELAI-VCL만 전이·재개선을 모두 달성해 평생 평균 76.4%**(GEPA 66.0 · Meta Harness 64.6 · baseline 58.7).
- 🛠 **에이전트 빌더 시사점:** 이득이 복리로 쌓인 유일한 조건은 **최적화 루프 안에 회귀 통제(regression control)**를 넣어, 일반화 못 하는 지름길 해답에 편향을 거는 것. 프롬프트/하네스를 튜닝할 때 **고정 회귀 세트**를 함께 돌려 과거 성능이 깎이지 않는지 매 라운드 확인하라.
- 한계: Terminal-Bench 2.0 하드 과제·특정 3개 방법 비교라 다른 도메인 일반화는 검증 필요.
- 링크: <https://arxiv.org/abs/2607.14004>

![옵티마이저 평생 평균 pass rate 막대: RELAI-VCL 76.4, GEPA 66.0, Meta Harness 64.6, baseline 58.7](/images/ai-news-2026-07-17/optimizer-compound.svg)
*회귀 통제가 있는 방법만 새 과제 유입 후에도 이득을 유지·누적한다. (직접 생성, 출처: arXiv 2607.14004)*

- **GEPA** — [github.com/gepa-ai/gepa](https://github.com/gepa-ai/gepa) · Python
  - 프롬프트·코드·에이전트 설정 같은 **텍스트 구성요소를 임의 평가지표에 맞춰 최적화**하는 프레임워크. 평가기가 에러 메시지·프로파일링·추론 로그 같은 **Actionable Side Information(ASI)**을 돌려주면, LLM이 그걸 읽고 실패를 진단해 수정안을 제안하는 reflective evolution 방식.
  - 어댑터 제공: DefaultAdapter(단일 시스템 프롬프트), DSPy Full Program, **TerminalBench**(Terminus 에이전트 시스템 프롬프트 최적화).
  - 용처: 당신의 eval 지표에 맞춰 에이전트 시스템 프롬프트를 자동 진화. **단, 위 2607.14004 결과대로 GEPA 단독은 새 과제에서 이득이 안 쌓일 수 있으니 회귀 세트와 병행하라.**
- **Terminal-Bench** — [github.com/laude-institute/terminal-bench](https://github.com/laude-institute/terminal-bench)
  - 터미널 사용 에이전트를 평가하는 벤치마크. 오늘 논문(2607.14004)의 연속학습 실험대이자, 코딩 에이전트 하네스를 **재현 가능하게 회귀 평가**하는 표준 후보. 자체 에이전트를 여기에 물려 평생 성능 곡선을 그려보는 게 실전 활용법.

# 🧑‍💻 Codex & Claude Code 실전 팁

### 🟩 Codex — Config 프로필로 워크로드별 승인·샌드박스 분리
- 카테고리: Workflow / Config · 출처: [Codex Configuration Reference](https://developers.openai.com/codex/config-reference)
- 💡 **핵심:** 프로젝트마다 승인 강도(approval)와 샌드박스를 매번 바꾸지 말고, **named 프로필**로 저장해 `--profile`로 스위칭한다.
- 📝 **방법:** `~/.codex/config.toml`을 베이스로 두고, 프로필별 오버레이를 얹는다.

```toml
# ~/.codex/config.toml (베이스)
approval_policy = "on-request"   # 명령 실행 전 승인 요청 (균형)

# --profile yolo 로 로드되는 오버레이
[profiles.yolo]
approval_policy = "never"        # full-auto
sandbox_mode    = "workspace-write"
```

```bash
# 격리된 git worktree에서만 full-auto
codex --profile yolo "리팩터링 후 테스트 통과시켜"
```

- 🎯 **언제:** "리뷰용 신중 모드"와 "격리 워크트리 full-auto"를 오가며 쓸 때.
- ⚠️ **주의:** `full-auto`는 최악의 경우 워크트리를 날리는 격리 환경에서만. 운영 디렉터리에 직접 걸지 말 것.
- ⭐ **추천:** 팀이면 최소 2개 프로필(신중/자동)을 표준화해두면 사고가 준다.

### 🟦 Claude Code — `--safe-mode`로 "내 설정이 범인인가" 5초 격리
- 카테고리: Debugging · 출처: [Claude Code updates](https://releasebot.io/updates/anthropic/claude-code)
- 💡 **핵심:** 이상 동작이 **CLAUDE.md·플러그인·스킬·훅·MCP 서버 중 무엇 때문인지** 한 방에 가른다.
- 📝 **방법:**

```bash
# 모든 커스터마이즈를 끈 상태로 시작
claude --safe-mode
# 또는
CLAUDE_CODE_SAFE_MODE=1 claude
```

- 🎯 **언제:** 훅이 이상하게 걸리거나, MCP 툴이 안 붙거나, 응답이 갑자기 이상해질 때 — 순정 상태에서 재현되는지부터 확인.
- ⚠️ **주의:** safe-mode에선 당신의 워크플로 자동화가 전부 꺼지므로 실작업용이 아니라 **진단 전용**.
- ⭐ **추천:** 재현되면 설정 문제, 안 되면 세션/모델 문제 — 이슈 리포트 전 항상 이 단계부터.

### 🟪 공통 — 최적화 루프에 "고정 회귀 세트"를 박아라
- 카테고리: Eval · 출처: [arXiv 2607.14004](https://arxiv.org/abs/2607.14004)
- 💡 **핵심:** 새 실패를 잡으려 프롬프트/하네스를 재최적화할 때, **과거에 통과하던 케이스가 깎이지 않는지**를 같은 루프에서 강제 검증.
- 📝 **방법:** 잠근 회귀 세트를 매 최적화 라운드의 목적함수에 함께 넣는다.

```text
optimize(new_tasks) 목적함수 = maximize(new_pass)  ⛔ 위험
optimize(new_tasks) 목적함수 = maximize(new_pass)  ✅ 안전
                              AND keep(regression_pass ≥ 이전값)
```

- 🎯 **언제:** 프롬프트 옵티마이저(GEPA 등)·에이전트 하네스를 반복 튜닝하는 모든 상황.
- ⚠️ **주의:** 회귀 세트가 작으면 지름길 해답을 못 걸러낸다 — 대표 실패 유형을 폭넓게.
- ⭐ **추천:** 1회성 벤치 점수 대신 **평생 평균**으로 방법을 고르면 배포 후 후회가 준다.
