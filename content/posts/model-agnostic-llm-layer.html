2026년 6월, Claude Code v2.1.187에 "조직이 지정한 모델 제한" 기능이 들어갔습니다. 관리자가 막아 둔 모델을 고르면 "조직 설정에 의해 제한됨"이라는 메시지가 뜨고, 그 모델은 더 이상 쓸 수 없습니다.

— 모델은 내가 통제할 수 없는 의존성입니다. 어느 날 갑자기 사라질 수 있습니다.

코드를 짤 때 우리는 보통 openai.chat.completions.create(model="gpt-...") 한 줄을 깊숙한 곳에 박아 둡니다. 데모에서는 잘 돕니다. 그런데 프로덕션에서 이 한 줄은 가장 위험한 의존성이 됩니다. 내가 만들지도, 호스팅하지도, 통제하지도 못하는 외부 서비스 하나에 서비스 전체의 운명을 거는 셈이거든요.

이번 주 화제가 된 Claude Code의 모델 제한 기능은 빙산의 일각입니다. 모델은 예고 없이 deprecated 되고, 회사 정책이나 규제로 막히고, 트래픽이 몰리면 429 Too Many Requests를 뱉고, 가격이 오르고, 어느 날 더 좋은 모델로 갈아타고 싶어집니다. 그때마다 코드 곳곳에 박힌 모델 이름을 찾아 고치고 다시 배포해야 한다면, 그건 엔지니어링이 아니라 도박입니다.

그래서 2026년의 AI 엔지니어링에서 빠지지 않는 패턴이 모델-애그노스틱 레이어(model-agnostic layer)입니다. "특정 모델에 종속되지 않는 한 겹"을 앱과 모델 사이에 끼워 넣는 것이죠. 이 글에서는 그 레이어를 왜 두는지, 어떻게 만드는지, 그리고 초보가 꼭 빠지는 함정까지 차근차근 풀어 보겠습니다.

1. 모델은 내가 통제 못 하는 의존성이다

먼저 문제를 정확히 보겠습니다. 우리가 외부 모델 API를 부를 때 의존하는 것은 단순한 라이브러리가 아닙니다. 버전을 고정(pin)해 둘 수도 없고, 내 서버에 복제해 둘 수도 없는, 완전히 통제 밖에 있는 실시간 서비스입니다. 깨질 수 있는 지점을 늘어놓아 보면 이렇습니다.

  • 모델 단종(deprecation) — 모델 회사가 구버전을 내리고 새 버전으로 강제 이전합니다. 출력 성향이 미묘하게 달라져 프롬프트가 깨질 수 있습니다.
  • 접근 제한(governance) — 위 인용처럼 조직·규제·지역 정책으로 특정 모델이 막힙니다.
  • 일시 장애 — 5xx 서버 오류, 타임아웃, 또는 트래픽 폭주로 인한 429 레이트 리밋.
  • 컨텍스트 초과 — 입력이 모델의 컨텍스트 창보다 길면 ContextWindowExceeded 오류가 납니다.
  • 콘텐츠 정책 거부 — 안전 필터가 요청을 막아 응답이 안 옵니다.
  • 가격·성능 변화 — 더 싸거나 더 좋은 모델이 나오면 갈아타고 싶어집니다.

핵심은 이겁니다. 이 사건들은 "혹시"가 아니라 "언제"의 문제입니다. 운영을 길게 하다 보면 거의 다 한 번씩 겪습니다. 그러니 "안 깨지게" 만드는 게 아니라 "깨져도 서비스는 굴러가게" 만드는 게 목표입니다. 데이터베이스에 백업과 복제본을 두듯, 모델에도 대비책이 필요합니다.

2. 핵심 발상: 앱과 모델 사이에 "경계" 한 겹

해법의 출발점은 소프트웨어 공학의 오래된 원칙입니다. 변하는 것을 한곳에 가두라. 모델은 변합니다. 그러니 모델을 부르는 코드를 앱 전체에 흩뿌리지 말고, 딱 한 겹의 경계(boundary) 뒤에 숨깁니다. 앱의 나머지는 그 경계하고만 이야기합니다.

이 경계를 부르는 이름이 여럿입니다. SDK 수준이면 프로바이더 추상화(provider abstraction), 별도 서버로 띄우면 LLM 게이트웨이(gateway) 또는 프록시(proxy)라고 합니다. 이름은 달라도 발상은 같습니다. 앱은 "모델 X에게 이 메시지를 보내 줘"가 아니라, 그냥 "내 채팅 엔드포인트에 이 메시지를 보내 줘"라고만 말합니다. 그 뒤에서 실제로 누구를 부를지는 경계가 정합니다.

이 한 겹 덕분에 따라오는 이점이 큽니다. 모델을 바꾸려면 설정 한 줄만 고치면 됩니다. 여러 프로바이더의 키·요금·레이트 리밋을 한곳에서 관리합니다. 비용·토큰·지연시간 로깅도 경계에서 한 번에 합니다. 그리고 다음 절에서 볼 폴백(fallback)도 바로 여기서 일어납니다.

3. 게이트웨이를 직접 써보기 (LiteLLM)

가장 널리 쓰이는 오픈소스 게이트웨이가 LiteLLM입니다(2026년 6월 기준 GitHub 별 약 5.1만 개). 100개가 넘는 프로바이더 API를 OpenAI 호환 형식 하나로 감싸 줍니다. 즉, 코드는 OpenAI를 부르는 것처럼 짜고, 뒤에서 실제 모델만 바꿔 끼우는 식입니다.

가장 간단한 형태는 파이썬 SDK입니다. completion() 한 함수로 어떤 프로바이더든 똑같이 부릅니다.

{{< highlight python >}} # pip install litellm from litellm import completion # 모델 이름 문자열 하나만 바꾸면 프로바이더가 바뀐다 — 호출부는 동일 messages = [{"role": "user", "content": "한 줄로 자기소개 해줘"}] resp = completion(model="openai/gpt-5.1", messages=messages) # OpenAI # resp = completion(model="anthropic/claude-opus-4-8", messages=messages) # Anthropic # resp = completion(model="gemini/gemini-3-flash", messages=messages) # Google print(resp.choices[0].message.content) # 응답 구조도 OpenAI 형식으로 통일된다 {{< /highlight >}}

여기서 핵심은 messages를 만들고 응답을 꺼내는 코드가 프로바이더와 무관하게 똑같다는 점입니다. 모델 문자열만 설정값으로 빼 두면, 모델 교체가 코드 수정이 아니라 설정 변경이 됩니다.

팀 규모로 가면 SDK보다 프록시 서버 형태가 낫습니다. LiteLLM을 별도 서버로 띄우고, 모든 앱이 그 서버 하나만 바라보게 하는 거죠. 그러면 키 관리·예산·로깅·폴백 정책을 앱과 분리해 한곳에서 운영할 수 있습니다. 설정은 YAML로 합니다.

{{< highlight yaml >}} # config.yaml — 게이트웨이가 알고 있는 모델 목록 model_list: - model_name: chat-primary # 앱이 부르는 "별칭" — 앱은 이 이름만 안다 litellm_params: model: openai/gpt-5.1 api_key: os.environ/OPENAI_API_KEY - model_name: chat-backup # 같은 역할의 백업 모델 (다른 프로바이더) litellm_params: model: anthropic/claude-opus-4-8 api_key: os.environ/ANTHROPIC_API_KEY {{< /highlight >}}

이제 앱은 모델 이름이 아니라 chat-primary라는 역할 별칭을 부릅니다. 실제로 어떤 모델이 그 역할을 맡는지는 게이트웨이 설정이 결정합니다. 다음 절에서 이 구조 위에 폴백을 얹어 보겠습니다.

4. 폴백 라우팅 — 무엇이 폴백을 부르나

폴백(fallback)은 1차 모델 호출이 실패했을 때 자동으로 2차, 3차 모델로 넘어가는 장치입니다. 핵심은 "어떤 실패에 어떤 대응을 할지"를 실패 종류별로 나눠 정의하는 것입니다. 모든 실패를 똑같이 다루면 안 됩니다. 예를 들어 입력이 너무 길어서 난 오류라면, 같은 입력을 다른 모델에 그대로 보내도 또 터집니다. 컨텍스트 창이 더 큰 모델로 보내야 하죠.

LiteLLM 프록시에서는 이 정책을 router_settings 아래에 적습니다. 키 이름이 곧 위 그림의 분기와 일대일로 대응합니다.

{{< highlight yaml >}} router_settings: num_retries: 2 # 폴백 전에 같은 모델을 2번까지 재시도(지수 백오프) timeout: 30 # 한 호출당 최대 대기 시간(초) # 1) 일반 폴백: chat-primary가 실패하면 순서대로 시도 fallbacks: - {"chat-primary": ["chat-backup", "chat-last-resort"]} # 2) 입력이 컨텍스트 창을 넘으면 → 더 큰 창을 가진 모델로 context_window_fallbacks: - {"chat-primary": ["chat-bigctx"]} # 3) 안전 필터에 막히면 → 정책이 다른 모델로 content_policy_fallbacks: - {"chat-primary": ["chat-backup"]} {{< /highlight >}}

동작 순서는 이렇습니다. 먼저 1차 모델을 num_retries만큼 재시도합니다(일시적인 429·5xx는 잠깐 기다리면 풀릴 때가 많으니까요). 그래도 안 되면 오류 종류를 보고 알맞은 폴백 목록의 다음 모델로 넘어갑니다. ContextWindowExceededError면 context_window_fallbacks를, 콘텐츠 정책 거부면 content_policy_fallbacks를, 그 외 일반 실패는 fallbacks를 탑니다.

왜 굳이 나눌까? 폴백을 하나로 뭉뚱그리면 비효율이 생깁니다. 입력이 길어서 난 오류인데 같은 길이를 다른 일반 모델에 보내면 또 ContextWindowExceeded가 납니다. 재시도·폴백을 헛돌게 만들 뿐이죠. "이 실패에는 이 대응"을 분리하는 게 폴백 설계의 핵심입니다.

5. 함정: 테스트 안 한 폴백은 폴백이 아니다

여기서 초보가 가장 많이 빠지는 함정을 짚겠습니다. 폴백을 걸어 두면 마음이 놓입니다. "1차가 죽어도 2차가 받아주겠지." 그런데 한 번도 검증한 적 없는 2차 모델은 사실 폴백이 아닙니다. 정작 1차가 죽는 위기의 순간에, 2차가 프롬프트를 엉뚱하게 해석하거나 도구 호출 형식을 못 맞춰서 더 큰 사고를 낼 수 있거든요.

모델마다 같은 프롬프트에 다르게 반응합니다. 시스템 프롬프트를 따르는 정도, JSON을 정확히 뱉는 능력, 도구 호출(tool calling) 형식, 거절하는 성향이 다 다릅니다. 그래서 "갈아탈 수 있는 상태"를 만들려면 평가(eval)로 폴백을 미리 검증해야 합니다. 이것이 모델-애그노스틱 평가(model-agnostic eval)입니다.

방법은 단순합니다. 내 작업을 대표하는 입력·기대출력 묶음, 즉 골든 데이터셋(golden dataset)을 만들어 두고, 똑같은 평가를 1차 모델과 폴백 후보에 각각 돌려 점수를 비교합니다. 폴백 후보가 합격선을 넘으면 그제야 "production-ready" 도장을 찍는 거죠.

{{< highlight python >}} # 같은 평가 세트를 여러 모델에 똑같이 돌려 비교한다 from litellm import completion CANDIDATES = ["openai/gpt-5.1", "anthropic/claude-opus-4-8"] # 1차와 폴백 후보 # golden: [{"input": ..., "check": <합격 판정 함수>}, ...] — 내 작업을 대표하는 예시들 def run_eval(model, golden): passed = 0 for case in golden: out = completion(model=model, messages=[{"role": "user", "content": case["input"]}]) text = out.choices[0].message.content if case["check"](text): # 정답 포함? 형식 맞음? JSON 파싱됨? 등 passed += 1 return passed / len(golden) # 합격률 for m in CANDIDATES: score = run_eval(m, golden) print(f"{m:32s} 합격률 {score:.1%}") # 폴백 후보가 합격선(예: 0.9) 미만이면 → 아직 폴백으로 쓰면 안 된다 {{< /highlight >}}

check 함수는 작업에 맞게 정합니다. "정답 키워드가 들어 있나", "JSON으로 파싱되나", "필수 필드가 있나" 같은 규칙 기반이면 가장 싸고 안정적입니다. 더 주관적인 품질(말투·정확성)을 볼 때는 다른 LLM에게 채점을 맡기는 LLM-as-a-judge 방식을 씁니다. 다만 여기에 함정이 하나 더 있습니다.

판정 모델(judge)도 모델이다. LLM-as-a-judge로 채점한다면, 채점하는 모델 버전을 고정(pin)해야 합니다. 어느 날 판정 모델이 조용히 업데이트되면, 코드는 그대로인데 점수만 들썩여 "성능이 떨어졌다"는 착각을 부릅니다. 판정 모델을 바꿀 때는 캘리브레이션 세트(정답을 아는 묶음)로 점수 차이를 먼저 재고, 그 교체를 "평가 스위트 마이그레이션"으로 신중히 다뤄야 합니다.

핵심 원칙은 이겁니다. "내 폴백은 검증됐는가?"에 숫자로 답할 수 없다면, 그건 폴백이 아니라 희망입니다.

6. 관측: 프로바이더를 공정하게 비교하기

모델을 갈아탈 수 있으려면, 갈아탔을 때 무슨 일이 벌어지는지 같은 잣대로 봐야 합니다. 그런데 프로바이더마다 응답 로그 형식이 제각각이면 비교가 안 됩니다. 그래서 2026년의 사실상 표준이 된 것이 OpenTelemetry GenAI 시맨틱 컨벤션입니다.

OpenTelemetry(OTel)는 원래 일반 소프트웨어의 로그·트레이스·메트릭을 표준화한 CNCF 프로젝트입니다. 여기에 LLM용 규약이 추가됐는데, gen_ai.*라는 통일된 속성 이름으로 요청 모델·프로바이더·토큰 수·비용·도구 호출·종료 사유 같은 걸 기록합니다. 어떤 프로바이더를 쓰든 같은 형식으로 남으니, 벤더에 묶이지 않고 공정하게 비교·전환할 수 있습니다.

알아둘 점. 2026년 기준 GenAI 시맨틱 컨벤션의 상당수는 아직 experimental(실험) 단계입니다. 그래도 Datadog 같은 주요 관측 도구가 네이티브로 지원하기 시작했고, OTLP를 받는 백엔드(Langfuse·Phoenix·Honeycomb 등)면 어디로든 보낼 수 있습니다. 한 번 OTel로 계측해 두면 관측 도구를 갈아탈 때도 코드를 다시 안 짜도 됩니다 — 모델뿐 아니라 관측 벤더에도 묶이지 않는 셈이죠.

실무에서 게이트웨이(LiteLLM 등)는 이런 로깅을 대신 해주는 경우가 많습니다. 그러니 직접 계측 코드를 다 짜기보다, 게이트웨이의 로깅·관측 연동 기능을 먼저 켜 보는 걸 권합니다. 폴백이 실제로 몇 번 발동했는지, 어느 모델이 더 싸고 빠른지가 한 대시보드에서 보이기 시작하면, 그때부터 모델 선택이 감이 아니라 데이터가 됩니다.

7. 무엇을 쓸까 — 선택지 비교

"모델-애그노스틱 레이어"를 만드는 길은 하나가 아닙니다. 상황에 맞게 고르면 됩니다. 대표 선택지를 정리했습니다.

8. 흔한 실수 · 팁

  • 프롬프트는 모델 간에 그대로 옮겨지지 않는다. A에 맞춰 튜닝한 프롬프트가 B에선 JSON을 깨거나 도구 호출 형식을 틀릴 수 있습니다. 폴백을 켤 거면 프롬프트도 후보 모델에서 검증하세요(5절).
  • 폴백 개수에 제한이 있는 서비스가 있다. 예컨대 OpenRouter는 fallbacks를 3개까지만 받습니다. "일단 다 넣어두자"가 안 통합니다.
  • 재시도 시 이중 청구를 조심하라. 같은 요청을 재시도·폴백하면 결제가 두 번 될 수 있습니다. 게이트웨이가 지원하면 idempotency key(멱등 키)를 붙여 같은 요청이 중복 과금되지 않게 하세요.
  • 판정 모델 버전을 고정하라. LLM-as-a-judge를 쓴다면 채점 모델을 pin 하고, 바꿀 땐 캘리브레이션부터(5절).
  • 폴백이 발동했다는 사실 자체를 알람으로 띄워라. 폴백은 "조용히 잘 막아줬다"가 아니라 "1차가 아프다"는 신호입니다. 발동 횟수를 관측(6절)하고 임계치를 넘으면 알림을 받으세요.
  • 비용·지연을 모델별로 기록하라. 폴백 모델이 훨씬 비싸거나 느리면, 장애가 길어질 때 비용 폭탄이나 지연 폭증으로 이어집니다. gen_ai.* 메트릭으로 미리 파악해 두세요.
  • 경계를 우회하지 마라. 급하다고 특정 파일에서 프로바이더 SDK를 직접 부르기 시작하면, 애써 만든 한 겹이 무너집니다. 모델 호출은 무조건 게이트웨이를 통하게 규칙으로 못 박으세요.

9. 마무리 (TL;DR)

  • 모델은 통제 밖 의존성이다. 단종·제한·장애·가격변화는 "혹시"가 아니라 "언제"의 문제다.
  • 앱과 모델 사이에 경계 한 겹을 둬라. 게이트웨이/프록시(LiteLLM 등)나 프로바이더 추상화(Vercel AI SDK 등)로, 앱은 역할 별칭만 부르게 한다.
  • 폴백은 실패 종류별로 나눠라. 일시 오류는 재시도, 일반 실패는 다른 프로바이더, 컨텍스트 초과는 더 큰 모델, 정책 거부는 정책이 다른 모델로.
  • 테스트 안 한 폴백은 폴백이 아니다. 골든 데이터셋으로 1차와 폴백을 같은 평가에 돌려, 합격선을 넘은 것만 "production-ready"로 인정한다. 판정 모델은 pin 한다.
  • OpenTelemetry GenAI 컨벤션으로 관측하라. gen_ai.* 표준 속성으로 프로바이더를 공정하게 비교하고, 폴백 발동을 알람으로 본다.
  • 도구보다 패턴이 본질이다. 무엇을 쓰든 "한 곳에 묶이지 않는다"는 원칙을 지키면, 모델이 사라져도 서비스는 굴러간다.
