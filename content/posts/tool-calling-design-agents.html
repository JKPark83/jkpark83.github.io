에이전트가 이상하게 굴 때, 열에 예닐곱은 모델이 멍청해서가 아닙니다. 도구 이름이 send_user와 send_channel처럼 헷갈리거나, 설명이 부실하거나, 에러가 그냥 빈 문자열로 돌아와서 모델이 다음에 뭘 해야 할지 알 수 없게 만든 겁니다.

— 함수 호출에서 모델이 하는 일은 실행이 아니라 "결정"입니다. 그 결정을 잘 하도록 판을 깔아 주는 게 우리 몫입니다.

에이전트(agent)를 처음 만들어 본 junior 개발자가 자주 부딪히는 벽이 있습니다. 날씨 봇을 만들려고 get_weather 함수를 붙였습니다. 데모 질문 "서울 날씨 알려줘"엔 잘 답합니다. 그런데 "이번 주말에 우산 챙겨야 해?"라고 물으면 도구를 안 부르고 지어내거나, 엉뚱한 도시를 넣어 부릅니다. "모델이 부족한가?" 싶지만, 같은 모델이 다른 사람 코드에선 멀쩡히 잘 돕니다.

차이는 대개 도구를 어떻게 설계했느냐에 있습니다. 함수 호출(tool calling, function calling이라고도 합니다)은 그냥 "LLM에 함수를 연결하는 기능"이 아니라, 모델과 우리 코드가 주고받는 하나의 프로토콜입니다. 이 프로토콜의 규칙과 함정을 모르면, 아무리 좋은 모델을 붙여도 에이전트는 도구를 헛짚습니다.

마침 이번 주(2026년 7월) 이 주제가 실무자 타임라인에서 다시 화제입니다. 도구 호출 능력을 재는 Berkeley Function Calling Leaderboard(BFCL)가 v4로 갱신되며 "이제는 단발 호출이 아니라 여러 도구를 계획해서 쓰는 agentic 능력을 본다"는 이야기가 돌고, Anthropic은 advanced tool use(도구 검색·프로그래밍적 도구 호출)를 정식으로 내놓으며 "도구 정의가 컨텍스트를 잡아먹는다"는 오래된 고민에 답을 냈습니다. 에이전트의 해라는 2026년에, 결국 병목은 "모델을 얼마나 잘 쓰느냐"가 아니라 "도구를 얼마나 잘 설계하느냐"로 옮겨 온 셈입니다.

이 글은 그 함수 호출을 처음부터 짚습니다. 실제로 어떻게 도는지, 도구 정의는 왜 API 문서처럼 써야 하는지, 도구가 많아지면 무슨 세금이 붙는지, 병렬 호출과 에러는 어떻게 설계하는지, 그리고 "그래서 잘 되는지"를 BFCL·tau-bench로 어떻게 재는지 — 쉽고 자세하게 살펴봅니다.

1. 함수 호출은 실제로 어떻게 도는가

가장 먼저 오해부터 풉시다. LLM은 함수를 실행하지 않습니다. 이게 함수 호출의 전부를 관통하는 핵심입니다. 모델이 하는 일은 딱 하나예요 — "지금은 get_weather를 {"city": "서울"} 인자로 부르는 게 맞겠다"라는 의도를 구조화된 데이터(JSON)로 뱉는 것. 실제로 그 함수를 호출해 인터넷에서 날씨를 긁어 오는 건 우리 코드(런타임)가 합니다.

왜 이렇게 나눌까요? 모델은 샌드박스 밖으로 나가 API를 때리거나 DB를 건드릴 수 없기 때문입니다. 그래서 역할을 쪼갭니다. 모델은 "무엇을, 어떤 인자로 부를지" 정하고(brain), 우리 코드는 "실제로 부르고, 결과를 받아 다시 모델에게 먹여 주는" 손발(hands) 노릇을 합니다. 이 왕복이 한 번으로 끝나지 않고 "더 부를 도구가 없다"고 모델이 판단할 때까지 반복되는데, 이걸 에이전트 루프(agentic loop)라고 부릅니다.

이 그림에서 꼭 기억할 두 가지. 첫째, ②와 ③은 다른 주체입니다 — 모델이 인자를 잘못 넣으면 그건 모델 탓이지만, 함수가 터지거나 엉뚱한 값을 돌려주면 그건 우리 코드 탓입니다. 둘째, ③의 결과를 ④에서 다시 모델에게 돌려줘야 루프가 이어집니다. 이 되먹임을 빼먹으면 모델은 자기가 부른 함수의 결과를 영영 모릅니다(초보가 가장 많이 하는 실수입니다).

2. 도구 정의 = API 문서다

모델은 도구의 구현을 못 봅니다. 오직 우리가 준 도구 정의(tool definition)만 보고 "이 도구가 뭘 하는지, 언제 써야 하는지, 인자가 뭔지" 판단합니다. 그래서 도구 정의는 사실상 모델에게 주는 API 문서입니다. 문서가 부실하면 아무리 똑똑한 개발자(모델)라도 API를 잘못 씁니다.

도구 정의는 보통 JSON Schema로 씁니다. 세 부분이 핵심입니다 — name(무엇), description(언제·왜), input_schema(어떤 인자). Anthropic 문서 기준으로 하나 보겠습니다.

{{< highlight json >}} { "name": "get_weather", "description": "특정 도시의 현재 날씨와 강수 확률을 조회한다. 사용자가 날씨, 기온, 비/눈, 우산 여부 등을 물을 때 사용하라. 미래 예보가 아니라 '현재' 관측값을 반환한다.", "input_schema": { "type": "object", "properties": { "city": { "type": "string", "description": "도시 이름. 한국 도시는 한글로 (예: 서울, 부산)." }, "unit": { "type": "string", "enum": ["celsius", "fahrenheit"], "description": "온도 단위. 기본값 celsius." } }, "required": ["city"] } } {{< /highlight >}}

여기서 정확도를 가르는 건 description입니다. Anthropic 엔지니어링 글은 대놓고 말합니다 — "도구 설명의 품질이 에이전트가 도구를 올바르게 쓰는지를 결정하는 가장 큰 단일 변수"라고요. 좋은 설명은 "무엇을 하는가"뿐 아니라 "언제 써야 하는가"와 "무엇을 반환하는가"까지 명시합니다. 위 예에서 "현재 관측값이지 미래 예보가 아니다"라는 한 줄이, 모델이 "주말 예보"를 이 도구로 착각하는 걸 막아 줍니다.

비슷한 이름은 재앙입니다. Anthropic이 관찰한 가장 흔한 실패가 잘못된 도구 선택과 잘못된 인자인데, 특히 notification-send-user와 notification-send-channel처럼 이름이 비슷할 때 심해집니다. 두 도구가 하는 일이 "거의 같은데 살짝 다르면" 모델은 매번 헷갈립니다. 규칙은 단순합니다 — "에이전트가 왜 A 대신 B를 골라야 하는지 한 문장으로 설명 못 하면, 경계가 불분명한 것"입니다. 합치거나, 확실히 다르게 만드세요. 어정쩡한 중간은 프로덕션에서 안 통합니다.

도구를 "사람이 아니라 모델이 읽는 함수"로 보면 설계 감각이 잡힙니다. 사람에겐 편한 list_users(filter)의 자유 형식 filter 문자열이 모델에겐 지옥입니다(형식을 지어내니까요). 차라리 status, created_after처럼 구조화된 파라미터로 쪼개고 enum으로 선택지를 못 박는 편이 훨씬 안정적입니다. 애매한 자유도를 주지 말고, 밟을 수 있는 길만 깔아 주세요.

3. 도구가 많아질 때 붙는 두 가지 세금

"도구를 많이 붙이면 만능 에이전트가 되겠지?" 초보의 흔한 착각입니다. 도구가 늘면 두 가지 세금이 붙습니다.

세금 ①: 토큰. 모든 도구 정의는 매 요청마다 컨텍스트(context, 모델이 한 번에 보는 입력)에 실려 갑니다. 도구가 많으면 사용자가 질문을 꺼내기도 전에 컨텍스트가 도구 설명으로 가득 찹니다. Anthropic 보고에 따르면 도구 정의가 5만 토큰 이상을 잡아먹는 경우가 흔하고, 극단적으로는 13만 4천 토큰까지 간 사례도 있습니다. 이건 그대로 비용·지연이고, 정작 대화에 쓸 공간을 깎아먹습니다.

세금 ②: 선택 정확도. 사람도 메뉴판이 세 장이면 못 고릅니다. 모델도 똑같아서, 선택지가 늘수록 올바른 도구를 고를 확률이 떨어집니다. 도구 20개 중 하나를 고르는 것과 200개 중 하나를 고르는 건 난이도가 다릅니다.

그래서 실무의 첫 처방은 화려한 기술이 아니라 줄이고 정리하기입니다.

  • 통합·차별화 — 겹치는 도구는 합치고, 남길 거면 경계를 뾰족하게. "왜 이걸 쓰지?"에 한 문장으로 답할 수 있어야 합니다.
  • 네이밍 접두사 — calendar_create, calendar_list처럼 도메인 접두사를 붙이면, 모델의 선택이 "어느 범주냐 → 그중 뭐냐" 두 단계로 쉬워집니다.
  • 필요할 때만 노출 — 워크플로 단계별로 켜고 끄면 한 번에 보이는 도구 수 자체가 줄어듭니다.
동적 도구 로딩은 따로 있습니다. 도구 수백 개를 다뤄야 한다면, 정의를 전부 업프론트로 올리지 않고 필요한 것만 검색해 불러오는 Tool Search(defer_loading)나, 도구를 코드로 조율하는 Programmatic Tool Calling 같은 기법이 있습니다. Anthropic 벤치마크에서 프로그래밍적 도구 호출은 75개 도구를 쓰는 프로젝트 관리 에이전트의 청구 입력 토큰을 약 38% 줄였고, 도구가 10~49개인 경우 대체로 20~40%를 아꼈다고 합니다. 이 "컨텍스트 폭증" 자체를 깊게 다룬 이전 글(도구 메뉴판이 너무 길다)이 있으니, 여기선 "도구가 많으면 세금이 붙는다"는 사실만 기억하고 넘어갑니다.

4. 병렬 호출과 에러 설계

병렬 호출 — 독립적인 일은 한 번에

"서울이랑 부산 날씨 둘 다 알려줘"라는 질문에, 모델이 get_weather(서울)을 부르고 → 결과 받고 → 다시 get_weather(부산)을 부르면 왕복이 두 번입니다. 느리죠. 요즘 모델은 서로 의존하지 않는 도구를 한 번에 여러 개 부를 수 있습니다. 한 응답에 tool_use 블록이 여러 개 담겨 오고, 우리 코드가 그걸 (가능하면) 동시에 실행합니다.

에러를 "값"으로 설계하라

초보 코드에서 도구가 실패하면 보통 예외가 터져 루프가 죽습니다. 하지만 함수 호출의 세계에선 에러도 모델에게 돌려주는 하나의 결과입니다. 핵심은, 모델이 그 에러를 읽고 스스로 복구할 수 있게 만드는 것입니다.

그러려면 에러를 사람 말이 아니라 기계가 해석 가능한 구조로 돌려줘야 합니다. 아래 둘을 비교해 보세요.

{{< highlight python >}} # 나쁜 에러 — 모델이 뭘 해야 할지 알 수 없다 return "Error: something went wrong" # 좋은 에러 — 무엇이·왜·다음에 뭘 할지가 담긴다 return { "error": "rate_limited", # 무슨 오류인지 (기계가 분기 가능) "message": "요청이 너무 잦습니다.", "retry_after": 30, # 30초 뒤 재시도하라는 신호 } {{< /highlight >}}

아래처럼 돌려주면 모델은 "아, 30초 기다렸다 다시 부르면 되는구나"를 추론합니다. 도구 결과를 모델에게 되먹일 때 "이건 에러다"라는 표시(예: is_error: true)를 함께 주면 더 확실합니다. 정리하면 도구의 출력은 성공이든 실패든 "모델이 다음 행동을 결정하기에 충분한 정보"여야 합니다.

5. 실전 — 도구 호출 루프 만들기

말보다 코드입니다. Anthropic SDK로 §1의 루프를 그대로 구현해 보겠습니다. 뼈대는 놀랄 만큼 단순합니다 — "모델이 도구를 부르면 실행해서 결과를 되먹이고, 안 부를 때까지 반복".

{{< highlight python >}} import anthropic client = anthropic.Anthropic() # ANTHROPIC_API_KEY 사용 # 1) 도구 정의 — 모델이 읽는 'API 문서' (§2 참고) tools = [{ "name": "get_weather", "description": "특정 도시의 현재 날씨와 강수 확률을 조회한다. " "날씨·기온·비/눈·우산 여부를 물을 때 사용. 현재 관측값만 반환한다.", "input_schema": { "type": "object", "properties": { "city": {"type": "string", "description": "도시 이름 (예: 서울)"}, }, "required": ["city"], }, }] # 2) 실제 실행부 — 이건 '우리 코드'의 몫 (모델이 아니라) def run_tool(name, args): if name == "get_weather": # 실제로는 여기서 날씨 API를 호출한다 return {"temp_c": 11, "rain_prob": 0.8} return {"error": "unknown_tool", "message": f"모르는 도구: {name}"} messages = [{"role": "user", "content": "이번 주말 서울, 우산 챙겨야 해?"}] # 3) 에이전트 루프 — 모델이 도구를 안 부를 때까지 반복 while True: resp = client.messages.create( model="claude-sonnet-5", max_tokens=1024, tools=tools, messages=messages, ) messages.append({"role": "assistant", "content": resp.content}) if resp.stop_reason != "tool_use": # 더 부를 도구가 없다 → 최종 답. 루프 종료 print(resp.content[-1].text) break # 응답에 담긴 tool_use 블록들을 실행해 결과를 되먹인다 (병렬 가능) results = [] for block in resp.content: if block.type == "tool_use": out = run_tool(block.name, block.input) results.append({ "type": "tool_result", "tool_use_id": block.id, # 어떤 호출의 결과인지 반드시 매칭 "content": str(out), }) messages.append({"role": "user", "content": results}) # 결과를 대화에 추가 {{< /highlight >}}

딱 세 덩어리입니다. 도구 정의(모델이 읽는 문서), 실행부 run_tool(우리 코드가 실제로 하는 일), 그리고 루프(stop_reason이 tool_use인 동안 결과를 되먹임). 여기서 놓치기 쉬운 두 지점 — tool_use_id로 "어떤 호출의 결과인지"를 반드시 매칭해야 하고, 결과는 assistant가 아니라 user 역할로 되먹입니다. OpenAI·Gemini도 이름만 다를 뿐(tool_calls/function_call) 구조는 똑같습니다 — 모델이 호출을 뱉고, 내가 실행하고, 결과를 되먹인다.

6. 그래서 잘 되는지 어떻게 아나 (BFCL·tau-bench)

도구를 잘 설계했는지 "느낌"으로 알 순 없습니다. 다행히 도구 호출 능력을 재는 표준 벤치마크가 있고, 여기서 쓰는 측정 방식을 이해하면 내 에이전트를 평가하는 감각도 생깁니다.

BFCL v4 — 무엇을, 어떻게 재나

Berkeley Function Calling Leaderboard(BFCL)는 UC Berkeley Gorilla 프로젝트가 만든, 가장 널리 인용되는 도구 호출 벤치마크입니다. 재밌는 건 채점 방식이에요. 함수를 실제로 실행하지 않고, 모델이 생성한 호출을 추상 구문 트리(AST, abstract syntax tree)로 파싱해 함수 이름·필수 인자·값·타입이 정답과 맞는지 비교합니다(인자 순서는 무시). "제대로 된 호출을 만들었는가"를 실행 없이 결정적으로 검증하는 거죠.

2026년 4월 나온 v4는 단발 호출을 넘어 여러 도구를 계획해서 쓰는 agentic 능력까지 보도록 재편됐습니다. 대략 이런 구성입니다(가중치는 공개 리더보드 기준).

여기서 junior가 얻을 교훈이 있습니다. 좋은 도구 호출은 "부를 때 잘 부르는 것"만이 아니라 "안 부를 때 참는 것"도 포함합니다. 관련 도구가 없는데 억지로 뭔가를 부르는 게 바로 도구 환각(hallucination)이고, BFCL이 이걸 따로 재는 이유입니다.

tau-bench — 진짜 대화에서 끝까지 되나

단발 호출을 잘해도 실제 서비스는 다릅니다. 사용자는 말을 바꾸고, 정보를 조금씩 흘리고, 도메인엔 지켜야 할 정책이 있죠. tau-bench(τ-bench)는 Sierra가 만든 벤치마크로, LLM이 흉내 낸 사용자와 에이전트가 여러 차례 대화하며 도메인 API와 규칙을 쓰게 합니다. 채점은 영리하게도 대화가 끝난 뒤 DB의 최종 상태가 목표 상태와 같은지로 판정합니다(말이 아니라 결과로).

tau-bench가 남긴 가장 중요한 개념이 pass^k입니다. 같은 과제를 k번 시켜서 매번 성공해야 점수를 주는 지표죠. 한 번 잘하는 것과 열 번 중 열 번 잘하는 건 프로덕션에서 완전히 다릅니다. 에이전트의 진짜 병목이 똑똑함이 아니라 신뢰성(consistency)이라는 걸 숫자로 못 박은 겁니다. 내 에이전트를 평가할 때도 이 감각을 빌려 오세요 — 한 번 돌려 보고 "된다"가 아니라, 라벨 붙인 작은 셋으로 여러 번 돌려 성공률과 그 편차를 보라는 뜻입니다.

7. 흔한 실수 · 팁

  • 결과를 되먹이지 않음 — 함수를 실행하고 그 tool_result를 대화에 다시 넣지 않으면, 모델은 자기가 부른 결과를 모른 채 헛돕니다. 초보 1순위 버그입니다(§1·§5).
  • 도구를 무작정 늘리기 — 많을수록 토큰이 불어나고 선택 정확도는 떨어집니다. 겹치는 도구는 합치고, 남길 건 경계를 뾰족하게(§3).
  • 이름·설명이 헷갈림 — get_data 같은 모호한 이름, 비슷한 두 도구는 오호출을 부릅니다. "언제 쓰는지"를 설명에 명시하세요(§2).
  • 자유 형식 인자 — 자유 문자열 filter는 모델이 형식을 지어냅니다. 구조화된 파라미터 + enum으로 선택지를 못 박으세요.
  • 에러를 예외로 죽이기 — 실패를 사람 말 한 줄로 던지지 말고, {"error": ..., "retry_after": ...}처럼 모델이 복구할 수 있는 구조로 되먹이세요(§4).
  • 병렬 가능한 걸 직렬로 — 서로 의존하지 않는 호출을 한 번에 처리하지 않으면 괜히 느립니다. 반대로 앞 결과가 뒤 입력이 되는 의존 관계를 병렬로 부르면 틀립니다.
  • 한 번 되면 "된다"고 착각 — 도구 호출은 확률적입니다. pass^k 관점으로 여러 번 돌려 성공률과 편차를 재세요(§6).
  • 보안을 모델에 맡기기 — 위험한 도구(삭제·결제·메일 발송)는 모델 판단만 믿지 말고, 최소 권한·읽기/쓰기 분리·사람 확인 같은 가드를 우리 코드에 두세요.
한 줄 원칙. 도구는 "모델이 읽는 API"이고, 모델은 그 API의 사용자입니다. 사람 개발자에게 좋은 API(명확한 이름, 좁은 인자, 친절한 에러 메시지, 필요 없을 땐 안 부를 자유)가 모델에게도 좋은 도구입니다. 도구 설계가 막힐 땐 "내가 이 API 문서만 보고 처음 쓴다면?"을 떠올리세요.

8. 마무리 (TL;DR)

  • 함수 호출(tool calling)에서 모델은 실행하지 않는다. 어떤 도구를 어떤 인자로 부를지 JSON으로 "정하기"만 하고, 실행은 우리 코드가 한다. 결과를 되먹여 도구가 필요 없을 때까지 반복하는 게 에이전트 루프다.
  • 도구 정의는 모델에게 주는 API 문서다. description의 품질이 정확도의 최대 변수 — "무엇을·언제·무엇을 반환"까지 써라. 비슷한 이름의 도구는 오호출의 근원이다.
  • 도구가 많으면 토큰(5만+ 토큰 흔함)과 선택 정확도 두 세금이 붙는다. 통합·네이밍·필요 시 노출로 줄여라. 수백 개면 Tool Search·Programmatic Tool Calling을 검토.
  • 서로 의존하지 않는 호출은 병렬로. 에러는 예외로 죽이지 말고 {"error":...,"retry_after":...}처럼 모델이 복구할 수 있는 값으로 되먹여라.
  • 루프의 뼈대는 단순하다 — stop_reason=="tool_use"인 동안 실행→tool_result 되먹임. tool_use_id 매칭을 잊지 마라.
  • 평가: BFCL v4는 AST로 호출을 검증하고 agentic·다회전·안 부르기(hallucination)까지 본다. tau-bench는 DB 최종 상태로 채점하고 pass^k로 신뢰성을 잰다. 한 번 말고 여러 번 돌려 성공률을 봐라.
