"내가 마크다운 대신 HTML을 쓰는 진짜 이유는, 그게 Claude와 함께 루프 안에 있다(in the loop)고 훨씬 더 강하게 느끼게 해주기 때문이다."

— Thariq Shihipar, Anthropic 기술 스태프(MTS)

Anthropic의 Thariq Shihipar가 쓴 The Unreasonable Effectiveness of HTML(HTML의 비합리적인 효과)이라는 글이 있습니다. 한 줄로 요약하면 이렇습니다.

"Claude Code에게 결과물을 만들게 할 때, 마크다운(.md)이 아니라 HTML(.html)로 받아라. 그게 훨씬 낫다."

처음 들으면 "출력 포맷 취향 얘기 아냐?" 싶습니다. 그런데 읽다 보면 이건 단순한 미관 문제가 아니라, AI와 일하는 방식 자체가 달라졌기 때문에 생긴 변화라는 걸 알게 됩니다. 그리고 마침 이 주장과 잘 맞는 형식이 있죠 — 그래서 이 블로그 글 자체를 HTML과 SVG 그림으로 썼습니다. 지금 보고 계신 도표들이 곧 글의 논거입니다.

1. 원문이 말하는 것 — 한 문장과 그 배경

Shihipar의 주장은 간단합니다. Claude Code에 "리포트 만들어줘", "이 코드 설명해줘", "스펙 정리해줘"라고 시킬 때, 결과를 마크다운 대신 HTML 파일로 받으라는 겁니다. 그가 든 이유의 출발점은 아주 개인적이고 솔직합니다.

"나는 100줄이 넘는 마크다운 파일은 읽기가 어렵다고 느낀다. (I find it difficult to read a Markdown file of more than a hundred lines.)"

여기서 핵심을 짚어야 합니다. 마크다운이 갑자기 나빠진 게 아닙니다. AI가 만들어내는 결과물이 예전과 비교할 수 없이 길고 풍부해진 것이 변화의 원인입니다. 예전엔 AI에게 짧은 답을 받았으니 마크다운으로 충분했습니다. 그런데 지금은 Claude가 표, 다이어그램, 색으로 강조한 코드, 단계별 설명을 한꺼번에 쏟아냅니다.

문제는 마크다운에는 그걸 다 담을 그릇이 없다는 거죠. 표는 그럭저럭 되지만, 다이어그램은 못 그리고 색도 못 입힙니다. 그래서 Claude는 "마크다운 안에서 할 수 있는 비효율적인 방법"에 의존하게 됩니다 — ASCII 아트로 다이어그램 흉내, 유니코드 문자로 색 흉내 같은 것들요. Shihipar의 표현으로는, "마크다운은 점점 더 제약이 많은(restrictive) 포맷이 되어가고 있다."

비유하자면, 마크다운은 좁은 빨대고 HTML은 넓은 파이프입니다. AI가 만든 풍부한 정보를 사람에게 전달하는 "대역폭(bandwidth)"이 다른 거죠. 예전엔 흘려보낼 정보가 적어서 빨대로도 충분했지만, 이제는 양이 많아져서 빨대가 병목이 됐습니다.

2. 기존(마크다운)과 무엇이 다르게 다가오나

"HTML이 더 예쁘다"는 건 누구나 압니다. 이 글이 신선한 지점은 따로 있습니다. 병목의 위치가 바뀌었다는 통찰입니다.

예전에는 결과물의 품질을 막는 한계가 모델 쪽에 있었습니다. "AI가 이걸 제대로 만들어낼 수 있느냐"가 관건이었죠. 그런데 모델이 똑똑해지고 출력이 풍부해지면서, 이제 한계는 사람 쪽으로 넘어왔습니다. "내가 이 긴 결과물을 끝까지 읽고 이해하느냐"가 새로운 병목입니다.

그러니 다르게 다가오는 장점은 이렇게 정리됩니다.

  • 마크다운의 장점은 "쓰기 편함"이었습니다 — 사람이 손으로 타이핑하기 좋은 포맷이죠. 그런데 이제 글을 쓰는 주체가 사람이 아니라 AI입니다. "쓰기 편함"이라는 장점이 무의미해진 것입니다.
  • 중요한 건 "읽기 좋음"으로 바뀌었습니다 — AI가 쓰고 사람이 읽는 구조이므로, 사람이 끝까지 읽게 만드는 포맷이 이깁니다. 시각적 위계, 표, 그림, 탭이 있는 HTML이 여기서 강합니다.
  • 그리고 "되돌려 줄 수 있음"이 새로 생겼습니다 — 마크다운은 일방통행(AI → 사람)이지만, HTML은 슬라이더·토글·복사 버튼으로 사람이 조정한 값을 다시 AI에게 돌려줄 수 있습니다. 즉 양방향이 됩니다.

이 세 번째가 바로 Shihipar가 말한 "in the loop(루프 안에 있다)"의 정체입니다. 결과물을 그냥 받아보는 게 아니라, 결과물 위에서 직접 만지고 그 변화를 다시 AI에게 먹이는 닫힌 고리가 만들어집니다.

이 "루프 안에 있다"는 감각은, 지난 글 Loop 엔지니어링에서 다룬 닫힌 루프(closed loop) 개념과 정확히 같은 결입니다. 거기선 테스트 결과를 에이전트에게 되먹였다면, 여기선 사람이 HTML 위에서 만진 값을 되먹입니다. 검증과 조정이 루프 안으로 들어온다는 점에서 같은 이야기죠.

3. 그래서 뭐가 좋아지나 — 5가지 이점

원문이 정리한 HTML의 이점을 다섯 가지로 추렸습니다. 주니어 입장에서 "그래서 내가 뭘 얻나"를 한 줄로 달아뒀습니다.

📊
1. 정보 밀도
표·SVG 다이어그램·차트·코드 하이라이팅을 한 화면에. ASCII 흉내가 필요 없다.
👀
2. 가독성
제목·탭·시각적 위계 덕에 길어도 끝까지 읽게 된다. "100줄의 벽"을 넘는다.
🔗
3. 공유
파일 하나를 링크로 던지면 끝. 동료가 어떤 브라우저에서든 그대로 본다.
🎛️
4. 양방향 상호작용
슬라이더·토글로 직접 조정하고, 복사 버튼으로 그 값을 Claude에게 되돌린다.
🧩
5. 맥락 통합
Claude Code는 파일·git·Slack·Linear(MCP)를 읽어 HTML에 녹인다. 일반 챗봇은 못 하는 것.

특히 다섯 번째 맥락 통합이 Claude Code만의 차별점입니다. 그냥 claude.ai 챗 창에서 HTML을 만드는 것과 달리, Claude Code는 당신의 파일시스템, git 히스토리, 연결된 MCP(Slack·Linear 등)를 직접 읽을 수 있습니다. 그래서 "우리 레포의 최근 변경과 Slack 논의를 합쳐서 설명서 HTML로 만들어줘" 같은 요청이 가능합니다.

토큰 이야기도 정직하게 짚습니다. HTML이 마크다운보다 토큰을 더 먹는 건 사실입니다. 하지만 Shihipar는 이렇게 말합니다.

"마크다운이 토큰을 덜 쓰는 경우가 많지만, HTML의 표현력이 더 크고 내가 그걸 실제로 읽을 확률이 훨씬 높기 때문에, 결과적으로 더 나은 출력을 얻는다."

게다가 요즘 모델의 컨텍스트 윈도우가 100만(1M) 토큰급으로 커지면서, 이 정도 토큰 추가 비용은 사실상 무시할 만한 수준이 됐습니다. "토큰 몇 개 아끼려다 안 읽으면 0점"이라는 거죠.

4. 내 프로젝트에 적용해보기 — 6가지 예시

이론은 됐고, 그래서 뭘 해볼 수 있을까요? 원문이 든 실제 사례 6가지를 지도처럼 펼쳤습니다. 전부 "~을 HTML 파일로 만들어줘" 한 마디로 시작합니다.

  1. 디자인 탐색 그리드 — "이 화면 UI 6가지 방향으로 각각 다르게 그려서 한 페이지에 나란히 보여줘." 마음에 드는 걸 골라 다음 단계로 확장합니다.
  2. 코드리뷰 · diff 설명 — PR diff를 여백 주석과 심각도별 색상, 흐름도와 함께 렌더링. 리뷰어에게 복잡한 로직을 설명할 때 좋습니다.
  3. 학습 explainer — 원문의 대표 예시는 rate limiter(요청 제한기) 설명서입니다. 토큰버킷 흐름도 + 주석 달린 코드 + "이런 함정 주의(gotchas)" 섹션을 한 HTML에. 복잡한 시스템을 처음 배우는 사람에게 딱입니다.
  4. 드래그 우선순위 보드 — 티켓 카드를 Now / Next / Later / Cut 칸으로 끌어다 놓고, 정리되면 마크다운으로 export. 백로그 정리용 일회용 도구죠.
  5. 피처플래그 편집기 — 폼으로 플래그를 켜고 끄면 의존성 경고가 뜨고, 결과를 diff로 내보냅니다.
  6. 프롬프트 튜닝 화면 — 프롬프트를 좌우로 놓고 편집하면 템플릿이 실시간 렌더링되고 토큰 수가 같이 표시됩니다.

핵심은 ④⑤⑥처럼 "이 문제 하나만 풀고 버리는 일회용 도구(single-use tool)"를 부담 없이 만들 수 있게 됐다는 점입니다. 예전엔 이런 작은 내부 도구는 "만들 시간이 아깝다"고 그냥 손으로 했지만, 이제 Claude가 30초 만에 HTML로 뱉어줍니다.

시작은 정말 단순합니다. 거창한 프롬프트가 필요 없습니다.

{{< highlight text >}} 이 백테스트 결과를 표·차트·요약과 함께 보기 좋은 HTML 파일로 만들어줘. 다크모드에서도 잘 보이게 하고, 핵심 지표는 위에 카드로 요약해줘. {{< /highlight >}}

(예를 들어 트레이딩/백테스트 프로젝트라면, 에이전트 실행 로그나 수익률 리포트를 이렇게 HTML로 받으면 매번 터미널 텍스트를 눈으로 훑는 것보다 훨씬 빨리 파악됩니다.) 의도와 "어떻게 상호작용하고 싶은지"를 한 줄 덧붙이면 결과가 더 좋아집니다.

5. 현실적인 균형 — 토큰과 마크다운의 자리

원문 저자는 "대부분의 작업에서 마크다운을 거의 버렸다"고까지 말하는데, 이건 솔직히 극단적인 입장입니다. 주니어 입장에서 균형 잡힌 가이드는 이렇습니다.

이럴 땐 HTML이럴 땐 마크다운 유지
길고 복잡한 리포트·설명서짧은 메모, README 한두 단락
표·차트·다이어그램이 필요할 때git에서 diff로 관리할 문서
직접 만지고 조정할 도구다른 도구가 .md를 입력으로 먹을 때
동료에게 링크로 공유할 결과물코드 주석·커밋 메시지

또 하나 주의할 점. HTML 결과물은 보기 좋은 만큼 "그럴듯해 보여서 검증을 건너뛰기 쉽다"는 함정이 있습니다. 예쁜 차트가 틀린 숫자를 담고 있어도 깜빡 속을 수 있죠. 포맷이 화려해진 것과 내용이 정확한 것은 별개라는 걸 잊지 마세요 — 숫자와 핵심 주장은 여전히 직접 확인해야 합니다.

6. 마무리 — '받아보는 사람'에서 '함께 만드는 사람'으로

이 글의 진짜 메시지는 "HTML이 예쁘다"가 아닙니다. AI 결과물을 대하는 태도에 관한 것입니다.

마크다운 결과물은 너무 길면 그냥 "오케이"하고 넘기게 됩니다. 비판적으로 검토하지 않고, 후속 질문도 안 하게 되죠. 반면 잘 만든 HTML은 시각적으로 명료하고 직접 만질 수 있어서, 사용자가 선택지를 비판적으로 뜯어보고, 질문하고, 방향을 직접 잡게 만듭니다.

그래서 Shihipar는 이렇게 결론짓습니다 — HTML은 Claude Code를 단순 작업 자동화 도구에서 '함께 생각하는 파트너(collaborative thinking partner)'로 바꿔준다고. 결과물을 멍하니 받는 게 아니라, 그 위에서 계속 관여하게 만들기 때문입니다. 처음에 인용한 "in the loop"가 바로 이 뜻입니다.

가장 짧은 시작은 다음 한 줄입니다. 다음에 Claude에게 뭔가 정리시킬 때, 끝에 이 말만 붙여보세요.
