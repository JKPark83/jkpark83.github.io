에이전트에게 "코드를 짜서 실행해 봐"라고 시키는 순간, 우리는 믿을 수 없는 프로그램을 우리 인프라 안에서 실행하는 겁니다. 문제는 "무엇을 짜느냐"가 아니라 "그게 대체 어디서 도느냐"입니다.

1. 왜 지금 '샌드박스'가 화제인가

요즘 에이전트는 답만 말하지 않습니다. 데이터를 분석해 달라고 하면 직접 파이썬 코드를 짜서 돌리고, 버그를 고치라고 하면 테스트를 실행하고, 웹을 조사하라고 하면 셸 명령을 칩니다. 코드 인터프리터, 코딩 에이전트, "computer use"가 다 이런 패턴입니다. 편리하죠. 그런데 여기엔 조용한 전제가 하나 깔려 있습니다 — 그 코드가 실제로 어딘가에서 실행된다는 것.

이게 왜 무서운지는 2026년 7월, Hugging Face가 공개한 보안 사건이 아주 잘 보여 줍니다. 자율 에이전트 프레임워크가 원격 데이터셋 로더의 코드 실행 취약점과 설정 템플릿 인젝션을 파고들어, 데이터 처리 워커에서 허가받지 않은 코드를 실행했습니다. 거기서부터 권한을 올리고, 클라우드·클러스터 크레덴셜을 긁어모으고, 주말 내내 내부 인프라를 옆으로 타고 다녔죠. 공격 측은 수천 개의 짧게 살았다 사라지는 샌드박스 위에서 수만 건의 행동을 자동으로 실행했다고 합니다. 사건을 조사한 쪽도, 17,000건이 넘는 공격 로그를 LLM 분석 에이전트로 돌려 몇 시간 만에 타임라인을 복원했고요.

같은 주에 보안 진영에서는 "당신의 AI 에이전트는 셸을 든 내부자(insider threat with a shell)"라는 표현이 돌았습니다. 프롬프트 인젝션 한 방, 문서에 숨겨진 악의적 지시 한 줄이면, 파일·네트워크·크레덴셜에 접근 권한을 가진 에이전트가 그대로 명령을 실행해 버린다는 겁니다. 요점은 분명합니다. 임의 코드 실행(arbitrary code execution)이 가능한 순간, "실행 경계"가 시스템에서 가장 지켜야 할 자산이 됩니다.

그래서 "에이전트가 짠 코드를 어디서, 어떻게 격리해서 돌릴 것인가"가 2026년 내내 실무의 뜨거운 주제였습니다. E2B·Modal·Daytona·Vercel 같은 샌드박스 서비스가 벤치마크로 맞붙었고, Anthropic은 아예 "두뇌(에이전트 루프)는 우리 쪽에, 손발(코드 실행)은 너희 인프라에"라는 self-hosted 샌드박스를 공개하며 Cloudflare·Daytona·Modal·Vercel을 파트너로 세웠죠. 이 글은 그 밑바닥, 즉 격리가 실제로 어떻게 동작하는지를 junior 개발자 눈높이에서 풀어 봅니다.

2. 위협 모델: LLM이 짠 코드가 할 수 있는 나쁜 짓

먼저 사고방식을 하나 바꿔야 합니다. LLM이 생성한 코드는 "믿을 수 없는 입력(untrusted input)"입니다. 사용자가 폼에 입력한 문자열을 SQL에 그대로 넣지 않는 것처럼, 모델이 뱉은 코드도 그대로 실행하면 안 됩니다. 모델이 나빠서가 아니라, 그 코드를 조종하는 게 누구인지 모르기 때문입니다. 프롬프트 인젝션으로 외부 문서·웹페이지·도구 결과에 숨은 지시가 모델을 움직이면, 코드는 얼마든지 악의적으로 바뀔 수 있습니다.

격리 없이 호스트에서 실행하면 무슨 일이 벌어질까요? 크게 네 가지입니다.

  • 파일시스템 파괴·유출 — rm -rf로 데이터를 날리거나, /etc·~/.aws/credentials·.env 같은 민감 파일을 읽어 갑니다.
  • 네트워크 유출(egress) — 읽어 낸 비밀을 공격자 서버로 curl 한 방에 보냅니다. 내부망을 훑는 SSRF도 여기 포함됩니다. 데이터 유출은 가장 피해가 큰 벡터인데, 네트워크를 막는 것만으로 즉시 차단됩니다.
  • 크레덴셜 탈취 — 프로세스 환경변수, 클라우드 메타데이터 서비스(169.254.169.254), 마운트된 토큰을 훔쳐 권한을 확대합니다. Hugging Face 사건이 정확히 이 경로였죠.
  • 리소스 고갈 — 무한 루프, fork 폭탄, 메모리 폭식으로 호스트를 마비시킵니다(가용성 공격).

여기에 하나 더, 가장 무서운 게 탈출(escape)입니다. 여러 사용자의 코드를 같은 머신에서 돌린다면, 한 샌드박스가 격리를 뚫고 나가 옆 테넌트의 데이터에 닿는 순간 서비스 전체가 무너집니다. 그래서 격리의 강함은 결국 "탈출이 얼마나 어려운가"로 판가름 납니다. 표준 컨테이너처럼 호스트 커널을 공유하는 방식이 에이전트 코드 실행에 부족하다고 말하는 이유가 이것입니다 — 커널 버그 하나면 벽이 사라지니까요.

핵심 원칙: 격리 설계의 기본값은 "deny by default"입니다. 파일도 네트워크도 권한도, 필요한 것만 열어 주고 나머지는 전부 막습니다. 열어 줄 것을 고르는 게 아니라, 막지 못한 것이 곧 공격 표면입니다.

3. 격리의 스펙트럼: 프로세스부터 microVM까지

격리는 "있다/없다"의 이분법이 아니라 스펙트럼입니다. 아래로 갈수록 안전하지만, 대체로 더 무겁고 느립니다. 핵심 질문은 딱 하나 — "호스트 커널을 공유하느냐, 전용 커널을 주느냐"입니다.

먼저 절대 하면 안 되는 것부터. 파이썬 eval()·exec()나 자바스크립트 eval로 모델이 준 코드를 그냥 실행하는 건 격리가 아예 없는 겁니다. "데모니까 괜찮겠지"도 위험합니다. 아래는 나쁜 예시입니다.

{{< highlight python >}} # ❌ 절대 하지 마세요 — 모델이 짠 코드를 호스트에서 그대로 실행 code = llm.generate("사용자 CSV를 분석하는 파이썬 코드를 짜 줘") exec(code) # rm -rf, 크레덴셜 읽기, 외부 유출… 뭐든 가능 {{< /highlight >}}

컨테이너(③)는 실무의 최소선입니다. Linux 네임스페이스로 프로세스·파일시스템·네트워크를 갈라 주고, seccomp로 위험한 시스템 콜을 걸러 냅니다. 하지만 모든 컨테이너는 호스트 커널을 공유합니다. 커널에 취약점이 하나 있으면 컨테이너 벽을 넘어 호스트로, 나아가 옆 컨테이너로 갈 수 있죠. "믿을 수 없는 코드"를 여러 테넌트가 나눠 쓰는 상황에선 이 공유 커널이 약한 고리입니다. 그래서 두 가지 강화 방향이 나옵니다 — 커널로 가는 통로를 좁히는 gVisor, 아예 커널을 따로 주는 microVM.

4. gVisor: 유저스페이스 커널로 syscall을 가로채기

gVisor는 구글이 만든 "application kernel(애플리케이션 커널)"입니다. 아이디어가 재밌어요. 컨테이너 안 프로그램이 read() 같은 시스템 콜을 부르면, 그 호출이 진짜 리눅스 커널로 가는 게 아니라 Sentry라는 프로그램으로 갑니다. Sentry는 Go로 리눅스 시스템 콜 인터페이스를 처음부터 다시 구현한, 일종의 가짜(대리) 커널입니다. 유저스페이스에서 돌면서 게스트 코드가 호스트 커널을 직접 만지지 못하게 중간에서 다 받아 처리합니다.

왜 이게 더 안전할까요? 공격 표면이 확 줄기 때문입니다. 리눅스 커널에는 시스템 콜이 약 350개 있고, 각각이 잠재적 취약점입니다. gVisor를 쓰면 게스트는 그 350개를 직접 부를 수 없고, Sentry가 대신 처리한 뒤 최대 68개의 호스트 시스템 콜만 씁니다. 게스트가 만질 수 있는 진짜 커널 표면이 350에서 68로 줄어드는 셈이죠. 커널 버그를 노리는 공격이 훨씬 어려워집니다.

공짜는 아닙니다. 모든 시스템 콜이 Sentry를 거치니 오버헤드가 붙습니다. 2019년 USENIX 연구에선 시스템 콜 종류에 따라 네이티브보다 2~11배 느린 경우가 측정됐습니다. 이후 gVisor는 기존 ptrace 방식을 대체하는 Systrap 플랫폼(2023)을 기본값으로 도입해 이 syscall 가로채기 성능을 크게 개선했습니다. 그래도 "커널을 많이 두드리는" I/O 집약 워크로드에선 microVM보다 느릴 수 있다는 점은 기억해 두세요. gVisor는 쿠버네티스와도 잘 붙어서, 기존 컨테이너 파이프라인에 격리를 한 겹 더 올리고 싶을 때 특히 매력적입니다.

5. Firecracker: 전용 커널을 주는 microVM

더 센 격리는 아예 가상 머신을 주는 겁니다. 단, 무겁고 느린 옛날 VM이 아니라 초경량 VM이요. Firecracker는 AWS가 Lambda와 Fargate의 격리를 위해 만든 오픈소스 VMM(virtual machine monitor)입니다. 리눅스 KVM 하드웨어 가상화 위에서 "microVM"을 띄우는데, 각 microVM이 자기만의 커널을 갖습니다. 게스트 코드가 아무리 날뛰어도, 호스트 커널이 아니라 자기 VM 안의 커널만 만질 수 있죠. 그 아래엔 CPU가 강제하는 하드웨어 경계(KVM)가 있어서, 공유 커널 방식이 겪는 탈출을 원천적으로 어렵게 만듭니다.

"VM이면 느리고 무겁지 않나?" 하는 걱정을, Firecracker는 숫자로 지웁니다. 검증된 수치를 볼까요.

Firecracker는 안전을 위해 일부러 기능을 뺐습니다. 불필요한 가상 장치와 게스트 기능을 제거해 메모리도 줄이고 공격 표면도 줄였죠. 여기에 jailer라는 동반 프로그램이 microVM을 리눅스 유저스페이스 보안 장벽으로 한 번 더 감싸, 가상화 경계가 뚫려도 두 번째 방어선이 남게 합니다. Kata Containers도 비슷하게 "컨테이너처럼 쓰지만 속은 경량 VM"인 접근이고요. 실무 컨센서스는 대체로 이렇습니다 — 믿을 수 없는 코드를 프로덕션 멀티테넌트로 돌린다면, microVM(Firecracker·Kata)이 사실상 안전한 최소선입니다.

6. 실전: 샌드박스 서비스 고르기와 코드

다행히 이걸 직접 다 만들 필요는 없습니다. Firecracker나 gVisor를 매니지드로 감싸 "샌드박스 API"로 파는 서비스가 여럿입니다. 2026년 기준 자주 비교되는 선택지를 정리하면 이렇습니다.

서비스격리 방식대략 콜드 스타트특징 · 메모
E2B Firecracker microVM ~150ms 에이전트 코드 실행 전용. Python·JS SDK. 누적 5억+ 샌드박스 실행, Fortune 100의 88% 사용을 표방. Perplexity·Hugging Face·Groq 등 사용.
Modal gVisor 가변 추론·학습·배치까지 아우르는 넓은 플랫폼. 샌드박스 안에서 GPU를 쓸 수 있는 거의 유일한 선택지.
Daytona 하드닝된 OCI 컨테이너 (seccomp·namespace) ~27–90ms 콜드 스타트가 가장 빠른 편. 2026년 6월 프로덕션 코드베이스를 클로즈드 소스로 전환(보안 사유).
Vercel Sandbox microVM 기반 가변 웹/JS 생태계 친화. Anthropic self-hosted 샌드박스 파트너 중 하나.

표에서 읽어야 할 트레이드오프가 보입니다. 콜드 스타트 vs 격리 강도가 핵심 축입니다. 컨테이너 기반(Daytona)은 빠르지만 커널 공유를 네트워크 정책·seccomp로 보완하는 방식이고, microVM 기반(E2B)은 조금 느려도 하드웨어 경계를 얻습니다. 여기에 GPU가 필요하면(Modal), 가격이 필요하면, SDK 편의성이 필요하면… 요구에 따라 답이 갈립니다. 참고로 CPU 단가는 E2B·Daytona가 vCPU-시간당 약 $0.05로 엇비슷하고, GPU를 쓰는 Modal은 대략 3배 배수가 붙습니다.

실제 사용은 놀랄 만큼 간단합니다. E2B의 파이썬 SDK로, 모델이 짠 코드를 격리된 샌드박스 안에서 실행하는 예를 보죠.

{{< highlight python >}} from e2b_code_interpreter import Sandbox # 1) 짧게 살았다 사라지는 microVM 샌드박스를 띄운다 (호스트와 완전히 분리) with Sandbox(timeout=60) as sandbox: # 60초 지나면 강제 종료 → 무한 루프 방어 code = llm.generate("업로드된 CSV의 월별 매출 합계를 구하는 파이썬 코드") # 2) 모델이 짠 코드를 '호스트가 아니라' 샌드박스 안에서 실행 execution = sandbox.run_code(code) # 3) 결과만 회수한다 — 표준출력, 에러, 생성된 차트(파일) 등 if execution.error: print("샌드박스 안에서 실패:", execution.error.name) else: print(execution.logs.stdout) # with 블록을 벗어나면 샌드박스는 통째로 파기된다 → 남는 상태 없음 {{< /highlight >}}

포인트는 세 가지입니다. ① 실행은 호스트가 아니라 격리된 microVM 안에서 일어납니다. ② timeout으로 리소스 고갈(무한 루프)을 원천 차단합니다. ③ 샌드박스는 일회용(ephemeral)이라, 작업이 끝나면 통째로 버려 상태가 남지 않습니다. 혹시 그 안에서 나쁜 짓을 했더라도, 폭발 반경이 그 microVM 하나로 갇힙니다.

왜 매니지드를 쓰나: Firecracker를 직접 운영하려면 KVM 튜닝, 커널 이미지 관리, 스냅샷/풀링, 네트워크 격리, 오토스케일을 다 떠안아야 합니다. 하루 수천~수만 개의 일회용 샌드박스를 안정적으로 찍어 내는 건 그 자체로 인프라 프로젝트예요. 그래서 대부분은 매니지드 서비스로 시작하고, 규모·비용·규정 때문에 필요할 때만 self-host로 내려갑니다.

7. 하드닝 체크리스트 (네트워크 deny-by-default)

샌드박스를 골랐다고 끝이 아닙니다. 격리 계층은 벽일 뿐, 문을 어떻게 여닫을지는 우리가 설정합니다. 가장 중요한 건 앞서 말한 네트워크입니다. 데이터 유출이 가장 큰 피해인데, 그 통로가 네트워크니까요. 기본을 "전부 차단"으로 두고 꼭 필요한 목적지만 허용 목록(allowlist)에 올리세요.

{{< highlight yaml >}} # 샌드박스 하드닝 개념 예시 — 서비스마다 설정 키는 다르지만 원칙은 같다 network: egress: deny # 기본값은 전부 차단 (데이터 유출 1순위 차단) allowlist: # 정말 필요한 곳만 연다 - "pypi.org" # 예: 패키지 설치가 꼭 필요하면 그 도메인만 block: - "169.254.169.254" # 클라우드 메타데이터 = 크레덴셜 훔치는 단골 경로, 반드시 차단 filesystem: root: read-only # 시스템 파일은 읽기 전용 writable: ["/tmp/work"] # 쓰기는 작업 폴더 한 곳만 secrets: none # .env, 토큰, 키를 샌드박스 안에 절대 넣지 않는다 resources: cpu: "1" # CPU·메모리 상한 → fork 폭탄/메모리 폭식 방어 memory: "512Mi" timeout: 60s # 벽시계 타임아웃 → 무한 루프 방어 {{< /highlight >}}

여기에 실무 규칙 몇 가지를 더합니다.

  • 크레덴셜은 샌드박스 안에 두지 않는다. API 키가 필요하면, 샌드박스가 아니라 우리 코드(오케스트레이터)가 대신 호출하고 결과만 넘겨 줍니다. 샌드박스는 비밀을 몰라야 훔쳐 갈 것도 없습니다.
  • 테넌트마다 샌드박스를 분리한다. 사용자 A의 코드와 B의 코드를 같은 샌드박스에서 돌리지 마세요. 일회용으로 매번 새로 띄웁니다.
  • 입력뿐 아니라 출력도 검사한다. 샌드박스가 뱉은 결과를 그대로 사용자에게 보여 주거나 다음 도구에 먹이기 전에, 민감정보·이상 패턴을 한 번 거릅니다.
  • 사람 승인 게이트를 둔다. 파일 삭제·결제·외부 전송처럼 되돌리기 어려운 행동은 자동 실행하지 말고 확인을 받습니다. 프롬프트 인젝션에 대한 마지막 안전망입니다.
기억할 문장: "샌드박스를 쓴다"는 격리 계층을 고르는 일이 절반, 네트워크·파일·크레덴셜·리소스를 deny-by-default로 좁히는 일이 나머지 절반입니다. 강한 microVM을 써도 네트워크를 활짝 열어 두면 유출은 그대로 일어납니다.

8. 흔한 실수 · 팁

실수무슨 일이 벌어지나대신 이렇게
데모라서 exec()로 실행 데모 코드가 그대로 프로덕션으로 승격 → 첫 프롬프트 인젝션에 뚫림 처음부터 컨테이너 이상으로 격리. 격리는 나중에 붙이는 기능이 아니라 기본값
표준 컨테이너면 충분하다고 믿음 호스트 커널 공유 → 커널 취약점 하나로 탈출·크로스테넌트 멀티테넌트·믿을 수 없는 코드엔 gVisor 또는 microVM
네트워크를 열어 둔 채 격리만 강화 유출 통로가 열려 있어, 강한 microVM도 소용없음 egress를 deny-by-default로. 메타데이터 IP는 반드시 차단
크레덴셜을 샌드박스에 주입 탈취 시 그대로 권한 확대(Hugging Face 사건 경로) 비밀은 오케스트레이터가 쥐고, 샌드박스엔 결과만 전달
타임아웃·리소스 상한 없음 무한 루프·메모리 폭식으로 가용성 붕괴, 비용 폭증 CPU·메모리·벽시계 타임아웃을 항상 설정
샌드박스를 재사용(상태 유지) 이전 작업의 잔여물·오염이 다음 작업에 새어 나감 작업마다 일회용으로 새로 띄우고 끝나면 파기
주의: "콜드 스타트가 제일 빠른 게 최고"라는 함정에 빠지지 마세요. 27ms짜리 하드닝 컨테이너가 150ms짜리 microVM보다 항상 나은 게 아닙니다. 무엇을 실행하느냐가 먼저입니다 — 신뢰할 수 있는 내부 코드라면 컨테이너로 충분하고, 완전히 믿을 수 없는 코드를 멀티테넌트로 돌린다면 몇십 ms를 더 내고 하드웨어 경계를 사는 게 맞습니다.

9. 마무리 (TL;DR)

  • 에이전트가 코드를 실행하는 순간, 실행 경계가 최우선 보안 자산이 됩니다. LLM이 짠 코드는 프롬프트 인젝션으로 조종될 수 있는 "믿을 수 없는 입력"입니다.
  • 위협은 넷 — 파일 파괴/유출, 네트워크 egress, 크레덴셜 탈취, 리소스 고갈, 그리고 최악은 격리를 뚫는 탈출입니다. Hugging Face 2026-07 사건이 코드 실행 취약점 → 크레덴셜 탈취 → 측면 이동의 교과서적 경로였습니다.
  • 격리는 스펙트럼입니다. eval()(금지) → 컨테이너(실무 최소선) → gVisor(유저스페이스 커널, syscall 표면 350→68) → Firecracker/Kata microVM(전용 커널 + KVM 하드웨어 경계, 가장 강함).
  • Firecracker는 ~125ms에 뜨고, microVM당 <5MiB, 초당 150개 생성이 가능해 "VM은 무겁다"는 통념을 깹니다. 일회용 에이전트 샌드박스에 적합합니다.
  • 직접 만들지 말고 매니지드부터. E2B(microVM)·Modal(gVisor·GPU)·Daytona(하드닝 컨테이너)·Vercel 중 격리 강도 vs 콜드 스타트 vs GPU/가격으로 고릅니다.
  • 격리 계층만큼 설정이 중요합니다. 네트워크 deny-by-default, 메타데이터 IP 차단, 크레덴셜 미주입, 리소스 상한·타임아웃, 일회용 샌드박스, 되돌리기 어려운 행동엔 사람 승인 게이트.
