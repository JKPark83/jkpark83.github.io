# 📚 주목 논문

## 1) E3 — "에이전트는 쉬운 과제인지 아는가?"

- 저자: Junjie Yin, Xinyu Feng
- arXiv: **2607.13034** (cs.AI, 2026-07-14)
- 핵심 기여: 실행 *전에* 과제 범위를 추정하는 **E3(Estimate·Execute·Expand)** + Agent Cognitive Redundancy Ratio(불필요 추론량 측정) 프레임워크.
- 방법(3줄): ① 과제를 보고 예상 범위(읽어야 할 파일/스텝)를 먼저 추정. ② 그 범위 안에서만 실행. ③ 부족하면 그때 확장. 쉬운 과제에서 "일단 다 읽고 보는" 낭비를 제거.
- 결과: MSE-Bench 121개 편집 과제(gpt-4o)에서 **성공률 100% 유지, 비용 -85%, 토큰 -91%, 검사 파일 -92%.**
- 🛠 **에이전트 빌더 시사점:** 매 요청을 최대 컨텍스트로 밀지 말고, "이건 1파일 수정" vs "리포 전역 추적"을 라우팅하는 **난이도 게이트**를 앞단에 둬라. 코드 편집 에이전트 비용의 대부분은 불필요한 파일 읽기다.
- 한계: 코드 편집 태스크·단일 모델(gpt-4o) 기준. 추정이 틀리면 확장 단계 오버헤드가 생김.
- 링크: <https://arxiv.org/abs/2607.13034>

![E3 리소스 절감 막대그래프: 비용 -85%, 토큰 -91%, 검사 파일 -92%](/images/ai-news-2026-07-16/e3-cost.svg)
*기존 대비 절감폭. 성공률은 100%로 유지된다. (직접 생성, 출처: arXiv 2607.13034)*

## 2) OAT — "성공의 흐름에서 실패를 추적하기"

- 저자: Samuel Yeh, Yiwen Zhu, Shaleen Deep, Sharon Li
- arXiv: **2607.12747** (cs.AI, 2026-07-14)
- 핵심 기여: **성공 궤적만** 학습하는 one-class 학습(Neural Controlled Differential Equations)으로 실패 궤적의 **오류 스텝**을 비지도 방식으로 지목.
- 방법(2줄): 정상 실행 로그로 "정상 흐름"을 모델링 → 새 실패 궤적을 이상치로 채점해 어디서 이탈했는지 표시. 학습에 **성공 궤적 100개**면 충분.
- 결과: 프롬프트 기반 실패-원인 판별 대비 **200~5000× 빠름**, F1 **+20%(인-도메인) / +7%(OOD)**.
- 🛠 **에이전트 빌더 시사점:** 실패 케이스를 사람이 라벨링하지 않아도, 이미 쌓인 **성공 로그**만으로 실패 원인 스텝 탐지기를 만들 수 있다. LLM-judge로 매 실패를 재실행·판정하는 비용을 크게 줄인다.
- 한계: 궤적 스키마가 일관돼야 하고, 분포 밖(OOD) 이득(+7%)은 인-도메인(+20%)보다 작다.
- 링크: <https://arxiv.org/abs/2607.12747>

![OAT 파이프라인: 성공 궤적 100개 → one-class NCDE 학습 → 실패 궤적 채점 → 이탈 스텝 지목](/images/ai-news-2026-07-16/oat-flow.svg)
*OAT 흐름과 결과. 라벨된 실패 없이 성공 로그만으로 동작한다. (직접 생성, 출처: arXiv 2607.12747)*

# 🧑‍💻 Codex & Claude Code 실전 팁

## 🟩 Codex — 원격 실행(모바일 승인) [Workflow]
- 소스: <https://releasebot.io/updates/openai/codex>
- 💡 **핵심:** Codex Remote GA로, 자리 비운 사이 폰에서 호스트 작업을 이어가고 위험 액션만 승인. 페어링은 기기-호스트 1:1 인증이라 아무 기기나 붙지 못한다.
- 📝 **방법:** 데몬이 떠 있는 호스트에서 수동 페어링 코드 생성 →

```bash
codex remote-control pair
# 출력된 코드를 ChatGPT 모바일 앱에 입력해 이 호스트와 1:1 페어링
```
- 🎯 **언제:** 장시간 빌드/리팩터를 돌려두고 이동 중 승인만 처리할 때.
- ⚠️ **주의:** 페어링은 기기별로 관리 — 분실/공용 기기는 즉시 해제. 호스트가 실제로 명령을 실행하므로 승인 대상 액션을 반드시 확인.
- ⭐ **추천:** 원격 승인은 편리하지만, 파괴적 작업은 auto-approve에서 빼고 "물어보게" 유지.

## 🟦 Claude Code — `/doctor`로 셋업 자가진단 [Debugging]
- 소스: <https://code.claude.com/docs/en/whats-new> (Week 28, v2.1.202–206)
- 💡 **핵심:** 환경·설정 문제를 감으로 찾지 말고 `/doctor`(별칭 `/checkup`)가 진단하고 상당수는 바로 고치게 한다. 비대해진 체크인 `CLAUDE.md` 정리 제안까지 포함.
- 📝 **방법:**

```text
/doctor      # 또는 /checkup — MCP/권한/업데이트/설정 점검 후 수정 제안
```
- 🎯 **언제:** MCP 서버가 안 붙거나, 권한 프롬프트가 과하거나, "왜 이렇게 느리지" 싶을 때 첫 명령.
- ⚠️ **주의:** 자동 수정 제안은 검토 후 적용. auto mode는 이제 트랜스크립트 변조를 막고 미해결 변수의 `rm -rf` 전에 재확인한다 — 그래도 파괴적 명령은 사람이 확인.
- ⭐ **추천:** 새 리포에서 세션을 시작하면 습관적으로 `/doctor` 한 번.

## 🟪 공통 — 성공 로그로 실패 탐지기 만들기 [Eval]
- 소스: <https://arxiv.org/abs/2607.12747>
- 💡 **핵심:** 실패를 사람이 라벨링하지 말고, 이미 쌓인 **정상 궤적**을 이상탐지 기준선으로 쓰자(OAT 아이디어).
- 📝 **방법(체크리스트):**
  - 툴 호출 궤적을 **일관된 스키마**로 로깅(스텝·입력·출력·상태).
  - 명백히 성공한 실행 ~100개를 "정상" 세트로 고정.
  - 새 실패 실행을 정상 세트와 비교해 **처음 이탈한 스텝**을 원인 후보로 표시.
- 🎯 **언제:** 실패 원인 판별을 매번 LLM-judge 재실행으로 돌려 비용이 큰 팀.
- ⚠️ **주의:** 궤적 포맷이 흔들리면 오탐 급증. OOD 상황은 이득이 작으니(+7%) 사람 검토 병행.
- ⭐ **추천:** E3(앞단 난이도 게이트)와 짝지으면 "싸게 실행 + 실패는 싸게 진단"이 된다.
