# 🔥 오늘의 핵심

축이 둘이다. **코딩 툴링**: Claude Code v2.1.215가 `/verify`·`/code-review`의 **자동 실행을 끄고 명시 호출로 전환**했고, v2.1.214는 `EndConversation` 툴과 진행 heartbeat, 권한 검사 다수 수정을 넣었다. **에이전트 내부**: 오늘의 논문·오픈소스는 하나를 가리킨다 — 성능보다 **기억과 평가의 견고성**이다. Terminal-Bench 2.0 지속학습 실험은 "회귀 제어를 넣은 최적화만 복리로 쌓인다"를 실측했다. 오늘 할 일: 코딩 에이전트의 리뷰/검증을 훅·CI로 **명시적으로 재배선**하고, 최적화 루프엔 **회귀 골든셋**을 붙여라.

# 🤖 모델 & 릴리즈 레이더

- **Claude Code v2.1.214 → 2.1.215 (7/18–19).** v2.1.215에서 `/verify`·`/code-review`가 **더 이상 자동 실행되지 않고**, 필요 시 사용자가 직접 호출해야 한다. v2.1.214는 `EndConversation` 툴(악용·탈옥 시도 세션 종료), 장시간 툴콜용 **진행 heartbeat**, 그리고 권한 관련 수정이 핵심 — 특히 `Edit(src/**)` 같은 단일 세그먼트 `dir/**` 규칙이 트리 어디든 중첩 디렉터리 쓰기를 자동 승인하던 버그와 PowerShell 5.1 권한 우회를 고쳤다. → **에이전트 빌더 시사점**: 자동 리뷰에 의존하던 파이프라인이라면 오늘 깨진다. 훅/CI에서 명시 호출로 재배선하고, `dir/**` allow 규칙을 다시 점검하라. 출처: [code.claude.com/docs/en/changelog](https://code.claude.com/docs/en/changelog)
- **Codex changelog (7월 중순).** GPT-5.6 **Sol·Terra·Luna 컨텍스트가 272,000 토큰으로 정정**됐고, 위험 명령 탐지가 강화됐다(강제 `rm` 변형 추가, 거부 사유 명확화). → **시사점**: 컨텍스트 예산 산정을 272K 기준으로 다시 잡고, 파괴적 명령은 여전히 auto-approve에서 빼두라. 출처: [developers.openai.com/codex/changelog](https://developers.openai.com/codex/changelog)
- **Gemini 3.5 Pro — 여전히 공식 GA 없음.** 루머 날짜 7/17이 지났지만 오늘 기준 공개 API에 `gemini-3.5-pro` **모델카드·가격·런치 포스트가 없다**. 보도는 코딩 품질 개선을 위해 시점을 미뤘다고 전한다. → **시사점**: 2M 컨텍스트·Deep Think 같은 스펙은 아직 '보도'다. 통합 코드에 모델 ID를 **하드코딩하지 말고** env로 빼둘 것. 출처: [9to5google](https://9to5google.com/2026/07/16/gemini-3-5-pro-delays/)

# 📚 주목 논문

## 1) 에이전트 최적화는 복리로 쌓이는가? (Terminal-Bench 2.0)
- **저자**: Wenxiao Wang, Priyatham Kattakinda, Soheil Feizi (arXiv `2607.14004`, cs.AI, 2026-07-15 제출)
- **핵심**: 에이전트 최적화 기법의 이득이 **새 태스크가 계속 들어오는 지속학습 환경**에서도 누적되는지를 2단계로 평가.
- **방법(요약)**: GEPA · Meta Harness · RELAI-VCL 세 접근을 Terminal-Bench 2.0 위에서 비교. 핵심 변수는 **회귀 제어(regression control)** — 지름길·꼼수 해법을 막는 장치의 유무.
- **결과(검증됨)**: 지속학습 평균 Pass Rate가 **RELAI-VCL 76.4% > GEPA 66.0% > Meta Harness 64.6% > baseline 58.7%**. 회귀 제어가 최적화 루프 안에 있을 때만 이득이 복리로 쌓였다.
- **🛠 agent-builder 시사점**: "프롬프트/하니스 자동 최적화"를 돌린다면, **개선분이 기존 케이스를 깨지 않는지 검사하는 노드**를 루프에 넣어라. 없으면 최적화가 국소 성능만 올리고 전체는 정체한다.
- **한계**: 단일 벤치(터미널 태스크) 기반이라 도메인 일반화는 별도 확인 필요.
- 링크: [arxiv.org/abs/2607.14004](https://arxiv.org/abs/2607.14004)

![지속학습 평균 Pass Rate 막대그래프: RELAI-VCL 76.4, GEPA 66.0, Meta Harness 64.6, baseline 58.7](/images/ai-news-2026-07-20/optimizer-compound.svg)
*회귀 제어를 넣은 RELAI-VCL만 성능이 복리로 누적. (자체 제작 SVG, 값 출처 arXiv 2607.14004)*

## 2) DeepStress — 근거가 나쁠 때 검색 에이전트는 얼마나 버티나
- **저자**: Ismael Rousseau, Geraldine Damnati, Frederic Bechet (arXiv `2607.13920`, cs.CL, 2026-07-15 제출)
- **핵심**: 문서 품질을 **신뢰성·관련성·사실성** 세 축으로 합성 통제해, 검색(딥서치) 에이전트가 나쁜 근거에 얼마나 흔들리는지를 스트레스 테스트.
- **결과(검증됨)**: HotpotQA·BrowseCompPlus에서 에이전트별 견고성 편차가 크게 드러났고, **파라메트릭 지식 vs 검색된 지식의 상호작용**을 더 잘 포착하는 지표를 제안.
- **🛠 agent-builder 시사점**: RAG/서치 에이전트 eval을 "정답률"만 보지 말고, **오염된 근거를 섞은 세트**로도 돌려라. 근거가 나쁠 때 모델이 자기 파라메트릭 지식으로 복구하는지, 아니면 오염을 그대로 삼키는지가 프로덕션 신뢰성을 가른다.
- **한계**: 합성 통제라 실제 웹 노이즈 분포와의 간극은 남는다.
- 링크: [arxiv.org/abs/2607.13920](https://arxiv.org/abs/2607.13920)

# 🛠 오픈소스 스포트라이트

## OpenViking — 에이전트를 위한 '자기 진화형 컨텍스트 DB'
- 리포: [github.com/volcengine/OpenViking](https://github.com/volcengine/OpenViking) · 라이선스 Apache-2.0
- ⭐ 스타 수: 이번 세션 GitHub 접근이 이 블로그 리포로만 스코프돼 API 조회 불가 → **정확한 스타 수 미검증**(도입 전 직접 확인 권장).
- **무엇을 하나(쉬운 말로)**: 기존 벡터 RAG의 파편화 대신, **기억·리소스·스킬을 하나의 파일시스템 트리**로 통합한다. 디렉터리 위치 + 의미 검색을 결합한 재귀 조회로 필요한 컨텍스트만 계층적으로(L0/L1/L2) 불러온다.
- **에이전트 개발자 활용처**: 세션이 길어질수록 커지는 컨텍스트 비용/누수를 잡는 **에이전트 장기기억 백엔드** 후보. mem0·LightRAG류를 쓰던 자리에 "파일시스템 패러다임"이 맞는지 A/B로 재볼 만하다.

![OpenViking 아키텍처: Memory·Resources·Skills를 파일시스템 트리로 통합하고 L0/L1/L2 계층으로 온디맨드 로딩](/images/ai-news-2026-07-20/openviking-arch.svg)
*기억·RAG·스킬을 파일 트리로 통합하고 계층별로 로딩. 수치는 프로젝트 자체 발표라 재현 검증 권장. (자체 제작 SVG)*

### 🟪 공통 — Eval
- 출처: [arxiv.org/abs/2607.14004](https://arxiv.org/abs/2607.14004) · [arxiv.org/abs/2607.13920](https://arxiv.org/abs/2607.13920)
- 💡 **핵심**: 자동 최적화엔 **회귀 골든셋**, 서치 에이전트엔 **오염 근거 세트**.
- 📝 **How-to**: (1) 과거 실패 5~10개를 골든셋으로 고정 → 최적화 후 pass rate가 떨어지면 릴리스 차단. (2) eval 세트에 신뢰성 낮은 문서를 섞어 "근거 나쁠 때 정확도"를 별도 지표로 추적.
- 🎯 **언제**: 프롬프트/하니스를 반복 최적화하거나 RAG를 굴리는 모든 팀.
- ⚠️ **주의**: 골든셋은 정기 갱신하지 않으면 과적합된다.
- ⭐ **추천도**: ★★★★★
