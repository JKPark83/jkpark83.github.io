# 🔥 오늘의 핵심

오늘(7/13)은 큰 모델 릴리즈가 없는 대신, 에이전트 빌더가 매일 의존하는 **"LLM 판사(judge)"** 를 정면으로 판 논문 두 편이 뜬다. 핵심 한 줄: **후보 응답을 그대로 둬도 평가자만 바꾸면 점수가 움직인다.** 즉 eval 점수와 RL 보상은 모델 실력이 아니라 *판사 선택*에 오염될 수 있다. 두 번째 논문은 반대 방향의 위안도 준다 — **프론티어 모델을 판사로 쓸 필요는 없다. mini급도 충분히 경쟁력 있고, 대신 '보정(calibration)'이 전제**다. 그래서 오늘 해볼 일: **내 eval/보상 판사에 골드 라벨 소량으로 F1·κ를 재고, F1만이 아니라 FPR·FNR·합격률 표류(방향 편향)를 함께 리포트**하는 것.

# 📚 주목 논문

오늘은 arXiv(`export.arxiv.org`)로 ID·저자·수치를 1차 확정했다. 두 편 모두 **"판사(LLM-as-judge)를 믿기 전에 감사하라"** 는 같은 결이다.

### 1) When the Judge Changes, So Does the Measurement — 판사 교체는 측정 변경이다
- 저자: Zongyou Yang, Yinghan Hou, Xiaokun Yang · **arXiv 2607.08535 (cs.CL)**
- 핵심 기여: LLM-judge 점수는 후보 응답이 고정돼도 **평가자가 바뀌면 이동**한다. 이를 *측정 타당성(measurement-validity)* 문제로 다룬다.
- 방법: 4개 판정 데이터셋에서 현실적 업그레이드 두 경로를 비교 — **Qwen3 dense 판사 1.7B→32B 스케일업**과 **MiniMax M2→M2.7 릴리즈 이동**.
- 결과(검증): **판사 업그레이드는 서로 호환되지 않는다.** Qwen3 **1.7B→4B만 견고한 인접 개선**을 주고, **MiniMax 인접 릴리즈는 그렇지 않다**. 더 강한 판사도 위치·장황함 편향을 *줄일 뿐 제거하진 못함*. 반복 샘플 배심은 오류가 상관될 때 이득이 거의 없고, 구조화된 토론은 결정을 크게 바꾸지만 **파서·폴백 로그가 없으면 그 변화를 '숙의' 덕으로 귀인할 수 없다**.
- 🛠 **빌더 시사점**: LLM-judge 리포트에 **① 데이터셋 슬라이스 ② 편향 프로브 ③ 오류 상관 추정 ④ 프로토콜 감사 로그**를 넣어라. "판사 버전 올렸으니 좋아졌겠지"는 금물.
- ⚠️ 한계: 판사 4개 데이터셋·특정 모델 패밀리(Qwen3·MiniMax) 기준.
- 링크: [arXiv:2607.08535](https://arxiv.org/abs/2607.08535)

![판사를 바꾸면 측정값이 바뀐다: 고정된 후보 응답, Qwen3 1.7B→4B는 견고하지만 MiniMax 인접 릴리즈는 교체 불가, 하단에 감사 로그 4항목](/images/ai-news-2026-07-13/judge-swap.svg)
*같은 응답도 판사가 바뀌면 점수가 갈린다 — 그래서 감사 로그(슬라이스·편향 프로브·오류 상관·프로토콜)가 리포트의 일부여야 한다. 값 출처: arXiv 2607.08535.*

### 2) Do You Need a Frontier Model as a Citation Verifier? — 싼 판사도 충분, 단 보정 전제
- 저자: Ethan Leung, Elias Lumer, Corey Feld 외 · **arXiv 2607.08700 (cs.CL)**
- 핵심: RL은 갈수록 **LLM 판사를 보상 모델**로 쓴다. 그 신호를 믿기 전에 *판사가 얼마나 유능하고 얼마나 편향됐는지*를 알아야 한다. 대상은 Deep-Research의 **인용 품질**(각 주장에 근거 출처를 다는 구조화 루브릭 — source relevance / factual support 두 축).
- 방법: 적대적 롱폼 벤치에서 **3개 패밀리 8개 오프더셸프 판사**를 골드 라벨 **1,248개 판정**(모두 사람 검수, **378개는 판사 불일치를 재심한 하드케이스**)으로 채점.
- 결과(검증): **싼 판사가 경쟁력**을 유지. source-relevance는 **GPT-5-mini가 pass-class F1 0.908(κ=0.636)로 최강**, factual-support는 판사 간 **통계적으로 구분 불가**(신뢰구간 겹침). 같은 F1이라도 **합격률 표류·FPR·FNR은 크게 다르며**, 스칼라 F1이 이 *방향 편향*을 가린다 — 그런데 RL 루프는 바로 그 방향을 강화한다.
- 🛠 **빌더 시사점**: 판사를 보상/게이트로 쓰기 전, **작은 골드셋으로 F1+κ를 먼저 재고 FPR·FNR·drift까지 리포트**하라. 그리고 **최상위 모델이 아니라 mini급으로 시작**해도 된다.
- ⚠️ 한계: 인용/출처 귀인 태스크·특정 벤치 기준. 다른 루브릭엔 재보정 필요.
- 링크: [arXiv:2607.08700](https://arxiv.org/abs/2607.08700)

![인용 판사 벤치: GPT-5-mini source-relevance F1 0.908(κ=0.636), factual-support는 판사 간 구분 불가, 스칼라 F1이 FPR·FNR·합격률 표류를 가림, 하단 빌더 체크리스트 4항목](/images/ai-news-2026-07-13/citation-judge.svg)
*프론티어 모델 없이도 보정된 mini급 판사면 충분 — 단 F1만 보면 방향 편향을 놓친다. 값 출처: arXiv 2607.08700.*

# 🛠 오픈소스 스포트라이트

투명하게 밝힌다 — **오늘도 임의 GitHub 레포의 star 수는 세션 네트워크 정책으로 API 검증이 막혀** 별점 랭킹을 1차 확인할 수 없다. 제로 팩트 원칙상, 검증 불가한 별점 대신 **오늘 바로 재현 가능한 검증된 오픈 아티팩트**를 가리킨다.

- **오늘의 논문 = 프레임워크 독립 레시피.** 두 논문의 결론(판사 감사 로그, F1 대신 방향 편향 리포트, mini급 판사 보정)은 **DSPy·자체 eval 하네스·LangGraph 어디든 개념만 이식**하면 된다. 별도 라이브러리가 필요 없다.
- **재현 팁**: 이미 굴리는 LLM-judge에 **골드 라벨 50~100개**만 붙여 F1·κ를 재고, 판사 모델을 한 단계 올렸을 때 점수가 *단조 개선*되는지 A/B로 확인하라. 논문 1의 핵심 경고("업그레이드 ≠ 호환")를 내 스택에서 즉시 체감할 수 있다.
- ⚠️ 별점 기반 레포 랭킹은 API 접근이 복구되는 세션에서 정확한 수치로 다시 다룬다.

### 🟦 Claude Code — 서브에이전트 + 구조화 출력으로 '보정된 판사'를 붙이기 (category: Eval)
- 출처: [Claude Code 서브에이전트 문서](https://code.claude.com/docs/en/sub-agents)
- 💡 **핵심**: 오늘 논문의 교훈을 실전에 — eval 판사를 **전용 서브에이전트**로 분리하고 출력을 **구조화 스키마**로 강제해, 점수뿐 아니라 근거·편향 신호를 함께 남긴다.
- 📝 **How-to**:
```markdown
# .claude/agents/judge.md
---
name: judge
description: 후보 답변을 루브릭으로 채점하는 보정된 판사
tools: Read
---
루브릭 각 항목을 pass/fail로 채점하고, JSON으로:
{ "score": 0-1, "per_criterion": [...],
  "confidence": 0-1, "position_bias_check": "...",
  "gold_agreement": "판정이 골드와 갈리면 이유" }
```
```bash
# 메인 세션에서 골드셋 일부로 판사 F1·κ를 먼저 재보정
claude -p "judge 서브에이전트로 gold/*.json을 채점하고 F1과 오판 방향(FPR/FNR)을 집계"
```
- 🎯 **언제**: eval/RL 보상에 LLM-judge를 넣기 전, 판사 자체를 감사하고 싶을 때.
- ⚠️ **주의**: 판사 서브에이전트는 **평가 대상과 같은 모델을 피해** 자기편향을 줄여라. 골드 라벨 없는 자평은 신뢰 금물.
- ⭐ **추천**: eval을 진지하게 굴린다면 필수 패턴.

### 🟪 공통 — 'F1만 보지 마라' 판사 감사 체크리스트 (category: Eval)
- 출처: [arXiv:2607.08535](https://arxiv.org/abs/2607.08535), [arXiv:2607.08700](https://arxiv.org/abs/2607.08700)
- 💡 **핵심**: 스칼라 점수 하나는 방향 편향을 가린다. 판사 리포트에 최소 네 가지를 남겨라.
- 📝 **체크리스트**:
  - [ ] 데이터셋 **슬라이스별** 점수 (전체 평균만 X)
  - [ ] **편향 프로브**: 위치·장황함(verbosity) 편향 측정
  - [ ] **방향 오류**: FPR·FNR·합격률 표류를 F1과 함께
  - [ ] **프로토콜 로그**: 파서·폴백·토론 경로 기록 (귀인 가능하게)
- 🎯 **언제**: LLM-judge를 eval 지표나 RL 보상으로 승격시키기 직전.
- ⚠️ **주의**: 판사 모델 버전 업이 자동 개선이라는 가정 금지 — 인접 릴리즈도 호환 안 될 수 있다(논문 1).
- ⭐ **추천**: 오늘 당장 기존 eval 스크립트에 FPR/FNR 집계 한 줄 추가부터.
