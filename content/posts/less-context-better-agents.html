대화 기록을 전부 들고 있던 에이전트는 작업의 71%를 끝냈습니다. 토큰 148만 개, 14.5시간을 쓰고도요.
최근 5개만 남기고 나머지를 버린 에이전트는 79%를 끝냈습니다. 토큰은 64% 적게 쓰고요.

— Microsoft, Less Context, Better Agents (arXiv:2606.10209, 2026)

"맥락(context)은 많을수록 좋다." 우리가 LLM을 다룰 때 거의 본능적으로 믿는 명제입니다. 모델에게 정보를 더 줄수록 더 똑똑하게 답할 것 같으니까요. 그래서 모델 회사들은 컨텍스트 창을 20만 토큰, 100만 토큰으로 경쟁하듯 키워 왔습니다.

그런데 Microsoft 연구진이 2026년 6월에 공개한 논문 "Less Context, Better Agents"는 정반대를 실험으로 보여줍니다. 긴 작업을 수행하는 도구 사용 에이전트에서는, 오래된 맥락을 과감히 잘라낸 쪽이 성능도 좋고 비용도 싸다는 것입니다. 그것도 "조금" 좋은 게 아니라 완료율 71% → 91.6%, 토큰 63% 절감이라는 큰 차이로요.

이 글에서는 다음을 다룹니다. 에이전트가 아직 낯선 주니어 개발자도 따라올 수 있게 기초부터 풀었습니다.

1. 도구 쓰는 에이전트는 왜 컨텍스트가 터지는가

먼저 용어부터 맞추겠습니다. 도구 사용 에이전트(tool-using agent)란, LLM이 혼자 글만 쓰는 게 아니라 실제 행동을 하는 시스템입니다. 검색을 하고, DB를 조회하고, 폼에 값을 입력하고, 버튼을 누릅니다. 이때 "어떤 도구를 어떻게 부를지"를 표준화한 규약이 MCP(Model Context Protocol)입니다.

MCP란? Anthropic이 2024년 11월에 공개한 개방형 규약으로, "AI를 위한 USB"에 비유됩니다. AI 앱 M개와 데이터 소스 N개를 잇는 데 M×N개의 일회용 커넥터를 짜는 대신, 양쪽이 MCP만 지키면 어떤 AI든 어떤 도구든 붙습니다. 2026년 기준 공개 MCP 서버만 9,000개가 넘고, Claude·ChatGPT·Cursor·VS Code 등이 기본 지원합니다.

에이전트가 도구를 쓰는 방식은 "생각 → 행동 → 관찰"이 끝없이 도는 루프입니다(ReAct 패턴이라 부릅니다). 한 바퀴마다 이런 일이 벌어집니다.

  1. 모델이 "다음에 뭘 할지" 생각한다
  2. 도구 호출을 내보낸다 (예: get_form_state())
  3. 도구가 실행되고 결과(관찰)를 돌려준다
  4. 그 결과가 대화 기록(history)에 그대로 덧붙는다
  5. 이제 늘어난 기록 전체를 다시 모델에 넣어 다음 한 바퀴를 돈다

문제는 4번과 5번에 있습니다. 매 턴마다 지금까지의 기록 전부를 다시 모델에 밀어 넣기 때문에, 토큰 비용이 단계 수에 거의 제곱으로 불어납니다. 10단계째는 1단계의 10배가 아니라, 1~9단계를 통째로 다시 읽는 비용까지 포함합니다.

게다가 MCP 도구의 응답은 설계상 장황합니다. 폼의 모든 필드 값, 메타데이터, 내비게이션 경로, 시스템 상태를 통째로 담아 돌려주거든요. 정확성과 디버깅엔 좋지만 토큰엔 독입니다. 어떤 팀은 도구 정의(스키마)만으로 20만 토큰 창의 72%를 써버렸다고 보고했습니다. 실제 추론에 쓸 공간은 5만 토큰밖에 안 남은 거죠.

2. "그냥 컨텍스트 창을 키우면 되잖아?" — 안 되는 이유

여기서 자연스러운 반문이 나옵니다. "2026년이면 컨텍스트 창이 100만 토큰인데, 그냥 다 집어넣으면 안 되나?" (현재 Gemini 계열 ~100만, GPT·Claude 계열 20만~100만 토큰 수준입니다.) 답은 "창 크기와, 모델이 실제로 잘 쓰는 양은 다르다"입니다.

이 격차를 가리키는 말이 "컨텍스트 부패(context rot)"입니다. 하드 한계에 닿기 한참 전부터, 입력이 길어질수록 모델의 실효 성능이 떨어지는 현상이죠. 2025년 6월 한 개발자가 붙인 이 표현은, 곧 벤치마크로 뒷받침됐습니다.

  • Chroma의 Context Rot 연구(2025) — GPT-4.1, Claude Opus 4, Gemini 2.5 Pro 등 프론티어 모델 18개 전부가 입력이 길어질수록 정확도(F1)가 단조 감소했습니다. 어려운 과제에서는 13개 중 11개 모델이 고작 32K 토큰에서 이미 기준 성능의 절반 아래로 떨어졌습니다.
  • "Lost in the Middle"(Liu et al., TACL 2024) — 정답이 컨텍스트의 앞·뒤가 아니라 중간에 묻히면 정확도가 30~50%p 떨어집니다. 모델 6종 전부 동일한 U자형 곡선을 보였고, 2026년 100만 토큰 모델에서도 여전히 재현됩니다.
  • 비용·지연 — 입력 토큰 비용은 길이에 정비례하고(일부 모델은 20만 토큰을 넘으면 단가가 2배), 어텐션 연산은 O(n²)라 입력이 길수록 첫 응답까지의 지연도 급격히 늘어납니다.
핵심 비유. 컨텍스트 창은 책상입니다. 책상이 넓다고 서류를 무한정 쌓아두면, 정작 지금 필요한 한 장을 못 찾습니다. "창을 키우는 것"은 책상을 넓히는 일이고, "컨텍스트 엔지니어링"은 지금 작업에 필요한 서류만 책상에 올려두는 일입니다. 후자가 없으면 책상을 아무리 넓혀도 일이 느려집니다.

그래서 등장한 분야가 컨텍스트 엔지니어링(context engineering)입니다. Anthropic은 이를 "LLM 추론 동안 최적의 토큰 집합을 선별·유지하는 전략"으로 정의하며, 목표는 "원하는 결과를 낼 확률을 최대화하는, 가능한 한 작은 고신호 토큰 집합"을 찾는 것이라고 말합니다. 이 논문은 바로 그 주장을, 실제 기업 워크플로에서 숫자로 증명한 사례입니다.

3. 실험 설계: 호텔 영수증을 0원까지 쪼개라

연구진이 고른 작업은 Microsoft Dynamics 365 Finance & Operations(D365 F&O)에서 호텔 경비를 정산하는 일입니다. 에이전트가 MCP로 ERP에 접속해, 호텔 영수증 한 장을 개별 항목(객실료, 세금, 식대 등)으로 쪼개 입력하는 거죠.

왜 이 작업이 까다로운 시험대일까요? "잔액이 정확히 $0.00이 되어야만 성공"이라는 엄격한 조건 때문입니다. 영수증 총액을 한 푼도 남김없이 항목들로 배분해야 합니다. 실제 ERP에서 1원이라도 남으면 경비 보고서가 마감되지 않고 컴플라이언스 검토로 넘어가니, 비즈니스 기준으로도 "완료=완벽"이어야 합니다. 항목 수는 4~23개(중앙값 8개)고, 같은 "Hotel Tax"가 금액만 다르게 두 번 나오거나, "Entertainment External"을 올바른 하위 카테고리로 매핑해야 하는 함정이 깔려 있습니다.

연구진은 GPT-5 에이전트로 50개 작업을 5번씩 반복하며, 컨텍스트를 다루는 방식만 바꾼 4가지 설정을 비교했습니다.

설정컨텍스트 정책한 줄 설명
C1유저 모델 없음비교를 위한 출발점(아래 설명). 사람 없는 환경에서 GPT-5 단독.
C2전체 기록 유지표준 관행 = 기준선. 대화 전부를 매 턴 그대로 들고 감.
C3최근 5개만 유지 (pruning)최근 도구 호출/응답 5쌍만 남기고 나머지는 버림.
C4최근 5개 + 요약버리기 전에, 버려질 내용을 한 줄로 요약해 끼워 넣음.
C1은 왜 따로 있나? 비대화형(사람이 없는) 환경에서 GPT-5는 중간에 "이거 추가할까요?"를 묻고 멈춰버리는 버릇이 있었습니다. 그래서 C2~C4에는 "유저 역할"을 하는 보조 LLM을 똑같이 넣어 이 멈춤을 막았습니다. 즉 컨텍스트 엔지니어링의 효과는 유저 모델을 고정한 C2 → C3 → C4 비교로만 본다는 게 핵심입니다. C1은 "유저 모델이 왜 필요한지"를 보여주는 곁가지입니다.

4. 결과 ① — 잘라냈더니 더 잘했다

먼저 가장 충격적인 비교, 전체 기록(C2) vs 최근 5개만(C3)입니다. 상식대로라면 정보가 더 많은 C2가 이겨야 합니다. 결과는 반대였습니다.

숫자로 정리하면 이렇습니다.

설정완료율잔액 ≤10%총 토큰시간
C2 (전체 기록)71.0%74.0%1,481K14.56h
C3 (최근 5개)79.0% (+8%p)87.0%535K (−63.9%)5.39h (−63%)

왜 이런 일이 벌어질까요? 핵심은 "낡은 상태(stale state)"입니다. 에이전트가 초반에 받은 도구 응답은 그 시점의 폼 상태를 담고 있습니다. 그런데 작업이 진행되면서 폼은 계속 바뀌죠. 옛 응답은 이미 덮어쓰인 과거인데, 전체 기록을 들고 있으면 모델이 이 낡은 스냅숏을 현재로 착각하고 엉뚱한 결정을 내립니다.

논문의 실제 실패 사례를 보면 직관적으로 이해됩니다. C2(전체 기록)에서 자주 나온 "낡은 상태 참조" 실수입니다.

{{< highlight text >}} A: get_form_state() T: {lines: [{Room, $180}], total_added: $180, remaining: $420} A: add_line(RoomTax, $40) ... (5번의 도구 호출이 더 지나간 뒤) ... A: # 맨 처음의 스냅숏을 다시 참조한다 A: add_line(Room, $180) # 중복! 폼엔 이미 Room이 들어가 있음 => 잔액: -$180 (초과 배분, Room 라인이 중복됨) {{< /highlight >}}

처음 스냅숏("Room이 아직 없네")을 보고, 이미 한참 전에 넣은 Room 라인을 또 넣어버린 겁니다. 책상에 낡은 메모가 쌓여 있으니, 이미 끝낸 일을 다시 하는 거죠. 최근 5개만 남기면 이 "낡은 메모"가 애초에 사라지므로 모델은 항상 현재 폼 상태에 집중하게 됩니다.

5. 결과 ② — 그런데 너무 잘라내면 길을 잃는다

그럼 무조건 많이 잘라낼수록 좋을까요? 아닙니다. 여기서 두 번째 반전이 나옵니다. C3(최근 5개만)는 또 다른 실수가 늘었습니다. 바로 "조기 종료(premature termination)"입니다.

최근 5개만 남기다 보면, 초반에 봤던 "남은 잔액 $240" 같은 전체 진행 상황이 창 밖으로 밀려납니다. 그러면 모델은 "내가 얼마나 했고 얼마가 남았는지"를 잊고, 일이 다 안 끝났는데 폼을 제출해버립니다.

{{< highlight text >}} A: get_form_state() T: {total_added: $340, remaining: $240} A: add_line(Parking, $45) # pruning 창이 앞쪽의 "remaining: $240" 줄을 밀어냄 A: # 에이전트가 더 이상 남은 잔액을 못 봄 A: submit_form() => 잔액: $195 미배분. 폼이 미완성 상태로 제출됨. {{< /highlight >}}

해결책이 C4 = 최근 5개 + 요약입니다. 오래된 내용을 그냥 버리는 게 아니라, 버리기 직전에 한 줄짜리 진행 요약으로 압축해 끼워 넣는 거죠. 논문에 실린 실제 요약 출력은 이렇습니다.

{{< highlight text >}} 이전 도구 호출 요약(Summary of previous tool calls): - 경비 보고서 폼을 열고 ER-00184(호텔, 영수증 총액 $612.40)로 이동함. - "Itemize" 버튼을 눌러 호텔 라인의 항목 분할 하위 폼을 엶. - add_line으로 Hotel-Room 라인 $180.00 추가함. - add_line으로 Hotel-Tax 라인 $14.40 추가함. {{< /highlight >}}

흥미로운 점은 이 요약이 잔액을 직접 계산하지 않는다는 것입니다. 그냥 "어떤 폼을 열었고, 어떤 버튼을 눌렀고, 무슨 라인을 넣었는지"를 일반적으로 서술할 뿐인데도, 이것만으로 에이전트가 "아 내가 여기까지 했지"라는 작업 수준의 인식을 회복해 조기 종료를 멈춥니다.

결과적으로 C4는 모든 지표에서 1등입니다.

C2 (전체 기록)71.0%74.0%92.03%1,481K
C3 (최근 5개)79.0%87.0%96.92%535K
C4 (5개+요약)91.6%99.6%99.64%553K

요약을 추가하는 비용은 C3 대비 토큰 3.4% 증가에 불과합니다(요약을 위한 LLM 호출이 퇴거 이벤트당 딱 한 번 추가될 뿐). 전체 기록(C2)과 비교하면 토큰 62.7% 절감, 시간 60.2% 단축, 완료율 +20.6%p. 통계적으로도 C3와 C4의 95% 신뢰구간이 깔끔하게 분리돼([73.5, 83.6] vs [87.5, 94.4]), 요약의 효과가 우연이 아님을 보여줍니다.

6. 메커니즘: 컨텍스트를 어떻게 구성하나

C4의 컨텍스트 구성 알고리즘은 의외로 단순합니다(논문의 Algorithm 1). 조절 항목은 두 가지뿐입니다.

  • N (pruning window) — 최근 도구 호출/응답을 몇 쌍이나 그대로 남길지. 논문은 N=5.
  • W (summary window) — 밀려나는(퇴거되는) 메시지 중 몇 개를 요약해 한 줄로 다시 끼워 넣을지. 논문은 W=3.

정리하면 C4는 이렇게 동작합니다. (1) 최근 N개 쌍은 원문 그대로 남긴다 → 현재 상태 파악. (2) 밀려나는 옛 메시지 중 가장 최근 W개를 한 줄 요약으로 압축해, 퇴거된 자리 맨 앞에 한 번 끼워 넣는다 → 전체 진행 파악. 이 단순한 분업이, 전체 기록도 못 한 91.6%를 만들어냅니다.

이 결과는 호텔 경비에만 국한되지 않았습니다. 연구진은 출장(렌터카·항공)과 식대·선물 카테고리에서도 같은 실험을 반복했고, 모든 카테고리에서 +19~20%p의 일관된 향상을 확인했습니다. 또 에이전트를 Claude Sonnet 4.5로 바꿔도 "요약이 pruning을 이긴다"는 결과가 그대로였습니다.

7. 실패의 해부: 두 가지 실수의 줄다리기

이 논문에서 가장 통찰적인 부분은 실패를 종류별로 분류한 표입니다. 정책마다 어떤 실수가 늘고 주는지를 보면, 왜 "pruning + 요약"의 조합이 정답인지가 선명해집니다.

5회 실행을 합산한 미완료 건수(총 50작업 × 5회 = 250런 기준)는 이렇게 움직입니다.

실패 모드C2C3C4
낡은 상태 참조 (stale-state)3464
조기 종료 (premature termination)9183
중복/누락된 반복 항목12115
잘못된 하위 카테고리 매핑896
도구/폼 내비게이션 오류652

총 미완료735321

이야기가 깔끔하게 들어맞습니다. pruning(C3)은 낡은 상태 참조를 34→6으로 84%나 줄이지만, 진행 상황을 잃어 조기 종료를 9→18로 두 배 늘립니다. 요약(C4)은 그 조기 종료를 18→3으로 6분의 1로 잡으면서, 낡은 상태도 다시 끌어올리지 않습니다. 두 기법이 서로의 약점을 정확히 메우는 거죠.

8. 그래서 내 에이전트엔 어떻게 적용하나

논문이 D365라는 특정 환경을 다루지만, 교훈은 바로 실무에 적용할 수 있습니다. 도구를 많이 쓰는 에이전트를 만든다면 이렇게 접근해 보세요.

  1. 전체 기록은 기본값이 아니라 선택지다. "대화 전부를 매 턴 재전송"하는 표준 관행을 의심하세요. 도구 응답이 큰 워크플로일수록, 그 기록의 상당수는 이미 지나간 상태입니다.
  2. 가장 먼저 시도할 건 "최근 N개만 남기기"다. 구현이 가장 단순하고(요약용 LLM 호출조차 없음), 토큰을 가장 크게 줄이며, 낡은 상태 오류까지 잡습니다. 출발점으로 N=5가 합리적입니다.
  3. 그다음, 가벼운 요약 한 줄을 더하라. pruning만으로 "에이전트가 일을 일찍 끝내버린다" 싶으면, 버려지는 내용을 진행 요약으로 압축해 끼워 넣으세요. 잔액 같은 걸 정밀 계산할 필요 없이, "무엇을 했는지" 서술만으로도 충분합니다.
  4. 토큰 단위로 자르지는 마라. LLMLingua류의 토큰 압축은 RAG/QA엔 좋지만, 폼 조작 에이전트에선 컨트롤 이름·숫자 같은 식별자를 망가뜨릴 수 있습니다. 잘라낼 단위는 토큰이 아니라 "도구 호출/응답 쌍"입니다.
  5. 측정으로 검증하라. N, W 값은 작업 구조에 따라 다릅니다. 이 논문도 "한 항목 입력에 도구 호출 2~3개가 필요하니 5개면 두 사이클 분량"이라는 분석에서 N=5를 골랐습니다. 당신의 작업 한 단위가 몇 번의 도구 호출인지 세어 보고 N을 정하세요.
좋은 소식 — 이미 도구가 있습니다. 이 논문의 "pruning + 요약"은 새 개념이 아니라 업계가 수렴 중인 패턴입니다. Anthropic API는 똑같은 두 기법을 기본 기능으로 제공합니다. tool-result clearing(오래된 도구 응답을 자리표시자로 치환 — 가장 가벼운 형태의 pruning)과 compaction(임계치 도달 시 대화를 요약으로 압축, Claude Code의 /compact가 이것)입니다. 직접 구현하기 전에, 쓰는 플랫폼에 이런 기능이 있는지부터 확인하세요.

학계도 같은 방향입니다. 같은 달 나온 ACON(arXiv:2510.00615, ICML 2026)은 압축 규칙 자체를 에이전트의 실패 사례로부터 학습시키고, JetBrains 연구(2025.12)는 "observation masking vs 요약"을 비교했습니다. 공통된 결론은 하나입니다 — 긴 작업 에이전트에서 컨텍스트는 관리 대상이지, 무한정 쌓아두는 로그가 아니다.

9. 한계 — 이건 만병통치약이 아니다

이 논문이 좋은 이유 중 하나는, 저자들이 스스로 범위를 분명히 못 박았다는 점입니다. 그대로 옮길 가치가 있습니다.

  • 단일 도메인 증거다. 호텔 경비 정산이라는, 반복 하위 카테고리·복잡한 매핑·엄격한 0원 조건을 가진 워크플로 한 부류를 다뤘습니다. 저자들은 이를 "보편적 일반화의 증명이 아니라, 한 부류의 기업 도구 사용 워크플로에 대한 강력한 증거"라고 명시합니다.
  • 잘라내도 안 풀리는 실수가 남는다. "Room tax vs Non-Room tax" 같은 유사 명칭의 하위 카테고리 매핑 오류는 컨텍스트 정책과 무관합니다(23개짜리 카테고리 사전이 부르는 별개의 난이도). pruning·요약이 잡는 건 어디까지나 "낡은 상태"와 "조기 종료"입니다.
  • 완료율 91.6%는 여전히 8.4%가 실패한다는 뜻이다. 프로덕션에서는 플래그된 케이스에 대한 사람의 검토가 필요합니다.

그래도 핵심 메시지는 도메인을 넘어 단단합니다. "더 많은 맥락 = 더 나은 성능"이라는 등식은, 적어도 도구를 많이 쓰는 장기 작업에서는 성립하지 않는다.

10. 마무리 (TL;DR)

"오래된 도구 응답은 중립적인 기록이 아니다. 그것은 적극적으로 해로운 노이즈다."

  • 문제 — 도구 쓰는 에이전트는 매 턴 전체 기록을 재전송해 토큰이 폭증하고, 옛 도구 응답("낡은 폼 상태")이 노이즈가 되어 성능까지 떨어뜨린다.
  • 해법 ① — 최근 N개(=5)만 남기고 잘라내면, 토큰 64%↓ + 완료율 +8%p. 낡은 상태 오류가 사라진다.
  • 해법 ② — 단, 너무 잘라내면 진행 상황을 잊고 일찍 끝낸다. 버려질 내용을 한 줄 요약으로 끼워 넣으면(C4) 완료율 91.6%, 토큰은 여전히 전체 기록의 1/3.
  • 왜 — 최근 메시지(지역 상태)와 요약(전역 진행)이라는 두 채널의 분업 덕분.
  • 실천 — 전체 기록을 의심하라 → 최근 N개만 남겨라 → 가벼운 요약을 더하라 → 토큰이 아니라 도구 쌍 단위로 잘라라 → N·W는 측정으로 정하라. Anthropic의 tool-result clearing/compaction처럼 이미 나온 기능부터 써라.

컨텍스트 창은 계속 커질 겁니다. 하지만 이 논문의 교훈은 그 반대 방향을 가리킵니다. 에이전트를 똑똑하게 만드는 건 더 큰 창이 아니라, 그 창에 무엇을 남길지 정하는 손입니다. 맥락은, 적을수록 똑똑합니다.
