# 🔥 오늘의 핵심

오늘은 새 파운데이션 모델보다 **하네스(harness)** 쪽 변화가 신호가 크다. 오픈소스 터미널 코딩 에이전트(gemini-cli 105.6k★, deer-flow 75k★)가 빠르게 커지고, Claude Code는 자동 모드에서 `git reset --hard` 같은 파괴적 명령을 기본 차단한다. 그리고 RigorBench는 "코드가 통과했는가"가 아니라 "에이전트가 어떻게 일했는가"를 측정한다 — 에이전트 빌더라면 오늘부터 평가 기준을 바꿀 만하다.

# 📚 주목 논문

### RigorBench — 코딩 에이전트의 '과정 규율'을 측정하다

- **저자**: Meher Bhaskar Madiraju, Meher Sai Preetam Madiraju
- **arXiv**: [2606.22678](https://arxiv.org/abs/2606.22678) · cs.SE · 2026-06 제출
- **핵심 기여**: 기존 코딩 에이전트 벤치마크는 "테스트 통과 / 이슈 해결"이라는 **결과**만 본다. RigorBench는 에이전트가 결과에 도달하는 **과정의 규율**을 처음으로 정량화한다.
- **방법(요약)**: 에이전트 하네스를 5개 축으로 평가 — 계획 충실도(Planning Fidelity), 검증 커버리지(Verification Coverage), 복구 효율(Recovery Efficiency), 기권 품질(Abstention Quality), 원자적 전이 무결성(Atomic Transition Integrity).
- **주요 결과**: 현행 에이전트는 **기본 설정에서 과정 규율이 낮다**. 즉 정답을 내더라도 검증을 건너뛰거나, 모르면서 기권하지 않고, 변경을 원자적으로 쪼개지 못하는 경향.
- 🛠 **에이전트 빌더 시사점**: 자체 eval에 "출력 정답률" 외에 **'검증 단계를 실제로 거쳤나 / 불확실하면 멈췄나'** 같은 과정 지표를 추가하면, 프로덕션에서의 사고를 결과 점수보다 잘 예측한다.
- ⚠️ **한계**: 평가 축이 다소 주관적이며, 점수화 방식이 특정 하네스 구조에 유리할 수 있어 절대 비교보다 같은 하네스 내 개선 추적에 적합.

# 🧑‍💻 Codex & Claude Code 실전 팁

### 🟦 Claude Code — 자동 모드 파괴적 명령 가드레일 (Debugging / Workflow)
- 출처: [Claude Code changelog](https://code.claude.com/docs/en/changelog) (v2.1.187, v2.1.193)
- 💡 **핵심**: 자동 모드에서 작업물을 날리는 명령은 사용자가 명시 요청하지 않는 한 분류기가 막는다. 별도 설정 없이 기본 적용.
- 📝 **how-to**: 모든 셸 명령을 분류기에 태우고, 거부 사유를 추적하려면:

```json
// .claude/settings.json
{
  "autoMode": { "classifyAllShell": true }
}
```

거부 내역은 `/permissions` 의 recent denials 탭과 거부 토스트, 트랜스크립트에서 확인.
- 🎯 **언제**: CI·원격 러너 등 사람이 매 명령을 승인하지 못하는 비대화형 자동 실행.
- ⚠️ **caveat**: 이건 권한 게이트일 뿐 파일시스템 롤백이 아니다. 진짜 안전망은 별도 브랜치 + 커밋. 실수로 날렸다면 `/rewind`(v2.1.191)로 `/clear` 이전 대화를 복구하는 것도 가능하지만 작업 트리 복구는 아님.
- ⭐ **추천**: 자동/원격 세션을 돌린다면 `classifyAllShell` 켜고 시작.

### 🟩 Codex — MCP 툴 검색 기본 활성화 (MCP)
- 출처: [Codex releases](https://github.com/openai/codex/releases) (0.142.2)
- 💡 **핵심**: 지원되는 MCP 서버에서 **툴 검색이 기본 켜짐**. 수십 개 툴을 모두 프롬프트에 펼치지 않고 필요할 때 검색해 불러온다 → 컨텍스트·토큰 절약.
- 📝 **how-to**: stable로 핀 고정해 공유 환경의 동작을 일정하게:

```bash
npm i -g @openai/codex@0.142.3
codex --version   # 0.142.3 확인
```

- 🎯 **언제**: MCP 서버를 여러 개 물려 툴 수가 많아 컨텍스트가 새는 에이전트.
- ⚠️ **caveat**: alpha 라인(0.143.x)은 동작이 자주 바뀌니 팀 공용 CI엔 stable만.
- ⭐ **추천**: 툴 10개 이상이면 검색 기반 디스커버리가 사실상 필수.

### 🟪 공통 — eval에 '과정 규율' 축을 추가하라 (Eval)
- 출처: [RigorBench 2606.22678](https://arxiv.org/abs/2606.22678)
- 💡 **핵심**: 정답률만 보던 eval에 RigorBench의 5축 중 2~3개(검증 수행 여부, 불확실 시 기권 여부, 변경의 원자성)를 더하면 프로덕션 사고를 더 잘 잡는다.
- 📝 **how-to**: eval 케이스마다 통과/실패 외에 아래 체크리스트를 기록.

```text
[ ] 코드 변경 전 계획을 명시했는가
[ ] 변경 후 스스로 검증(테스트/실행)을 돌렸는가
[ ] 정보가 부족할 때 추측 대신 멈췄는가(기권)
[ ] 한 번에 한 가지 변경만 커밋했는가(원자성)
```

- 🎯 **언제**: 자동화 비중을 높이기 전, 하네스 신뢰도를 비교·추적할 때.
- ⚠️ **caveat**: 과정 지표는 주관적이라 절대 점수보다 **같은 하네스의 개선 추세**로 해석.
- ⭐ **추천**: "정답은 맞는데 가끔 사고 친다"는 에이전트에 특히 효과적.
