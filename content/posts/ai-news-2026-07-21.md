# 🔥 오늘의 핵심

오늘 신선한 프런티어 모델 릴리스는 없다. 대신 **아키텍처 결정을 감(感)이 아니라 근거로 내리라**는 논문 두 편이 강하다. 하나는 "멀티에이전트가 언제 이득인가"를 정보병목으로 정량화했고(항상 이득 아님 — **릴레이 용량이 충분하고 모델이 약할 때만**), 다른 하나는 월드모델로 실행 없이 결과를 예측해 **데이터과학 에이전트 학습을 ~14× 가속**했다. 오늘 할 일: 멀티에이전트로 쪼개기 전에 단일 에이전트 베이스라인을 먼저 측정하고, 툴 실행이 비싼 루프엔 "예측→선택" 계층을 넣어라.

- **Codex — Guardian 자동리뷰 정책 복구(7/16).** 프롬프트 회귀로 흐트러졌던 Guardian 자동리뷰의 정책·요청 포맷·툴 동작이 이전 상태로 복구됐고, 위험 명령 탐지도 강화됐다(강제 `rm` 변형 추가, 거부 사유 명확화). GPT-5.6 컨텍스트는 272,000 토큰으로 정정된 값이 유지된다. → **시사점**: Guardian 리뷰를 CI에 붙여둔 팀은 정책이 원복됐으니 억제/허용 규칙을 다시 확인하라. 출처: [developers.openai.com/codex/changelog](https://developers.openai.com/codex/changelog)

# 📚 주목 논문

## 1) 멀티에이전트는 언제 이득인가 — 정보병목 관점
- **저자**: Wendi Yu, Lianhao Zhou, Xiangjue Dong, Sai Sudarshan Barath, Declan Staunton, Byung-Jun Yoon, Xiaoning Qian, James Caverlee, Shuiwang Ji (arXiv `2607.16133`, cs.LG, 2026-07-17 제출)
- **핵심**: "멀티에이전트가 단일 에이전트보다 낫다"를 무조건 참으로 두지 않고, **에이전트 간 메시지(릴레이)를 정보병목**으로 모델링해 이득이 나는 조건을 규명.
- **방법(요약)**: 에이전트를 고립된 로컬 컨텍스트 + 제한된 상호 메시지로 두면 압축 트레이드오프가 생긴다 — 중복을 줄이면 효율↑, 그러나 태스크에 필요한 정보까지 잃을 수 있다.
- **결과(검증됨)**: **18개 통제 실험 · 벤치 5종 · 모델 스케일 3종**에서, 멀티에이전트는 **릴레이 용량이 '거의 충분'할 때, 그리고 약한 모델에서** 가장 크게 도움이 됐다. 강한 모델은 중복 컨텍스트에서 스스로 가치를 뽑아내 이득이 줄었다.
- **🛠 agent-builder 시사점**: 멀티에이전트로 쪼개기 전에 **단일 에이전트 베이스라인**을 먼저 재라. 쪼갤 거라면 에이전트 간 메시지 대역을 넉넉히(=릴레이 충분) 설계하고, 강한 모델을 쓸수록 분할 이득이 작다는 걸 전제하라.
- **한계**: 정보병목 추상화 기반이라 실제 프로덕션 오케스트레이션의 지연/비용 요소는 별도 검증 필요.
- 링크: [arxiv.org/abs/2607.16133](https://arxiv.org/abs/2607.16133)

![멀티에이전트 이득 결정 가이드: x축 릴레이 용량, y축 이득. 약한 모델은 릴레이가 충분해지면 이득이 크고, 강한 모델은 대체로 낮게 평탄하다](/images/ai-news-2026-07-21/mas-decision.svg)
*논문의 정성적 결론을 결정 가이드로 재구성(수치가 아니라 방향성). 스위트스팟은 "릴레이 ≈ 충분·약한 모델". (자체 제작 SVG)*

## 2) DSWorld — 데이터과학 에이전트를 위한 월드모델
- **저자**: Zherui Yang, Fan Liu, Hao Liu (arXiv `2607.15901`, cs.AI, 2026-07-17 제출)
- **핵심**: 데이터과학 연산의 결과를 **실제 실행 없이 예측**하는 월드모델. 비싼 실행을 예측으로 대체해 학습·탐색 비용을 크게 낮춘다.
- **방법(요약)**: 구조화된 상태 표현 + 비용 인지 라우팅 + (비싼 연산엔) LLM 시뮬레이터. **8K 규모 전이 궤적**으로 학습하고, 오류 인지형 RL인 Reflective World Model Optimization을 적용.
- **결과(검증됨)**: **RL 에이전트 학습 ~14× 가속**, 탐색 기반 추론 **3–6× 가속**, 전이 예측 정확도는 최강 LLM 베이스라인 대비 **+35.6%**.
- **🛠 agent-builder 시사점**: 툴 실행이 느리거나 비싼 에이전트(코드 실행·쿼리·시뮬레이션)라면, **"실행 전 결과 예측 → 유망 경로만 실제 실행"** 계층을 붙여 탐색 비용을 줄일 여지가 있다.
- **한계**: 데이터과학 도메인 기반이라 일반 툴 환경으로의 이식성은 별도 확인 필요.
- 링크: [arxiv.org/abs/2607.15901](https://arxiv.org/abs/2607.15901)

![DSWorld 가속 막대: RL 학습 ~14배, 탐색 추론 3-6배, 전이 예측 정확도 +35.6%](/images/ai-news-2026-07-21/dsworld-speedup.svg)
*월드모델로 실행을 예측으로 대체해 얻은 가속(검증된 수치). (자체 제작 SVG, 값 출처 arXiv 2607.15901)*

# 🛠 오픈소스 스포트라이트

## 오늘의 논문과 잇는 프레임워크 선택 관점
- **무엇을**: MAS 논문의 결론은 곧 프레임워크 선택 기준이 된다. **멀티에이전트 오케스트레이터**(예: OpenManus, Semantic Kernel+AutoGen 계열)를 도입할지 말지는 "우리 태스크의 릴레이 용량이 충분한가 / 우리가 쓰는 모델이 강한가"로 결정하라.
- **에이전트 개발자 활용처**: 강한 모델 + 단순 태스크라면 멀티에이전트 오케스트레이션이 비용만 늘릴 수 있다. 반대로 약한/저비용 모델로 복잡 태스크를 풀 때, 메시지 대역을 충분히 준 멀티에이전트가 실질 이득을 낼 확률이 높다.

> ⚠️ **검증 메모**: 이번 세션은 GitHub 접근이 이 블로그 리포로만 스코프돼 외부 리포의 **스타 수·최신 커밋을 API로 검증할 수 없었다**. 그래서 오늘은 특정 리포의 스타 수치를 싣지 않고, 논문 근거에 기반한 **선택 기준**만 제시한다. 도입 전 각 리포의 활동성 지표는 직접 확인하길.

# 🧑‍💻 Codex & Claude Code 실전 팁

## 🟦 Claude Code — `/fork`로 안전하게 분기 탐색하기
- 카테고리: Workflow
- 출처: [code.claude.com/docs/en/changelog](https://code.claude.com/docs/en/changelog) (v2.1.213)
- 💡 **핵심**: 재설계된 `/fork`는 **현재 세션을 유지한 채 대화를 백그라운드 세션으로 복제**한다. "이 방향이 맞나?"를 원본 손상 없이 실험할 수 있다.
- 📝 **How-to**:
  ```text
  # 원본 세션에서
  /fork            # 현재 대화를 백그라운드 세션으로 복제 (원본은 그대로 진행 가능)
  /subtask         # (구 in-session fork) 같은 세션 안에서 서브에이전트로 위임할 때
  # 병렬 팬아웃이 많다면 상한을 명시
  export CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSION=50
  export CLAUDE_CODE_MAX_WEB_SEARCHES_PER_SESSION=100
  ```
- 🎯 **언제**: 리팩터링 방향이 둘 이상이라 각각을 격리해 실험하고 싶을 때.
- ⚠️ **주의**: 과거 `/fork`의 in-session 동작에 의존하던 매크로/스크립트는 `/subtask`로 옮겨야 한다.
- ⭐ **추천**: 위험한 대규모 변경 전, `/fork`로 "실험용 백그라운드 세션"을 먼저 띄우는 습관.

- 🎯 **언제**: 무인 실행(CI·배치)에서 에이전트가 위험 명령을 낼 수 있는 파이프라인.

- ⭐ **추천**: destructive 명령은 항상 사람 승인 + Guardian 이중 게이트.

## 🟪 공통 — 멀티에이전트는 "측정 후" 도입
- 카테고리: Eval
- 출처: [arxiv.org/abs/2607.16133](https://arxiv.org/abs/2607.16133)
- 💡 **핵심**: 멀티에이전트는 릴레이 용량이 충분하고 모델이 약할 때만 유리하다. 아키텍처를 감으로 늘리지 말고 **A/B로 재라**.
- 📝 **How-to**:
  ```text
  1) 단일 에이전트 베이스라인 점수 고정 (골든셋 20~50 태스크)
  2) 멀티에이전트 변형 = 동일 태스크, 에이전트 간 메시지 대역만 넉넉히
  3) 델타가 유의미하지 않으면 → 단일 에이전트 유지(비용·지연 이득)
  ```
- 🎯 **언제**: "에이전트를 더 쪼갤까?" 결정 직전.
- ⚠️ **주의**: 강한 모델일수록 분할 이득이 작다 — 최신 프런티어 모델에선 특히.
- ⭐ **추천**: 멀티에이전트 PR엔 항상 "단일 에이전트 대비 델타" 수치를 첨부.
