# 🔥 오늘의 핵심

오늘 신호는 **"에이전트 훈련·평가 파이프라인"** 한 줄기로 모입니다. 프런티어 모델은 잠잠하지만(아래), 그 아래층 — *어떻게 RL로 멀티턴 에이전트를 학습시키고, 무엇으로 평가할 것인가* — 가 빠르게 정리되고 있습니다. 핵심 교훈은 하나: **평가는 "정답 맞췄나"가 아니라 "몇 스텝·몇 호출로 맞췄나"를 재야 한다**는 것. 오늘은 헤드리스 실행(`claude -p` / `codex exec`)으로 과정 지표를 모으는 루프부터 만들어 보세요.

*오늘의 변화 지도 — 훈련·스캐폴드·실행·평가가 하나의 닫힌 루프. 오늘 다루는 항목들이 각 단계에 정확히 들어맞습니다.*

# 🤖 모델 & 릴리즈 레이더

- **Hugging Face `agent-eval` — 라이브러리 단위 에이전트 평가 (6/18):** 일반 LLM 벤치마크는 *최종 정답*만 보지만, 실제 에이전트 실패는 **과정에서 일어난다**는 주장. 단일 CLI(`agent-eval`)로 모델·라이브러리 리비전·태스크별로 "정답 여부 + 얼마나 일했는가(스텝·호출 수)"를 측정하고, HF Jobs로 팬아웃 후 결과를 HF Space로 발행합니다. 사례 연구는 `transformers` 라이브러리. 🎯 내 에이전트 회귀 테스트에 **"성공률"뿐 아니라 "평균 도구 호출 수"** 지표를 추가하는 직접적 동기. → [HF 블로그](https://huggingface.co/blog/is-it-agentic-enough)
- **프런티어 모델은 소강 상태:** Gemini 3.5 Pro는 여전히 Vertex 제한 프리뷰(GA 미정), Claude Code는 v2.1.185에서 멈춰 있고, Codex의 Appshots·Goal mode GA는 5월 발표분입니다. 오늘은 **큰 신규 모델 소식 없음** — 그래서 오늘은 모델보다 *훈련·평가 인프라*에 집중합니다.

## 1. ProRL Agent: Rollout-as-a-Service for RL Training of Multi-Turn LLM Agents
- **소속:** NVIDIA (NeMo Gym 통합) · **arXiv:** [2603.18815](https://arxiv.org/abs/2603.18815) · cs.LG · 제출 2026-03
- **핵심 기여:** 멀티턴 에이전트 RL 학습에서 **롤아웃을 트레이너에서 분리해 독립 HTTP 서비스로** 제공. 환경 초기화 → 에이전트 실행 → 결과 평가까지 롤아웃 전 과정을 서비스화합니다.
- **방법:** 파이프라인 전체를 **token-in / token-out**으로 통신 → 트레이너가 토큰 단위 trajectory를 그대로 소비해 *re-tokenization drift*를 없앱니다. rootless HPC에서 도는 표준·확장형 샌드박스 환경 제공.
- **검증:** 소프트웨어 엔지니어링·수학·STEM·코딩 태스크 RL 학습으로 검증, 오픈소스 공개([NVIDIA-NeMo/ProRL-Agent-Server](https://github.com/NVIDIA-NeMo/ProRL-Agent-Server)).
- 🛠 **빌더 관점:** 자체 에이전트를 RL로 튜닝하려 할 때, 롤아웃을 서비스로 빼면 **새 도구·환경을 트레이너 코드 수정 없이** 붙일 수 있다는 설계 청사진.
- ⚠️ **한계:** HPC·대규모 학습 전제의 인프라 논문 — 소규모 팀엔 과한 복잡도일 수 있음.
- 🔗 [arXiv abs](https://arxiv.org/abs/2603.18815)

![ProRL Agent 구조도: RL 트레이너와 롤아웃 HTTP 서비스를 token-in/token-out으로 연결하고, 서비스가 SWE·코딩·수학 샌드박스를 호출하는 다이어그램](/images/ai-news-2026-06-22/prorl-rollout.svg)

*1번 논문 한 장 요약 — 롤아웃을 독립 서비스로 빼 트레이너 GPU 유휴를 줄이고 환경을 모듈식으로 교체.*

## 2. SWE-rebench V2: Language-Agnostic SWE Task Collection at Scale
- **arXiv:** [2602.23866](https://arxiv.org/abs/2602.23866) · cs.SE · 제출 2026-02
- **핵심 기여:** 실제 저장소에서 **실행 가능한 SWE 태스크를 언어 무관하게 대량 수집**하는 자동 파이프라인. 단순 데이터셋이 아니라 *RL 학습 환경 자체*를 스케일로 구성합니다.
- **방법:** 대화형 *setup agent*가 저장소별 설치·테스트 절차를 합성 → 사람이 일일이 환경을 세팅하지 않아도 실행·검증 가능한 태스크가 만들어집니다.
- 🛠 **빌더 관점:** 1번(ProRL)이 "어떻게 학습 돌리나"라면 이건 "무엇으로 학습·평가하나"의 공급원. 내 코딩 에이전트 평가셋을 *특정 언어·프레임워크에 가두지 않고* 확장하는 레퍼런스.
- ⚠️ **한계:** 자동 수집 태스크의 품질·난이도 분포는 직접 샘플 검수 필요.
- 🔗 [arXiv abs](https://arxiv.org/abs/2602.23866)

# 🧑‍💻 Codex & Claude Code 실전 팁

오늘 테마(평가 루프)에 맞춰 **헤드리스 실행** 3종.

### 🟦 Claude Code — `claude -p`로 평가/배치 자동화
- **category:** Workflow
- **source:** [Claude Code Headless Docs](https://code.claude.com/docs/en/headless)
- 💡 **core idea:** `-p`(`--print`)를 붙이면 대화형 없이 한 번 실행하고 stdout으로 결과를 뱉고 종료 → CI·야간 배치·병렬 평가의 토대.
- 📝 **how-to:**
```bash
# 구조화 출력 + 사용 도구 제한 + 빠른 시작(bare)
claude -p "auth.py의 버그를 찾아 고쳐줘" \
  --allowedTools "Read,Edit,Bash" \
  --output-format json --bare
```
`--bare`는 hooks·skills·MCP·CLAUDE.md 자동탐색을 건너뛰어 시작을 빠르게 합니다. 세션은 헤드리스 호출 간에도 유지돼 `--continue`로 이어집니다.
- 🎯 **when:** 동일 프롬프트를 여러 모델·리비전에 팬아웃해 회귀 평가를 돌릴 때.
- ⚠️ **caveat:** 무인 실행에서 권한 프롬프트는 멈춤 → `--dangerously-skip-permissions`는 **격리 러너 안에서만**.
- ⭐ **추천:** 평가 하니스의 1순위 실행기. JSON 출력으로 스텝·도구 호출 수까지 파싱하세요.

### 🟩 Codex — `codex exec`로 CI 파이프라인에 끼우기
- **category:** Workflow
- **source:** [Codex Non-interactive Docs](https://developers.openai.com/codex/noninteractive)
- 💡 **core idea:** `codex exec`(=`codex e`)는 사람 개입 없이 끝나는 스크립트/CI용 실행. 진행은 stderr로, **최종 메시지만 stdout**으로 나와 파이프 연결이 깔끔합니다.
- 📝 **how-to:**
```bash
# CI 전용 키로 단발 실행, JSON으로 결과 파싱
CODEX_API_KEY=$KEY codex exec --json \
  "열린 버그 리포트를 트리아지해줘"
```
`--ephemeral`은 세션 롤아웃 파일을 디스크에 남기지 않습니다.
- 🎯 **when:** pre-merge 체크·예약 작업·릴리스 노트 생성처럼 출력을 다른 도구로 넘길 때.
- ⚠️ **caveat:** `--yolo`(승인·샌드박스 우회)는 위험 → 격리 러너 전용. `CODEX_API_KEY`는 `codex exec`에서만 동작.
- ⭐ **추천:** Claude `-p`와 같은 인터페이스로 맞춰두면 두 도구를 같은 평가 하니스에서 교체 비교 가능.

### 🟪 공통 — "정답"이 아니라 "과정"을 평가하라
- **category:** Eval
- **source:** [HF: Is it agentic enough?](https://huggingface.co/blog/is-it-agentic-enough)
- 💡 **core idea:** 성공률만 보면 *비싸게 맞히는* 에이전트를 놓칩니다. **스텝 수·도구 호출 수·토큰 비용**을 함께 기록하세요.
- 📝 **how-to:** 위 헤드리스 JSON 출력에서 `(성공여부, 도구호출수, 총토큰)`을 뽑아 모델·리비전별로 표로 비교 → 회귀 시 *정확도 유지하면서 호출이 늘었는지* 감시.
- 🎯 **when:** 모델 업그레이드·프롬프트 변경의 실제 비용 영향을 판단할 때.
- ⚠️ **caveat:** 과정 지표는 태스크 난이도에 민감 → 고정 평가셋(SWE-bench 등)에서 비교.
- ⭐ **추천:** 어제의 "stop을 1급 행동으로" 정책과 짝 — 호출 수를 줄이는 설계의 *측정 도구*.
