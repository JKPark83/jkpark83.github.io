에이전트 하나에게 큰 일을 통째로 맡기면, 처음엔 잘하다가 컨텍스트가 길어질수록 점점 멍해집니다. 그래서 일을 잘게 쪼개 여러 에이전트가 각자 맡는 멀티 에이전트가 답처럼 보입니다. 그런데 누군가는 "그렇게 했더니 +90%"라고 하고, 누군가는 "절대 그렇게 짓지 마라"고 합니다.

— 둘 다 진심이고, 사실 둘 다 맞습니다. 갈림길은 '읽기냐 쓰기냐'입니다.

이번 주 실무자 타임라인에는 같은 그림이 반복해서 올라왔습니다. 가운데에 오케스트레이터(orchestrator) 하나, 그 주위로 일을 나눠 받는 서브에이전트(subagent) 여럿. "이제는 프롬프트를 짜는 게 아니라 오케스트라를 지휘한다"는 표현이 돌아다녔고, 'From Conductor to Orchestrator' 같은 가이드와 Addy Osmani의 'The Code Agent Orchestra' 글이 함께 회자됐습니다. 한쪽에서는 Anthropic이 멀티 에이전트로 단일 에이전트를 90.2% 앞섰다는 숫자를, 다른 쪽에서는 Cognition이 "멀티 에이전트를 짓지 마라(Don't Build Multi-Agents)"는 글을 들고 나오면서 논쟁이 다시 뜨거워졌습니다.

이 글은 그 논쟁을 정리합니다. 멀티 에이전트가 왜 필요해졌는지(컨텍스트 병목), 핵심 구조인 오케스트레이터·워커가 어떻게 도는지, 왜 통하는지(토큰을 병렬로 쏟기 때문에), 서브에이전트에게 일을 맡길 때 꼭 정해 줘야 하는 것들, 그리고 무엇보다 언제 쪼개고 언제 참아야 하는지를 코드와 그림으로 짚겠습니다. 코딩 기본기가 있는 주니어라면 그대로 따라올 수 있습니다.

1. 왜 쪼개나 — 컨텍스트 윈도우라는 병목

먼저 용어 하나. 컨텍스트 윈도우(context window)는 모델이 한 번에 들여다볼 수 있는 토큰의 총량입니다. 시스템 프롬프트, 대화 기록, 도구 정의, 검색해 온 문서, 중간 추론까지 전부 여기에 욱여넣습니다. 에이전트가 일을 오래 할수록 이 창은 계속 채워집니다.

문제는 창을 키운다고 똑똑해지지 않는다는 점입니다. 오히려 토큰이 많아질수록 정작 중요한 정보가 묻히고 답이 흐려지는 현상이 자주 보고됩니다(흔히 "context rot"이라 부릅니다). 이번 주 화제가 된 Martin Fowler 쪽 글의 한 줄 요약도 똑같습니다 — "이제 코딩 에이전트의 병목은 컨텍스트다. 창을 키운다고 저절로 나아지지 않는다."

한 에이전트에 일을 다 몰아주면 어떻게 될까요? 리포지토리 열 개를 조사하는 작업이라면, 첫 번째 리포의 잡다한 로그·코드·검색 결과가 창에 쌓이고, 그 쓰레기들이 세 번째·네 번째 리포를 볼 때까지 따라다니며 자리를 차지합니다. 정작 결론을 쓸 때쯤이면 창은 노이즈로 가득합니다.

핵심 직관. 일을 쪼개 서브에이전트에게 맡기면, 각 서브에이전트는 자기만의 깨끗한 컨텍스트 윈도우에서 일합니다. 리포 A를 뒤지며 5만 토큰을 쓰더라도, 리드 에이전트에게는 1~2천 토큰짜리 응축된 요약만 돌려보냅니다. 잡음은 서브에이전트의 창에 가두고, 리드의 창은 깨끗하게 유지하는 것 — 이게 멀티 에이전트의 출발점입니다.

즉 멀티 에이전트는 "여러 두뇌를 모은다"기보다, "하나의 거대한 작업을 여러 개의 작은 깨끗한 창으로 나눈다"에 더 가깝습니다. 이 관점을 잡고 가면 나머지가 전부 쉽게 풀립니다.

2. 오케스트레이터·워커 패턴과 컨텍스트 격리

가장 널리 쓰이는 구조는 오케스트레이터·워커(orchestrator–worker)입니다. 허브 앤 스포크(hub-and-spoke)라고도 부릅니다. 가운데 허브에 리드 에이전트(오케스트레이터)가 있고, 바깥 스포크에 워커(서브에이전트)들이 달려 있습니다.

  • 리드 에이전트 — 사용자 요청을 받아 계획을 세우고, 일을 잘게 나눠 워커에게 분배하고, 돌아온 요약들을 종합합니다. 직접 검색하거나 코드를 짜기보다 "지휘"에 집중합니다.
  • 서브에이전트 — 명확히 정의된 작은 하위 작업 하나를 받아, 자기 컨텍스트 안에서 도구를 부르고 결과를 만든 뒤, 응축된 결과만 리드에게 돌려줍니다.

여기서 가장 중요한 개념이 컨텍스트 격리(context isolation)입니다. 서브에이전트는 리드의 대화 기록을 물려받지 않습니다. 각자 새 컨텍스트 윈도우에서 시작하고, 옆에 다른 서브에이전트가 있다는 사실조차 모릅니다. 받은 건 "목표 + 출력 형식 + 도구 안내 + 경계"가 적힌 자기 임무서 한 장뿐입니다.

실제 Anthropic의 리서치 기능이 정확히 이 구조입니다. 리드 리서처(LeadResearcher, Opus)가 계획을 세워 메모리에 저장하고, 병렬로 여러 서브에이전트(Sonnet)를 띄워 각자 조사하게 한 뒤, 마지막에 별도의 CitationAgent가 모든 주장에 출처를 다시 붙입니다. 큰 모델은 지휘에, 작은 모델은 병렬 실행에 쓰는 조합이죠.

3. 왜 통하나 — 토큰을 병렬로 쏟는다

가장 인상적인 숫자부터 보겠습니다. Anthropic은 내부 리서치 평가에서 Opus를 리드로, Sonnet을 서브에이전트로 둔 멀티 에이전트가 단일 에이전트 Opus보다 90.2% 더 잘했다고 밝혔습니다. 흔히 "더 많은 두뇌라서 똑똑해졌나" 싶지만, 그들이 분석한 진짜 이유는 더 단순합니다.

성능 변동의 원인을 분해해 보니 토큰 사용량 하나가 변동의 약 80%를 설명했습니다. 여기에 도구 호출 횟수(약 10%)와 모델 선택(약 5%)을 더하면 셋이서 약 95%를 설명합니다. 풀어 말하면 — 문제를 풀 만큼 충분히 토큰을 쓰면 잘 풀린다는 겁니다. 그리고 단일 에이전트는 컨텍스트 창 하나에 토큰을 쌓는 데 한계가 있지만, 멀티 에이전트는 여러 창에 병렬로 토큰을 쏟아부을 수 있습니다. 멀티 에이전트가 통하는 본질은 바로 이 "토큰의 병렬 확장"입니다.

공짜가 아닙니다. 같은 분석에서 멀티 에이전트는 일반 채팅 대비 약 15배의 토큰을 썼습니다. 90% 성능과 15배 비용을 함께 봐야 합니다. 그래서 결론은 단순합니다 — 작업의 가치가 토큰 비용보다 클 때만 멀티 에이전트가 경제적으로 말이 됩니다. 깊은 리서치처럼 한 번에 큰 가치를 만드는 일에는 맞고, 자잘한 질의응답에는 과합니다.

4. 서브에이전트 계약 — 네 가지를 못 박아라

멀티 에이전트가 무너지는 가장 흔한 이유는 모델이 멍청해서가 아니라, 오케스트레이터가 일을 충분히 명확하게 시키지 않아서입니다. 서브에이전트는 리드의 맥락을 모른 채 임무서 한 장만 들고 일을 시작합니다. 그 임무서에 다음 네 가지가 빠지면 서브에이전트는 엉뚱한 길로 샙니다.

  1. 목표(objective) — 무엇을 알아내거나 만들어야 하는가. 한 문장으로 분명하게.
  2. 출력 형식(output format) — 결과를 어떤 모양으로 돌려줘야 하는가(예: 항목별 요약 5줄, JSON 스키마). 리드가 곧장 종합할 수 있는 형태여야 합니다.
  3. 도구·소스 안내(tools & sources) — 어떤 도구를 쓰고 어디를 봐야 하는가. 방치하면 엉뚱한 도구를 남발합니다.
  4. 경계(boundaries) — 어디까지가 이 임무이고 어디부터는 아닌가. "이건 하지 마라"가 있어야 일이 끝나는 지점을 압니다.

이 네 가지 중 하나라도 빠지면 서브에이전트가 표류합니다. "done이 무엇인지"를 모델이 스스로 정하게 두지 말고, 임무서에 박아 두는 게 핵심입니다.

또 하나, 서브에이전트를 몇 개나 띄울지도 리드가 작업 난이도에 맞춰 정해야 합니다. Anthropic은 리드 프롬프트에 아예 규칙을 넣어 둡니다 — 단순 사실 확인은 1개, 직접 비교는 2~4개, 복잡한 리서치는 10개 이상. 무턱대고 많이 띄우면 비용만 폭증합니다.

5. 실전: 작은 오케스트레이터·워커 만들기

개념을 코드로 옮겨 보겠습니다. 프레임워크 없이 의사코드에 가깝게, 핵심 흐름만 담았습니다. 먼저 서브에이전트에게 줄 임무서(계약)를 구조체로 만듭니다.

{{< highlight python >}} from dataclasses import dataclass @dataclass class SubAgentTask: objective: str # 목표: 한 문장으로 분명하게 output_format: str # 출력 형식: 리드가 곧장 종합할 수 있는 모양 tools: list[str] # 쓸 수 있는 도구/소스 boundaries: str # 경계: 여기까지만, 이건 하지 마라 def build_prompt(task: SubAgentTask) -> str: # 서브에이전트는 리드의 맥락을 모른다. 네 가지를 임무서에 못 박는다. return f"""너는 독립적으로 일하는 서브에이전트다. [목표] {task.objective} [출력 형식] {task.output_format} [사용 도구] {", ".join(task.tools)} [경계] {task.boundaries} 규칙: 위 출력 형식만 지켜서, 응축된 결과만 돌려줘라. 과정은 길게 쓰지 마라.""" {{< /highlight >}}

다음은 리드가 일을 쪼개 서브에이전트들을 병렬로 띄우고, 돌아온 요약만 종합하는 부분입니다. 멀티 에이전트의 힘이 "병렬"에서 나오므로 동시에 실행하는 게 중요합니다.

{{< highlight python >}} import asyncio async def run_subagent(task: SubAgentTask) -> str: prompt = build_prompt(task) # 핵심: 각 서브에이전트는 '새 대화'로 시작한다 → 컨텍스트 격리 # llm()은 도구 호출 루프를 포함한 한 번의 독립 실행이라고 보면 된다. summary = await llm(prompt, fresh_context=True) # 5만 토큰 쓰고 1.5천 토큰만 반환 return summary async def orchestrate(user_request: str) -> str: # 1) 리드가 계획을 세워 하위 작업으로 쪼갠다 (난이도에 맞춰 개수 결정) tasks = plan_subtasks(user_request) # 예: 비교 작업이면 2~4개 # 2) 서브에이전트를 병렬로 실행 — 토큰을 여러 창에 동시에 쏟는다 summaries = await asyncio.gather(*(run_subagent(t) for t in tasks)) # 3) 리드는 '요약들'만 받아 종합한다. 잡음은 서브에이전트 창에 갇혀 있다 return await llm(synthesize_prompt(user_request, summaries), fresh_context=True) {{< /highlight >}}

핵심은 세 가지입니다. ① plan_subtasks로 일을 쪼개고, ② asyncio.gather로 병렬 실행하고, ③ 리드는 큰 원본이 아니라 요약만 받아 종합합니다. 이 골격만 지키면 LangGraph든 직접 구현이든 본질은 같습니다.

실전에서 빼먹기 쉬운 마무리 한 가지 — Anthropic 사례의 CitationAgent처럼, 종합한 결과의 사실·출처를 다시 검증하는 단계를 따로 두면 신뢰성이 크게 올라갑니다. 서브에이전트들이 각자 본 것을 리드가 합치는 과정에서 출처가 뒤섞이기 쉽기 때문입니다.

6. 반대편의 경고 — "멀티 에이전트 짓지 마라"

여기까지 보면 "무조건 쪼개는 게 답"처럼 들립니다. 그런데 같은 시기 Cognition(코딩 에이전트 Devin을 만든 곳)의 Walden Yan은 정반대 제목의 글을 냈습니다 — "Don't Build Multi-Agents." 핵심 주장은 이렇습니다.

맥락 공유가 안 되면 결정이 충돌한다. 서브에이전트들이 서로의 맥락을 모른 채 각자 결정을 내리면, 합칠 때 앞뒤가 안 맞는 결과가 나옵니다. 한 서브에이전트는 버튼을 파란색으로 만들고 다른 서브에이전트는 같은 버튼을 빨간색으로 만드는 식이죠. 컨텍스트 격리가 장점이자 동시에 약점이라는 겁니다.

그래서 Cognition의 처방은 "기본은 단일 스레드(single-threaded) 에이전트"입니다. 멀티 에이전트는 아주 높은 기준을 통과할 때만 쓰라는 거죠. 이게 Anthropic의 +90.2%와 모순처럼 보이지만, 자세히 보면 두 진영이 서로 다른 종류의 일을 이야기하고 있습니다.

실제로 Cognition도 이후 입장을 더 날카롭게 다듬었습니다 — "쓰기(write)는 단일 스레드로 유지하고, 추가되는 에이전트는 '행동'이 아니라 '지능'을 보태게 하라." 즉 여러 에이전트가 동시에 같은 결과물을 고쳐 쓰게 두지 말고, 조사·검토 같은 읽기·판단만 나눠 맡기라는 겁니다. 바로 여기서 도입 기준이 갈립니다.

7. 그래서 언제 쪼개나 — 읽기 vs 쓰기

두 진영을 한 줄로 화해시키면 이렇습니다. 읽기(read-heavy) 작업은 쪼개기 좋고, 쓰기(write-heavy) 작업은 쪼개기 위험하다.

  • 읽기 작업 — 리서치, 정보 수집, 여러 소스 비교처럼 각자 따로 봐도 되는 일. 서브에이전트들이 독립적으로 토큰을 쏟고 요약만 합치면 됩니다. 충돌할 결과물이 없으니 멀티 에이전트가 빛납니다(Anthropic 리서치).
  • 쓰기 작업 — 하나의 코드베이스를 함께 수정하는 일처럼 모두가 같은 결과물·맥락을 공유해야 하는 일. 여기서 컨텍스트를 격리하면 결정이 충돌합니다. 쓰기는 단일 스레드로 묶는 게 안전합니다(Cognition).

읽기 작업 안에서도 한 번 더 갈립니다. 워커들이 서로의 발견을 공유하고 반박하며 조율해야 한다면 단순 서브에이전트로는 부족합니다. 이럴 때 쓰는 게 에이전트 팀(agent teams)입니다. 판단 기준은 한 문장 — "워커끼리 소통이 필요한가?" 필요 없으면 서브에이전트(빠르게 일하고 요약만 반환), 필요하면 에이전트 팀(서로 조율). 대부분의 일은 서브에이전트로 충분합니다.

8. 흔한 실수 · 팁

  • 가치 없는 일에 멀티 에이전트. 15배 비용을 기억하세요. 간단한 질의응답이나 한 파일 수정에 오케스트레이터를 얹으면 돈과 지연만 늘어납니다. 단일 에이전트로 먼저 시도하고, 막힐 때만 쪼개세요.
  • 임무서가 부실하다. 표류의 90%는 모델 탓이 아니라 목표·형식·도구·경계 중 하나가 빠진 탓입니다. 4장의 네 가지를 체크리스트로 쓰세요.
  • 쓰기를 병렬로 쪼갠다. 같은 코드베이스를 여러 서브에이전트가 동시에 고치면 결정이 충돌합니다. 쓰기는 단일 스레드, 추가 에이전트는 조사·검토 같은 '지능'만 맡기세요.
  • 서브에이전트가 원본을 통째로 반환한다. 그러면 격리의 의미가 사라지고 리드의 창이 다시 터집니다. 반드시 응축된 요약만 돌려주도록 출력 형식을 강제하세요.
  • 출처가 뒤섞인다. 여러 서브에이전트의 결과를 합치면 "누가 어디서 봤는지"가 사라집니다. CitationAgent처럼 출처 재검증 단계를 따로 두세요.
  • 관측이 어렵다. 에이전트가 여럿이면 어디서 잘못됐는지 추적이 까다롭습니다. 각 서브에이전트의 입력 임무서와 반환 요약을 트레이스로 남겨 디버깅 가능하게 만드세요.

9. 마무리 (TL;DR)

  • 병목은 컨텍스트다. 창을 키운다고 똑똑해지지 않는다. 일을 쪼개 각자 깨끗한 컨텍스트 윈도우에서 일하게 하는 게 멀티 에이전트의 본질이다.
  • 오케스트레이터·워커. 리드가 계획·분배·종합을 맡고, 서브에이전트는 격리된 창에서 일한 뒤 응축된 요약만 돌려준다.
  • 왜 통하나. 토큰 사용량이 성능 변동의 약 80%를 설명한다. 멀티 에이전트는 토큰을 병렬로 쏟아 단일 창의 한계를 넘는다 → 단일 Opus 대비 +90.2%. 단, 토큰은 15배.
  • 서브에이전트 계약. 목표·출력 형식·도구·경계 네 가지를 못 박아라. 빠지면 표류한다.
  • 언제 쪼개나. 읽기(리서치·비교)는 쪼개기 좋고, 쓰기(같은 코드 수정)는 단일 스레드로 묶어라. "충돌할 결과물이 없으면 쪼개고, 모두가 같은 걸 만지면 묶어라."
  • 작게 시작하라. 단일 에이전트로 먼저, 막힐 때만 쪼개라. 멀티 에이전트는 가치가 비용을 넘을 때만 경제적이다.
