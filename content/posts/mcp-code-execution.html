에이전트에 MCP 서버를 하나둘 붙이다 보면, 정작 첫 메시지를 보내기도 전에 컨텍스트가 수만 토큰씩 차 있습니다. 도구를 쓰기도 전에 도구 설명서만으로 자리를 다 잡아먹은 거죠.

— 그래서 2026년의 화두는 "도구를 어떻게 부를 것인가"입니다. 호출(call) 대신 코드(code)로요.

에이전트에게 도구를 쥐여 주는 표준은 이제 MCP(Model Context Protocol)로 거의 굳었습니다. Notion, Slack, GitHub, 사내 DB… MCP 서버 하나만 연결하면 그 안의 도구 수십 개가 한꺼번에 딸려 옵니다. 편하죠. 그런데 이 편함에는 조용한 청구서가 붙습니다. 도구를 많이 연결할수록, 아무 일도 하기 전에 컨텍스트가 먼저 터집니다.

이번 주 실무자 타임라인에는 "우리가 MCP를 잘못 쓰고 있었다"는 글이 줄줄이 올라왔습니다. 핵심은 하나입니다. 도구 50개를 연결하면 그 정의(definition)만으로 컨텍스트의 77,000토큰가량이 사라지고, 도구가 돌려주는 큰 결과까지 모델 컨텍스트를 왕복하면서 비용·지연·정확도가 동시에 무너진다는 것이죠. 그리고 이걸 푸는 세 가지 해법 — Anthropic의 Tool Search Tool, Code Execution with MCP, 그리고 Cloudflare의 Code Mode — 이 한데 묶여 "MCP 컨텍스트 다이어트"라는 주제로 화제가 됐습니다.

이 글은 그 문제와 해법을 기초부터 정리합니다. MCP를 깊이 안 써 본 주니어 개발자도 따라올 수 있게, 왜 터지는지부터 그림으로 시작하겠습니다.

1. 먼저, 도구 호출은 어떻게 컨텍스트를 먹나

도구 호출(tool calling, 함수 호출이라고도 합니다)의 흐름부터 짚겠습니다. 모델은 사실 도구를 직접 실행하지 못합니다. 대신 "이 도구를 이런 인자로 불러 줘"라고 JSON을 뱉을 뿐이고, 실행은 바깥의 코드(에이전트 런타임)가 합니다.

그러려면 모델이 "어떤 도구가 있고, 각 도구가 어떤 인자를 받는지"를 알아야 합니다. 그래서 매 요청마다 모든 도구의 정의(이름·설명·파라미터 스키마)를 시스템 프롬프트에 통째로 넣어 줍니다. 도구 하나의 스키마가 보통 수백 토큰입니다. 도구가 다섯 개면 별것 아니지만, MCP 서버 몇 개를 붙여 도구가 50~100개가 되면 이야기가 달라집니다.

왜 이게 문제일까요? 컨텍스트 윈도우(context window)는 모델이 한 번에 볼 수 있는 작업 공간입니다. 도구 정의가 이 공간의 앞자리를 미리 다 차지하면 ① 입력 토큰값이 매 호출마다 비싸지고, ② 긴 입력 한가운데 정보를 모델이 자주 놓치며(흔히 'lost in the middle'), ③ 비슷한 이름의 도구가 많아질수록 엉뚱한 도구를 고릅니다. 즉 토큰만 새는 게 아니라 정확도까지 같이 떨어집니다.

실제 숫자로 보면 체감이 됩니다. MCP 도구 50개를 그냥 다 올리면 대략 77,000토큰이 도구 정의로만 소모됩니다. 모델 컨텍스트가 200K라면, 일 시작도 전에 3분의 1 이상이 "메뉴판"으로 채워지는 셈입니다.

2. 두 가지 누수 — 정의 누수와 결과 누수

컨텍스트가 새는 통로는 사실 둘입니다. 이 둘을 구분해야 해법이 보입니다.

  • 정의 누수: 이번 작업에서 쓸 도구는 두세 개인데, 연결된 도구 100개의 설명서를 매 호출마다 전부 다시 보냅니다. → 해법 1·3이 겨냥하는 지점입니다.
  • 결과 누수: list_users()가 1만 건을 돌려주면, 그 1만 건이 통째로 모델 컨텍스트에 들어왔다가 다음 도구로 넘어갑니다. 정작 모델에 필요한 건 "활성 사용자 5명"뿐인데도요. → 해법 2·3이 겨냥하는 지점입니다.

3. 해법 1: 점진적 공개 (Tool Search Tool)

가장 직관적인 해법은 "안 쓸 도구는 안 보여 주기"입니다. 식당에 비유하면, 100쪽짜리 메뉴판을 통째로 안기는 대신 "찾으시는 메뉴 말씀하시면 그 페이지만 펼쳐 드릴게요"라고 하는 거죠. 이걸 점진적 공개(progressive disclosure)라고 부릅니다.

Anthropic이 2025년 11월에 내놓은 Tool Search Tool이 정확히 이 방식입니다(beta 헤더 advanced-tool-use-2025-11-20). 도구 정의를 다 올리되, 당장 안 쓸 도구에는 defer_loading: true 표시를 답니다. 그러면 모델 컨텍스트에는 그 도구의 전체 정의 대신 '검색 가능한 후보'만 남고, 모델이 필요할 때 검색해서 그때 펼쳐 옵니다.

효과는 분명합니다. Anthropic 보고로는 MCP 평가에서 도구 정의 토큰이 약 85% 줄고(50개 기준 77K → 8.7K), 컨텍스트 윈도우의 95%가 실제 작업용으로 보존됐습니다. 정확도도 같이 올랐습니다. 도구가 많을 때 Tool Search를 켜니 Opus 4.5의 도구 사용 평가 점수가 79.5% → 88.1%로 올랐죠(Opus 4는 49% → 74%). 도구가 적게 보이니 모델이 헷갈릴 일도 준 겁니다.

의사코드로 개념만 보면 이렇습니다.

{{< highlight python >}} # 직접 호출(Before): 도구 100개 정의를 매 요청마다 전부 전송 tools = [crm_tool_1, crm_tool_2, ..., crm_tool_100] # ≈ 150K tokens # 점진적 공개(After): 검색 도구 + '자주 쓰는 핵심 도구'만 먼저, 나머지는 보류 response = client.beta.messages.create( betas=["advanced-tool-use-2025-11-20"], tools=[ tool_search_tool, # 도구를 찾아 주는 메타 도구 {**create_record, "defer_loading": False}, # 핵심 도구는 바로 노출 {**list_users, "defer_loading": True}, # 나머지는 검색될 때만 펼침 {**send_message, "defer_loading": True}, # ... 도구 97개 더, 전부 defer_loading: True ], messages=messages, ) # 모델은 필요할 때 tool_search로 도구를 찾고, 찾은 정의만 컨텍스트에 들어온다 {{< /highlight >}}

같은 릴리스에는 형제 기능이 둘 더 있습니다. Tool Use Examples(도구마다 현실적인 호출 예시를 붙여 복잡한 파라미터 정확도를 72% → 90%로 끌어올림)와, 다음 절의 주인공인 Programmatic Tool Calling입니다.

4. 해법 2: 코드 실행 (Code Execution with MCP)

점진적 공개는 '정의 누수'를 막습니다. 하지만 '결과 누수' — 큰 도구 결과가 모델을 통째로 거쳐 가는 문제 — 는 그대로입니다. 여기서 더 과감한 발상이 나옵니다. 도구를 하나씩 "호출"하게 하지 말고, 모델이 "코드를 써서" 도구를 부르게 하자.

핵심 아이디어는 이렇습니다. MCP 서버의 도구들을 모델에게 JSON 함수 목록으로 들이미는 대신, 파일시스템 위의 코드 API처럼 보여 줍니다. 예를 들어 CRM 서버는 ./servers/crm.ts, Slack 서버는 ./servers/slack.ts 같은 식이죠. 모델은 필요한 함수만 import 하는 코드를 작성하고, 그 코드를 샌드박스에서 실행합니다.

이 구조가 두 누수를 동시에 막습니다.

  • 정의 누수 차단: 모델은 도구 100개의 스키마를 미리 받지 않습니다. 파일 목록을 훑고, 이번에 쓸 함수의 시그니처만 그때 읽습니다(이것도 일종의 점진적 공개입니다).
  • 결과 누수 차단: list_users()가 돌려준 1만 건은 샌드박스 안에 머뭅니다. 거기서 코드로 필터링·집계한 뒤, 모델에는 "활성 5명"이라는 작은 결과만 돌려줍니다.

Anthropic은 이 방식으로 한 벤치마크에서 같은 작업의 토큰을 150,000 → 2,000으로 줄였다고 합니다. 98.7% 절감입니다. 게다가 중간 데이터가 모델을 거치지 않으니, 개인정보 같은 민감 데이터를 컨텍스트에 노출하지 않고 흘려보낼 수 있다는 부가 효과까지 있습니다.

Anthropic API의 Programmatic Tool Calling이 바로 이 발상을 제품화한 기능입니다. 모델이 코드를 써서 여러 도구를 엮고, 그 출력 중 무엇을 자기 컨텍스트에 들일지 직접 고릅니다. 실제로 Claude for Excel은 이 방식으로 수천 행짜리 스프레드시트를 컨텍스트를 넘치게 하지 않고 읽고 고칩니다.

5. 해법 3: Cloudflare Code Mode

Cloudflare도 거의 같은 시기에 같은 결론에 도달했습니다. 이름은 Code Mode. 발상은 한 문장으로 요약됩니다. "모델은 도구를 직접 호출하는 것보다, 도구를 호출하는 코드를 작성하는 데 훨씬 능하다." LLM은 학습 데이터에서 JSON 함수 호출보다 실제 코드를 압도적으로 많이 봤으니까요.

Code Mode는 연결된 MCP 서버들을 타입이 붙은 TypeScript API로 변환해 모델에게 보여 줍니다. 그리고 모델에게 노출되는 도구는 사실상 딱 두 개입니다 — search()(어떤 API가 있는지 찾기)와 execute()(작성한 코드 실행하기). 모델이 쓴 코드는 인터넷과 격리된 V8 isolate 샌드박스(Dynamic Worker)에서 돌고, 바깥세상으로 나가는 통로는 연결된 MCP 서버의 TS API뿐입니다.

숫자로 보면: 이 두 도구만으로 Cloudflare의 전체 API를 MCP로 제공하면서 컨텍스트 소비는 약 1,000토큰에 그칩니다. MCP 서버 하나를 TS API로 바꾸기만 해도 토큰을 약 81% 절감한다고 보고합니다. 샌드박스는 파일시스템도, 환경 변수 노출도 없고, 바깥 요청은 명시적으로 허용된 핸들러로만 나갑니다.

정리하면 셋은 강조점이 조금씩 다릅니다. Tool Search는 '정의를 미루는' 데, Code Execution과 Code Mode는 '호출을 코드로 바꾸는' 데 무게를 둡니다. 뒤의 둘은 결과 필터링까지 한 번에 가져갑니다.

6. 왜 '코드로 부르기'가 이기나

도구를 한 번에 하나씩 호출하는 방식의 근본 한계는, 매 호출마다 모델을 한 바퀴 돌아야 한다는 데 있습니다. "list_users 호출 → 결과 1만 건이 컨텍스트로 → 모델이 그중 5명 추림 → create_record 호출"처럼, 도구 사이를 오가는 데이터가 전부 모델을 경유합니다. 단계가 늘수록 토큰도, 왕복 지연도, 실수할 자리도 함께 늘죠.

코드로 부르면 이 흐름이 한 번에 돕니다.

{{< highlight python >}} # 직접 호출 방식 — 모델을 3번 왕복하고, 1만 건이 컨텍스트를 통과한다 rows = list_users() # ← 결과 1만 건이 모델 컨텍스트로 active = [r for r in rows if r.active][:5] # ← 모델이 머릿속으로 추림 create_record(active) # ← 또 한 번 왕복 # 코드 실행 방식 — 모델은 아래 '코드'만 한 번 내보내고, 실행은 샌드박스가 한다 def task(): rows = list_users() # 1만 건은 샌드박스 안에만 존재 active = [r for r in rows if r.active][:5] # 거르기도 샌드박스에서 return create_record(active) # 모델엔 최종 결과 한 줄만 반환 {{< /highlight >}}

차이는 셋입니다. ① 도구 정의를 필요할 때만 읽으니 정의 누수가 막히고, ② 큰 중간 결과가 샌드박스에 머무니 결과 누수가 막히며, ③ 반복·조건·필터 같은 제어 흐름을 코드가 처리하니 모델 왕복이 줄어 더 빠르고 싸집니다. 모델은 "무엇을 할지"만 코드로 정하고, "어떻게 데이터를 주무를지"는 코드가 알아서 합니다.

(Cloudflare)		 MCP를 타입 붙은 TS API로 바꾸고 V8 isolate에서 실행 정의 + 결과 누수 전체 API ≈ 1,000 토큰
변환 시 ~81%↓		 Workers/Cloudflare 생태계, 강한 샌드박스 격리가 필요할 때

고르는 기준은 간단합니다. 도구가 적으면 그냥 직접 호출하세요. 도구가 많아 정의가 부담이면 Tool Search로 시작하세요(코드를 거의 안 고쳐도 됩니다). 결과가 크거나 여러 도구를 엮어야 하면 코드 실행 계열로 넘어가세요. 세 가지는 배타적이지 않습니다 — 실제로는 "코드 실행 + 도구 검색"을 함께 쓰는 게 가장 강력합니다.

8. 흔한 실수 · 팁

  • 작은 도구셋에 코드 실행을 끌어오기. 도구가 5개면 코드 실행 인프라(샌드박스·파일 API)가 오히려 짐입니다. 누수가 실제로 아플 때 도입하세요. 측정 먼저, 최적화 나중.
  • 샌드박스 보안을 가볍게 보기. 모델이 쓴 코드를 실행한다는 건, 신뢰할 수 없는 코드를 실행한다는 뜻입니다. 반드시 격리된 환경(컨테이너, V8 isolate 등)에서 돌리고, 네트워크·파일·환경 변수 접근을 최소 권한으로 잠그세요. Code Mode가 "인터넷에서 격리, FS·env 없음"을 기본값으로 둔 데는 이유가 있습니다.
  • 모델이 코드를 못 쓸 거라 단정하기. 요즘 모델은 JSON 호출보다 코드를 더 잘 씁니다. 다만 작은 모델일수록 편차가 크니, 코드 실행 경로는 반드시 평가(eval)로 검증하고 실패 시 직접 호출로 떨어지는 폴백을 두세요.
  • 디버깅이 더 어려워짐을 잊기. "도구 한 번 호출"이 "코드 한 덩어리 실행"으로 바뀌면, 어디서 틀렸는지 추적이 까다로워집니다. 실행한 코드·입력·출력을 트레이스(trace)로 남겨 두세요. 관측(observability) 없이 코드 실행을 켜는 건 눈 감고 운전하는 격입니다.
  • 도구 이름·네임스페이스를 대충 짓기. 점진적 공개든 코드 실행이든, 모델이 "필요한 도구를 찾는" 단계가 생깁니다. crm.create_record처럼 서비스·동작으로 네임스페이스를 명확히 두면 검색·선택 정확도가 올라갑니다.
  • 전부 보류로 돌리기. Tool Search에서 자주 쓰는 핵심 도구까지 defer_loading: true로 두면, 매번 검색하느라 왕복이 늘 수 있습니다. 1군 도구는 바로 노출하고, 꼬리(long tail)만 보류하세요.

9. 마무리 (TL;DR)

  • MCP로 도구를 많이 붙이면 두 가지 누수로 컨텍스트가 샙니다 — 안 쓸 도구까지 미리 올리는 정의 누수, 큰 결과가 모델을 거쳐 가는 결과 누수.
  • 점진적 공개(Tool Search Tool): defer_loading으로 도구를 검색 시점에만 펼쳐 정의 누수를 막습니다. 도구 토큰 약 85%↓, 정확도도 ↑.
  • 코드 실행(Code Execution with MCP): 도구를 호출하는 대신 모델이 코드를 써서 부르고, 큰 결과는 샌드박스에서 걸러 요약만 반환합니다. Anthropic 보고 150K → 2K(98.7%↓).
  • Code Mode(Cloudflare): MCP를 타입 붙은 TS API로 바꿔 V8 isolate에서 실행. 전체 API를 약 1,000토큰에, 변환만으로 ~81%↓.
  • 고르는 법: 도구 적으면 직접 호출 → 정의가 부담이면 Tool Search → 결과가 크거나 여러 도구를 엮으면 코드 실행. 측정 먼저, 그리고 샌드박스 보안과 트레이스는 필수.
