간단한 에이전트 루프를 처음 짜면 감동합니다. LLM이 도구를 부르고, 결과를 보고, 다시 도구를 부르고… 알아서 문제를 풉니다. 그런데 "이 주제로 리서치해서 보고서 써 줘" 같은 진짜 일을 맡기는 순간, 열 몇 번 도구를 부르다 컨텍스트가 꽉 차고, 처음에 뭘 하려 했는지 잊고, 엉뚱한 결론으로 끝납니다.

— 얕은 에이전트는 데모에서 빛나고 긴 작업에서 무너진다. 그 간극을 메우는 게 Deep Agent이다.

에이전트를 조금 다뤄 본 junior 개발자라면 이 장면이 익숙할 겁니다. while 루프 안에서 LLM에게 도구 몇 개를 쥐여 주면, 짧은 작업은 정말 잘 해냅니다. "이 파일에서 버그 찾아 줘" 정도는 훌륭합니다. 그런데 단계가 대여섯 개를 넘어가는 일 — 코드베이스 전체 리팩터링, 여러 소스를 뒤지는 리서치, 스펙 하나를 처음부터 구현 — 을 맡기면 갑자기 바보가 됩니다.

이걸 실무자들은 얕은 에이전트(shallow agent)라고 부릅니다. 도구를 부르는 단순 루프는 맞지만, 긴 시간 축을 버틸 구조가 없다는 뜻입니다. 반대말이 Deep Agent(딥 에이전트)입니다. 이름은 거창해도 마법은 아닙니다. 네 가지 평범한 장치를 더했을 뿐입니다 — 상세한 시스템 프롬프트, 계획 도구, 서브에이전트(sub-agent), 파일시스템.

이 주제가 2026년 여름에 부쩍 화제입니다. Claude Sonnet 5가 "가장 에이전트다운 모델"을 내세우며 나왔고, 프런티어 랩들이 며칠 간격으로 비슷한 에이전트 아키텍처를 내놓으면서 "결국 에이전트에 필요한 뼈대는 같더라"는 공감대가 형성됐습니다. 그 뼈대를 그대로 오픈소스로 묶은 LangChain의 deepagents 라이브러리는 이번 주에도 새 버전(0.7 계열 프리릴리스)을 올리며 활발히 갱신 중입니다. Claude Code가 실제로 이 네 기둥으로 동작한다는 사실이 알려지면서, "왜 Claude Code는 긴 작업을 그렇게 잘 버티나"의 답으로 이 패턴이 다시 조명받고 있습니다.

이 글은 그 네 기둥을 하나씩 뜯어봅니다. 왜 얕은 에이전트가 무너지는지, 각 기둥이 정확히 어떤 문제를 푸는지, deepagents로 리서치 에이전트를 직접 만드는 법, 그리고 흔한 실수까지 — 쉽고 자세하게 살펴봅니다.

1. 왜 얕은 에이전트는 긴 작업에서 무너지나

가장 단순한 에이전트는 이런 루프입니다. LLM에게 도구 목록을 주고, "도구를 부르든 최종 답을 내든 하라"고 시킵니다. 도구를 부르면 그 결과를 대화에 이어 붙이고, 다시 LLM을 호출합니다. 최종 답이 나올 때까지 반복합니다.

{{< highlight python >}} # 얕은 에이전트의 뼈대 — 짧은 작업엔 이걸로 충분하다 messages = [{"role": "user", "content": task}] while True: reply = llm(messages, tools=tools) # 도구를 부를지, 답할지 모델이 결정 if reply.tool_calls: for call in reply.tool_calls: result = run_tool(call) # 도구 실행 messages.append(result) # 결과를 '대화'에 그대로 이어 붙임 ← 문제의 씨앗 else: return reply.content # 도구를 안 부르면 그게 최종 답 {{< /highlight >}}

문제는 messages.append(result) 한 줄입니다. 도구를 부를 때마다 그 전체 결과가 대화에 쌓입니다. 웹 검색 결과 20개, 파일 3천 줄, 로그 뭉치… 전부 컨텍스트 창(context window, 모델이 한 번에 볼 수 있는 토큰의 양)에 그대로 눌러 담깁니다. 단계가 길어질수록 세 가지가 동시에 나빠집니다.

  • 컨텍스트가 터진다. 도구 결과가 누적돼 창을 가득 채우고, 결국 한계에 부딪힙니다.
  • 목표가 흐려진다. 처음에 준 지시가 대화의 맨 위로 밀려나고, 최근 도구 결과가 모델의 주의를 독차지합니다. 초반 계획을 잊어버립니다.
  • 정확도가 떨어진다. 토큰이 많아질수록 그 안에서 필요한 정보를 정확히 꺼내 오는 능력이 떨어지는 현상 — 이른바 context rot(컨텍스트 부식) — 이 심해집니다. 창이 크다고 다 채워 쓰면 오히려 성능이 나빠집니다.

핵심을 한 문장으로 줄이면 이렇습니다. 얕은 에이전트에겐 "지금 이 순간의 대화" 말고는 아무 구조가 없습니다. 계획도, 작업 기억을 덜어 둘 곳도, 지저분한 일을 대신 맡길 조수도 없습니다. Deep Agent의 네 기둥은 정확히 이 빈자리를 하나씩 채웁니다.

2. 배경 — 에이전트 루프와 컨텍스트 창이라는 병목

본격적으로 들어가기 전에 딱 두 가지만 짚겠습니다. 이미 안다면 3장으로 건너뛰어도 됩니다.

에이전트 = LLM + 루프 + 도구. 에이전트는 특별한 모델이 아닙니다. 평범한 LLM을 루프에 넣고, 바깥세상과 닿을 도구(웹 검색, 파일 읽기·쓰기, 코드 실행 등)를 쥐여 준 것입니다. 모델은 매 턴 "지금 무슨 도구를 부를까, 아니면 끝낼까"를 정합니다.

컨텍스트 창 = 에이전트의 작업 기억(working memory). 흔한 비유가 "컨텍스트 창은 저장소가 아니라 RAM"입니다. 지금 이 순간 모델이 동시에 붙들고 있는 정보의 총량이고, 유한하며, 꽉 차면 밀려납니다. 그래서 에이전트 엔지니어링의 상당 부분은 결국 이 유한한 RAM에 무엇을 올리고 무엇을 내릴지를 설계하는 일 — 컨텍스트 엔지니어링(context engineering) — 입니다.

이 관점을 쥐고 보면 네 기둥이 왜 그렇게 생겼는지가 선명해집니다. 계획 도구는 목표를 RAM에 다시 띄우고, 서브에이전트는 지저분한 작업을 다른 RAM에서 처리하고, 파일시스템은 지금 안 쓰는 것을 RAM 밖으로 내려 둡니다. 상세한 시스템 프롬프트는 언제 그렇게 할지를 알려 주는 사용 설명서입니다.

3. Deep Agent의 네 기둥 한눈에 보기

먼저 전체 그림입니다. 얕은 에이전트와 Deep Agent의 결정적 차이는 "도구 결과를 어떻게 다루느냐"에 있습니다.

네 기둥을 한 줄씩 요약하면 이렇습니다. 뒤에서 하나씩 자세히 봅니다.

기둥푸는 문제구체적 장치
① 상세한 시스템 프롬프트"언제 계획하고 언제 위임하나"를 모름도구 사용 규칙·절차를 길고 구체적으로
② 계획 도구긴 작업 중 목표를 잊음write_todos / TodoWrite — 체크리스트를 컨텍스트에 유지
③ 서브에이전트지저분한 조사가 메인 창을 오염task 도구 — 별도 컨텍스트에서 처리, 요약만 반환
④ 파일시스템긴 결과·상태를 둘 곳이 없음ls/read_file/write_file/edit_file — 창 밖 저장소

4. 기둥 ① 상세한 시스템 프롬프트

가장 지루하지만 가장 효과가 큰 기둥입니다. 얕은 에이전트가 실패하는 이유 중 하나는, 우리가 "알아서 잘해"라고만 말하기 때문입니다. 모델은 언제 계획을 세우고, 언제 하위 작업을 위임하고, 언제 결과를 파일에 저장할지 스스로 정할 단서가 없습니다.

Deep Agent의 시스템 프롬프트는 짧지 않습니다. 도구 하나하나의 사용 규칙, 작업을 시작할 때의 절차, 위임 판단 기준까지 구체적으로 적습니다. Claude Code나 deepagents의 기본 프롬프트를 열어 보면 놀랄 만큼 깁니다. 예를 들면 이런 식입니다.

{{< highlight text >}} 너는 리서치 에이전트다. 복잡한 요청을 받으면: 1. 먼저 write_todos로 작업을 단계별 체크리스트로 쪼개라. 2. 한 번에 하나의 항목만 in_progress로 두고, 끝나면 곧바로 completed로 표시하라. 3. 깊은 조사가 필요한 하위 작업은 직접 하지 말고 task 도구로 서브에이전트에게 위임하라. - 서브에이전트에는 '무엇을 조사할지'와 '어떤 형식으로 보고할지'를 명확히 전달하라. 4. 웹 페이지 원문·검색 결과 뭉치처럼 긴 내용은 대화에 남기지 말고 write_file로 파일에 저장한 뒤, 필요할 때 read_file로 다시 읽어라. 5. 최종 답을 내기 전, todos가 모두 completed인지 확인하라. {{< /highlight >}}

포인트는 "규칙"이 아니라 "언제 그 도구를 쓰는지"까지 못 박는다는 점입니다. 도구만 쥐여 주고 설명이 부실하면 모델은 그 도구를 거의 안 씁니다. 반대로 절차를 구체적으로 주면, 모델은 매 턴 이 프롬프트를 나침반 삼아 스스로를 조율합니다.

왜 이게 먹히나. 시스템 프롬프트는 대화가 아무리 길어져도 항상 컨텍스트 맨 앞에 상주합니다. 다른 내용이 밀려나도 이 "행동 지침"만은 안 밀려납니다. 그래서 긴 작업일수록 상세한 프롬프트의 값어치가 커집니다.

5. 기둥 ② 계획 도구 (write_todos / TodoWrite)

두 번째 기둥은 놀랄 만큼 단순합니다. 할 일 목록을 쓰는 도구 하나입니다. deepagents에선 write_todos, Claude Code에선 TodoWrite라고 부릅니다. 항목마다 상태가 있습니다 — pending(대기), in_progress(진행 중), completed(완료).

"이게 왜 도구씩이나?"라고 생각하기 쉽습니다. 핵심은 이겁니다. 이 체크리스트는 에이전트 상태에 저장되고, 매 턴 컨텍스트에 다시 실립니다. 즉 목표와 진행 상황을 계속 모델 눈앞에 띄워 두는 장치입니다. 긴 작업 중 초반 계획이 대화 위로 밀려나도, todos만은 최신 상태로 앞에 남습니다.

재미있는 사실 하나. 이 todos는 대단한 스케줄러가 아닙니다. 그냥 모델에게 자기 계획을 소리 내어 말하게 하고, 그 말을 다음 턴까지 붙들어 두는 장치에 가깝습니다. 그런데 이 단순한 트릭이 "스텝을 건너뛰는" 실패를 극적으로 줄입니다. 작업 첫머리에 "먼저 전체 단계를 todos로 적어라"를 강제하는 것만으로도, 중간에 빼먹는 스텝이 눈에 띄게 사라집니다.

6. 기둥 ③ 서브에이전트 — 컨텍스트 격리

세 번째 기둥이 Deep Agent의 진짜 힘입니다. 서브에이전트(sub-agent)는 메인 에이전트가 task라는 도구로 불러내는 또 다른 에이전트입니다. 새로 태어나고, 자기만의 깨끗한 컨텍스트 창을 갖고, 맡은 일을 끝까지 하고, 결과를 딱 한 줄(또는 한 문단) 요약으로 돌려주고 사라집니다.

왜 이게 중요할까요. "이 라이브러리의 인증 흐름을 조사해"라는 하위 작업을 생각해 봅시다. 제대로 하려면 파일 열 개를 읽고, 함수를 따라가고, 문서를 뒤져야 합니다. 그 과정에서 나오는 원문 수천 줄이 메인 컨텍스트에 그대로 쌓이면 앞에서 본 오염이 그대로 재현됩니다. 하지만 서브에이전트에게 맡기면, 그 지저분한 읽기는 딴 창에서 벌어지고, 메인은 "인증은 JWT를 쓰고 만료는 15분, 갱신은 /refresh 엔드포인트" 같은 결론만 받습니다.

Claude Code의 task 도구가 정확히 이렇게 동작합니다. 서브에이전트마다 별도 컨텍스트 창, 별도 시스템 프롬프트, 별도 도구 권한을 갖습니다. 여러 개를 동시에 띄울 수도 있어서(한 번에 최대 10개까지 병렬), 서로 무관한 조사를 병렬로 돌리면 속도도 크게 빨라집니다. deepagents도 create_deep_agent(..., subagents=[...])로 커스텀 서브에이전트를 정의할 수 있고, 기본으로 범용 서브에이전트를 부르는 task 도구를 제공합니다.

병렬은 빠르지만 싸지는 않다. 서브에이전트를 여러 개 띄우면 속도는 빨라져도 비용은 줄지 않습니다. 창마다 토큰을 따로 소모하기 때문입니다. 그러니 서브에이전트는 "컨텍스트를 격리해야 하거나 병렬이 정말 이득일 때"만 쓰세요. 둘 다 아니라면, 메인에서 순차로 도구를 부르는 편이 더 싸고 디버깅도 쉽습니다.

7. 기둥 ④ 파일시스템 — 컨텍스트 오프로드

마지막 기둥은 파일시스템입니다. Deep Agent에는 대개 네 개의 파일 도구가 붙습니다 — ls, read_file, write_file, edit_file. 역할은 하나입니다. 지금 당장 안 보는 정보를 컨텍스트 밖으로 내려 두는 것입니다.

비유하자면 이렇습니다. 컨텍스트 창이 책상 위라면, 파일시스템은 서랍입니다. 검색 결과 원문 20개를 책상에 늘어놓으면 정작 일할 공간이 없습니다. 대신 write_file("sources.md", 원문)으로 서랍에 넣고, 책상엔 "자료는 sources.md에 있음"이라는 쪽지 한 장만 남깁니다. 나중에 특정 대목이 필요하면 그때 read_file로 꺼내 봅니다.

여기서 junior가 자주 헷갈리는 지점 하나. deepagents의 기본 파일시스템은 진짜 디스크가 아닙니다. LangGraph의 상태(State) 객체 안에 파일을 흉내 낸 가상 파일시스템입니다. 그래서 격리가 안전하고, 필요하면 진짜 파일시스템이나 저장소로 갈아 끼울 수도 있습니다. 핵심은 "실제 디스크냐"가 아니라, 컨텍스트 창과 별개인 저장 공간이 있다는 것 자체입니다.

이 파일시스템은 서브에이전트와 특히 잘 어울립니다. 여러 서브에이전트가 같은 가상 파일시스템을 공유하면, 그 파일이 서로의 협업 게시판이 됩니다. A가 findings_auth.md를 쓰고, B가 그걸 읽어 이어서 작업하는 식입니다 — 서로의 지저분한 대화 원문을 주고받지 않고도요.

8. 실전 — deepagents로 리서치 에이전트 만들기

이제 네 기둥을 한꺼번에 써 봅니다. deepagents는 이 네 가지(계획 도구·서브에이전트·파일시스템·상세 프롬프트)를 기본 탑재한 라이브러리라, 우리가 직접 배선할 필요가 없습니다. 우리는 "도메인 도구"(여기선 웹 검색)와 "지시"만 얹으면 됩니다.

{{< highlight bash >}} # Python 3.11 이상 필요 pip install deepagents {{< /highlight >}}

가장 작은 예제입니다. 웹 검색 도구 하나를 만들어 넘기고, 리서치 절차를 시스템 프롬프트로 줍니다.

{{< highlight python >}} import os from deepagents import create_deep_agent from tavily import TavilyClient # 예시용 웹 검색 (아무 검색 도구나 무방) tavily = TavilyClient(api_key=os.environ["TAVILY_API_KEY"]) def internet_search(query: str) -> str: """웹을 검색해 상위 결과의 제목·요약·URL을 돌려준다.""" res = tavily.search(query, max_results=5) # 원문 전체가 아니라 요약만 반환 — 그래도 긴 건 에이전트가 파일에 저장한다 return "\n\n".join(f"{r['title']}\n{r['content']}\n{r['url']}" for r in res["results"]) # 리서치 절차를 '언제 무엇을 하라'까지 구체적으로 못 박는다 (기둥 ①) research_prompt = """너는 꼼꼼한 리서치 에이전트다. - 시작하면 먼저 write_todos로 조사 단계를 체크리스트로 쪼개라. - 자료를 많이 읽어야 하는 하위 주제는 task 도구로 서브에이전트에게 위임하라. - 검색 원문처럼 긴 내용은 write_file로 파일에 저장하고, 대화에는 요약만 남겨라. - 마지막에 final_report.md 파일로 근거 링크를 포함한 보고서를 작성하라.""" # 네 기둥(계획·서브에이전트·파일시스템·프롬프트)은 create_deep_agent이 기본 제공한다 agent = create_deep_agent( model="anthropic:claude-sonnet-5", # 아무 모델이나 — 모델에 종속되지 않는다 tools=[internet_search], # 우리가 얹는 건 도메인 도구뿐 system_prompt=research_prompt, ) result = agent.invoke({"messages": [ {"role": "user", "content": "2026년 벡터 DB의 하이브리드 검색 동향을 조사해 보고서로 정리해 줘"} ]}) # 에이전트가 만든 가상 파일(final_report.md 등)은 결과 상태에서 꺼내 볼 수 있다 print(result["messages"][-1].content) {{< /highlight >}}

이 몇 줄이 내부에서 하는 일을 풀어 보면 이렇습니다. 요청을 받은 메인 에이전트는 먼저 write_todos로 "① 하위 주제 나누기 → ② 각 주제 조사 → ③ 종합 → ④ 보고서 작성" 같은 계획을 세웁니다. 조사 단계에선 주제별로 task 서브에이전트를 띄워 internet_search를 돌리게 하고, 각 서브에이전트는 긴 검색 원문을 write_file로 저장한 뒤 요약만 메인에 돌려줍니다. 메인은 그 요약들을 모아 final_report.md를 씁니다. 우리가 배선한 건 아무것도 없습니다 — 네 기둥이 라이브러리 안에 이미 서 있으니까요.

모델에 안 묶인다. deepagents는 LangGraph 위에 있어서 model= 한 줄만 바꾸면 Claude·GPT·오픈 모델 사이를 오갈 수 있습니다. "Claude Code 같은 걸 특정 모델에 안 묶여서 쓰고 싶다"는 요구가 이 라이브러리가 뜬 이유 중 하나입니다.

9. 흔한 실수 · 팁

1) 안 그래도 되는 일에 Deep Agent를 쓴다. 네 기둥은 공짜가 아닙니다. 프롬프트가 길어지고, 서브에이전트는 토큰을 더 쓰고, 파일 왕복은 턴을 늘립니다. "파일 하나에서 함수 찾기"처럼 짧고 곧은 작업엔 얕은 루프가 더 빠르고 쌉니다. Deep Agent는 단계가 길고, 곁가지 조사가 많고, 상태를 오래 들고 가야 하는 작업을 위한 도구입니다.
2) 서브에이전트를 "그냥 병렬이니까" 남발한다. 병렬은 빠르지만 비용은 그대로입니다(6장). 게다가 서브에이전트가 많아질수록 무엇을 위임했고 무엇이 돌아왔는지 추적이 어려워집니다. 위임 기준을 프롬프트에 명확히 적고("파일을 3개 이상 읽어야 하면 위임하라"), 서브에이전트에는 무엇을 조사하고 어떤 형식으로 보고할지를 구체적으로 지시하세요.
3) 계획을 세우게만 하고 갱신은 안 시킨다. write_todos를 첫 턴에만 쓰고 방치하면 체크리스트가 현실과 어긋납니다. 프롬프트에 "한 항목을 끝내면 즉시 completed로 바꾸고, 새 하위 작업이 생기면 목록에 추가하라"를 넣어, 계획이 살아 있는 문서가 되게 하세요.
4) 파일시스템을 진짜 디스크로 착각한다. deepagents의 기본 파일시스템은 상태 안의 가상 저장소입니다(7장). 프로세스가 끝나면 사라집니다. 결과를 정말로 보존하려면, 마지막에 실제 디스크나 저장소로 내보내는 단계를 따로 두거나, 실 파일시스템 백엔드로 갈아 끼우세요.
5) 서브에이전트에 도구·권한을 통째로 넘긴다. 서브에이전트의 장점 하나는 권한을 좁힐 수 있다는 점입니다. 조사만 하는 서브에이전트에 파일 쓰기·코드 실행까지 다 주면, 격리의 안전성이 무너집니다. "이 서브에이전트는 읽기·검색만" 하는 식으로 최소 권한을 주세요.
6) 이미 쓰고 있을지도 모른다. Claude Code를 써 봤다면 이 네 기둥을 이미 경험한 겁니다 — TodoWrite로 계획을 세우고, Task로 서브에이전트를 띄우고, 파일을 읽고 씁니다. Deep Agent는 새로운 마법이 아니라, 잘 되는 에이전트가 이미 하고 있던 것에 이름을 붙이고 라이브러리로 묶은 것입니다.

10. 마무리 (TL;DR)

  • 얕은 에이전트(단순 while 루프)는 짧은 작업엔 좋지만, 긴 작업에선 도구 결과가 컨텍스트에 쌓여 창이 터지고, 목표를 잊고, 정확도가 떨어진다.
  • Deep Agent은 마법이 아니라 네 기둥의 조합이다 — ① 상세한 시스템 프롬프트, ② 계획 도구, ③ 서브에이전트, ④ 파일시스템.
  • 계획 도구(write_todos/TodoWrite)는 목표와 진행 상황을 매 턴 컨텍스트 앞에 띄워, 스텝을 건너뛰는 실패를 줄인다.
  • 서브에이전트는 지저분한 조사를 딴 컨텍스트 창에서 처리하고 요약만 돌려준다(컨텍스트 격리). 병렬이면 빠르지만 토큰은 그대로이니 필요할 때만 쓴다.
  • 파일시스템은 지금 안 보는 긴 결과를 창 밖 서랍에 내려 둔다(컨텍스트 오프로드). deepagents 기본은 가상 파일시스템이라 프로세스가 끝나면 사라진다.
  • 직접 배선할 필요는 없다. deepagents의 create_deep_agent가 네 기둥을 기본 탑재하고, 모델에도 묶이지 않는다. Claude Code가 이미 이 패턴으로 긴 작업을 버틴다.
  • 단, 짧은 작업엔 얕은 루프가 더 싸고 빠르다. Deep Agent는 길고 곁가지 많은 작업을 위한 도구다.
