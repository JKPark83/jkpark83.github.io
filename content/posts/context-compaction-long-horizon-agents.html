에이전트에게 한 시간짜리 작업을 맡기면, 매 턴마다 대화 기록이 쌓입니다. 도구 호출, 긴 파일 내용, 에러 로그… 결국 컨텍스트 창은 꽉 찹니다. 그때 앞부분을 버리면 무엇을 하던 중이었는지 잊고, 다 들고 있으면 창이 넘칩니다.

— compaction은 이 딜레마를 푸는 방법입니다. 오래된 대화를 통째로 버리는 대신, "과제·결정·다음 할 일"을 담은 짧은 요약으로 갈아 끼워 작업을 이어 가게 하죠.

짧은 질문 하나에 답하는 챗봇이라면 컨텍스트 창(context window, 모델이 한 번에 볼 수 있는 토큰의 양) 걱정은 없습니다. 그런데 요즘 우리가 만드는 건 긴 작업을 스스로 굴리는 에이전트입니다. 코드를 고치고, 도구를 수십 번 부르고, 문서를 읽어 오고… 이렇게 여러 턴을 도는 long-horizon(장기) 작업에서는 대화 기록이 계속 불어납니다. 200K, 심지어 1M 토큰짜리 창이라도 몇 시간짜리 작업 앞에서는 결국 바닥이 납니다.

창이 꽉 차기 직전, 에이전트는 갈림길에 섭니다. 오래된 메시지를 그냥 잘라 버리면(truncation) 방금 정한 규칙이나 파일 경로를 잊어 엉뚱한 짓을 합니다. 반대로 전부 끌고 가려 하면 창이 넘쳐 요청 자체가 실패하고, 넘치기 전에도 context rot(맥락이 길수록 모델 성능이 떨어지는 현상)로 판단이 흐려집니다. 둘 다 답이 아닙니다.

compaction(컴팩션)은 이 딜레마의 실전 해법입니다. 창이 한계에 다다르면 오래된 대화 앞부분을 하나의 요약 블록으로 압축해 갈아 끼우는 동작이죠. 원문 800개 메시지 대신 "지금까지 한 일, 내린 결정, 다음 할 일"을 담은 짧은 요약을 두면, 창에 다시 여유가 생겨 작업을 이어 갈 수 있습니다. Claude Code나 Codex CLI 같은 코딩 에이전트가 "몇 시간을 돌아도 안 죽는" 비결이 바로 이것입니다.

이번 주 실무자 타임라인에서 compaction이 다시 화제가 된 데는 이유가 있습니다. 2026년 들어 Anthropic이 서버 사이드 Compaction API를 정식 기능으로 내놓고 OpenAI Codex·Microsoft Agent Framework도 각자의 compaction을 붙이면서, "이제 요약 루프를 직접 짜지 않아도 된다"는 이야기가 퍼졌습니다. 동시에 "compaction은 임계값(threshold)이 아니라 결정(decision)"이라는 관점과, "compaction이 안전 제약을 소리 없이 지워 버린다"는 경고(Governance Decay)가 함께 오르내렸죠. 잘 쓰면 긴 작업을 살리지만, 잘못 쓰면 에이전트가 조용히 기억을 잃습니다.

이 글은 junior 개발자가 compaction을 원리부터 이해하고 실수 없이 적용할 수 있게 정리합니다. 무엇을 어떻게 압축하는지 그림으로 풀고, 좋은 요약과 나쁜 요약의 차이를 짚은 뒤, 프로바이더 API를 코드로 비교하고, 직접 만드는 간단한 compaction 루프와 자주 밟는 함정까지 다룹니다.

1. 긴 작업은 왜 컨텍스트가 터지는가

먼저 문제를 정확히 봅시다. 에이전트는 매 턴 대화 기록 전체를 모델에 다시 보냅니다. 모델은 상태를 기억하지 못하니까요. 1턴에 시스템 프롬프트+질문을 보내고, 2턴에는 거기에 1턴의 답과 도구 결과를 더해서 보내고, 3턴에는 또 그 위에 쌓아서 보냅니다. 대화가 길어질수록 매 요청의 입력이 눈덩이처럼 커집니다.

도구를 쓰는 에이전트는 특히 심합니다. 파일 하나 읽으면 수천 토큰, 검색 한 번에 문서 몇 장, 실패한 명령의 스택 트레이스까지 — 이런 도구 결과가 대화 기록의 대부분을 차지합니다. 정작 지금 판단에 필요한 건 그중 극히 일부인데도 전부 창을 채우죠.

이게 왜 문제일까요? 세 가지가 겹칩니다.

  • 창 초과 — 기록이 창 크기를 넘으면 요청 자체가 실패합니다. 몇 시간짜리 작업이면 아무리 큰 창도 결국 넘칩니다.
  • 비용·지연 — 입력 토큰에 비례해 돈과 시간이 듭니다. 매 턴 수십만 토큰을 다시 보내면, 뒤로 갈수록 한 턴이 점점 느리고 비싸집니다.
  • context rot — 창을 넘지 않아도, 맥락이 길수록 모델은 중간에 묻힌 정보를 놓칩니다(이른바 lost in the middle). 정보를 더 넣었는데 판단은 더 흐려지는 역설이죠.

"그럼 오래된 앞부분을 잘라 버리면?"이라고 생각하기 쉽습니다. 하지만 단순 truncation(잘라내기)은 위험합니다. 30턴 전에 사용자가 "프로덕션 DB는 절대 건드리지 마"라고 했는데 그 메시지가 잘려 나가면, 에이전트는 그런 말을 들은 적이 없는 것처럼 행동합니다. 오래됐다고 안 중요한 게 아니거든요. 필요한 건 "버리기"가 아니라 "압축하기"입니다.

2. compaction이 하는 일 — 요약해서 이어붓기

compaction의 아이디어는 한 문장입니다. 창이 꽉 차기 직전, 오래된 대화 앞부분을 읽어 하나의 요약 블록으로 만들고, 그 블록으로 원문을 갈아 끼운다. 그러면 800개 메시지가 몇백 토큰짜리 요약 하나로 줄고, 창에 다시 일할 공간이 생깁니다. 최근 몇 개의 메시지는 원문 그대로 남겨 두어 "방금 무슨 일이 있었나"의 디테일은 지키고요.

비슷해 보이는 세 용어를 먼저 구분합시다. 실무에서 자주 섞여 쓰이지만 다릅니다.

  • truncation(잘라내기) — 오래된 메시지를 그냥 버립니다. 싸지만 정보가 사라집니다. 최후의 수단.
  • summarization(요약) — 오래된 메시지를 요약으로 바꿉니다. compaction의 핵심 재료.
  • compaction(컴팩션) — 요약을 대화 흐름 안에 자연스럽게 갈아 끼워 에이전트가 끊김 없이 이어 가게 만드는 것. 요약 + 최근 원문 보존 + 도구 결과 정리를 묶은 운영 개념입니다.
메모리와 헷갈리지 마세요. compaction은 "지금 이 세션의 창을 어떻게 비우느냐"의 문제입니다. 반면 에이전트 메모리(agent memory)는 세션이 끝나도 남는 외부 저장소(벡터 DB 등)에서 필요할 때 꺼내 오는 것이죠. compaction은 세션 안, 메모리는 세션 밖이라고 기억하면 편합니다. 실전에서는 둘을 함께 씁니다 — compaction으로 창을 비우면서, 잘려 나갈 중요한 사실은 메모리에도 적어 두는 식으로요.

3. 좋은 요약 vs 나쁜 요약 — anchored iterative summarization

compaction의 성패는 결국 요약의 품질에서 갈립니다. 요약이 파일 경로 하나를 빠뜨리면, 에이전트는 그 파일을 처음부터 다시 찾아 헤맵니다. "프로덕션 DB 금지" 한 줄을 놓치면 사고로 이어지고요. 그래서 무엇을 요약에 넣느냐가 전부입니다.

여기서 두 가지 방식이 갈립니다.

Factory가 실제 엔지니어링 세션으로 두 방식을 평가한 결과, 이어붙이는(anchored iterative) 방식이 처음부터 다시 쓰는 방식보다 일관되게 나았습니다. 특히 파일 경로·에러 메시지 같은 기술적 디테일을 여러 번의 compaction을 거치며 얼마나 지켜 내는지에서 격차가 컸죠(Factory 자체 평가에서 정확도 4.04로, 다른 서버 사이드 방식들 3.74·3.43보다 높게 나옴). 왜 그럴까요? 처음부터 다시 요약하면 매번 조금씩 다른 걸 빠뜨립니다. 반면 이미 요약된 부분을 앵커로 고정하면, 한 번 담긴 디테일은 계속 살아남습니다.

한 가지 더. 좋은 요약은 구조화(structured)돼 있습니다. "이번 세션 목표", "수정한 파일", "내린 결정", "제약", "다음 할 일" 같은 고정된 섹션을 두면, 요약이 특정 정보를 소리 없이 빠뜨릴 수 없습니다. 각 섹션이 체크리스트처럼 작동하니까요. 자유 서술로 "알아서 요약해"라고 맡기는 것보다 훨씬 안정적입니다.

실전 규칙. 요약 프롬프트에 "다음 섹션을 반드시 채워라: (1) 현재 과제 (2) 확정된 결정과 제약 (3) 수정/생성한 파일 경로 (4) 미해결 문제 (5) 다음에 할 일"처럼 빈칸 양식을 주세요. 자유 요약보다 디테일 보존이 눈에 띄게 좋아집니다.

4. 프로바이더 compaction API 비교 (코드 포함)

예전에는 이 요약 루프를 직접 짰습니다. 토큰을 세고, 임계값을 넘으면 요약 요청을 따로 보내고, 메시지 배열을 손으로 갈아 끼우고… 2026년에는 프로바이더가 이걸 서버 사이드 기능으로 제공하기 시작했습니다. 대표 셋을 봅시다.

Anthropic — 명시적 compact_20260112

Anthropic은 Messages API에 context_management.edits로 compaction 전략을 얹는 방식입니다. 요청에 베타 헤더 compact-2026-01-12를 붙이고, trigger로 "입력이 몇 토큰을 넘으면 compact할지"를 정하면 됩니다.

{{< highlight python >}} # pip install anthropic from anthropic import Anthropic client = Anthropic() resp = client.beta.messages.create( betas=["compact-2026-01-12"], # compaction 베타 활성화 model="claude-sonnet-5", max_tokens=1024, context_management={ "edits": [ { "type": "compact_20260112", # 입력이 150K 토큰을 넘으면 자동 요약 (최소 50K, 기본 150K) "trigger": {"type": "input_tokens", "value": 150000}, # (선택) 요약 지침을 직접 지정 — 없으면 기본 프롬프트 사용 "instructions": ( "다음 섹션을 반드시 채워라: 현재 과제 / 확정된 결정·제약 / " "수정한 파일 경로 / 미해결 문제 / 다음 할 일." ), } ] }, messages=conversation, # 그동안 쌓인 긴 대화 기록 ) # 응답에 compaction 블록이 있으면, 다음 요청 때 그대로 되돌려줘야 한다. # API가 그 블록 '앞'의 원문 메시지는 알아서 떨궈 준다. conversation.append({"role": "assistant", "content": resp.content}) {{< /highlight >}}

핵심 동작 두 가지를 기억하세요. 첫째, compaction이 일어나면 응답에 type: "compaction" 블록이 들어옵니다. 이 블록에는 "지금까지의 상태·결정·다음 할 일"이 담겨 있고, 다음 요청 때 이 블록을 그대로 되돌려줘야 API가 그 앞의 원문 메시지들을 자동으로 떨궈 줍니다. 둘째, compaction은 공짜가 아닙니다. 요약을 만들려면 모델이 긴 원문을 한 번 읽어야 하니, 그만큼의 입력 토큰이 별도로 청구됩니다. 응답의 usage.iterations에 compaction 단계와 실제 답변 단계가 따로 찍혀 나오니 확인하세요.

OpenAI Codex — 불투명한 서버 사이드 compaction

OpenAI 쪽은 Codex가 자체 엔드포인트(POST /v1/responses/compact)로 대화를 암호화된 불투명(opaque) 표현으로 압축합니다. 요약 텍스트를 우리가 직접 보고 만지는 대신, 서버가 압축본을 돌려주고 그걸 다음 턴에 넘기는 식이죠. 손댈 손잡이가 적은 대신 통합이 단순합니다.

Claude Code / Codex CLI — 코딩 에이전트의 자동 compaction

우리가 매일 쓰는 코딩 에이전트에도 compaction이 내장돼 있습니다. Claude Code는 창의 약 98%가 차면 auto-compact이 돌아 오래된 대화를 요약으로 갈아 끼웁니다(이미지·PDF 블록은 빼고, 큰 도구 출력은 잘라 냅니다). /compact 명령으로 수동 실행도 가능하고요. 흥미로운 건 microcompact인데, 모델 호출 없이 오래돼 쓸모없어진 도구 결과만 조용히 비웁니다 — 가장 값싼 개입이죠. 요약이 필요 없는 청소는 요약 없이 처리하는 겁니다.

5. 한 발 더 — 임계값이 아니라 '결정'

지금까지 본 방식은 대부분 임계값(threshold) 기반입니다. "입력이 150K를 넘으면 compact." 단순하고 예측 가능하지만, 한 가지 약점이 있습니다. 타이밍이 내용을 고려하지 않는다는 점입니다.

생각해 보세요. 임계값에 딱 걸린 순간이 하필 복잡한 추론의 한복판이면 어떻게 될까요? 방정식을 반쯤 풀던 중, 또는 여러 파일을 엮어 디버깅하던 중에 compaction이 끼어들어 맥락을 요약으로 바꿔 버리면, 에이전트는 방금 쌓던 논리를 잃고 헤맵니다. 반대로 하위 과제 하나를 막 끝낸 깔끔한 경계라면, 그때 compact하는 게 손실이 가장 적죠.

그래서 최근 담론은 "compaction은 임계값이 아니라 결정이다"로 옮겨 가고 있습니다. 토큰 수만 보고 기계적으로 자르지 말고, 지금이 요약하기 좋은 순간인지를 판단하자는 겁니다. 한 발 더 나아가 모델이 스스로 "지금 하위 과제를 끝냈으니 compact하겠다 / 지금은 추론 중이니 참겠다"를 결정하게 하는 self-compaction 연구도 나왔습니다.

또 하나 주목할 갈래는 ACON(Optimizing Context Compression for Long-horizon LLM Agents)입니다. 요약 지침 자체를 자동으로 다듬는 접근이죠. 방법이 재밌습니다 — 원문을 다 주면 성공하는데 압축본을 주면 실패한 사례들을 모아, "왜 실패했지? 요약이 무엇을 빠뜨렸지?"를 LLM이 분석하게 하고, 그 교훈을 요약 지침에 반영합니다. 파인튜닝 없이 자연어 지침만 고쳐서요. ACON은 이렇게 최적화한 압축으로 최대 메모리(peak) 토큰을 약 26~54% 줄이면서 성능은 대체로 지켰고, 그 압축기를 작은 모델로 증류(distill)해도 원본 성능의 95% 이상을 유지했다고 보고합니다.

정리하면 스펙트럼입니다. 가장 단순한 건 고정 임계값, 그다음은 "경계에서만 자르는" 결정 기반, 그 위는 "모델이 언제·무엇을 자를지 스스로 정하는" self-compaction, 그리고 "요약 지침을 실패로부터 학습하는" ACON. 대부분의 서비스는 임계값으로 충분하지만, 긴 추론이 중요한 작업일수록 결정 기반이 이득을 봅니다.

6. 실전: 직접 만드는 compaction 루프

원리를 손에 익히려면 직접 짜 보는 게 최고입니다. 프로바이더 API 없이도 compaction의 뼈대는 간단합니다. 아래는 "임계값을 넘으면 오래된 앞부분을 구조화 요약으로 갈아 끼우는" 최소 루프입니다.

{{< highlight python >}} KEEP_RECENT = 6 # 최근 메시지 6개는 원문 그대로 보존 TOKEN_LIMIT = 150_000 # 이 선을 넘으면 compaction 발동 SUMMARY_PROMPT = """다음 대화를 이어서 작업할 수 있도록 요약하라. 반드시 아래 섹션을 채워라(빈칸 금지): 1) 현재 과제 2) 확정된 결정·제약 3) 수정/생성한 파일 경로 4) 미해결 문제 5) 다음에 할 일 파일 경로·에러 메시지·사용자가 건 금지사항은 절대 빠뜨리지 마라.""" def maybe_compact(messages, prior_summary=None): if count_tokens(messages) < TOKEN_LIMIT: return messages, prior_summary # 아직 여유 → 그대로 system, recent = messages[0], messages[-KEEP_RECENT:] old = messages[1:-KEEP_RECENT] # 요약 대상(오래된 앞부분) # anchored: 이전 요약을 고정하고, 새로 밀려난 old만 요약해 병합 to_summarize = old if prior_summary is None else \ [{"role": "system", "content": f"[이전 요약]\n{prior_summary}"}] + old new_summary = llm(SUMMARY_PROMPT, to_summarize) # 요약 생성(모델 호출 1회) # 원문 앞부분을 요약 블록 하나로 갈아 끼운다 compacted = [system, {"role": "system", "content": f"[진행 요약]\n{new_summary}"}, *recent] return compacted, new_summary {{< /highlight >}}

포인트 셋입니다. 첫째, KEEP_RECENT로 최근 메시지는 원문 보존합니다 — 방금의 디테일까지 요약으로 뭉개면 안 되니까요. 둘째, prior_summary를 넘겨 anchored 방식으로 이어붙입니다. 셋째, 요약 프롬프트가 빈칸 양식이라 파일 경로·금지사항을 강제로 남깁니다.

주의할 함정 하나. 이 new_summary를 어딘가 안전한 곳에도 적어 두세요. 요약이 또 요약되는 과정에서 초기의 중요한 제약이 서서히 옅어질 수 있거든요. 사용자의 "프로덕션 DB 금지" 같은 절대 규칙은 요약에 맡기지 말고 시스템 프롬프트에 고정해 두는 게 안전합니다. 그래야 compaction을 몇 번 거쳐도 사라지지 않습니다.

7. 흔한 실수 · 팁

  • 절대 규칙을 요약에 맡긴다. compaction이 반복되면 초기 제약이 조용히 사라질 수 있습니다(Governance Decay가 경고하는 바로 그 위험). 안전·권한·금지사항은 요약 대상이 아니라 시스템 프롬프트에 고정하세요.
  • 프롬프트 캐싱을 통째로 깬다. compaction은 대화 앞부분을 통째로 바꾸는 동작이라, 그 뒤의 프롬프트 캐시(prompt cache)가 전부 무효화됩니다. 시스템 프롬프트는 cache_control로 따로 캐시해 두면, compaction이 일어나도 시스템 부분 캐시는 살릴 수 있습니다.
  • 추론 한복판에 compact한다. 임계값에만 의존하면 하필 복잡한 추론 중에 끼어들어 맥락이 끊깁니다. 가능하면 하위 과제가 끝난 경계에서 compact하도록 유도하세요(§5).
  • 도구 결과까지 요약으로 뭉갠다. 오래돼 쓸모없는 큰 도구 출력은 요약할 필요 없이 그냥 비우면(microcompact) 됩니다. 모든 걸 요약하려 들지 마세요 — 청소로 될 일에 모델 호출을 낭비하지 마세요.
  • compaction 비용을 계산에서 뺀다. 요약을 만들 때 긴 원문을 한 번 읽으므로 추가 입력 토큰이 듭니다. compaction이 너무 잦으면(임계값이 너무 낮으면) 오히려 비용·지연이 늘 수 있습니다. usage.iterations로 실제 비용을 확인하세요.
  • 요약 품질을 평가하지 않는다. "요약이 잘 되고 있겠지"는 위험한 가정입니다. compaction 전/후로 에이전트가 파일 경로·결정을 여전히 기억하는지 작은 평가 세트로 점검하세요(자유 요약보다 구조화 요약이 이 테스트에 강합니다).

8. 마무리 (TL;DR)

  • 문제 — 긴 작업을 도는 에이전트는 대화가 쌓여 컨텍스트 창이 결국 꽉 찹니다. 앞부분을 그냥 버리면 기억을 잃고, 다 들고 있으면 창이 넘치고 context rot로 판단이 흐려집니다.
  • compaction — 창이 한계에 다다르면 오래된 앞부분을 "과제·결정·제약·다음 할 일" 요약 블록으로 갈아 끼우고, 최근 메시지는 원문으로 남겨 작업을 이어 갑니다. 세션 안의 창 관리이며, 세션 밖의 메모리와는 별개입니다.
  • 좋은 요약 — 매번 처음부터 다시 쓰지 말고(full reconstruction) 이전 요약을 앵커로 고정해 새로 밀려난 부분만 이어붙이세요(anchored iterative). 고정 섹션이 있는 구조화 요약이 디테일 보존에 강합니다.
  • API — Anthropic compact_20260112(명시적·요약 노출), OpenAI Codex(불투명·단순), Claude Code auto-compact·microcompact(자동). 요약이 필요 없는 청소는 microcompact로 값싸게.
  • 한 발 더 — "임계값이 아니라 결정" — 추론 한복판이 아니라 과제 경계에서 compact하는 게 이득. self-compaction·ACON은 언제·무엇을 자를지, 요약 지침 자체를 최적화합니다.
  • 함정 — 절대 규칙은 요약에 맡기지 말고 시스템 프롬프트에 고정, 시스템 프롬프트는 따로 캐시해 캐시 붕괴 방지, compaction 비용도 계산에 포함, 요약 품질은 작은 평가로 점검.

compaction은 화려하지 않지만, "몇 시간을 돌아도 안 죽는 에이전트"의 밑바닥을 떠받치는 기술입니다. 창을 키우는 경쟁이 아니라 창을 잘 관리하는 쪽이 긴 작업을 이깁니다. 오늘 만드는 에이전트가 자꾸 "앞에서 한 말을 까먹는다"면, 메모리를 붙이기 전에 compaction부터 점검해 보세요.
