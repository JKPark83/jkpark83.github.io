고객마다 말투와 규칙이 다르다고 모델을 하나씩 파인튜닝했더니, 어느새 "이 모델 다 어디서 돌리지?"가 진짜 문제가 됩니다. 7B 모델 하나가 GPU 메모리를 14GB 넘게 먹는데, 고객이 100명이면 GPU가 100대라는 계산이 나오니까요.

— 멀티-LoRA 서빙은 "모델을 더 싸게 돌리자"가 아니라, "고객마다 다른 건 사실 몇 MB뿐이니 그것만 갈아끼우자"는 발상의 전환입니다.

파인튜닝(fine-tuning)을 조금이라도 해 봤다면 LoRA라는 단어는 익숙할 겁니다. 거대한 모델 전체를 다시 학습시키는 대신, 작은 어댑터(adapter)만 얹어서 값싸게 특화시키는 방법이죠. 그런데 막상 이렇게 만든 특화 모델을 여러 개 동시에 서비스하려는 순간, 많은 사람이 똑같은 벽에 부딪힙니다 — "이걸 다 무슨 수로 돌리지?"

이번 주(2026년 7월) arXiv에는 이 문제를 정면으로 다룬 서빙 논문들이 잇달아 올라왔습니다. base 모델 추론과 LoRA 계산을 아예 분리해 버리는 InfiniLoRA, 멀티-LoRA 에이전트를 위한 copy-on-write KV 캐시를 제안한 ForkKV 같은 것들이죠. 새로 나오는 base 모델이 대부분 MoE(mixture-of-experts) 구조라 어댑터 메모리 비용이 커지면서, "수많은 어댑터를 어떻게 싸게 서빙하느냐"가 다시 실무의 뜨거운 화두가 됐습니다. 그 뿌리에 있는 개념이 바로 멀티-LoRA 서빙(multi-LoRA serving)입니다.

이 글은 왜 순진하게 어댑터를 합치면 안 되는지, 여러 어댑터를 한 배치(batch)에 섞어 돌리는 트릭이 무엇인지, 그리고 vLLM·SGLang으로 실제로 어떻게 서빙하고 어디서 발이 걸리는지까지 junior 개발자 눈높이로 하나씩 풀어 보겠습니다.

1. 문제 — 특화 모델이 늘수록 GPU가 폭발한다

상황을 그려 봅시다. SaaS를 운영하는데 고객사마다 요구가 조금씩 다릅니다. A사는 법률 문서 톤, B사는 캐주얼한 상담 톤, C사는 자사 제품 용어를 알아야 하고요. 각 고객의 데이터로 base 모델을 통째로 파인튜닝(full fine-tuning)하면, 고객마다 7B 파라미터짜리 모델이 하나씩 생깁니다.

문제는 서빙입니다. 7B 모델은 반정밀도(FP16)로도 파라미터에만 약 14GB가 듭니다. 이걸 GPU에 올려 두고 요청을 받아야 하는데, 고객이 100명이면 서로 다른 100개의 모델을 동시에 대기시켜야 합니다. GPU 한 장에 다 못 올리니 결국 고객 수에 비례해 GPU를 늘려야 하죠. 대부분의 고객은 트래픽이 적어 GPU가 놀고 있는데도 말입니다. 돈이 이렇게 샙니다.

여기서 핵심 관찰이 하나 있습니다. 고객마다 파인튜닝했다고 해도, 결과 모델 100개는 거의 다 똑같습니다. 원래 base 모델에서 아주 조금씩만 달라졌을 뿐이죠. "다른 부분"만 따로 떼어내 작게 저장할 수 있다면, 무거운 공통 부분(base)은 딱 한 번만 올려 두고 공유하면 됩니다. 그 "다른 부분"을 값싸게 만들어 주는 도구가 바로 LoRA입니다.

2. 배경 — LoRA 30초 복습

LoRA(Low-Rank Adaptation)는 모델의 가중치 행렬 W를 직접 바꾸지 않습니다. 대신 그 옆에 작은 곁가지를 하나 답니다. 원래 출력이 W·x였다면, LoRA는 여기에 (B·A)·x를 더합니다. 이때 A와 B는 아주 납작한(low-rank) 행렬입니다.

예를 들어 W가 4096×4096(약 1,600만 개 원소)이라면, LoRA는 A를 4096×r, B를 r×4096으로 둡니다. 여기서 r(rank)은 보통 8, 16, 32처럼 작은 수입니다. r=16이면 두 행렬을 합쳐 봐야 원본의 0.4%밖에 안 됩니다. 학습할 때는 무거운 W는 얼려 두고(freeze) 이 작은 A, B만 업데이트합니다. 그래서 싸고 빠릅니다.

서빙 관점에서 중요한 결론은 이겁니다. base 모델은 모두가 공유하는 14GB짜리 무거운 공통 자산이고, 어댑터는 고객마다 다른 몇 MB짜리 작은 파일이라는 것. 그렇다면 base는 GPU에 딱 한 번만 올리고, 어댑터는 SSD나 CPU 메모리에 잔뜩 쌓아 뒀다가 요청이 올 때마다 필요한 것만 꺼내 쓰면 되지 않을까요? 바로 그 아이디어가 멀티-LoRA 서빙입니다.

3. 왜 순진하게는 안 되나 — 어댑터를 합치면 배치가 깨진다

가장 먼저 떠오르는 방법은 이겁니다. "요청이 오면 그 어댑터를 base에 합쳐서(merge) 완성된 모델을 만들고, 그걸로 답한 뒤 되돌린다." W' = W + B·A를 미리 계산해 두면 추론할 때 곁가지가 사라져 속도 손해도 없습니다. 실제로 어댑터를 하나만 쓸 거라면 이게 정답입니다.

그런데 어댑터가 여러 개가 되는 순간 이 방법이 무너집니다. 원인은 LLM 서빙 성능의 핵심인 배치(batching)에 있습니다. GPU는 요청 하나를 처리하든 서른 개를 한꺼번에 처리하든 걸리는 시간이 크게 다르지 않습니다. 그래서 최신 서빙 엔진은 여러 요청을 한 배치로 묶어 GPU에 한 번에 흘려보내 처리량(throughput)을 끌어올립니다.

그런데 어댑터를 base에 합쳐 버리면, 한 배치 안의 모든 요청이 똑같은 합쳐진 가중치를 써야 합니다. 고객 A의 요청과 고객 B의 요청은 서로 다른 어댑터가 필요한데, 가중치를 하나로 굳혀 버리면 둘을 같은 배치에 넣을 수가 없죠. 결국 어댑터마다 배치를 따로 만들어야 하고, 요청이 올 때마다 GPU에서 가중치를 합쳤다 되돌렸다 하며 값비싼 시간을 허비합니다. 트래픽이 여러 고객에 골고루 흩어져 있으면, 배치가 잘게 쪼개져 처리량이 바닥을 칩니다.

여기가 핵심 갈림길입니다. "어댑터 1개 = merge해서 서빙"은 완전히 정상적인 최적화입니다. 문제는 어댑터가 많을 때입니다. 서로 다른 어댑터를 같은 배치 안에서 함께 굴리지 못하면, 멀티-LoRA의 경제성은 사라집니다.

4. 핵심 아이디어 — 안 합치고 배치로 함께 계산하기

해법은 발상을 뒤집는 것입니다. 어댑터를 base에 합치지 않고 따로 둔 채로, 한 배치 안에서 요청마다 다른 어댑터를 적용해 함께 계산합니다. 출력이 W·x + B·A·x라는 걸 떠올려 보세요. 공통 부분 W·x는 배치 전체를 한 번에 계산하면 됩니다. 문제는 뒤의 B·A·x — 요청마다 A, B가 다르다는 것뿐입니다.

여기서 Punica가 제안한 SGMV(Segmented Gather Matrix-Vector Multiplication) 커널이 등장합니다. 아이디어는 단순합니다. 배치 안의 토큰들을 같은 어댑터끼리 이어 붙여 세그먼트(segment)로 묶고, 커널 호출 한 번으로 모든 세그먼트를 병렬 처리합니다. 어댑터가 10개든 100개든, 곁가지 계산을 GPU 커널 한 방에 끝내는 거죠. Punica 논문은 이 방식으로 기존 서빙 시스템 대비 최대 12배 높은 처리량을 보고했습니다.

여기에 S-LoRA가 메모리 관리를 얹습니다. 핵심은 Unified Paging(통합 페이징)입니다. rank가 제각각인 어댑터 가중치와, 길이가 제각각인 KV 캐시를 하나의 통합 메모리 풀에서 페이지 단위로 관리합니다. 덕분에 어댑터를 GPU와 CPU 메모리 사이에서 조각(fragmentation) 걱정 없이 자유롭게 올렸다 내렸다 할 수 있습니다. S-LoRA는 이 방식으로 GPU 한 대에서 어댑터 2,000개를 동시에 서빙했고, LoRA를 순진하게 지원하는 HuggingFace PEFT나 vLLM 대비 처리량을 최대 4배 높였다고 보고했습니다.

정리하면 멀티-LoRA 서빙 엔진의 두 기둥은 이렇습니다. (1) 이질적 배치(heterogeneous batching) — 서로 다른 어댑터를 한 배치에서 함께 계산하는 SGMV류 커널, (2) 어댑터 페이징 — base는 GPU에 고정하고 어댑터는 CPU↔GPU를 오가며 필요한 것만 상주시키는 메모리 관리. 우리가 직접 이걸 구현할 필요는 없습니다. vLLM과 SGLang이 이미 안에 넣어 뒀으니까요.

5. 실전 — vLLM으로 멀티-LoRA 서빙하기

가장 널리 쓰이는 vLLM으로 시작해 봅시다. 먼저 오프라인(파이썬 코드)에서 어떻게 쓰는지 보면 개념이 명확해집니다.

{{< highlight python >}} from vllm import LLM, SamplingParams from vllm.lora.request import LoRARequest # base 모델은 딱 한 번만 GPU에 올립니다. # enable_lora=True 로 멀티-LoRA를 켜고, 한 배치에 최대 몇 개까지 섞을지(max_loras), # 어댑터 rank 상한(max_lora_rank)을 지정합니다. llm = LLM( model="meta-llama/Llama-3.1-8B", enable_lora=True, max_loras=4, # 한 배치 안에서 동시에 쓸 어댑터 최대 개수 max_lora_rank=16, # 올릴 어댑터들의 rank 중 가장 큰 값에 맞춤 ) sampling = SamplingParams(temperature=0.7, max_tokens=256) # LoRARequest(사람이 읽는 이름, 전역 고유 정수 ID, 어댑터 경로) sql_lora = LoRARequest("sql-adapter", 1, "/adapters/sql") chat_lora = LoRARequest("chat-adapter", 2, "/adapters/chat") # 요청마다 다른 어댑터를 지정 — 엔진이 알아서 한 배치로 묶어 줍니다. out1 = llm.generate("SELECT ...", sampling, lora_request=sql_lora) out2 = llm.generate("안녕하세요, 반품하고 싶어요", sampling, lora_request=chat_lora) {{< /highlight >}}

포인트는 LoRARequest의 두 번째 인자인 전역 고유 정수 ID입니다. 엔진은 이 ID로 어댑터를 식별해 캐시하므로, 같은 어댑터에는 항상 같은 ID를 주어야 합니다. 다른 어댑터에 같은 ID를 재사용하면 엉뚱한 가중치로 답하는 조용한 버그가 납니다.

실제 서비스는 보통 OpenAI 호환 API 서버로 띄웁니다. 이때는 어댑터를 서버 시작 시 등록해 두고, 요청의 model 필드에 어댑터 이름을 넣어 고릅니다.

{{< highlight bash >}} # base 모델 하나 + 어댑터 두 개를 등록해 서버를 띄웁니다. vllm serve meta-llama/Llama-3.1-8B \ --enable-lora \ --lora-modules sql-adapter=/adapters/sql chat-adapter=/adapters/chat \ --max-loras 4 \ --max-lora-rank 16 \ --max-cpu-loras 32 # GPU엔 못 올려도 CPU 메모리에 캐시해 둘 어댑터 수 {{< /highlight >}} {{< highlight bash >}} # 요청의 "model"에 base 이름이 아니라 어댑터 이름을 넣으면 그 어댑터로 응답합니다. curl http://localhost:8000/v1/completions \ -H "Content-Type: application/json" \ -d '{"model": "sql-adapter", "prompt": "주문 테이블에서 어제 매출 합계를", "max_tokens": 128}' {{< /highlight >}}

여기서 두 개의 max 옵션이 성능을 가릅니다. --max-loras는 한 배치 안에서 동시에 쓸 수 있는 어댑터 개수입니다. 이 값이 작으면, 서로 다른 어댑터 요청이 몰려도 한 번에 4종류밖에 못 섞어 배치가 쪼개집니다. --max-cpu-loras는 CPU 메모리에 미리 캐시해 둘 어댑터 수입니다. 여기 없는 어댑터는 요청이 올 때 디스크에서 새로 읽어야 해서 첫 응답이 느려집니다(cold start).

규칙 하나만 기억하세요. 동시에 트래픽이 붙는 어댑터 수는 --max-loras로, 전체 보유하지만 가끔 쓰는 어댑터 수는 --max-cpu-loras로 커버합니다. 활성 어댑터가 --max-loras를 넘으면 엔진이 알아서 순번을 돌리지만, 그만큼 배치 효율이 떨어집니다.

6. SGLang은 어떻게 다른가

또 다른 주력 엔진 SGLang도 멀티-LoRA를 1급 기능으로 지원합니다. 내부적으로 앞서 본 S-LoRA·Punica의 기법을 가져와 한 배치에 여러 어댑터를 섞습니다. 플래그 이름만 조금 다릅니다.

개념vLLMSGLang
멀티-LoRA 켜기--enable-lora--lora-paths 주면 자동 활성
어댑터 등록--lora-modules 이름=경로--lora-paths 이름=경로
한 배치 동시 어댑터 수--max-loras--max-loras-per-batch
CPU에 캐시할 어댑터 수--max-cpu-loras--max-loaded-loras
요청에서 어댑터 선택model 필드 = 어댑터 이름lora_path 파라미터

한 가지 SGLang 특유의 함정이 있습니다. SGLang의 강력한 RadixCache(프롬프트 접두사를 공유해 재계산을 줄이는 캐시)는 LoRA와 아직 완전히 호환되지 않습니다. 어댑터마다 KV 캐시를 따로 둬야 하기 때문인데, 그래서 LoRA를 켤 때 --disable-radix-cache가 필요한 경우가 있습니다. 접두사 캐시의 이점을 일부 포기하는 셈이라, 도입 전 자신의 SGLang 버전에서 이 조합이 어떻게 동작하는지 반드시 확인하세요. --max-loaded-loras는 --max-loras-per-batch 이상이어야 한다는 제약도 있습니다.

어느 걸 고를까. 어댑터가 몇 개뿐이라면 둘 다 무난합니다. 수십 개 이상을 이질적 트래픽으로 굴린다면 배치 스케줄링 성숙도를 각자의 최신 릴리스에서 벤치마크해 보고 고르는 게 정답입니다. 엔진은 빠르게 바뀌므로, "누가 더 빠르다"는 옛날 글보다 내 워크로드로 직접 잰 숫자를 믿으세요.

7. 최신 흐름 — 분리형(disaggregated) 서빙

지금까지는 base 추론과 LoRA 계산이 같은 GPU, 같은 엔진 안에서 함께 도는 구조였습니다. 이번 주 arXiv에 올라온 InfiniLoRA는 여기서 한 걸음 더 나갑니다. LoRA 실행을 base 모델 추론에서 떼어 내(disaggregate) 별도의 "LoRA 서버"로 분리하자는 제안입니다.

왜 이런 방향이 나왔을까요? 요즘 나오는 base 모델이 대부분 MoE 구조라서입니다. MoE는 토큰마다 일부 전문가(expert)만 활성화하는데, 각 전문가에 LoRA를 붙이면 어댑터가 붙을 자리가 확 늘어 어댑터 메모리 비용이 폭증합니다. 이러면 base와 LoRA를 한 덩어리로 묶어 둔 기존 설계가 확장성 한계와 꼬리 지연(tail latency) 문제에 부딪힙니다. InfiniLoRA는 LoRA 서버를 따로 두고 GPU 주도 통신·전용 커널로 이 경로를 최적화해, 엄격한 지연 목표(SLO) 아래에서 처리 가능한 요청률을 평균 3.05배 높였다고 보고합니다.

같은 주에 나온 ForkKV는 멀티-LoRA 에이전트 시나리오에 초점을 맞춰, 공통 프롬프트에서 갈라지는 KV 캐시를 copy-on-write로 공유합니다. 방향은 달라도 메시지는 같습니다 — 어댑터가 많아지고 base가 무거워질수록, "무엇을 공유하고 무엇을 분리할까"가 서빙 설계의 핵심이 된다는 것이죠. 당장 이걸 직접 구현할 일은 없겠지만, 앞으로 서빙 엔진이 어느 방향으로 진화할지 감을 잡아 두면 좋습니다.

8. 흔한 실수 · 팁

  • rank를 제각각으로 학습했다. --max-lora-rank는 올릴 어댑터 중 가장 큰 rank에 맞춰야 합니다. 이 값이 크면 어댑터마다 잡아 두는 메모리가 늘어 손해입니다. 가능하면 팀 안에서 rank를 하나로 통일하세요(예: 전부 16).
  • 어댑터 하나만 쓰는데 멀티-LoRA로 돌린다. 어댑터가 딱 하나면 base에 merge해 서빙하는 게 더 빠릅니다. 멀티-LoRA의 곁가지 계산은 여러 어댑터를 섞을 때 값을 합니다.
  • cold start를 간과한다. --max-cpu-loras(SGLang은 --max-loaded-loras)에 없는 어댑터는 요청 시 디스크에서 읽어 올려야 해서 첫 응답이 눈에 띄게 느립니다. 자주 쓰는 어댑터는 미리 캐시 범위 안에 넣어 두세요.
  • 어댑터 ID를 재활용한다. vLLM LoRARequest의 정수 ID는 어댑터마다 전역 고유여야 합니다. 다른 어댑터에 같은 ID를 주면 캐시가 꼬여 엉뚱한 가중치로 답하는, 잡기 어려운 버그가 납니다.
  • SGLang에서 RadixCache를 켠 채 LoRA를 쓴다. 버전에 따라 충돌하거나 --disable-radix-cache가 필요합니다. 접두사 캐시 이점이 큰 워크로드라면 도입 전 반드시 확인하세요.
  • 활성 어댑터가 많은데 --max-loras가 작다. 동시에 트래픽이 붙는 어댑터 종류가 이 값을 넘으면 배치가 쪼개져 처리량이 급락합니다. 실제 동시 활성 어댑터 수를 재서 맞추세요.

9. 마무리 (TL;DR)

  • 고객·업무마다 모델을 통째로 파인튜닝하면 GPU가 고객 수에 비례해 폭발합니다. 하지만 결과 모델들은 대부분 똑같고, 다른 건 몇 MB짜리 LoRA 어댑터뿐입니다.
  • 멀티-LoRA 서빙 = 무거운 base는 GPU에 딱 한 번 올리고, 요청마다 필요한 작은 어댑터만 갈아끼워 GPU 한 대로 수십~수천 개를 처리하는 것.
  • 어댑터를 base에 merge하면 배치가 깨집니다. 그래서 합치지 않고, SGMV 같은 이질적 배치 커널로 여러 어댑터를 한 배치에서 함께 계산하고, Unified Paging으로 어댑터를 CPU↔GPU로 오가게 합니다.
  • 실전에선 vLLM(--enable-lora, --lora-modules, --max-loras)이나 SGLang(--lora-paths, --max-loras-per-batch)으로 켭니다. 요청은 model 필드(vLLM)나 lora_path(SGLang)로 어댑터를 고릅니다.
  • 함정: rank 통일, cold start 캐시, 어댑터 ID 고유성, SGLang의 RadixCache 충돌, --max-loras와 동시 활성 어댑터 수 맞추기.
  • 최신 흐름은 base 추론과 LoRA를 분리(disaggregated)하는 방향(InfiniLoRA·ForkKV). MoE base가 어댑터 메모리를 키우면서 "무엇을 공유하고 무엇을 분리할까"가 서빙의 핵심 질문이 됐습니다.
