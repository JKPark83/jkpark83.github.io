# 🔥 오늘의 핵심

오늘은 **"에이전트 보안"** 한 주제로 신호가 모였습니다. OpenClaw가 **38만 스타**로 GitHub 역사상 최고 속도로 확산하는 동안, prompt injection은 *데이터-명령 분리 같은 단일 방어로는 원리상 막을 수 없다*는 연구가 굳어지고 있습니다. 마침 Claude Code는 v2.1.183에서 **auto 모드의 파괴적 명령(git reset --hard, terraform destroy 등)을 차단**하는 가드레일을 넣었습니다. 결론은 분명합니다 — "프롬프트로 막기"가 아니라 **권한 최소화 + 정책 게이트 + 감사**를 층층이 쌓는 것.

## 1. AI Agents May Always Fall for Prompt Injections
- **arXiv:** [2605.17634](https://arxiv.org/abs/2605.17634) · cs.CR/cs.AI · 제출 2026-05-17
- **저자:** Sahar Abdelnabi (ELLIS Institute Tübingen & MPI-IS & Tübingen AI Center), Eugene Bagdasarian (UMass Amherst)
- **핵심 기여:** prompt injection을 **Contextual Integrity(맥락 무결성)** 관점으로 재정의해, 사실상 *불가능성 결과*를 제시. 공격자는 차단된 정보 흐름을 "정당해 보이는" 맥락으로 항상 재구성할 수 있고, 방어자가 규범을 조이면 정상 흐름까지 막히게 됩니다.
- **방법:** 에이전트가 규범을 위반하게 만드는 (1) 흐름 위장, (2) 규범 조작, (3) 다중 흐름 혼합 시나리오를 구성해, 기존 "데이터-명령 분리" 패러다임이 맥락 조작형 공격을 못 잡고 정상 동작까지 떨어뜨림을 보임.
- 🛠 **빌더 관점:** "완벽한 입력 필터"를 목표로 삼지 말 것. injection을 *못 막는다*고 가정하고 권한·격리로 피해 반경을 줄이는 설계로 전환하세요.
- ⚠️ **한계:** 이론·시나리오 중심 분석 → 모델별 정량 방어율은 추가 측정 필요.
- 🔗 [arXiv abs](https://arxiv.org/abs/2605.17634)

## 2. Prompt Injection Attacks on Agentic Coding Assistants (SoK)
- **arXiv:** [2601.17548](https://arxiv.org/abs/2601.17548) · cs.CR · Systematization of Knowledge
- **저자:** Narek Maloyan 외
- **핵심 기여:** **Claude Code · GitHub Copilot · Cursor** 등 코딩 어시스턴트를 대상으로, 흩어진 공격 분류를 **3차원(전달 벡터 · 양식 · 전파 행위)**으로 통합. tool poisoning, 프로토콜(MCP) 악용, 멀티모달 injection, cross-origin 컨텍스트 오염을 망라.
- **핵심 결과:** 다수 방어가 *적응형 공격자* 앞에서 급격히 무너짐. 공격 페이로드는 값싸고 폴리모픽이라 탐지-회피가 쉬워 공격/방어 비용 비대칭이 구조적이라는 점을 강조.
- 🛠 **빌더 관점:** 프롬프트 필터링 한 겹 대신 **defense-in-depth** — 정책 게이트, 최소 권한, 감사를 아키텍처로 분리하라는 직접적 처방. 오늘 본 Claude Code/Codex 가드레일과 정확히 같은 방향.
- ⚠️ **한계:** SoK(메타 분석) 성격 → 신규 단일 방어의 정량 평가라기보다 지형도.
- 🔗 [Hugging Face Paper](https://huggingface.co/papers/2601.17548)

![프롬프트 인젝션 방어 다층화 다이어그램: 최소 권한, 정책 게이트, 감사·격리의 3개 레이어](/images/ai-news-2026-06-20/defense-in-depth.svg)

*두 논문의 공통 결론을 한 장으로 — 한 겹이 뚫려도 다음 겹이 막는 방어 다층화.*

## 🟦 Claude Code — Hook/Workflow: auto 모드 가드레일 이해하고 쓰기
- **source:** [Claude Code Changelog v2.1.183](https://code.claude.com/docs/en/changelog)
- 💡 **핵심:** v2.1.183부터 auto 모드는 *명시 요청이 없으면* 파괴적 명령을 스스로 거부합니다. "에이전트가 실수로 rm/reset" 사고를 한 겹 줄여줍니다.
- 📝 **how-to:** 업데이트 후 동작 확인 — 특정 스택만 의도적으로 허용하려면 그 스택을 명시하세요.

```bash
claude --version   # 2.1.183 이상인지 확인
# auto 모드에서 destroy가 필요하면 "대상"을 명시해야 통과됨
# 예: "terraform destroy를 staging 스택에 대해 실행해줘"
```

- 🎯 **언제:** 자율 실행/긴 작업 루프를 자주 돌릴 때.
- ⚠️ **주의:** 가드레일은 *안전망*이지 정책이 아닙니다. CI에 권한 분리·승인 단계를 별도로 두세요.
- ⭐ **추천:** 강력 추천 — 비용 0, 다운사이드 거의 없음.

## 🟩 Codex — Context/Security: sandbox + approval 2단 통제
- **source:** [Codex Sandboxing Docs](https://developers.openai.com/codex/concepts/sandboxing)
- 💡 **핵심:** Codex 보안은 **샌드박스(무엇을 할 수 있나)**와 **승인 정책(언제 물어보나)** 두 축으로 분리됩니다. 둘을 함께 조이는 게 핵심.
- 📝 **how-to:** 로컬 자동화에 권장되는 저위험 프리셋:

```bash
codex --sandbox workspace-write --ask-for-approval on-request
```

```toml
# config.toml
sandbox_mode  = "workspace-write"   # read-only | workspace-write | danger-full-access
approval_policy = "on-request"

[sandbox_workspace_write]
writable_roots = ["./"]             # 쓰기 범위를 작업 폴더로 제한
# network_access = false (기본) — 외부 입력 처리 시 네트워크 차단 유지
```

- 🎯 **언제:** 신뢰할 수 없는 입력(웹/파일/MCP 응답)을 다루는 에이전트 작업 전부.
- ⚠️ **주의:** `danger-full-access`는 이름 그대로 위험 — 일회성 디버깅 외엔 피하세요.
- ⭐ **추천:** 강력 추천 — 오늘 논문들이 가리키는 "권한 최소화"의 가장 실용적 구현.

## 🟪 공통 — Review/Security: injection을 "못 막는다" 전제로 설계
- **source:** [arXiv:2601.17548](https://huggingface.co/papers/2601.17548)
- 💡 **핵심:** 탐지/필터는 적응형 공격에 무너집니다. 피해 반경을 줄이는 데 투자하세요.
- 📝 **체크리스트:**
  - [ ] 에이전트 도구를 allowlist로 제한 (필요한 것만)
  - [ ] 외부 입력 처리 경로는 비밀정보·자격증명과 격리
  - [ ] 파괴적/되돌릴 수 없는 작업은 휴먼인루프 또는 정책 게이트
  - [ ] 모든 도구 호출 로깅 → 사후 추적 가능성 확보
- 🎯 **언제:** 도구·MCP·파일 접근을 가진 모든 프로덕션 에이전트.
- ⚠️ **주의:** "권한 최소화"는 UX와 트레이드오프 — 마찰이 큰 지점만 선택적으로 푸세요.
- ⭐ **추천:** 추천 — 신규 에이전트 설계의 기본 골격으로.
