데모에서 잘 돌던 에이전트가 프로덕션에서 무너지는 이유는 대개 모델이 갑자기 멍청해져서가 아닙니다. 8단계짜리 작업의 7번째에서 서버가 재배포되거나 API가 잠깐 삐끗해 프로세스가 죽으면, 지금까지의 진행과 이미 지불한 토큰이 통째로 날아가고, 재시작하면 처음부터 다시 시작하기 때문입니다.

— 내구성 있는 실행은 "죽지 않게" 만드는 기술이 아니라, "죽어도 이어서 할 수 있게" 만드는 기술입니다.

여러 단계를 거치는 에이전트(agent)를 처음 프로덕션에 올린 junior 개발자가 반드시 부딪히는 벽이 있습니다. 로컬에서, 데모에서, 짧은 작업에서는 멀쩡합니다. 그런데 "리서치해서 초안 쓰고, 검토하고, 승인 기다렸다가, 메일 보내고, 기록 남기는" 식으로 오래 걸리고 단계가 많은 작업을 실제 트래픽에서 돌리기 시작하면, 어느 순간부터 이상하게 자주 실패합니다. 절반쯤 하다 만 주문, 두 번 나간 결제, 사라진 진행 상황.

이게 개별 버그처럼 보이지만, 사실은 구조적인 문제입니다. 그리고 2026년 들어 이 문제를 정면으로 다루는 이름이 하나 자리를 잡았습니다 — 내구성 있는 실행(durable execution)입니다. 이번 주(2026년 7월)에도 실무자 뉴스레터에서 "에이전트 신뢰성(reliability)과 컴퓨트 경제학"이 화두로 다시 올라왔고, LangGraph·Pydantic AI·OpenAI Agents SDK 같은 주류 프레임워크가 앞다투어 durable execution을 1급 기능으로 품는 흐름이 이어지고 있습니다. Temporal이 "에이전틱 AI를 프로덕션으로"라는 명분으로 2026년 2월 $50억 밸류에 $3억을 조달한 것도 같은 맥락이고요.

왜 갑자기 다들 이 이야기를 할까요? 그리고 우리 에이전트에는 어떻게 적용할까요? 그 문제부터, 두 가지 구현 방식, LangGraph로 직접 해보는 실전, 그리고 초보를 태우는 함정까지 — 쉽고 자세하게 짚어 보겠습니다.

1. 왜 에이전트는 프로덕션에서 무너지나

먼저 왜 "단계가 많으면" 위험한지 숫자로 봅시다. 각 단계가 99% 확률로 성공한다고 해봅시다. 꽤 믿음직하죠. 그런데 단계는 곱셈으로 엮입니다. 전체가 성공하려면 모든 단계가 성공해야 하니까요.

  • 5단계: 0.995 ≈ 95.1%
  • 10단계: 0.9910 ≈ 90.4%
  • 단계당 성공률이 95%로 조금만 떨어지면? 10단계면 0.9510 ≈ 59.9% — 열에 넷은 실패

이걸 신뢰성 복리(compounding)라고 부릅니다. 단계 하나하나는 멀쩡한데, 길게 이으면 전체가 무너지는 겁니다. LLM 호출, 도구 실행, 외부 API — 하나하나가 확률적으로 삐끗할 여지가 있고, 오래 도는 에이전트일수록 그 여지가 쌓입니다.

여기서 두 번째 문제가 나옵니다. 그냥 실패도 아프지만, "진행 중에 프로세스가 통째로 죽는" 실패는 특히 아픕니다. 이유는 세 가지입니다.

  • 진행 상황이 메모리에만 있다. 8단계 중 7단계까지의 결과가 대부분 프로세스 메모리(변수)에 들어 있습니다. 프로세스가 죽으면 그대로 증발합니다.
  • 이미 쓴 토큰을 또 쓴다. 재시작하면 처음부터 다시 도니까, 앞 단계의 비싼 LLM 호출을 다시 지불합니다. 긴 에이전트일수록 이 낭비가 큽니다.
  • 부수효과가 중복된다. 앞에서 이미 결제를 했거나 메일을 보냈는데, 처음부터 다시 돌면 또 결제하고 또 메일을 보냅니다.

그래서 오래 도는 에이전트에는 "재시도(retry)"만으로는 부족합니다. 재시도는 "다시 처음부터"이기 쉽거든요. 필요한 건 "마지막으로 멀쩡했던 지점부터 이어서"입니다. 그게 내구성 있는 실행이 주는 것입니다.

2. 내구성 있는 실행이란 무엇인가

내구성 있는 실행(durable execution)은 워크플로의 각 단계 결과를 실행하는 즉시 바깥의 저장소(DB, 로그)에 기록해 두고, 프로세스가 죽었다 살아나면 그 기록을 이용해 마지막 성공 지점부터 이어서 실행하는 방식입니다. 한마디로 충돌에 강한(crash-tolerant) 함수 실행이죠.

가장 쉬운 비유는 게임의 세이브 지점(save point)입니다. 보스전에서 죽어도 마을 입구가 아니라 보스 앞 체크포인트에서 다시 시작하잖아요. 이미 깬 던전을 다시 깨지 않고요. durable execution은 에이전트에 이 세이브 지점을 자동으로 깔아 주는 것입니다.

이걸 성립시키는 두 가지 핵심 성질이 있습니다. 이름만 기억해도 절반은 이해한 겁니다.

① 상태 지속성(state persistence) — 각 단계가 끝날 때마다 그 결과를 프로세스 바깥(DB 등)에 저장합니다. 그래서 프로세스가 죽어도 진행 상황이 살아남습니다.

② 정확히 한 번(exactly-once) 실행 의미론 — 재개했을 때 이미 끝난 단계는 다시 실행하지 않고 저장된 결과를 그대로 돌려줍니다. 그래서 결제·메일 같은 부수효과가 딱 한 번만 일어납니다.

이 둘이 합쳐지면, 겉으로 보기엔 "한 번도 멈추지 않고 끝까지 실행된 것처럼" 보이는 워크플로가 됩니다. 실제로는 중간에 세 번 죽었다 살아났더라도요. 개발자 입장에서는 그냥 평범한 함수를 쓰는 것처럼 코드를 짜는데, 런타임이 뒤에서 저장·재개를 알아서 해주는 겁니다.

참고. "재시도(retry)"와 헷갈리기 쉽습니다. 재시도는 실패한 것을 다시 하는 것이고, durable execution은 성공한 것을 다시 하지 않는 것입니다. 좋은 시스템은 둘 다 합니다 — 실패한 단계는 재시도하되, 성공한 단계는 저장된 결과로 건너뜁니다.

3. 두 갈래 — 저널 리플레이 vs DB 체크포인트

"각 단계 결과를 저장했다가 이어서 실행한다"는 아이디어를 구현하는 방식은 크게 두 갈래입니다. 도구를 고를 때 이 차이를 알아야 하니 짚고 갑시다.

저널 기반 리플레이(journal-based replay) — Temporal, Restate가 대표적입니다. 워크플로가 실행한 모든 단계를 "이벤트 히스토리(event history)"라는 저널에 차곡차곡 기록합니다. 크래시 후 재개하면 워크플로 함수를 처음부터 다시 실행하는데, 이때 저널에 이미 있는 단계는 진짜로 돌리지 않고 저장된 결과를 즉시 반환합니다. 그러다 저널에 없는 첫 단계를 만나면 그때부터 실제로 실행합니다.

여기서 아주 중요한 규칙이 하나 나옵니다. 함수를 "처음부터 다시 실행"해도 매번 똑같은 길을 가야 저널과 아귀가 맞습니다. 그래서 LLM 호출처럼 매번 결과가 달라지는(비결정적) 작업은 반드시 "액티비티(activity)" 같은 단계로 감싸서, 그 결과를 저널에 남기고 재실행 때는 절대 다시 부르지 않게 해야 합니다. (5장에서 이 지점을 자세히 봅니다.)

DB 체크포인트(database checkpointing) — LangGraph, DBOS가 대표적입니다. 워크플로를 노드(node) 단위로 쪼개고, 각 노드가 끝날 때마다 상태 전체를 DB에 저장합니다. 크래시 후에는 함수를 처음부터 돌리는 게 아니라, 마지막으로 저장된 상태를 불러와 그 다음 노드부터 실행합니다. 리플레이 방식보다 직관적이지만, "노드 하나의 경계"가 곧 세이브 지점이라, 노드 안에서 죽으면 그 노드는 통째로 다시 돕니다.

둘은 경쟁 관계가 아니라 겹쳐 쓰기도 합니다. 실제로 견고한 시스템은 안쪽에서 LangGraph가 에이전트의 추론·도구 계획을 담당하고, 바깥에서 Temporal이 전체 워크플로를 감싸 재시도·상태 내구성·장기 조율을 맡는 식으로 조합하곤 합니다.

4. 실전 — LangGraph로 죽었다 살아나는 에이전트

말로만 하면 감이 안 오니, LangGraph로 직접 만들어 봅시다. LangGraph에서는 체크포인터(checkpointer)를 붙이는 순간 durable execution이 켜집니다. 별도 마법이 아니라, "노드마다 상태를 저장하는 저장소"를 꽂아 주는 것뿐입니다.

핵심 개념 두 가지만 알면 됩니다.

  • checkpointer — 상태를 저장할 곳. 개발용 InMemorySaver, 로컬 SqliteSaver, 프로덕션용 PostgresSaver 등이 있습니다. (메모리 세이버는 프로세스가 죽으면 같이 죽으니 진짜 내구성은 없습니다 — 개념 확인용입니다.)
  • thread_id — "이 대화/작업 한 건"을 식별하는 키. 같은 thread_id로 다시 호출하면, 그 지점의 저장된 상태를 이어받습니다. 세이브 파일 이름이라고 생각하면 됩니다.

먼저 기본형입니다. 체크포인터를 붙이고, thread_id를 넘겨 실행합니다.

{{< highlight python >}} from langgraph.graph import StateGraph, START, END from langgraph.checkpoint.memory import InMemorySaver # 프로덕션이라면: from langgraph.checkpoint.postgres import PostgresSaver # builder = StateGraph(...) 로 노드/엣지를 구성했다고 하자. checkpointer = InMemorySaver() graph = builder.compile(checkpointer=checkpointer) # ← 이 한 줄로 durable 모드 config = {"configurable": {"thread_id": "order-4127"}} # 작업 1건의 세이브 키 graph.invoke({"order": order}, config) # 노드가 끝날 때마다 상태가 저장된다 {{< /highlight >}}

이제 이 작업이 3번째 노드까지 마치고 4번째에서 프로세스가 죽었다고 합시다. 다시 켠 뒤 같은 thread_id로 다시 호출하면, LangGraph는 저장된 상태(3번째까지의 결과)를 불러와 4번째부터 이어서 실행합니다.

{{< highlight python >}} # 프로세스를 새로 켠 뒤 — 같은 config(thread_id)로 다시 호출 graph.invoke(None, config) # 입력이 None이어도, 저장된 지점부터 이어서 실행된다 {{< /highlight >}}

여기서 실전 노브가 하나 있습니다. "얼마나 자주, 얼마나 꼼꼼히 저장할지"를 고르는 durability 모드입니다. LangGraph는 세 가지를 제공합니다.

모드언제 저장하나성격
"exit"워크플로가 끝날 때만 저장가장 빠름 · 내구성 가장 약함 (중간에 죽으면 진행분 날아감)
"async"다음 단계가 도는 동안 비동기로 저장성능·내구성 균형 (저장 직전 크래시 시 그 단계는 유실될 소지)
"sync"다음 단계 시작 전에 동기로 저장 완료가장 안전 · 약간의 오버헤드 (모든 단계가 반드시 기록됨)
{{< highlight python >}} # 돈이 오가거나 절대 유실되면 안 되는 작업이면 sync로. graph.invoke({"order": order}, config, durability="sync") {{< /highlight >}}

정리하면 이렇습니다. 체크포인터를 붙이고 → 작업마다 thread_id를 주고 → 중요도에 맞는 durability 모드를 고른다. 이 세 가지가 LangGraph durable execution의 뼈대입니다. 그런데 이대로 프로덕션에 올리면 조용히 당하는 함정이 있습니다. 다음 장이 이 글에서 제일 중요합니다.

5. 리플레이가 당신을 태우는 지점 (결정성·부수효과)

durable execution을 처음 쓰는 사람이 반드시 데는 함정이 있습니다. 이름하여 리플레이 위험(replay risk)입니다. 핵심을 한 문장으로 못박겠습니다.

재개는 "죽은 줄 다음부터"가 아니라 "그 단계(노드)를 통째로 다시"입니다. 체크포인트 이후의 노드가 다시 돌 때, 그 안의 LLM 호출·API 요청·결제 같은 부수효과(side effect)도 같이 다시 실행됩니다. 소스 코드의 다음 줄부터 이어지는 게 아닙니다.

왜 이게 무섭냐면, 코드가 "맞아 보이는데도" 중복이 나기 때문입니다. 예를 들어 한 노드 안에서 "① 결제하고 → ② 사람 승인을 기다린다(interrupt)"고 짰다고 합시다. 승인이 오면 그 노드가 다시 실행되는데, interrupt 앞의 "① 결제"가 또 나갑니다. 사용자 눈엔 안 보이는 이중 결제가 생기는 거죠. 실제로 LangGraph의 human-in-the-loop에서 흔히 보고되는 함정입니다.

해법은 두 가지 원칙으로 요약됩니다.

원칙 1 — 부수효과와 비결정 연산은 task로 감싼다. LangGraph의 함수형 API에서는 @task로 감싼 함수의 결과가 체크포인트에 저장됩니다. 재개하면 그 task는 다시 실행되지 않고 저장된 결과를 즉시 돌려줍니다. 결제·메일 발송·외부 API 호출·랜덤값·현재 시각처럼 "다시 하면 안 되거나 매번 값이 달라지는" 건 전부 task로 감싸세요.

{{< highlight python >}} from langgraph.func import entrypoint, task from langgraph.checkpoint.memory import InMemorySaver @task def call_llm(prompt: str) -> str: # 매번 값이 달라지는 LLM 호출 → task로 감싸 결과를 고정 저장 return llm.invoke(prompt).content @task def charge_card(user_id: str, amount: int) -> str: # '딱 한 번'이어야 하는 결제 → 반드시 task로 return payments.charge(user_id, amount) # 재개 시 재실행되지 않음 @entrypoint(checkpointer=InMemorySaver()) def workflow(order: dict) -> str: plan = call_llm(f"주문 처리 계획을 세워라: {order}").result() receipt = charge_card(order["user"], order["amount"]).result() # 여기서 프로세스가 죽어도, 재개하면 call_llm·charge_card는 # '다시 실행되지 않고' 저장된 결과를 그대로 돌려받는다. return f"완료: {receipt}" {{< /highlight >}}

원칙 2 — 그래도 부수효과는 멱등(idempotent)하게 설계한다. 멱등이란 "같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 같은" 성질입니다. task로 감싸는 것과 별개로, 결제·발송 API 자체에 멱등 키(idempotency key)를 붙여 두면 어떤 이유로 두 번 불려도 실제로는 한 번만 처리됩니다. durable execution은 "가능한 한 다시 안 부르게" 해주지만, 재시도가 얽히는 분산 환경에서 절대 두 번 안 부른다고 보장하긴 어렵습니다. 그래서 안전망을 하나 더 두는 겁니다.

{{< highlight python >}} # 결제 API에 멱등 키를 함께 보낸다 (thread_id·단계명으로 고유 키를 만든다) payments.charge( user_id, amount, idempotency_key=f"order-4127:charge", # 같은 키면 두 번째 요청은 무시된다 ) {{< /highlight >}}
기억할 것. "체크포인트가 있다 = 안전하다"가 아닙니다. 체크포인트는 상태를 저장할 뿐, 노드 안의 부수효과가 저절로 재실행-안전해지는 건 아닙니다. 노드(단계)의 경계를 리플레이 경계로 의식해서 설계해야 비로소 안전해집니다.

6. 언제 무엇을 쓰나

그럼 무조건 Temporal 같은 무거운 걸 깔아야 할까요? 아닙니다. 작업의 성격에 맞춰 고르면 됩니다.

7. 흔한 실수 · 팁

  • InMemorySaver를 프로덕션에 쓴다. 이름 그대로 메모리에 저장하니 프로세스가 죽으면 같이 죽습니다. 진짜 내구성은 PostgresSaver 같은 외부 저장소에서만 나옵니다. 개발·데모용으로만 쓰세요.
  • 부수효과를 노드에 날것으로 둔다. 이 글 5장의 핵심. 재개 때 결제·메일이 중복됩니다. task로 감싸고, API엔 멱등 키를 붙이세요.
  • 비결정 코드를 리플레이 경로에 둔다. 저널 리플레이(Temporal)에서는 random·현재 시각·정렬 안 된 딕셔너리 순회처럼 매번 달라지는 코드가 리플레이를 깨뜨립니다. 이런 건 반드시 액티비티/task 안으로 넣으세요.
  • thread_id를 재사용하거나 겹치게 쓴다. 서로 다른 작업이 같은 thread_id를 쓰면 상태가 뒤섞입니다. 작업 1건 = 고유 thread_id를 지키세요.
  • 노드를 너무 크게 짠다. 한 노드에 여러 부수효과를 몰아넣으면, 재개 시 그 노드가 통째로 다시 돌며 전부 재실행됩니다. 부수효과 하나 = task 하나로 잘게 나누는 게 안전합니다.
  • durability 모드를 안 따진다. 기본값이 항상 최선은 아닙니다. 돈·외부 부수효과가 걸린 작업은 sync로, 유실돼도 그만인 작업은 성능을 위해 async/exit로 — 의식적으로 고르세요.

8. 마무리 (TL;DR)

  • 다단계 에이전트는 신뢰성 복리 때문에 태생적으로 가끔 실패한다(단계당 95%×10단계 ≈ 60%). 그리고 진행 중 크래시는 진행분·토큰·부수효과를 통째로 날린다. 목표는 "안 죽기"가 아니라 "죽어도 이어서 하기"다.
  • 내구성 있는 실행(durable execution)은 각 단계 결과를 저장(상태 지속성)해 두고, 재개 시 끝난 단계는 다시 안 돌린다(exactly-once). 게임 세이브 지점과 같다.
  • 구현은 두 갈래 — 저널 리플레이(Temporal·Restate: 처음부터 다시 부르되 저널값 반환)와 DB 체크포인트(LangGraph·DBOS: 노드마다 상태 저장, 다음부터 잇기). 겹쳐 쓰기도 한다.
  • LangGraph는 checkpointer를 붙이고 작업마다 thread_id를 주면 durable 모드가 켜진다. 중요도에 따라 exit/async/sync durability를 고른다.
  • 제일 큰 함정은 리플레이 위험 — 재개는 "노드 통째 재실행"이라 부수효과가 중복된다. 부수효과·비결정 연산은 task로 감싸고, 결제·발송 API엔 멱등 키를 붙여 안전망을 둬라.
  • 선택 기준: 짧으면 그냥 재시도, 다단계·승인이면 LangGraph 체크포인터, 장기·엄격한 조율이면 Temporal 같은 엔진, 둘 다면 조합.
