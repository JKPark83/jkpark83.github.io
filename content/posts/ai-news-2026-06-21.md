# 🔥 오늘의 핵심

오늘은 **"에이전트 오케스트레이션 효율"** 한 줄기로 신호가 모입니다. 핵심 질문은 하나 — *"언제 도구를 호출하고, 언제 멈추고, 누구에게 시킬 것인가."* Codex는 6월 업데이트로 **반복 워크플로를 녹화해 재사용 Skill로 굳히는** 방향을 잡았고, 논문 쪽은 **respond·retrieve·tool call·verify·stop을 효용으로 점수화**해 불필요한 호출을 끊는 정책을 내놓았습니다. 결론은 분명합니다 — 더 많은 도구·호출이 아니라, **호출을 줄이고 작업을 쪼개는 설계**가 비용과 컨텍스트를 동시에 살립니다.

## 1. Utility-Guided Agent Orchestration for Efficient LLM Tool Use
- **arXiv:** [2603.19896](https://arxiv.org/abs/2603.19896) · cs.AI · 제출 2026-03
- **핵심 기여:** 에이전트 오케스트레이션을 **명시적 결정 문제**로 정식화. 매 스텝에서 `respond / retrieve / tool call / verify / stop` 다섯 행동을 **기대 이득, 스텝 비용, 불확실성, 중복**으로 점수화해 고르는 효용 기반 정책을 제안합니다.
- **방법:** "도구를 더 부를수록 좋다"는 가정 대신, 추가 호출의 *한계 이득이 비용보다 작으면 stop*하도록 설계 → 정확도를 크게 떨어뜨리지 않으면서 호출 수를 줄이는 게 목표.
- 🛠 **빌더 관점:** 에이전트 루프에 **명시적 stop/verify 단계와 호출 예산**을 넣는 직접적 근거. 무한 retrieve·tool-call 루프를 깨는 가장 싼 안전장치.
- ⚠️ **한계:** 효용 추정의 정확도가 정책 품질을 좌우 → 도메인별 보정 필요.
- 🔗 [arXiv abs](https://arxiv.org/abs/2603.19896)

## 2. The Evolution of Tool Use in LLM Agents: From Single-Tool Call to Multi-Tool Orchestration
- **arXiv:** [2603.22862](https://arxiv.org/abs/2603.22862) · cs.AI · 제출 2026-03
- **핵심 기여:** 단발 도구 호출과 **장기 다중 도구 오케스트레이션**을 구분해 통합 정리한 서베이. 추론시 계획·실행, 학습/궤적 구성, 안전·제어, 자원 제약하 효율, 개방환경 능력, 벤치마크 설계의 **6개 축**으로 문헌을 조직화합니다.
- 🛠 **빌더 관점:** 내 에이전트가 "단발 호출형"인지 "장기 오케스트레이션형"인지부터 진단하고, 그 축에 맞는 평가·제어 기법을 고르는 체크리스트로 쓰기 좋음.
- ⚠️ **한계:** 서베이라 신규 방법의 정량 비교가 아닌 지형도.
- 🔗 [arXiv abs](https://arxiv.org/abs/2603.22862)

# 🧑‍💻 Codex & Claude Code 실전 팁

## 🟩 Codex · Workflow
- **출처:** [Record & Replay 공식 문서](https://developers.openai.com/codex/record-and-replay)
- 💡 **핵심:** 반복적이고 *말로 설명하기보다 보여주는 게 빠른* 작업은 한 번 시연해 재사용 Skill로 굳힌다.
- 📝 **방법:** Record & Replay 시작 → 평소처럼 워크플로 1회 수행 → Codex가 시퀀스를 캡처해 Skill로 변환 → 다음부터 새 입력으로 같은 흐름 재실행. 팀 공유 가능.
- 🎯 **언제:** 정기 리포트 생성, 반복 데이터 정리 등 단계가 고정된 작업.
- ⚠️ **주의:** macOS 한정이며 Computer Use 활성 필요, 초기엔 EEA·영국·스위스 제외. 화면을 보고 동작하므로 민감정보 노출 화면은 녹화 전에 정리.
- ⭐ **추천:** 반복 사무 워크플로가 있는 팀이면 우선 시도, 일회성 작업엔 과함.

## 🟦 Claude Code · Subagent
- **출처:** [Claude Code 서브에이전트 문서](https://code.claude.com/docs/en/sub-agents)
- 💡 **핵심:** 검색·로그·파일 탐색처럼 *메인 대화를 오염시킬* 곁다리 작업은 **별도 컨텍스트의 서브에이전트**에 넘기고 요약만 받는다. 도구를 제한하고 모델을 낮추면 비용까지 절감.
- 📝 **방법:** `.claude/agents/`(또는 `~/.claude/agents/`)에 YAML frontmatter 파일을 둔다.

```yaml
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: haiku
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.
```

- 🎯 **언제:** 대량 코드 리뷰·리서치 등 결과 요약만 필요하고 중간 산출물을 다시 안 볼 때.
- ⚠️ **주의:** `tools`를 비우면 모든 도구를 상속한다 — 격리가 목적이면 꼭 최소 목록만 명시. 서브에이전트엔 메인 시스템 프롬프트가 전달되지 않는다.
- ⭐ **추천:** 컨텍스트 절약 + 비용 절감 둘 다 노리는 팀에 강력 추천.

## 🟪 공통 · Orchestration
- **출처:** [arXiv 2603.19896](https://arxiv.org/abs/2603.19896)
- 💡 **핵심:** 에이전트 루프에 **`stop`을 1급 행동**으로 넣어라. "한 번 더 부를 한계 이득 < 비용"이면 멈춘다.
- 📝 **방법:** 매 스텝 전 간단한 게이트를 둔다 — (1) 남은 호출 예산 확인, (2) 직전 결과가 답에 새 정보를 더했는지 점검, (3) 아니면 종료·응답. 호출 카운터 + "이번 호출의 기대 가치" 한 줄 자기평가만으로도 무한 루프가 크게 줄어든다.
- 🎯 **언제:** retrieve/tool-call이 답 없이 반복되거나 토큰이 폭증하는 에이전트.
- ⚠️ **주의:** stop 임계치를 너무 낮추면 조기 종료로 정확도 하락 → 도메인별로 보정.
- ⭐ **추천:** 자율 실행 범위를 넓힌 팀이라면 오늘 바로 넣을 가장 싼 안전장치.
