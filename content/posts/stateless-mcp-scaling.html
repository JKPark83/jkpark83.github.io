MCP 서버를 원격으로 띄우고 트래픽이 늘어 인스턴스를 2대로 늘린 순간, 갑자기 요청이 이상하게 실패하기 시작합니다. 원인은 대개 세션입니다. 서버 A가 기억하던 세션을, 로드 밸런서가 다음 요청을 서버 B로 보내면서 잃어버리는 거죠.

— stateless MCP는 "세션을 잘 공유하자"가 아니라, "애초에 프로토콜이 세션을 안 갖게 하자"는 방향의 전환입니다.

MCP(Model Context Protocol)는 이제 대부분의 AI 개발자에게 익숙한 단어입니다. 에이전트(agent)가 외부 도구·데이터에 연결하는 표준 방식으로 자리를 잡았고, Claude Code·Codex·Cursor 같은 도구가 모두 같은 프로토콜을 씁니다. 그런데 MCP 서버를 개인 노트북에서만 쓰다가 여러 사람이 붙는 원격 서비스로 올리려는 순간, 많은 사람이 똑같은 벽에 부딪힙니다 — 확장(scaling)이 이상하게 까다롭다는 것입니다.

이번 주(2026년 7월) 이 벽을 정면으로 허무는 발표가 나왔습니다. MCP의 다음 스펙인 2026-07-28 릴리스 후보(RC, release candidate)가 공개됐고, 헤드라인 변경은 딱 하나로 요약됩니다 — MCP가 프로토콜 계층에서 무상태(stateless)가 됐다. 공식 SDK들도 이 스펙을 지원하는 베타 버전을 같은 주에 함께 냈습니다. 이 변화가 왜 중요하고, MCP 서버를 만드는 우리에게 실제로 무엇을 바꾸는지 하나씩 풀어 보겠습니다.

참고로 지금은 정식 확정 전의 RC 단계입니다. RC는 2026년 5월 21일에 잠겼고(lock), SDK 검증·피드백을 위한 약 10주의 창을 거쳐 2026년 7월 28일 정식 발표가 예정돼 있습니다. 즉 이 글을 읽는 지금이 딱 "곧 표준이 될 변경을 미리 이해해 둘" 타이밍입니다.

1. 왜 MCP 서버는 확장이 어려웠나 — 세션이라는 족쇄

먼저 "세션(session)"이 뭔지 눈높이를 맞춥시다. 세션은 서버가 한 클라이언트를 기억하는 상태입니다. 지금까지의 MCP는 클라이언트가 서버에 처음 붙을 때 initialize라는 악수(handshake)를 한 번 나눴습니다. 이때 "나는 프로토콜 버전 몇을 쓰고, 이런 기능을 지원해"라는 정보를 주고받고, 서버는 그 클라이언트에게 세션 ID(Mcp-Session-Id 헤더)를 발급합니다. 이후 모든 요청은 이 세션 ID를 달고 다니고, 서버는 그 ID에 묶인 상태를 기억한 채로 응답합니다.

노트북에서 서버 한 대만 돌릴 때는 아무 문제가 없습니다. 그런데 사용자가 늘어 서버를 여러 대로 늘리는 순간 문제가 터집니다. 웹 서비스를 키우는 가장 기본적인 방법이 "같은 서버를 여러 대 세우고 로드 밸런서(load balancer)로 요청을 나눠 주는 것"인데, 세션이 있으면 이게 깨집니다.

문제의 핵심: 세션 상태는 initialize를 처리한 그 서버 한 대의 메모리에만 있습니다. 로드 밸런서가 다음 요청을 다른 서버로 보내면, 그 서버는 이 클라이언트를 전혀 모릅니다. "너 누구야? initialize부터 다시 해"가 되어 버리죠.

그래서 지금까지는 두 가지 우회로를 써야 했습니다. 둘 다 운영을 무겁게 만듭니다.

  • 고정 라우팅(sticky session) — 로드 밸런서에게 "이 클라이언트의 요청은 항상 서버 A로만 보내라"고 시킵니다. 코트 보관소에서 받은 번호표를 그 보관소에서만 쓸 수 있는 것과 같죠. 문제는 서버 A가 죽거나 재배포되면 세션이 통째로 날아가고, 부하도 고르게 안 퍼집니다.
  • 공유 세션 저장소(shared session store) — 세션을 Redis 같은 외부 저장소에 넣고 모든 서버가 공유합니다. 이러면 확장은 되지만, 저장소라는 인프라가 하나 더 늘고, 그 저장소가 새 병목이자 장애 지점이 됩니다.

2. 배경 — MCP 전송(transport) 30초 복습

세션 이야기를 제대로 이해하려면 MCP가 데이터를 어떻게 실어 나르는지, 즉 전송(transport)을 알아야 합니다. 딱 두 가지만 기억하면 됩니다.

  • stdio — 클라이언트가 서버를 자식 프로세스로 띄우고, 표준 입출력(stdin/stdout)으로 대화합니다. 로컬 도구에 제격입니다. 프로세스가 곧 세션이라 확장 이야기가 나올 일이 없습니다.
  • Streamable HTTP — 서버를 원격 HTTP 서비스로 띄우고, 여러 클라이언트가 네트워크로 붙습니다. 2025년 11월 스펙에서 기존의 SSE(Server-Sent Events) 전송을 대체하며 들어온 방식이고, 오늘 이야기하는 "확장"은 전부 이 원격 HTTP 서버에 관한 것입니다.

정리하면, 우리가 확장하고 싶은 대상은 Streamable HTTP로 뜬 원격 MCP 서버입니다. 그리고 지금까지 이 서버는 initialize 핸드셰이크와 Mcp-Session-Id로 클라이언트별 세션을 유지했기 때문에, 앞 장에서 본 확장 문제를 그대로 안고 있었던 것이죠.

비유: 지금까지의 MCP 원격 서버는 전화 통화 같았습니다. 한 번 연결하면 그 통화 내내 같은 교환원에게 매여 있죠(세션). stateless MCP는 이걸 우편엽서로 바꿉니다. 엽서 한 장 한 장에 보내는 이·받는 이·용건이 다 적혀 있으니, 어느 우체국이 처리하든 상관없습니다.

3. 무엇이 바뀌나 — stateless 코어(SEP-2575)

이제 본론입니다. 2026-07-28 스펙의 핵심 변경은 SEP-2575: Make MCP Stateless입니다. (SEP는 MCP의 표준 제안서를 뜻하는 Specification Enhancement Proposal입니다. 파이썬의 PEP를 떠올리면 됩니다.) 딱 세 덩어리로 나눠 보겠습니다.

① initialize 핸드셰이크 제거 → 매 요청이 자기소개를 들고 온다

연결 시 한 번 주고받던 정보(프로토콜 버전, 클라이언트 정보, 지원 기능)를, 이제 모든 요청의 _meta 필드에 실어 매번 함께 보냅니다. 클라이언트가 서버 기능을 미리 알고 싶으면 server/discover라는 새 메서드로 필요할 때 물어보면 됩니다. 앞의 우편엽서 비유 그대로입니다 — 엽서마다 보내는 이 정보가 적혀 있으니, 서버는 이전 대화를 기억할 필요가 없습니다.

② Mcp-Session-Id 제거(SEP-2567) → sticky도 공유 저장소도 불필요

프로토콜 계층의 세션 자체가 사라졌습니다. 세션 ID가 없으니 "이 요청은 어느 세션에 묶였나"를 서버가 따질 일이 없고, 그래서 어떤 요청이든 아무 서버 인스턴스로나 갈 수 있습니다. 앞 장에서 애먹었던 sticky 라우팅과 공유 세션 저장소가 프로토콜 차원에서 필요 없어진 겁니다.

③ 라우팅용 헤더 추가(SEP-2243) → 게이트웨이가 몸통을 안 열어 봐도 된다

Streamable HTTP 전송은 이제 Mcp-Method와 Mcp-Name 헤더를 요구합니다(프로토콜 버전은 Mcp-Protocol-Version 헤더). 로드 밸런서·게이트웨이·레이트 리미터가 요청 본문(JSON)을 뜯어보지 않고 헤더만 보고 "이건 tools/call이구나", "이 도구구나" 하고 라우팅·제한을 걸 수 있게 하려는 것입니다.

여기에 캐시 힌트도 한 겹 더해졌습니다(SEP-2549). tools/list 같은 목록·리소스 응답이 이제 ttlMs(얼마나 오래 신선한지)와 cacheScope(사용자 간 공유해도 되는지)를 함께 실어 줍니다. HTTP의 Cache-Control을 떠올리면 됩니다. 세션이 없어져 서버가 가벼워진 만큼, 이런 캐시 힌트로 반복 조회를 줄이는 여지가 커진 셈입니다.

4. 그럼 상태가 진짜 필요할 땐? — 명시적 핸들

여기서 초보가 가장 헷갈리는 지점입니다. "프로토콜이 무상태면, 장바구니처럼 상태가 꼭 필요한 도구는 어떻게 만들죠?" 좋은 질문이고, 답은 명확합니다.

프로토콜이 무상태라고 해서, 여러분의 애플리케이션까지 무상태여야 하는 건 아닙니다. 상태가 필요하면, 평범한 HTTP API가 늘 하던 방식을 그대로 쓰면 됩니다 — 서버가 명시적인 핸들(handle)을 하나 발급하고, 모델이 그 핸들을 다음 호출의 일반 인자로 다시 넘겨주는 것입니다.

예를 들어 장바구니 도구라면 이렇게 흐릅니다. create_cart를 호출하면 서버가 cart_id: "c_92af"를 돌려주고, 이후 add_item·checkout은 그 cart_id를 인자로 받습니다. 상태는 여러분의 DB에 cart_id를 키로 저장돼 있으니, 어느 서버 인스턴스가 요청을 받든 그 DB만 보면 됩니다. 숨은 세션이 아니라 눈에 보이는 인자로 상태를 이어 가는 것이죠.

이 패턴은 사실 REST API 세계가 20년 넘게 써 온 방식입니다. 무상태 HTTP 위에서 order_id·upload_id 같은 핸들을 주고받으며 상태를 이어 온 것과 똑같습니다. MCP가 뒤늦게 이 검증된 길로 합류한 셈이죠.

5. 실전 — 베타 SDK로 stateless 서버 만들기

말로만 하면 감이 안 오니 실제 SDK를 봅시다. 스펙 RC와 함께 공식 SDK 4종(파이썬·타입스크립트·Go·C#)의 베타가 같은 주에 나왔습니다. 우선 설치부터.

{{< highlight bash >}} # Python (Tier 1 베타) — 클래스 기반 MCPServer가 기존 FastMCP를 대체 pip install "mcp[cli]==2.0.0b1" # TypeScript v2 베타 — 서버/클라이언트 패키지가 분리됨 npm install @modelcontextprotocol/server@beta # Go — 사전 릴리스 go get github.com/modelcontextprotocol/go-sdk@v1.7.0-pre.1 {{< /highlight >}}

가장 눈여겨볼 곳은 Go SDK입니다. HTTP 전송에서 stateless 동작을 명시적으로 켜야 하기 때문입니다. 옵션 하나로 "이 서버는 세션을 안 쓴다"를 선언합니다.

{{< highlight go >}} // Go SDK: Streamable HTTP를 stateless 모드로 여는 핵심 스위치 opts := &mcp.StreamableHTTPOptions{ Stateless: true, // ← 세션 없이 동작. 아무 인스턴스나 요청 처리 가능 } // 이 옵션으로 뜬 서버는 라운드로빈 LB 뒤에 그냥 여러 대 세우면 된다. {{< /highlight >}}

애플리케이션 상태가 필요할 땐 4장에서 본 명시적 핸들 패턴을 그대로 코드로 옮깁니다. 아래는 개념을 보여 주는 파이썬 예시입니다(등록 방식은 SDK 버전에 따라 조금씩 다르지만, 핵심은 핸들을 발급하고 인자로 되받는 것입니다).

{{< highlight python >}} # 상태가 필요한 도구는 '숨은 세션'이 아니라 '보이는 핸들'로 잇는다. def create_cart() -> dict: cart_id = new_id() # 예: "c_92af" db.put(cart_id, {"items": []}) # 상태는 공유 DB에 저장 (서버 메모리 아님) return {"cart_id": cart_id} # ← 핸들을 모델에게 돌려준다 def add_item(cart_id: str, sku: str, qty: int) -> dict: cart = db.get(cart_id) # ← 어느 인스턴스가 받든 DB에서 복원 cart["items"].append({"sku": sku, "qty": qty}) db.put(cart_id, cart) return {"ok": True, "count": len(cart["items"])} def checkout(cart_id: str) -> dict: cart = db.get(cart_id) # 세 호출이 서로 다른 서버로 가도 문제없음 return {"order_id": place_order(cart)} {{< /highlight >}}
핵심: db가 서버 프로세스 메모리가 아니라 공유 저장소라는 점이 전부입니다. 도구가 상태를 인스턴스 로컬 변수(전역 dict 등)에 담는 순간, 다시 sticky 세션 시절로 돌아갑니다. stateless의 이득을 살리려면 상태를 반드시 밖(DB·캐시)으로 빼세요.

6. 세션 말고 또 바뀐 것들 (짧게)

2026-07-28은 세션만 손본 게 아니라 MCP 출범 이래 가장 큰 개정입니다. 실무에서 알아 둘 만한 나머지 변경을 표로 훑겠습니다.

영역무엇이 바뀌나왜 중요한가
Extensions (SEP-2133) 역-DNS 식별자로 스펙과 별개로 버전 관리되는 확장 프레임워크. 공식 확장으로 MCP Apps(SEP-1865, 샌드박스 iframe에 서버 렌더 UI)와 Tasks(SEP-2663, tasks/get·tasks/update·tasks/cancel)가 들어옴. 긴 작업의 진행/취소를 표준 방식으로 다루고, 코어를 안 건드리며 기능을 얹을 길이 열림.
서버→클라이언트 통신 (SEP-2322) 계속 열어 두던 SSE 스트림 대신 다중 왕복 요청. 서버가 InputRequiredResult로 추가 입력을 요청하면 클라이언트가 다시 호출. 무상태와 맞물리는 변화. 오래 붙어 있는 스트림이 없으니 확장·재시도가 단순해짐.
인가(auth) 강화 OAuth 2.0/OpenID Connect 정렬을 다지는 SEP 6종. 예: iss 검증(RFC 9207, SEP-2468), 동적 등록 시 application_type 선언(SEP-837). 여러 사용자가 붙는 원격 서버라면 인증·인가가 필수. 표준이 촘촘해짐.
폐기(deprecation) (SEP-2577) Roots·Sampling·Logging이 폐기 예정. 각각 도구 파라미터/리소스 URI, LLM 공급자 API 직접 호출, stdio는 stderr·구조적 관측은 OpenTelemetry로 대체. 12개월 유예. 이 기능들에 의존하는 서버라면 이전 계획을 세워야 함. 다만 1년 유예라 급하진 않음.
JSON Schema (SEP-2106) 도구 스키마가 JSON Schema 2020-12 전체(조합·조건부)를 지원. 도구 입력 스키마를 훨씬 정교하게 정의 가능(예: 조건부 필수 필드).
에러 코드 (SEP-2164) 없는 리소스 에러가 MCP 고유 -32002에서 JSON-RPC 표준 -32602(Invalid Params)로 변경. 표준 JSON-RPC 에러 처리에 자연스럽게 편입.

이 중 실무에 곧바로 영향이 큰 건 Sampling 폐기입니다. 서버가 클라이언트의 LLM을 빌려 쓰던 방식을 접고, 서버가 필요하면 LLM 공급자 API를 직접 부르라는 방향입니다. 관측성(observability)도 표준 로그 대신 OpenTelemetry로 가라는 신호라, 에이전트 관측성을 이미 붙여 뒀다면 결이 잘 맞습니다.

7. 마이그레이션과 하위 호환

가장 걱정되는 질문은 "그럼 내 기존 서버·클라이언트가 다 깨지나?"일 겁니다. 결론부터 말하면 자동 협상으로 부드럽게 넘어갑니다.

새 스펙(2026-07-28)을 말하는 클라이언트가 옛 스펙(2025-11-25 이하) 서버를 만나면, 알아서 예전의 initialize 핸드셰이크로 되돌아갑니다(fallback). 그래서 업그레이드했다고 기존 연결이 곧바로 깨지진 않습니다. 파이썬 베타 서버도 새 server/discover에 응답하면서 동시에 옛 initialize 핸드셰이크를 계속 지원합니다.

타입스크립트 v2에는 코드 자동 변환 도구(codemod)도 딸려 옵니다.

{{< highlight bash >}} # TS v1 → v2 자동 마이그레이션 (현재 폴더에 적용) npx @modelcontextprotocol/codemod@beta v1-to-v2 . {{< /highlight >}}
날짜단계
2026-05-21RC 잠금(lock) — 이 시점의 사양으로 SDK·적합성 테스트 검증 시작
2026-07 (지금)Tier 1 SDK 베타 공개 · 약 10주의 검증/피드백 창
2026-07-28정식 사양 발표 예정

지금 당장 프로덕션 서버를 뒤엎을 필요는 없습니다. 다만 새로 만드는 원격 MCP 서버라면 처음부터 stateless를 전제로 설계하는 편이 훨씬 유리하고, 기존 서버도 "상태를 서버 메모리에서 공유 저장소로 빼는" 준비를 미리 해 두면 전환이 매끄럽습니다.

8. 흔한 실수 · 팁

  • 전역 변수에 상태 담기. stateless 서버라면서 도구가 프로세스 전역 dict에 카트를 담으면, 인스턴스를 늘리는 순간 다시 깨집니다. 상태는 반드시 공유 DB·캐시로.
  • "프로토콜 무상태 = 앱 무상태"로 오해. 둘은 별개입니다. 앱 상태는 명시적 핸들로 얼마든지 유지합니다. 무상태가 된 건 프로토콜 코어지 여러분의 비즈니스 로직이 아닙니다.
  • RC를 확정본으로 착각. 아직 RC입니다. 정식 발표(2026-07-28) 전까지 세부가 바뀔 수 있으니, 프로덕션 반영은 정식본과 안정 SDK를 기다리는 게 안전합니다.
  • 새 라우팅 헤더 무시. 게이트웨이·LB 앞단을 직접 운영한다면 Mcp-Method·Mcp-Name 헤더로 라우팅·레이트리밋을 걸 수 있는 새 여지를 챙기세요. 본문을 파싱하던 로직을 걷어낼 기회입니다.
  • 폐기 기능에 새로 의존하기. 지금 새 서버를 만들며 Sampling·Roots·Logging에 기대는 설계는 피하세요. 12개월 유예가 있어도 새로 짓는 코드가 폐기 예정 기능에 묶일 이유는 없습니다.
  • 핸들을 추측 가능하게 발급. cart_id가 인자로 오간다는 건, 남의 핸들을 넘겨 상태를 훔쳐볼 수도 있다는 뜻입니다. 순번(1, 2, 3…) 대신 추측 어려운 값을 쓰고, 호출자 권한을 반드시 확인하세요.

9. 마무리 (TL;DR)

  • 무엇: MCP 2026-07-28 스펙 RC의 핵심은 stateless 코어(SEP-2575). initialize 핸드셰이크와 Mcp-Session-Id를 없애 프로토콜에서 세션을 걷어냈다.
  • 왜: 세션 때문에 원격 MCP 서버는 sticky 라우팅이나 공유 세션 저장소 없이는 수평 확장이 어려웠다. 세션이 사라지니 어떤 요청이든 아무 인스턴스로나 갈 수 있어, 평범한 라운드로빈 LB 뒤에 그냥 여러 대 세우면 된다.
  • 어떻게: 연결 시 한 번 주던 정보(버전·클라이언트·기능)를 매 요청 _meta로 보내고, 필요하면 server/discover로 조회. 라우팅용 Mcp-Method·Mcp-Name 헤더가 추가됐다.
  • 상태가 필요하면: 프로토콜은 무상태지만 앱은 아니어도 된다. 서버가 cart_id 같은 명시적 핸들을 발급하고 모델이 인자로 되돌려주는, REST가 늘 쓰던 방식으로 상태를 잇는다.
  • 호환: 새 클라이언트는 옛 서버를 만나면 initialize로 자동 폴백. Python 2.0.0b1·TS v2 베타·Go v1.7.0-pre.1 등 SDK 베타가 나왔고, TS에는 v1→v2 codemod가 있다. 정식 발표는 2026-07-28.
  • 지금 할 일: 새 원격 서버는 stateless 전제로 설계하고, 상태는 서버 메모리 대신 공유 저장소로 빼 두자. 그 외 Sampling·Roots·Logging 폐기(12개월 유예)도 체크.
