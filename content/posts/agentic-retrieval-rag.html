"RAG는 죽었다." 2026년에도 이 문장은 분기마다 한 번씩 타임라인을 덮습니다. 그런데 정작 현장에서 RAG를 걷어낸 팀은 거의 없습니다. 대신 모양이 바뀌고 있습니다.

— 죽은 건 "문서를 미리 다 쪼개 벡터DB에 넣고, 질문 한 번에 한 방으로 꺼내 쓰던" 그 방식이지, 외부 지식을 끌어다 쓰는 일 자체가 아닙니다.

외부 지식을 LLM에 붙이는 가장 흔한 방법이 RAG(Retrieval-Augmented Generation, 검색 증강 생성)입니다. 모델이 학습하지 못한 사내 문서나 최신 정보를, 질문할 때 찾아서(retrieval) 프롬프트에 끼워 넣고 답하게 하는(generation) 패턴이죠. 챗봇에 "우리 회사 환불 규정"을 물어보면 그럴듯하게 답하는 서비스 대부분이 이 구조입니다.

그런데 이번 주에도 "AI 에이전트는 더 이상 벡터 검색이 필요 없다", "컨텍스트 아키텍처가 RAG를 대체한다" 같은 글이 실무자 사이에서 돌았습니다. 결정적인 사연 하나가 자주 인용됩니다. Anthropic이 Claude Code에서 벡터 검색(임베딩·벡터DB·청킹)을 통째로 들어내고 그냥 grep으로 갈아탔는데, "압도적으로 더 잘 됐다"는 겁니다. 동시에 RAG 진영에서는 "그건 코드 검색이라 특수한 경우다, grep은 토큰을 너무 태운다"는 반박이 올라옵니다.

이 글은 그 논쟁의 지형을 junior 개발자가 따라올 수 있게 정리합니다. 먼저 클래식 벡터 RAG가 왜 등장했고 어디서 깨지는지 보고, 품질을 끌어올리는 고전 레버(하이브리드 검색·리랭킹)를 짚은 뒤, 요즘 새 기본값이 됐다는 에이전틱 검색(agentic retrieval)이 어떻게 다른지 코드와 그림으로 비교합니다. 결론부터 말하면 정답은 "둘 중 하나"가 아니라 "언제 무엇을"입니다.

1. RAG가 풀던 문제 — 1분 복습

LLM은 학습 시점에 본 것만 압니다. 어제 올라온 사내 위키, 비공개 PDF, 방금 바뀐 가격표는 모릅니다. 그렇다고 매번 새 정보로 모델을 다시 학습시키는 건 비싸고 느립니다. 그래서 나온 절충이 RAG입니다. 모델을 바꾸지 말고, 답할 때 필요한 자료를 찾아 프롬프트에 넣어 주자는 발상이죠.

클래식 벡터 RAG는 두 단계로 나뉩니다. 미리 해 두는 색인(indexing)과, 질문이 올 때마다 도는 검색·생성(retrieval & generation)입니다.

여기서 핵심 부품이 임베딩(embedding)입니다. 텍스트를 의미가 비슷할수록 가까이 놓이는 숫자 벡터로 바꾸는 모델이죠. "환불"과 "결제 취소"는 글자가 달라도 벡터 거리가 가깝습니다. 그래서 키워드가 정확히 일치하지 않아도 의미로 관련 문서를 찾을 수 있습니다. 이게 벡터 검색(semantic search, 의미 검색)의 힘입니다.

2. 클래식 벡터 RAG는 어디서 깨지나

데모는 잘 됩니다. 문제는 프로덕션입니다. 클래식 파이프라인은 "질문 한 번 → 검색 한 번 → 그걸로 끝"이라 한 번에 잘못 꺼내면 그대로 틀린 답이 나갑니다. 대표적인 약점을 정리하면 이렇습니다.

  • 청킹이 의미를 자른다. 문서를 기계적으로 500자씩 자르면, 표 하나가 두 청크로 갈라지거나 "단, 다음 경우는 제외한다" 같은 결정적 단서가 잘려 나갑니다. 검색은 성공해도 핵심이 빠진 조각이 옵니다.
  • 의미 검색이 정확한 단어에 약하다. 제품 코드 SKU-7741, 함수명 parseInvoice, 오류 코드 E-302처럼 "정확히 그 토큰"이 중요한 질의에서 벡터 검색은 의외로 헛다리를 짚습니다. 의미가 비슷한 다른 코드를 가까이 놓아 버리거든요.
  • top-k의 함정. 무조건 k개를 꺼내므로, 관련 문서가 없어도 "그나마 가까운" 무관한 청크를 채워 줍니다. 모델은 그걸 근거로 그럴듯한 거짓말(환각)을 합니다.
  • 긴 컨텍스트의 'lost in the middle'. 청크를 많이 끼워 넣으면 정작 정답이 프롬프트 가운데 묻혀 모델이 놓칩니다. 이건 Liu 등(arXiv 2307.03172)이 보여 준 현상으로, 모델은 입력의 앞·뒤는 잘 보지만 가운데 정보는 흘립니다. 토큰이 늘수록 품질이 떨어지는 '컨텍스트 부패(context rot)'와 같은 결입니다.
  • 색인 운영 비용. 문서가 바뀔 때마다 다시 청킹·임베딩·업서트해야 합니다. 임베딩 모델을 더 좋은 걸로 바꾸면 전체를 다시 임베딩해야 하고, 청킹 전략을 손대도 마찬가지입니다. 이 파이프라인 자체가 운영 부담입니다.
가장 흔한 착각: "검색이 됐으니 맞는 청크다." 벡터 검색은 가까운 걸 줄 뿐, 맞는 걸 준다고 보장하지 않습니다. 검색 품질을 따로 측정(eval)하지 않으면, 잘못 꺼낸 근거 위에 쌓인 그럴듯한 오답을 잡을 수 없습니다.

3. 고전 레버 — 하이브리드 검색 + 리랭킹

에이전틱 검색으로 넘어가기 전에, 클래식 RAG를 "그냥 잘 만드는" 두 레버를 알아야 합니다. 이건 2026년에도 여전히 품질 향상의 8할을 차지하는 기본기입니다.

① 하이브리드 검색(hybrid search). 의미 검색(dense)과 키워드 검색(sparse, 보통 BM25)을 둘 다 돌려 결과를 합칩니다. 의미 검색은 "환불≈결제 취소"를 잡고, 키워드 검색은 SKU-7741 같은 정확한 토큰을 잡습니다. 서로의 약점을 메우죠. 실무에서 "검색이 영 이상하다" 싶을 때 첫 번째로 켜 보는 스위치입니다.

② 리랭킹(reranking). 1차로 후보를 넉넉히 꺼낸 뒤, 더 정교한 모델(cross-encoder reranker)로 질문과 각 청크를 쌍으로 비교해 다시 점수를 매기고 상위 몇 개만 남깁니다. 1차 벡터 검색은 빠르지만 거칠고, 리랭커는 느리지만 정확합니다. 둘을 깔때기처럼 쌓는 게 핵심입니다.

실무 경험칙(rule of thumb): RAG 실무 가이드들이 공통으로 권하는 출발점은 "하이브리드로 20~50개 꺼내 → 리랭커로 5개로 → 3~5개를 LLM에." 리랭커만 끼워도 답변 품질 지표가 눈에 띄게 오른다는 보고가 많습니다(가이드에 따라 대략 15~30% 개선). 정확한 수치는 데이터마다 다르니, 반드시 직접 측정하세요. 숫자 자체보다 "넓게 꺼내 정밀하게 거른다"는 구조가 핵심입니다.

4. 에이전틱 검색 — 에이전트가 직접 찾는다

여기까지가 "검색을 잘하는 법"이었습니다. 그런데 클래식 RAG에는 구조적 한계가 하나 남습니다. 검색이 딱 한 번, 그것도 LLM이 답을 보기 전에 일어난다는 점입니다. 사람은 안 그러죠. 한 번 찾아보고 부족하면 검색어를 바꿔 다시 찾고, 찾은 걸 읽다가 새 단서가 나오면 또 찾습니다.

에이전틱 검색(agentic retrieval)은 바로 이 "다시 찾기"를 LLM에게 맡깁니다. 검색을 파이프라인의 고정 단계가 아니라 에이전트가 호출하는 도구(tool)로 노출하고, 언제·무엇을·몇 번 검색할지 모델이 스스로 정하게 합니다. 검색·평가·재검색의 루프죠. 학술적으로는 reflection(자기 점검)·planning(계획)·tool use(도구 사용) 같은 에이전트 패턴을 RAG에 얹은 것으로, Singh 등의 Agentic RAG 서베이(arXiv 2501.09136)가 그 갈래를 정리합니다.

가장 화제가 된 사례가 Claude Code입니다. Anthropic은 초기엔 코드베이스를 임베딩해 벡터DB에 넣는 RAG를 썼지만, 결국 들어내고 파일시스템 도구(glob으로 파일 찾기, grep으로 내용 검색, read로 특정 파일 읽기)로 바꿨습니다. Anthropic은 이를 agentic search(에이전틱 검색)라 부릅니다. Boris Cherny(Anthropic)는 "압도적으로 더 잘 됐고, 그게 놀라웠다"고 말했습니다. 이유는 이렇습니다.

  • 색인이 필요 없다. 임베딩 모델, 벡터DB, 재색인 파이프라인이 통째로 사라집니다. 코드가 바뀌어도 다시 임베딩할 일이 없습니다. grep은 언제나 지금의 파일을 봅니다.
  • just-in-time 검색. 미리 다 끌어다 놓지 않고, 필요한 순간에 필요한 만큼만 읽습니다. Anthropic의 컨텍스트 엔지니어링 글(2025-09)은 이를 "가벼운 식별자(파일 경로 등)만 들고 다니다가 런타임에 도구로 동적으로 불러온다"고 설명합니다.
  • 코드와 궁합이 좋다. 코드 검색은 정확한 식별자(함수명·심볼)가 결정적이라, 의미 검색보다 키워드/구조 검색이 잘 맞습니다.

5. 실전: 코드로 두 방식 비교

말로만 들으면 추상적이니 파이썬 의사코드로 골격만 봅시다. 먼저 클래식 RAG + 리랭킹입니다. 검색이 한 번, 흐름이 직선인 게 보입니다.

{{< highlight python >}} # 클래식 RAG: 검색 1회 → 리랭킹 → 생성 (흐름이 직선) def classic_rag(question: str) -> str: q_vec = embed(question) # 질문을 벡터로 # ① 하이브리드로 넓게: 의미(dense) + 키워드(BM25) dense = vector_db.search(q_vec, top_k=25) sparse = bm25.search(question, top_k=25) candidates = dedup(dense + sparse) # 후보 ~50개 # ② 리랭커로 정밀하게 좁히기 (질문↔청크 쌍 점수) ranked = reranker.score(question, candidates) top = ranked[:5] # 상위 5개만 # ③ 한 번 모은 근거로 곧장 답 생성 — 다시 찾지 않는다 context = "\n\n".join(c.text for c in top) return llm(f"근거:\n{context}\n\n질문: {question}") {{< /highlight >}}

이제 에이전틱 검색입니다. 검색을 도구로 주고, 모델이 충분하다고 판단할 때까지 루프를 돕니다. 핵심은 while과 "검색을 모델이 호출한다"는 점입니다.

{{< highlight python >}} # 에이전틱 검색: 모델이 도구로 직접·반복 검색 (흐름이 루프) TOOLS = [grep_tool, read_file_tool, search_docs_tool] # 검색을 '도구'로 노출 def agentic_retrieval(question: str) -> str: messages = [{"role": "user", "content": question}] for _ in range(MAX_STEPS): # 무한 루프 방지용 상한 # 모델이 스스로 "검색할지/무엇으로/이제 답할지"를 결정 resp = llm_with_tools(messages, tools=TOOLS) if resp.tool_calls: # 모델이 검색을 더 하기로 함 for call in resp.tool_calls: result = run_tool(call) # 예: grep("parseInvoice") messages.append(tool_result(call, result)) continue # 결과를 보고 다시 판단 ↺ return resp.content # 충분 → 최종 답 {{< /highlight >}}

두 코드의 차이가 한눈에 보입니다. 클래식은 개발자가 검색 시점·횟수를 코드에 박아 넣고, 에이전틱은 모델이 런타임에 정합니다. 후자는 똑똑한 대신 토큰을 더 쓰고 지연이 길며, 루프가 폭주하지 않도록 MAX_STEPS 같은 안전장치가 필요합니다.

둘은 적이 아닙니다. 에이전트가 호출하는 search_docs_tool 안에 ③장의 하이브리드+리랭킹 RAG를 그대로 넣을 수 있습니다. 즉 "에이전틱 검색 vs RAG"가 아니라 "RAG를 도구로 감싸 에이전트가 반복 호출"하는 형태가 실무의 현실적인 절충입니다.

6. "grep이 더 낫다"는 진짜인가 — 논쟁 정리

Claude Code 사례가 워낙 강렬해서 "이제 벡터DB는 끝"이라고 단정하기 쉽지만, 반론도 만만치 않습니다. 대표적으로 벡터DB 진영(예: Milvus 블로그)은 "grep-only 검색은 토큰을 너무 태운다"고 지적합니다. 양쪽 주장을 공정하게 놓으면 이렇습니다.

쟁점에이전틱 검색(grep·도구)클래식 벡터 RAG
최신성항상 현재 파일 — 재색인 불필요색인 시점 기준 — 갱신 파이프라인 필요
정확한 토큰(코드·ID)강함(키워드·구조 검색)약할 수 있음(의미 검색의 약점)
운영 복잡도낮음 — 인프라가 적음높음 — 임베딩·벡터DB·청킹 관리
대규모 코퍼스여러 번 검색하며 토큰·지연 증가한 번에 의미로 좁혀 효율적
의미 검색(동의어·개념)약함 — 같은 뜻 다른 표현을 놓침강함 — 임베딩의 본령
지연·비용루프마다 LLM 호출 → 누적검색 1회 → 예측 가능

정리하면, Claude Code가 grep으로 이긴 건 코드 검색이라는 특수성이 큽니다. 코드는 정확한 식별자가 결정적이고, 파일시스템이라는 잘 구조화된 인덱스가 공짜로 있으며, 모델이 코드를 읽고 다음 검색어를 잘 고릅니다. 반대로 방대한 산문 지식베이스에서 개념적 동의어로 찾아야 하는 고객지원·법무·연구 문서 검색이라면, 의미 검색과 리랭킹의 가치가 여전히 큽니다. "RAG는 죽었다"가 아니라 "도메인에 맞는 검색을 골라라"가 맞습니다.

7. 그래서 뭘 써야 하나 — 결정 기준

처음부터 에이전틱 루프와 지식그래프를 얹지 마세요. 실무 가이드들의 공통 조언은 "가장 단순한 것부터, 지표가 부족함을 증명할 때만 복잡도를 더하라"입니다. 아래 순서로 올라가면 대체로 안전합니다.

  1. 기본값: 하이브리드 검색(dense + BM25) + 리랭커. 대부분의 RAG는 여기서 충분히 좋아집니다.
  2. 정확한 토큰이 중요(코드·로그·ID)하고, 잘 구조화된 인덱스(파일시스템·DB)가 있다면: 에이전틱 검색(grep·도구) 우선. 색인 운영을 통째로 덜 수 있습니다.
  3. 질문이 복잡해 여러 번 찾아야 풀린다면: 검색을 도구로 감싸 에이전트가 반복 호출. ①의 RAG를 도구 안에 그대로 넣으세요.
  4. 코퍼스가 자주 바뀌거나 작다면: 색인 부담이 큰 벡터DB 대신, 긴 컨텍스트에 바로 넣거나 just-in-time으로 읽는 편이 단순합니다.
무엇을 고르든 평가(eval)가 먼저입니다. "검색이 옳은 청크를 가져왔는가"(retrieval)와 "그걸로 답이 맞았는가"(generation)를 따로 측정하세요. 작은 골든 질문셋이라도 있으면, 리랭커 추가나 에이전틱 전환이 정말 이득인지 숫자로 판단할 수 있습니다. 측정 없는 아키텍처 교체는 그냥 취향입니다.

8. 흔한 실수 · 팁

  • 리랭커를 빼먹는다. 가장 가성비 좋은 한 수인데 자주 건너뜁니다. 1차 검색을 넓게(20~50) 꺼내고 리랭커로 5개로 좁히는 깔때기만 끼워도 체감 품질이 달라집니다.
  • 청킹을 대충 한다. 고정 길이로 자르면 표·목록·"단, 예외" 조항이 갈라집니다. 문단·섹션 경계나 의미 변화 지점에서 자르고, 청크에 출처·제목 같은 메타데이터를 함께 넣으세요.
  • 벡터 검색만 믿는다. 정확한 코드·ID·고유명사가 섞인 도메인이라면 BM25를 반드시 함께 켜세요. 의미 검색 단독은 정확 일치에 약합니다.
  • 에이전틱 루프에 상한이 없다. 모델이 "한 번만 더"를 반복하면 토큰과 지연이 폭주합니다. MAX_STEPS와 호출 횟수 제한, 그리고 트레이스 로깅으로 무엇을 몇 번 검색했는지 꼭 남기세요.
  • 검색 품질을 측정하지 않는다. 스키마처럼 "검색이 됐다"는 사실은 정답을 보장하지 않습니다. retrieval과 generation을 분리해 평가해야 어디가 깨졌는지 보입니다.
  • "RAG는 죽었다"를 곧이곧대로 받는다. 죽은 건 특정 구현(한 방 벡터 검색)이지 검색 증강 자체가 아닙니다. 에이전틱 검색도 결국 외부 지식을 끌어와 답을 만드는 RAG의 한 형태입니다.

9. 마무리 (TL;DR)

  • RAG는 모델을 다시 학습시키지 않고, 답할 때 외부 지식을 찾아 프롬프트에 넣는 패턴입니다. 죽지 않았습니다 — 모양이 바뀌었을 뿐입니다.
  • 클래식 벡터 RAG는 "미리 색인 → 질문 한 번에 top-k 한 방 검색"이라, 청킹 손실·정확 토큰 약점·top-k 함정·lost in the middle에서 깨집니다.
  • 클래식을 살리는 두 레버는 하이브리드 검색(dense+BM25)과 리랭킹입니다. "넓게 꺼내(20~50) 정밀하게 거른다(→5)"가 공식입니다.
  • 에이전틱 검색은 검색을 도구로 노출하고, 언제·무엇을·몇 번 찾을지 모델이 정해 루프를 돕니다. Claude Code가 벡터DB를 grep으로 갈아탄 게 대표 사례입니다.
  • grep이 이긴 건 코드 검색의 특수성(정확한 식별자·공짜 인덱스) 덕이 큽니다. 방대한 산문·개념 검색에선 의미 검색·리랭킹이 여전히 강합니다.
  • 실무 절충은 "RAG를 도구로 감싸 에이전트가 반복 호출"입니다. 둘은 적이 아닙니다.
  • 무엇을 고르든 retrieval과 generation을 따로 평가하세요. 측정 없는 아키텍처 교체는 취향일 뿐입니다.
