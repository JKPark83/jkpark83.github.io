## 1. Cordon: Semantic Transactions for Tool-Using LLM Agents
- **arXiv:** [2606.17573](https://arxiv.org/abs/2606.17573) · cs.AI · 제출 2026-06-17 (오늘자 최신)
- **핵심 기여:** 멀티스텝 툴 호출을 고립된 RPC가 아니라 **"의미 트랜잭션"**으로 감싸, 태스크 단위 commit·rollback·복구·감사(audit) 경계를 부여하는 런타임.
- **방법:** 외부·비가역 효과를 commit 전에 스테이징 → 툴 의도와 결과 계보(lineage)를 되돌릴 수 있는 로컬 상태·위임 권한·감사 메타데이터에 결속 → commit 직전 스텝 간 정책 경계 검증.
- **결과(원문 보고):** 위험 동반 멀티툴 워크플로 45개 전부에서 정책 위반 효과를 commit 전 차단. 태스크 시간 24.6~27.9%·토큰 23.6~28.4% 절감, rollback 지연 중앙값 ~4.17ms, 일반 벤치에선 평범한 에이전트와 측정 오차 내.
- 🛠 **빌더 관점:** 파일·명령·외부 API를 *실제로 건드리는* 에이전트에 staging+rollback+audit을 정확도 손실 없이 얹는 구체적 청사진.
- ⚠️ **한계:** 자체 구성 45개 위험 워크플로 + 벤치 2종 평가라 실제 효과 유형·적대적 케이스 커버리지는 아직 좁음. (저자·소속은 인덱스 미확인 → 생략)

## 2. SING: Synthetic Intention Graph for Scalable Active Tool Discovery in LLM Agents
- **arXiv:** [2606.16591](https://arxiv.org/abs/2606.16591) · cs.AI/cs.CL · 제출 2026-06-15 (v2 06-16)
- **저자/소속:** Qiao Xiao(Cornell), Haochen Shi·Yisen Gao·Wenbin Hu·Huihao Jing·Tianshi Zheng 외(HKUST), Ziheng Zhang(Ohio State), Jiaxin Bai(HKBU).
- **핵심 기여:** 사용자 의도·툴 능력·툴 협업 패턴을 잇는 **"의도-툴 그래프"**로, 전체 스키마를 한꺼번에 주입하지 않고 서브골이 생길 때마다 동적으로 툴을 능동 탐색.
- **방법:** 오프라인에서 합성 의도 그래프 구축 → 런타임에 태스크 분해·관찰에 따라 그래프를 순회하며 관련 툴만 노출(open-world·long-horizon 대응).
- **결과:** 3개 벤치·툴 7,471개 통합 코퍼스에서 Global Recall@5 최대 **+59.8%**, 다운스트림 성공률 최대 **+28.9%**. 스키마 노출 토큰 평균 693,574 → 1,298 (**99.8%↓**).
- 🛠 **빌더 관점:** 툴이 수천 개로 늘어도 컨텍스트는 작게, 정답 툴 적중률은 높게 — MCP/대규모 툴 생태계의 정통 고통점 정조준.
- ⚠️ **한계:** 합성 의도 그래프 품질에 의존. 툴 변경 시 그래프 유지·갱신 비용은 미해결.

## 3. MyPCBench: A Benchmark for Personally Intelligent Computer-Use Agents
- **arXiv:** [2606.16748](https://arxiv.org/abs/2606.16748) · cs.AI/cs.CL · 제출 2026-06-15
- **저자/소속:** Lawrence Keunho Jang, Andrew Keunwoo Jang, Jing Yu Koh, Ruslan Salakhutdinov (Carnegie Mellon).
- **핵심 기여:** 17개 모의 실서비스 웹앱 + 풀 데스크톱 스택을 한 캐논 페르소나(드라마 The Office의 "Michael Scott")로 시드한 리눅스 데스크톱에서, 컴퓨터-유즈 에이전트의 **"개인 비서" 역량**을 평가.
- **방법:** OpenClaw 커뮤니티 실제 요청에서 영감을 얻은 184개 태스크를 동일한 computer+bash 툴 표면 위에서 폐쇄·오픈웨이트 6개 모델로 측정.
- **결과(정성):** 현재 컴퓨터-유즈 에이전트는 일반 태스크는 처리하지만, 사용자 맥락·로그인·이력 데이터를 활용해야 하는 **개인화 의존** 태스크에서 뚜렷이 약함. (모델별 수치는 원문 확인 권장)
- 🛠 **빌더 관점:** 데스크톱/어시스턴트 에이전트라면 "일반 성공률"이 아니라 *저장된 개인 컨텍스트 활용*을 평가축으로 가져가야 함을 시사.
- ⚠️ **한계:** 단일 캐논 페르소나·모의 데스크톱이라 다양한 실사용자·라이브 앱 일반화는 미검증.

## 1. 🟪 🎨 `npx skills`로 스킬 배포 — 단, `.agents/skills` 심볼릭링크 함정
크로스 에이전트(Claude Code·Codex·Cursor) 공용으로 스킬을 평문 파일로 add/list/version 하는 CLI가 사실상 배포 채널로 자리잡는 중. 💡 핵심은 *git 없이 팀에 컨벤션을 표준 배포*인데, ⚠️ **CC는 `.claude/skills/`를 읽지만 이 CLI는 `.agents/skills/`에 쓰는** 불일치 버그가 있음([issue #1355](https://github.com/vercel-labs/skills/issues/1355)).

```bash
# 스킬 스캐폴드 + 설명 자동 연결
npx skills add <source>
# Claude Code 타깃, 전역 스코프
npx skills add <source> -a claude-code -g
```

🎯 혼합 툴 팀에서 컨벤션 공유 시 유용. ⚠️ `-a claude-code`가 `.claude/skills/<skill>` 심볼릭링크를 안 만들어 CC가 못 보는 경우가 있으니, 설치 직후 `/skills`로 가시성 확인 또는 수동 심볼릭링크. ⭐ 배포엔 채택하되 #1355 패치 전까진 검증 한 번. → [repo](https://github.com/vercel-labs/skills)

## 2. 🟦 🤖 서브에이전트 "토큰 상각" — 메인 컨텍스트를 작게 유지하는 최고 레버리지

## 3. 🟪 🧩 컴팩션에 핵심 규칙을 맡기지 마라 — 영속 시스템 프롬프트로 이전
💡 Anthropic 컴팩션은 100턴 eval에서 토큰을 ~84% 줄이며 완수율을 유지했지만, 규칙은 단 하나: **중요 규칙을 컴팩션에 의존하지 말고 영속 시스템 프롬프트로 재배치**. 2026 갱신점은 토큰 한계에서 반응적으로 압축하지 말고 **태스크 사이에서 전략적으로** 압축. 📝 불변식("끝내기 전 테스트", "생성 파일 편집 금지")은 대화 중간이 아니라 CLAUDE.md/AGENTS.md/시스템 프롬프트에. 문서는 미리 주입하지 말고 "전부 파일로 노출" 후 `read_file`/`grep`로 점진 인출(A-RAG). 🎯 컴팩션을 자주 치는 멀티턴 에이전트, 장기 세션 후 행동 드리프트. ⚠️ 전략적 압축은 하네스 지원 필요; 자동 컴팩션에만 의존하면 시스템 프롬프트에 고정 안 된 건 유실. ⭐ 휘발성 채팅에만 있는 규칙을 감사해 영속 설정으로 이동. → [awesome-harness-engineering](https://github.com/ai-boost/awesome-harness-engineering)

## 4. 🟪 🧩 단위테스트식 eval 졸업 — 트래젝토리 평가 + LLM-as-Judge
💡 에이전트엔 단위테스트식 평가가 안 맞는다. Anthropic 프레이밍은 **커버리지(무엇을 시도)·정확성(성공했나)·정렬(성공이 의도와 맞나)** 3축을 멀티턴 트래젝토리 전체에서 측정. 📝 툴 선택 정확성·인자 유효성·스텝 수·시간/비용·정책 준수를 트래젝토리로 평가하고, 계산적 통제(린터/테스트)와 추론적 통제(LLM 심판)를 레이어링. 반복 관찰을 자동 judge로 만들어 이후 모든 트레이스에 적용. 🎯 프로덕션 출시 전, "테스트는 통과하는데 실제론 이상하게" 행동할 때. ⚠️ LLM 심판은 사람 라벨로 캘리브레이션 안 하면 편향을 내재화; 트래젝토리 평가는 단위테스트보다 비쌈. ⭐ 실제 트래젝토리 골든셋 소규모 + 툴선택·정렬용 LLM-judge 2~3개로 릴리즈 게이팅 시작. → [awesome-harness-engineering](https://github.com/ai-boost/awesome-harness-engineering)
