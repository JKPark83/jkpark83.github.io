리즈닝 모델에게 "더 깊이 생각해"라고 시키는 건 공짜가 아닙니다. 눈에 보이지 않는 생각 토큰에 매 호출마다 돈과 시간을 지불하고 있고, 어느 지점을 넘으면 그 돈으로 산 건 더 나은 답이 아니라 더 늦고 더 비싼 같은 답입니다.

— "얼마나 똑똑한가"만큼 "얼마나 생각하게 둘 것인가"가 엔지니어링 문제입니다.

리즈닝 모델(reasoning model)을 처음 붙여 본 junior 개발자가 흔히 겪는 일이 있습니다. 데모는 놀랍습니다. 어려운 수학 문제도, 복잡한 코드 디버깅도 척척 풀죠. 그런데 실제 서비스에 올리고 청구서를 받아 보면 눈을 의심하게 됩니다. 분명 짧은 답이 나왔는데 비용은 예상의 몇 배고, 응답은 답답할 만큼 느립니다.

원인은 대개 하나입니다. 리즈닝 모델은 답을 내기 전에 속으로 길게 생각하고, 그 생각도 전부 토큰이라 돈이 나갑니다. 실제로 한 팀은 40자짜리 짧은 텍스트를 처리하는 데 Claude의 extended thinking을 켰다가, 입력 30 토큰·출력 18 토큰인데 생각 토큰만 1,200개가 붙어 비용이 예상의 30배가 나온 사례를 보고했습니다. o3에 500 토큰짜리 답을 요청해도 뒤에서 2,000~5,000개의 reasoning 토큰이 소모되기도 합니다.

더 얄궂은 건, 이렇게 많이 생각한다고 답이 꼭 좋아지지도 않는다는 점입니다. 이걸 오버씽킹(overthinking)이라고 부릅니다. 그래서 요즘 AI 엔지니어링의 화두 하나는 "모델을 얼마나 생각하게 둘 것인가"를 코드로 조절하는 일입니다. 이번 주(2026년 7월 9일) OpenAI가 GPT-5.6를 정식 출시하면서 none부터 max까지 여섯 단계의 reasoning effort와 호출 사이에 추론 상태를 이어 가는 persisted reasoning을 API 전면에 내놓은 것도 같은 흐름입니다. 이 글에서는 그 손잡이들을 하나씩 잡아 보겠습니다.

1. 리즈닝 모델은 왜 비싸고 느린가

먼저 용어부터 정리합시다. 리즈닝 모델은 최종 답을 쓰기 전에 사고 과정(chain-of-thought)을 스스로 길게 펼치도록 훈련된 모델입니다. OpenAI의 o 계열·GPT-5 계열, Google Gemini의 thinking 모드, Anthropic Claude의 extended thinking이 모두 여기에 해당합니다. 사람으로 치면 답을 말하기 전에 머릿속으로 이것저것 따져 보는 단계를 글로 적어 내려가는 셈입니다.

여기서 junior 개발자가 놓치기 쉬운 핵심이 하나 있습니다. 이 "머릿속 계산"도 전부 출력 토큰이라는 점입니다. 화면에 최종 답만 보여도, 그 답을 만들기 위해 생성한 생각 토큰(reasoning token, thinking token이라고도 합니다)은 과금 대상이고, 생성하는 데 실제 시간이 걸립니다. 보이지 않을 뿐 엄연히 존재하는 비용입니다.

왜 토큰 하나가 시간을 잡아먹나? LLM은 토큰을 한 번에 하나씩, 순차적으로 만들어 냅니다(autoregressive). 생각 토큰이 3,000개면 최종 답을 시작하기도 전에 토큰 3,000개를 순서대로 뽑아야 합니다. 그래서 생각이 길어질수록 첫 글자가 나오기까지의 지연이 그대로 늘어납니다.

정리하면 리즈닝 모델의 비용·지연은 보이는 답 길이가 아니라, 보이지 않는 생각 길이가 좌우합니다. 그리고 이 생각 길이는 우리가 어느 정도 조절할 수 있는 값입니다. 그 조절값이 바로 이 글의 주제인 추론 예산(reasoning budget)입니다.

2. 추론 예산이란 무엇인가 — 정확도-토큰 곡선

추론 예산은 말 그대로 "이 문제에 생각 토큰을 최대 얼마까지 쓰게 할 것인가"를 정하는 상한선(또는 강도)입니다. 예산을 늘리면 모델이 더 오래 궁리하고, 줄이면 빨리 결론으로 갑니다.

자연스러운 직관은 "예산을 많이 줄수록 정확해지겠지"입니다. 절반은 맞습니다. 하지만 2026년 들어 여러 연구가 이 직관의 나머지 절반을 깼습니다. 생각 토큰과 정확도의 관계는 계속 오르는 직선이 아니라, 정점을 찍고 다시 내려오는 언덕에 가깝습니다.

연구들이 관찰한 구체적인 현상은 이렇습니다. 한 연구는 대략 7,000 토큰을 넘어서면 모델이 새로운 정답을 발견하기보다 이미 맞힌 답을 뒤집을 확률이 더 높아지는 "오버씽킹 구간"에 들어간다고 봤습니다. AIME 같은 수학 벤치마크에서도 생각 토큰을 늘릴수록 평균 정확도가 오히려 떨어지는 구간이 관찰됐고, 쉬운 문제일수록 그 하락 지점이 더 일찍 찾아왔습니다. 반대로 어떤 기법은 "Wait", "Hmm" 같은 자기 되짚기 토큰을 억제해 사고 길이를 27~51%나 줄였는데도 정확도는 그대로였습니다.

핵심 교훈은 이렇습니다. 생각 토큰은 무조건 많이 주는 게 아니라, 문제 난이도에 맞게 주는 자원입니다. 쉬운 문제에 큰 예산을 주면 비용만 태우고 오히려 틀릴 수 있고, 어려운 문제에 예산을 조이면 성급하게 틀립니다. 그래서 "얼마를 줄까"가 엔지니어링 결정이 됩니다.

3. 세 가지 손잡이 — OpenAI · Gemini · Claude

다행히 주요 API는 저마다 예산을 조절하는 손잡이를 제공합니다. 접근 방식이 조금씩 달라서, 세 가지를 나란히 보면 감이 잡힙니다.

OpenAI — reasoning_effort 단계 고르기

OpenAI는 토큰 숫자 대신 강도 단계를 고르게 합니다. GPT-5.6 기준으로 none · minimal · low · medium · high · xhigh에 더해, 이번에 가장 어려운 작업용 max가 추가됐습니다. GPT-5.5의 기본값은 medium입니다. 숫자를 직접 정할 필요 없이 "이 정도로 생각해"만 알려 주면 되는 방식입니다.

{{< highlight python >}} from openai import OpenAI client = OpenAI() # 라우팅·분류·간단 추출처럼 가벼운 작업 → 낮은 강도로 빠르고 싸게 resp = client.responses.create( model="gpt-5.6", reasoning={"effort": "low"}, # none/minimal/low/medium/high/xhigh/max input="이 문의를 '환불/배송/기타' 중 하나로 분류만 해줘: ...", ) # 어려운 디버깅·다단계 설계 → 높은 강도로 품질 우선 resp = client.responses.create( model="gpt-5.6", reasoning={"effort": "high"}, # 필요할 때만 올린다 input="이 동시성 버그의 근본 원인을 단계별로 진단해줘: ...", ) {{< /highlight >}}
고르는 기준. 추출·라우팅·분류·간단한 재작성이면 low(또는 minimal), 문제를 진단하거나 선택지를 비교하거나 계획을 세우고 코드를 따져야 하면 medium~high. max는 정말 품질이 전부인 가장 어려운 작업에만 아껴 씁니다. 지연에 민감한 자동완성·실시간 UI라면 none/minimal이 정답입니다.

Gemini — thinkingBudget 토큰으로 지정

Gemini 2.5는 반대로 토큰 숫자를 직접 적습니다. 상한이 눈에 보여 통제가 직관적입니다. 특수값 두 개만 외우면 됩니다 — 0은 생각을 아예 끄고(가장 빠르고 쌈), -1은 동적 모드로 모델이 문제 난이도를 보고 스스로 정합니다. 2.5 Flash 기준 최대는 24,576 토큰입니다.

{{< highlight python >}} from google import genai from google.genai import types client = genai.Client() resp = client.models.generate_content( model="gemini-2.5-flash", contents="다음 SQL 쿼리가 왜 느린지 분석하고 인덱스를 제안해줘: ...", config=types.GenerateContentConfig( thinking_config=types.ThinkingConfig( thinking_budget=2048, # 0=끄기, -1=동적, 여기선 2048 토큰 상한 ) ), ) {{< /highlight >}}

Claude — budget_tokens로 목표치 지정

Claude의 extended thinking은 budget_tokens로 목표치를 줍니다. 두 가지만 기억하면 됩니다. 최소 1,024이고, 반드시 max_tokens보다 작아야 합니다. 그리고 이 값은 딱딱한 상한이 아니라 목표(target)라서, 쉬운 문제면 예산을 다 안 쓰기도 합니다. 그래서 공식 권장도 "작게 시작해 조금씩 올려 가며 스위트 스팟을 찾으라"입니다.

{{< highlight python >}} import anthropic client = anthropic.Anthropic() msg = client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, # thinking + 최종 답의 총합보다 커야 함 thinking={ "type": "enabled", "budget_tokens": 2048, # 최소 1024, max_tokens 보다 작게 }, messages=[{"role": "user", "content": "이 증명의 논리적 허점을 찾아줘: ..."}], ) {{< /highlight >}}
흔한 함정. budget_tokens를 max_tokens와 같거나 크게 잡으면 최종 답을 쓸 토큰이 남지 않습니다. max_tokens = 생각 예산 + 실제 답 길이 + 여유로 넉넉히 잡으세요. 또 32k를 넘기는 큰 예산은 요청이 오래 걸려 타임아웃에 걸리기 쉬우니 배치 처리가 권장됩니다.

4. budget forcing — 직접 늘리고 줄이기

API 손잡이가 없는 오픈 모델을 직접 서빙하거나, 예산을 더 세밀하게 다루고 싶을 때 쓰는 고전적인 기법이 budget forcing입니다. 2025년 s1 논문(Muennighoff 외)이 제안한, 놀랄 만큼 단순한 방법입니다.

아이디어는 이렇습니다. 리즈닝 모델은 생각을 마칠 때 특별한 사고 종료 토큰(예: </think>)을 냅니다. budget forcing은 이 경계를 인위적으로 건드립니다.

  • 줄이기: 정해 둔 토큰 수에 도달하면 사고를 강제로 끝내고 종료 토큰을 밀어 넣어 곧장 답을 쓰게 합니다.
  • 늘리기: 모델이 생각을 끝내려 하면 종료 토큰을 막고 대신 "Wait"를 덧붙여 생각을 이어 가게 합니다. 그러면 모델이 자기 답을 다시 점검하다가 틀린 단계를 바로잡기도 합니다.

개념을 코드로 옮기면 이런 모습입니다. vLLM 같은 엔진으로 오픈 리즈닝 모델을 서빙한다고 할 때, 생각을 늘리는 쪽을 단순화하면 이렇게 됩니다.

{{< highlight python >}} # budget forcing '늘리기'의 핵심 아이디어 (개념 코드) MAX_WAITS = 2 # 최대 몇 번까지 더 생각하게 밀어붙일지 think_end = "" prompt = build_prompt(question) + "" waits = 0 while True: # 사고 종료 토큰 직전까지 생성 (stop 시퀀스로 앞에서 멈춤) out = model.generate(prompt, stop=[think_end]) prompt += out.text if waits < MAX_WAITS: # 끝내려는 걸 막고 'Wait'를 붙여 다시 생각하게 한다 prompt += " Wait" waits += 1 else: # 예산 소진 → 종료 토큰을 넣고 최종 답을 받는다 prompt += think_end answer = model.generate(prompt) break {{< /highlight >}}
언제 쓰나. OpenAI·Gemini·Claude처럼 손잡이를 제공하는 상용 API라면 굳이 이걸 손으로 구현할 필요는 없습니다. budget forcing은 손잡이가 없는 오픈 모델(예: 자체 서빙하는 R1 계열)을 다룰 때, 또는 "Wait 몇 번" 같은 정밀한 제어가 필요할 때 진가를 발휘합니다.

5. 실전 — 난이도에 따라 예산 자동 배분

여기까지 왔으면 자연스러운 다음 질문이 나옵니다. "그럼 요청마다 예산을 어떻게 정하지?" 모든 요청에 high를 박으면 비용이 터지고, 전부 low로 두면 어려운 요청에서 틀립니다. 정답은 요청의 난이도를 먼저 가늠하고, 거기에 맞는 예산을 배분하는 것입니다. 이 패턴을 흔히 적응형 예산(adaptive budget) 또는 라우팅이라고 부릅니다.

가장 실용적인 형태는 "일단 싸게 시도하고, 미덥지 않으면 예산을 올려 재시도"하는 에스컬레이션(escalation) 패턴입니다. 대부분의 요청은 쉬운 쪽에 몰려 있으니, 이렇게만 해도 평균 비용이 크게 내려갑니다.

{{< highlight python >}} def solve_with_adaptive_budget(question: str) -> str: # 1단계: 낮은 강도로 싸고 빠르게 먼저 시도 ans, confident = try_solve(question, effort="low") if confident: return ans # 대부분 여기서 끝난다 (싸게) # 2단계: 미덥지 않으면 예산을 올려 재시도 ans, confident = try_solve(question, effort="medium") if confident: return ans # 3단계: 그래도 아니면 최고 강도 (드물게 발생 → 비용 통제됨) ans, _ = try_solve(question, effort="high") return ans def try_solve(question, effort): resp = client.responses.create( model="gpt-5.6", reasoning={"effort": effort}, input=f"{question}\n\n확신이 없으면 답 끝에 [UNSURE]를 붙여라.", ) text = resp.output_text return text, "[UNSURE]" not in text # 자기 신고를 확신 신호로 사용 {{< /highlight >}}
왜 효과적인가. 실제 트래픽은 쉬운 요청이 압도적으로 많습니다. 그 요청들은 1단계에서 싸게 끝나고, 진짜 어려운 소수만 비싼 단계로 올라갑니다. "전부 high"와 비교하면 품질은 비슷하게 지키면서 평균 비용·지연을 크게 줄일 수 있습니다. Gemini의 thinking_budget=-1(동적 모드)은 이 아이디어를 모델 안에 내장한 버전이라고 볼 수 있습니다.

6. 흔한 실수 · 팁

실수무슨 일이 벌어지나대신 이렇게
모든 요청에 최고 강도 쉬운 요청까지 생각 토큰을 잔뜩 태워 비용·지연 폭증, 오버씽킹으로 되레 틀리기도 기본은 낮게, 어려운 요청만 올리는 적응형 예산·에스컬레이션
생각 토큰을 비용에서 빼먹음 보이는 답만 보고 원가를 계산 → 청구서에서 몇 배 차이로 놀람 응답의 usage에서 reasoning/thinking 토큰을 꼭 로깅·모니터링
budget_tokens ≥ max_tokens 생각하다 예산을 다 써서 최종 답을 쓸 토큰이 없음 → 답이 잘리거나 빔 max_tokens를 생각 예산 + 답 길이 + 여유로 넉넉히
쉬운 작업에 리즈닝 모델 포맷팅·단순 추출에도 수천 생각 토큰 → 40자 작업 비용 30배 사례 가벼운 작업은 effort=none/minimal 또는 일반(비추론) 모델
지연에 민감한 경로에 큰 예산 생각 토큰이 순차 생성돼 첫 응답까지 몇 초씩 지연 실시간 UI·자동완성은 예산을 끄거나 최소로
팁: 예산은 "튜닝 대상"입니다. 한 번 정하고 잊지 마세요. 대표 요청 몇십 개를 골라 예산을 low → medium → high로 바꿔 가며 정확도와 비용을 같이 재 보면, 우리 서비스의 스위트 스팟이 보입니다. 정확도가 더 안 오르기 시작하는 지점 바로 앞이 대개 최적입니다.

7. 마무리 (TL;DR)

  • 리즈닝 모델의 비용·지연은 보이지 않는 생각 토큰이 좌우합니다. 짧은 답에도 수천 토큰을 태울 수 있어, 쉬운 작업에 잘못 쓰면 비용이 수십 배로 뜁니다.
  • 생각은 많을수록 좋은 게 아닙니다. 정확도-토큰 곡선에는 정점(스위트 스팟)이 있고, 그 뒤 오버씽킹 구간에서는 정확도가 떨어지고 비용만 늘어납니다.
  • 세 API의 손잡이: OpenAI reasoning_effort(none~max 단계), Gemini thinkingBudget(토큰 수, 0=끄기·−1=동적), Claude budget_tokens(최소 1024, max_tokens보다 작게).
  • budget forcing은 사고 종료 토큰을 조작해(Wait 덧붙이기 / 강제 종료) 예산을 직접 늘리고 줄이는 기법으로, 손잡이 없는 오픈 모델에 유용합니다.
  • 실전 정답은 적응형 예산. 낮은 예산으로 먼저 시도하고, 확신이 낮을 때만 예산을 올려 재시도하면 품질을 지키면서 평균 비용을 크게 낮춥니다.
  • 예산은 한 번 정하고 끝이 아니라, 대표 요청으로 정확도·비용을 재 가며 튜닝하는 값입니다.
