에이전트는 매 턴마다 똑같은 시스템 프롬프트와 똑같은 도구 정의를 처음부터 다시 모델에 보냅니다. 수천 토큰짜리 지침을 100번 대화하면 100번 다시 계산하는 셈이죠.

— 그 반복되는 앞부분을 한 번만 계산하고 재사용하자는 게 프롬프트 캐싱입니다. 비용의 문제만이 아니라, "무엇을 어디에 두느냐"가 캐시가 먹히느냐 깨지느냐를 가릅니다.

LLM에 요청을 보낼 때마다 우리는 같은 말을 반복합니다. "너는 사내 규정 도우미야"로 시작하는 시스템 프롬프트, 열몇 개의 도구 정의, 참고 문서 몇 장 — 이런 고정된 앞부분은 사용자가 무엇을 묻든 매번 똑같이 앞에 붙습니다. 그런데 LLM은 입력 전체를 매번 처음부터 다시 계산합니다. 똑같은 8,000토큰짜리 지침을 요청마다 새로 처리하면, 그만큼 돈과 시간을 반복해서 태우는 겁니다.

프롬프트 캐싱(prompt caching)은 이 낭비를 없앱니다. 반복되는 앞부분을 처음 한 번만 계산해 저장해 두고, 다음 요청부터는 그 저장본을 재사용합니다. 그 결과 반복 토큰은 원래 값의 최대 90% 싼값에 처리되고, 응답이 나오기 시작하는 시간(TTFT, time-to-first-token)도 짧아집니다. 2026년 현재 프로덕션 LLM 엔지니어링에서 가장 레버리지가 큰 비용 절감 수단으로 꼽히는 이유죠.

이번 주에도 이 주제가 실무자 타임라인에 다시 올라왔습니다. "토큰을 많이 태울수록 생산적"이라는 tokenmaxxing 문화가 꺾이고 "적은 토큰으로 잘하자"는 token discipline 담론이 번지면서, 프롬프트 캐싱을 제대로 쓰는 법이 다시 화제가 됐습니다. 특히 에이전트처럼 대화가 길어지는 상황에서 "캐시를 깨지 않게 설계하라(Don't break the cache)"는 이야기가 논문과 프레임워크 블로그에서 함께 오르내렸습니다. 마침 naive하게 전부 캐싱하면 오히려 지연이 늘 수 있다는 실험 결과도 자주 인용됐고요.

이 글은 junior 개발자가 프롬프트 캐싱을 원리부터 이해하고 실수 없이 적용할 수 있게 정리합니다. 먼저 캐싱이 무엇을 재사용하는지 그림으로 풀고, 세 프로바이더의 방식 차이를 코드로 비교한 뒤, 에이전트 루프에서 캐시를 지키는 설계와 자주 밟는 함정까지 다룹니다.

1. 프롬프트 캐싱이 푸는 문제

구체적인 숫자로 감을 잡아 봅시다. 사내 규정 챗봇이 8,000토큰짜리 시스템 프롬프트와 문서를 매 메시지에 함께 보낸다고 하죠. 입력 토큰 단가가 100만 토큰당 3달러라면, 이 반복 부분만으로 100만 메시지에 24달러가 나갑니다. 사용자의 실제 질문(수십 토큰)은 정작 얼마 안 되는데, 바뀌지 않는 앞부분에 돈이 다 나가는 구조입니다.

프롬프트 캐싱을 켜면 그 반복 토큰이 캐시 읽기 단가로 청구됩니다. 예를 들어 Anthropic은 캐시 읽기를 기본 입력값의 0.1배로 매깁니다. 방금 24달러가 나가던 부분이 2.4달러 남짓으로 줄어드는 거죠. 출력은 글자 하나 안 바뀌고, 지연은 오히려 짧아집니다.

왜 이게 가능한지 이해하려면, LLM이 입력을 어떻게 소화하는지 한 겹만 들어가면 됩니다.

핵심 한 줄. 프롬프트 캐싱은 "매번 똑같이 반복되는 앞부분(프리픽스)의 계산 결과를 저장해 두고 재사용하는 것"입니다. 새로 쓰는 것도 아니고, 답을 저장하는 것도 아닙니다. 계산 중간산물을 아껴 쓰는 겁니다.

2. 어떻게 동작하나 — KV 캐시 프리픽스 재사용

LLM은 입력 토큰을 하나씩 읽으면서 각 토큰마다 KV 캐시(key-value cache)라는 내부 상태를 만듭니다. 이건 어텐션(attention)이 "지금까지 본 토큰들"을 다시 참조하려고 쌓아 두는 계산 결과예요. 핵심은 이겁니다. 어떤 토큰의 KV 값은 그 앞에 온 토큰들에만 의존합니다. 즉 프롬프트의 앞부분이 완전히 똑같다면, 그 부분의 KV 값도 완전히 똑같습니다.

그러니 앞부분(프리픽스)이 지난 요청과 글자 하나까지 같다면, 굳이 다시 계산할 이유가 없습니다. 저장해 둔 KV 캐시를 그대로 끼워 넣고, 새로 붙은 꼬리 부분만 계산하면 됩니다. 이게 프롬프트 캐싱의 전부입니다.

여기서 두 가지 성질이 따라 나옵니다. 첫째, 캐시는 프리픽스 단위로만 맞습니다. 앞에서부터 정확히 일치하는 구간까지만 재사용되고, 처음으로 달라지는 지점부터는 전부 새로 계산합니다. 둘째, 처음 저장하는 캐시 쓰기는 오히려 기본보다 약간 비쌉니다(Anthropic 기준 1.25배). 그래서 재사용이 거의 없는 프리픽스를 캐싱하면 이득은커녕 손해입니다. 캐싱은 "여러 번 반복될 안정적인 앞부분"에만 켜야 합니다.

3. 캐시를 깨는 것들 — 무효화 규칙

캐시가 프리픽스 단위라는 성질은 곧 가장 흔한 함정으로 이어집니다. 프롬프트 맨 앞에 매번 바뀌는 내용을 두면, 그 뒤가 아무리 고정이어도 프리픽스가 매번 달라져 전부 캐시 미스가 납니다.

가장 흔한 실수. 시스템 프롬프트 맨 위에 오늘 날짜: 2026-07-02를 넣고 매일 바꾸면, 그 아래 수천 토큰의 고정 지침까지 매번 캐시가 깨집니다. 자주 바뀌는 값은 프롬프트 뒤로 보내세요.

Anthropic의 캐시 무효화는 tools → system → messages라는 계층을 따릅니다. 위쪽을 건드리면 그 지점과 그 아래 전부가 무효가 됩니다. 도구 정의를 하나 고치면 시스템·메시지 캐시까지 다 날아가고, 시스템 프롬프트를 바꾸면 메시지 캐시가 날아갑니다. 그러니 "잘 안 바뀌는 것을 위에, 자주 바뀌는 것을 아래에" 배치하는 게 설계의 제1원칙입니다.

캐시에는 수명(TTL)도 있습니다. Anthropic은 기본 5분, 옵션으로 1시간을 제공하고, 캐시가 재사용될 때마다 수명이 갱신됩니다. 트래픽이 뜸해 5분 넘게 요청이 안 오면 캐시가 사라져 다음 요청은 다시 캐시 쓰기부터 시작합니다. 요청이 드문드문 오는 서비스라면 이 점을 감안해야 합니다.

4. 프로바이더 3사 비교 (코드 포함)

큰 프로바이더는 모두 프롬프트 캐싱을 지원하지만, "내가 직접 켜야 하는지"와 "어디까지 자동인지"가 다릅니다. 핵심 차이를 먼저 표로 정리합니다.

Anthropic은 가장 세밀합니다. 캐시할 지점을 cache_control로 직접 찍어야 하는 대신, 안정성이 다른 여러 구간을 각각 캐싱하는 식의 정교한 제어가 가능합니다. 도구와 시스템 프롬프트를 캐싱하는 기본형은 이렇게 씁니다.

{{< highlight python >}} import anthropic client = anthropic.Anthropic() # 매 요청 똑같은 수천 토큰짜리 고정 지침 SYSTEM_PROMPT = "당신은 사내 규정 도우미입니다. ... (긴 지침 + 참고 문서)" resp = client.messages.create( model="claude-opus-4-8", max_tokens=1024, system=[ { "type": "text", "text": SYSTEM_PROMPT, # 여기까지를 캐시로 저장(기본 TTL 5분). 이 지점이 '고정 프리픽스의 끝' "cache_control": {"type": "ephemeral"}, } ], messages=[ # 변동하는 사용자 질문은 뒤에 — 프리픽스를 깨지 않는다 {"role": "user", "content": "환불은 며칠 안에 신청해야 하나요?"} ], ) # 캐시가 실제로 먹혔는지 usage로 확인 — 추측하지 말 것 u = resp.usage print(u.cache_creation_input_tokens) # 첫 요청: 새로 캐시에 쓴 토큰 print(u.cache_read_input_tokens) # 두 번째 요청부터: 캐시에서 읽은 토큰 print(u.input_tokens) # 캐시 뒤의 비캐시(질문) 토큰 {{< /highlight >}}

첫 요청에서는 cache_creation_input_tokens가 크고 cache_read_input_tokens가 0입니다. 5분 안에 같은 프리픽스로 두 번째 요청을 보내면 반대로 cache_read_input_tokens가 채워집니다. 이 값을 로그로 남겨 캐시 히트율을 지켜보는 습관이 중요합니다. 캐시는 항상 보장되는 게 아니라 best-effort라서, "먹혔겠지"라고 가정하면 안 됩니다.

OpenAI는 정반대로 손이 안 갑니다. 1,024토큰 이상이면 자동으로 캐싱되고, 프리픽스가 이전 요청과 정확히 일치하면 128토큰 단위로 히트합니다. 코드로 켤 건 없고, 대신 순서만 잘 지키면 됩니다.

{{< highlight python >}} from openai import OpenAI client = OpenAI() # OpenAI는 자동 캐싱 — 켤 필요가 없다. # 대신 '고정을 앞, 변동을 뒤' 순서만 지키면 프리픽스가 캐시된다. resp = client.responses.create( model="gpt-...", input=[ {"role": "system", "content": LONG_STATIC_INSTRUCTIONS}, # 고정 → 앞 {"role": "user", "content": user_question}, # 변동 → 뒤 ], prompt_cache_key="refund-bot", # 같은 캐시로 라우팅 유도 → 히트율↑ ) # 캐시 히트한 토큰 수 확인 print(resp.usage.input_tokens_details.cached_tokens) {{< /highlight >}}

Gemini는 두 모드를 함께 줍니다. 암묵적 캐싱은 유료 프로젝트에서 기본으로 켜져 저장 비용 없이 자동으로 겹치는 부분을 할인해 주고, 명시적 캐싱은 캐시할 내용을 직접 만들어 TTL을 정하고 보장된 할인을 받습니다. 대신 명시적 캐시는 저장 기간만큼 보관 비용이 붙습니다. 아주 큰 문서를 오래 반복 참조한다면 명시적 캐싱이, 트래픽이 알아서 반복된다면 암묵적 캐싱이 편합니다.

자체 서빙도 원리는 같다. vLLM의 automatic prefix caching이나 SGLang의 RadixAttention도 "같은 프리픽스의 KV 캐시를 재사용"하는 같은 아이디어입니다. RAG나 멀티턴처럼 프리픽스가 겹치는 워크로드에서 큰 이득을 봅니다. API를 쓰든 직접 서빙하든, "프리픽스를 안정적으로 유지하라"는 원칙은 똑같습니다.

5. 캐시를 의식한 에이전트 루프

프롬프트 캐싱이 진짜 빛나는 곳은 에이전트입니다. 에이전트는 한 작업을 위해 도구를 부르고 결과를 받아 다시 생각하기를 수십 번 반복합니다. 그때마다 같은 시스템 프롬프트 + 같은 도구 정의 + 지금까지의 대화 전체를 다시 보냅니다. 대화가 길어질수록 반복되는 프리픽스가 어마어마해지죠. 여기에 캐싱이 먹히면 매 턴이 싸지고 빨라집니다.

비결은 하나입니다. 히스토리를 앞에서부터 건드리지 말고, 뒤에만 이어 붙이세요(append-only). 대화의 앞부분이 그대로 유지되는 한 프리픽스는 매 턴 캐시 히트합니다. 반대로 중간 히스토리를 요약하거나 지우거나 순서를 바꾸면, 바로 그 지점부터 프리픽스가 달라져 캐시가 깨집니다.

{{< highlight python >}} # 캐시를 의식한 에이전트 루프 — 앞을 고정하고 뒤에만 이어 붙인다 system = [{"type": "text", "text": SYSTEM_PROMPT, "cache_control": {"type": "ephemeral"}}] # 고정 프리픽스 messages = [] # 히스토리는 append-only while not done: messages.append({"role": "user", "content": next_input}) resp = client.messages.create( model="claude-opus-4-8", max_tokens=1024, system=system, # 매 턴 동일 → 캐시 히트 messages=messages, # 뒤에 계속 붙이기만 한다 ) messages.append({"role": "assistant", "content": resp.content}) # ⚠️ 중간 히스토리를 요약·삭제·재정렬하면 그 지점부터 캐시가 깨진다. # 컨텍스트 압축이 꼭 필요하면 '가끔 한 번에 크게' — 매 턴 조금씩 바꾸지 말 것. {{< /highlight >}}

여기서 캐시 최적화와 컨텍스트 관리가 살짝 충돌합니다. 대화가 길어지면 컨텍스트를 줄이려고 히스토리를 요약하고 싶은데, 그게 바로 캐시를 깨는 행동이니까요. 실무의 절충은 "매 턴 조금씩 다듬지 말고, 임계치를 넘을 때 가끔 한 번에 크게 압축"하는 겁니다. 그러면 대부분의 턴에서는 캐시가 유지되고, 압축은 어쩌다 한 번만 캐시를 리셋합니다.

이 균형을 실증적으로 다룬 게 이번에 자주 인용된 논문 "Don't Break the Cache"(arXiv 2601.06007)입니다. 10,000토큰짜리 시스템 프롬프트로 500개가 넘는 에이전트 세션을 돌려 보니, 캐싱으로 API 비용이 41~80% 줄고 TTFT가 13~31% 개선됐습니다. 다만 무작정 전부 캐싱하면 오히려 지연이 늘 수 있고, 캐시 경계(브레이크포인트)와 TTL을 의도적으로 설계하는 편이 언제나 낫다고 보고합니다. 실제로 브레이크포인트를 잘 두면 히트율이 한 자릿수에서 84%까지 올랐다고 하죠.

6. 흔한 실수 · 팁

  • 날짜·타임스탬프를 맨 위에 두기. 가장 흔하고 뼈아픈 실수입니다. 프롬프트 맨 앞의 오늘 날짜: ... 한 줄이 매일 바뀌면 그 아래 전부가 캐시 미스입니다. 변동 값은 무조건 뒤로.
  • 최소 토큰 미달. Anthropic·OpenAI 모두 대략 1,024토큰 이상이어야 캐싱이 걸립니다. 짧은 프롬프트에 cache_control을 붙여 놓고 "왜 캐시가 안 되지?" 하는 경우가 많습니다.
  • 재사용 없는 캐시 쓰기. 캐시 쓰기는 기본보다 비쌉니다(1.25배). 한 번 쓰고 마는 프리픽스를 캐싱하면 손해입니다. 여러 번 반복될 안정적인 부분에만 켜세요.
  • TTL 만료를 잊기. 기본 5분 안에 재요청이 없으면 캐시가 사라집니다. 트래픽이 뜸하면 캐시 히트율이 뚝 떨어지니, 필요하면 1시간 TTL 옵션을 검토하세요.
  • 도구 목록을 매번 재정렬. 도구 정의(순서 포함)가 조금만 바뀌어도 그 아래 캐시가 전부 무효화됩니다. 도구 리스트는 요청 간에 바이트 단위로 고정하세요.
  • 캐시 히트를 눈으로 확인 안 하기. 캐시는 best-effort라 항상 먹힌다는 보장이 없습니다. cache_read_input_tokens(또는 각 사의 상응 필드)를 로그로 남겨 실제 히트율을 모니터링하세요.
  • 스키마가 유효한 것 ≠ 캐시가 먹힌 것. 요청이 200 OK로 잘 돌아온다고 캐시가 걸린 건 아닙니다. 비용이 줄었는지는 usage 필드로만 알 수 있습니다.

7. 마무리 (TL;DR)

  • 프롬프트 캐싱 = 반복되는 프리픽스의 KV 캐시를 재사용하는 것. 새로 만드는 것도, 답을 저장하는 것도 아니다. 계산 중간산물을 아껴 쓰는 것.
  • 캐시는 프리픽스 단위. 앞에서부터 정확히 일치하는 데까지만 재사용된다. 처음 달라지는 지점부터는 전부 다시 계산.
  • 고정을 앞, 변동을 뒤. 날짜·사용자 질문 같은 변동 값을 맨 위에 두면 캐시가 통째로 깨진다. 설계 제1원칙.
  • 프로바이더 차이: Anthropic은 명시적 브레이크포인트(읽기 0.1×), OpenAI는 자동(50%↓), Gemini는 암묵+명시(2.5+ 90%↓). 셋 다 대략 1,024토큰 이상, TTL이 있다.
  • 에이전트는 append-only. 히스토리를 뒤에만 이어 붙이면 매 턴 캐시 히트. 중간 요약·삭제는 그 지점부터 캐시를 깬다 — 압축은 가끔 크게 한 번에.
  • 무작정 전부 캐싱은 답이 아니다. 캐시 쓰기는 더 비싸고, 경계·TTL을 잘못 두면 손해다. usage 필드로 히트율을 꼭 확인하자.
