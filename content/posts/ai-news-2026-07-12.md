# 🔥 오늘의 핵심

오늘(7/12) 신호는 한 방향이다 — **"툴 결과를 매번 모델에 왕복시키지 마라."** OpenAI가 지난주 GA한 **GPT-5.6 (Sol/Terra/Luna)**의 핵심은 3티어 가격이 아니라 **Programmatic Tool Calling(PTC)**: 모델이 툴을 하나씩 호출하는 대신 **JS 프로그램을 짜서 격리 런타임에서 여러 툴을 돌리고, 필요한 상태만 컨텍스트로 되돌린다.** OpenAI 공지 기준 **동일 품질에 출력 토큰 -24%, 완료 시간 -28%**. 같은 결로 오늘 검증된 arXiv 논문 두 편도 **코딩 에이전트의 검색(ProjAgent)** 과 **에이전트 서빙 효율(SMetric)** 을 판다. 그래서 오늘 개발자가 해볼 일: **바운디드·툴-헤비 워크플로를 turn-by-turn 루프에서 코드-오케스트레이션으로 옮겨** 토큰·왕복을 깎아보는 것.

# 📚 주목 논문

오늘도 arXiv(`export.arxiv.org`)로 ID·저자·수치를 1차 확정했다. 둘 다 "코딩 에이전트를 실전에서 더 싸고 정확하게" 쪽이다.

### 1) ProjAgent — 코드를 '절차 유사도'로 검색
- 저자: QiHong Chen, Aaron Imani, Iftekhar Ahmed · **arXiv 2607.08691 (cs.SE/cs.AI/cs.IR)**
- 핵심 기여: 리포지토리-레벨 코드 생성에서 기존 검색은 **어휘·구조·의미 유사도**에 치우쳐, **식별자는 달라도 절차 로직이 비슷한 함수**를 놓친다. 이를 **절차 유사도(procedural similarity)** 라는 새 검색 신호로 잡는다.
- 방법: 목표 함수를 중간 추론 단계로 분해 → **각 단계마다 비슷한 절차의 리포 함수를 에이전트 워크플로로 검색** → 의미 검색과 결합해 더 풍부한 컨텍스트 구성. 여기에 **컴파일러·정적분석 피드백 루프**로 생성 코드를 보수적으로 반복 수정.
- 결과(검증): **REPOCOD에서 41.14% Pass@1**, 기존 검색 기반 베이스라인 상회.
- 🛠 **빌더 시사점**: 코드 RAG를 임베딩 유사도만으로 채우지 말고, **"이 함수가 뭘 하는 절차인가"** 를 별도 검색 축으로 추가하라. 정적분석 피드백 루프는 지금 스택에도 바로 붙는다.
- ⚠️ 한계: 절차 분해·다중 검색으로 **호출·지연 비용 증가**, 평가는 REPOCOD 단일 벤치.
- 링크: [arXiv:2607.08691](https://arxiv.org/abs/2607.08691)

### 2) SMetric — 에이전트 서빙은 사람 채팅과 다르다
- 저자: Jiahao Wang, Kaizhan Lin, Xingda Wei 외 · **arXiv 2607.08565 (cs.DC/cs.AI)**
- 핵심: 에이전트 요청은 (1) **완성 응답만 소비**해 클러스터 **TPS**가 목표가 되고, (2) 세션 내 요청이 **KV 캐시를 크게 공유**한다 — 프로덕션 트레이스에서 **KV 재사용 >80%** (채팅은 54~62%). 기존 스케줄러는 캐시 있는 인스턴스로 몰아 **일부 과부하·나머지 유휴**로 TPS를 깎는다.
- 방법: **세션 첫 요청만 순수 로드밸런싱**으로 흩고, 후속 요청은 **캐시-어웨어 라우팅** — 세션 turn 정보만으로 스테이트리스하게 결정. (balanced session-centric scheduling)
- 결과(검증): **클러스터 TPS +10~16%**(prefill-decode 코로케이션), **prefill TPS +2~34%**(분리 배치), per-token 지연도 개선.
- 🛠 **빌더 시사점**: 자체 추론 서빙을 굴린다면 라우팅 키를 요청이 아니라 **"에이전트 세션"** 단위로 바꿔라. 첫 요청만 밸런싱해도 지역 KV 재사용을 대부분 지킨다.
- 링크: [arXiv:2607.08565](https://arxiv.org/abs/2607.08565)

# 🛠 오픈소스 스포트라이트

투명하게 밝힌다 — **오늘도 GitHub API가 세션 네트워크 정책으로 차단**돼 임의 레포의 star 수를 1차 검증할 수 없다. 제로 팩트 원칙상 검증 불가한 별점 대신 **오늘 바로 재현 가능한 검증된 오픈 아티팩트**를 가리킨다.

- **REPOCOD (코드 생성 벤치)** — ProjAgent가 41.14% Pass@1을 낸 그 벤치. 리포-레벨 코드 에이전트를 만든다면, "절차 유사도 검색"을 켜고 끈 **A/B를 REPOCOD로** 돌려 내 스택에서도 이득이 나는지 바로 확인 가능. ([논문](https://arxiv.org/abs/2607.08691))
- **오늘의 논문 = 프레임워크 독립 레시피.** ProjAgent(절차 검색+정적분석 루프)와 SMetric(세션 중심 라우팅)은 특정 프레임워크에 묶이지 않는다. LangGraph·자체 루프·사내 게이트웨이 어디든 **개념만 이식**하면 된다.
- ⚠️ 별점 기반 레포 랭킹은 API 접근이 복구되는 세션에서 정확한 수치로 다시 다룬다.

# 🧑‍💻 Codex & Claude Code 실전 팁

### 🟩 Codex — Programmatic Tool Calling으로 툴 왕복 줄이기 (category: Workflow)
- 소스: [GPT-5.6 발표](https://openai.com/index/gpt-5-6/)
- 💡 **핵심**: 툴이 많고 **중간 판단이 필요 없는 바운디드 작업**은 모델에 매 결과를 되먹이지 말고, 모델이 짠 코드가 툴을 오케스트레이션하게 하라.
- 📝 **how-to**: Responses API에서 PTC를 켜고 대상 툴을 등록하면, 모델이 아래처럼 **한 번에** 흐름을 짠다.
```js
// 모델이 격리 런타임에서 생성·실행 (개념 예시)
const rows = await db.query(sql);              // 툴 1
const top = rows.filter(r => r.spend > 1000)   // 중간결과 필터
                .slice(0, 20);
const enriched = await Promise.all(            // 툴 2 병렬
  top.map(r => vendor.lookup(r.id)));
return summarize(enriched);                     // 유용한 상태만 반환
```
- 🎯 **언제**: 리포트 집계, 대량 레코드 정제, 다단계 조회처럼 **단계마다 새 추론이 불필요한** 툴-헤비 파이프라인.
- ⚠️ **주의**: 런타임은 **네트워크 없음**이고, 단계마다 사람이 개입할 판단이 필요한 워크플로엔 부적합. 실행 코드 로깅으로 감사 남길 것.
- ⭐ **추천**: 툴콜 왕복이 병목인 파이프라인부터 시범 적용 → 토큰·지연 A/B.

### 🟦 Claude Code — 2.1.207 Auto 모드 기본화, 플러그인은 exec form (category: Hook/Security)
- 소스: [What's new](https://code.claude.com/docs/en/whats-new)
- 💡 **핵심**: Bedrock·Vertex·Foundry에서 Auto 모드가 **기본 활성**됐다. 자동화는 늘지만 **플러그인 변수 확장 규칙이 조여져** 기존 플러그인이 조용히 깨질 수 있다.
- 📝 **how-to**: shell-form `${user_config.*}`는 이제 거부. 아래처럼 **exec form / 전용 env**로 바꾼다.
```toml
# 이전 (거부됨): shell-form 확장
# command = "deploy.sh ${user_config.env}"

# 이후: exec form + CLAUDE_PLUGIN_OPTION_ env
command = ["deploy.sh"]
# 스크립트 안에서 $CLAUDE_PLUGIN_OPTION_ENV 로 읽기
```
- 🎯 **언제**: 사내 배포/CI 플러그인을 쓰는 팀 — 업데이트 후 플러그인 스모크 테스트 필수.
- ⚠️ **주의**: Auto 모드를 원치 않으면 settings에 `disableAutoMode`. 기본 모델이 **Opus 4.8**로 바뀐 점도 비용 계산에 반영.
- ⭐ **추천**: 업데이트 → 플러그인 exec form 점검 → Auto 모드 정책 팀 합의.

### 🟪 공통 — 라우팅 키를 '요청'이 아니라 '세션'으로 (category: Context/Infra)
- 소스: [arXiv:2607.08565](https://arxiv.org/abs/2607.08565)
- 💡 **핵심**: 에이전트 트래픽은 세션 내 KV 공유가 크다(>80%). **첫 요청만 로드밸런싱**하고 후속은 캐시-어웨어로 붙이면 밸런스와 재사용을 동시에 잡는다.
- 📝 **how-to**: 게이트웨이/프록시에서 sticky 키를 `request_id`가 아니라 `session_id`로. 세션의 **첫 turn**만 최저부하 인스턴스로, 이후 turn은 같은 인스턴스로 고정.
- 🎯 **언제**: 자체 vLLM/SGLang 등으로 다중 에이전트를 서빙하고 TPS가 병목일 때.
- ⚠️ **주의**: 세션 길이 편차가 크면 hot-spot 위험 — global KV 티어나 주기적 리밸런싱 병행.
- ⭐ **추천**: 세션 스티키 라우팅 켜고 TPS/토큰·지연을 계측해 이득 확인.
