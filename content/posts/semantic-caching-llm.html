프롬프트 캐시는 글자 하나까지 똑같은 질문만 알아봅니다. 그런데 사람은 같은 걸 물어도 매번 조금씩 다르게 묻죠. "환불 어떻게 해요?"와 "환불은 어떤 절차인가요?"는 답이 같은데, 프리픽스가 달라 캐시가 전혀 안 먹힙니다.

— 시맨틱 캐시는 이 벽을 넘습니다. 글자가 아니라 뜻을 보고, 비슷하면 저장된 답을 곧바로 꺼내 줍니다.

챗봇이나 에이전트를 조금만 키워 운영해 보면 두 가지가 동시에 무섭게 늘어납니다. 돈과 기다림입니다. 에이전트는 사용자의 요청 한 건을 처리하면서 LLM을 8번, 20번씩 부릅니다. 그런데 실제 트래픽을 들여다보면 사람들이 묻는 건 생각보다 몇 가지로 뻔합니다. "배송 언제 와요?", "환불 어떻게 해요?", "이거 무슨 뜻이에요?" — 같은 질문이 표현만 바뀌어 하루에도 수천 번씩 들어옵니다.

여기서 자연스러운 생각. 같은 답을 또 만들 거면, 저번에 만든 답을 그냥 다시 쓰면 안 되나? 이게 바로 캐싱의 아이디어입니다. 문제는 LLM 세계의 캐시가 대부분 "완전히 똑같을 때만" 작동한다는 점입니다. 프롬프트 캐시(prompt cache)는 요청 앞부분(프리픽스)이 토큰 하나까지 동일해야 재사용합니다. 지난 글에서 다룬 프롬프트 캐싱이 바로 그것이죠. 훌륭하지만, 사람의 말은 매번 조금씩 다릅니다. "오늘 날씨 어때?"와 "지금 날씨 어떤가요?"는 뜻이 사실상 같은데, 글자가 달라서 프롬프트 캐시 입장에선 완전히 다른 요청입니다.

시맨틱 캐시(semantic cache)는 이 틈을 메웁니다. 핵심은 한 줄입니다. 질문을 임베딩(embedding, 문장을 숫자 벡터로 바꾼 것)으로 변환해서, 예전 질문들과 "의미가 얼마나 가까운지" 재고, 충분히 가까우면 그때 저장해 둔 답을 그대로 돌려준다. 글자가 아니라 뜻을 기준으로 삼는 겁니다. 잘 맞으면 LLM 호출을 아예 건너뛰어 비용을 0으로, 응답 시간을 수 초에서 수 밀리초로 줄입니다.

이 주제가 요즘 다시 화제입니다. 2026년 내내 "토큰을 많이 태우자(tokenmaxxing)"에서 "토큰을 아껴 쓰자(token discipline)"로 무게중심이 옮겨 오면서, 비용·지연을 어떻게든 깎으려는 실무자들이 프롬프트 캐시 다음 카드로 시맨틱 캐시를 꺼내 들었습니다. TrueFoundry는 아예 "Beyond Prefix Caching(프리픽스 캐싱 그 너머)"이라는 제목으로 이 흐름을 정리했고, Bifrost·Gravitee·Portkey 같은 게이트웨이들이 앞다퉈 시맨틱 캐시 기능을 기본 탑재했습니다. 학계에서도 2026년 "From Exact Hits to Close Enough"(arXiv 2603.03301)처럼 '정확 일치에서 근사 일치로'를 정면으로 다룬 연구가 나왔고요.

그런데 이 캐시는 양날의 검입니다. "비슷함"을 잘못 판단하는 순간, 엉뚱한 답을 그것도 아주 자신 있게 내놓거든요. 이 글은 junior 개발자가 시맨틱 캐시를 원리부터 이해하고, 직접 만들어 보고, 위험한 함정을 피할 수 있도록 쉽고 자세하게 정리합니다.

1. LLM 캐시 3형제 — KV · 프롬프트 · 시맨틱

"캐시"라는 한 단어에 서로 다른 세 가지가 섞여 있어서 처음엔 헷갈립니다. 먼저 이 셋을 갈라 놓아야 시맨틱 캐시가 어디에 있고 무엇을 재사용하는지 또렷해집니다.

  • KV 캐시(KV cache) — 모델 안쪽에 있습니다. LLM이 토큰을 하나씩 만들 때, 앞 토큰들을 처리하며 계산해 둔 어텐션의 Key·Value를 다시 안 계산하려고 GPU 메모리에 들고 있는 것입니다. 한 번의 생성 안에서만 쓰이고, 요청이 끝나면 보통 사라집니다.
  • 프롬프트 캐시 / 프리픽스 캐시(prompt cache) — 서빙 엔진 앞에 있습니다. 여러 요청이 똑같은 프리픽스(예: 긴 시스템 프롬프트)를 공유하면, 그 부분의 KV 계산을 요청 사이에 재사용합니다. 정확 일치라서 무손실이고, 출력이 한 글자도 안 바뀝니다. 하지만 모델은 여전히 호출됩니다 — 앞부분 계산만 아낄 뿐이죠.
  • 시맨틱 캐시(semantic cache) — 모델보다 훨씬 바깥, 애플리케이션 맨 앞에 있습니다. 새 질문이 예전 질문과 의미가 비슷하면, 저장된 답을 통째로 돌려주고 모델 호출 자체를 건너뜁니다. 정확 일치가 아니라 근사 일치라서, 잘 맞으면 가장 크게 절약하지만 틀릴 수도 있습니다.

한 문장으로 요약하면 이렇습니다. KV 캐시는 한 답 안에서 아끼고, 프롬프트 캐시는 계산을 아끼고, 시맨틱 캐시는 호출 자체를 아낍니다. 셋은 경쟁 관계가 아니라 같이 쓰는 층입니다.

헷갈리지 말기. "프롬프트 캐시로 90% 아꼈다"와 "시맨틱 캐시로 호출을 건너뛴다"는 완전히 다른 얘기입니다. 앞은 같은 프리픽스의 계산을 아끼는 것(모델은 돈다)이고, 뒤는 같은 뜻의 질문에 모델을 아예 안 부르는 것입니다. 둘은 함께 켜면 서로를 방해하지 않고 곱해집니다.

2. 시맨틱 캐시는 어떻게 '비슷함'을 재는가

"의미가 비슷하다"를 컴퓨터가 어떻게 알까요? 핵심 도구가 임베딩(embedding)입니다. 임베딩 모델은 문장을 받아 수백~수천 개의 숫자로 이뤄진 벡터 하나로 바꿉니다. 이 벡터 공간에는 마법 같은 성질이 있습니다. 뜻이 비슷한 문장은 벡터도 서로 가깝게 놓인다는 것이죠. "환불 어떻게 해요?"와 "환불 절차가 뭔가요?"는 글자는 달라도 벡터로는 거의 겹칩니다.

두 벡터가 얼마나 가까운지는 보통 코사인 유사도(cosine similarity)로 잽니다. 두 벡터가 이루는 각도를 보고, 완전히 같은 방향이면 1.0, 무관하면 0에 가깝습니다. 시맨틱 캐시는 이 값이 미리 정한 임계값(threshold), 예컨대 0.95를 넘으면 "충분히 비슷하다 = 캐시 히트"로 판정합니다.

비유. 임베딩은 모든 문장에 "뜻 좌표"를 찍는 일입니다. 도서관에서 비슷한 주제의 책을 가까운 서가에 꽂아 두는 것과 같죠. 새 질문이 들어오면 그 좌표 근처에 예전 질문(과 그 답)이 있는지 둘러보고, 코앞에 있으면 "아, 저번에 답한 그거네" 하고 그 답을 꺼내 옵니다. 임계값은 "얼마나 코앞이어야 같은 걸로 쳐 줄까"를 정하는 자입니다.

여기서 중요한 감각 하나. 코사인 유사도 0.95는 "95% 정답"이라는 뜻이 절대 아닙니다. 그냥 두 문장의 임베딩이 가리키는 방향이 그만큼 비슷하다는 기하학적 수치일 뿐입니다. 방향이 비슷해도 결정적인 한 단어가 다르면 답은 완전히 달라질 수 있습니다. "서울 날씨"와 "부산 날씨"는 임베딩이 꽤 가깝지만 답은 전혀 다르죠. 이 간극이 나중에 다룰 모든 함정의 뿌리입니다.

3. 읽기 경로 한 바퀴 — 히트와 미스

동작을 한 바퀴 따라가 봅시다. 시맨틱 캐시에는 두 개의 길이 있습니다. 질문이 들어올 때의 읽기(read) 길과, 새 답을 만들었을 때의 쓰기(write) 길입니다.

여기서 눈여겨볼 점. 미스가 났을 때 LLM을 부르고 그 결과를 반드시 캐시에 저장해야 다음번 비슷한 질문이 히트가 됩니다. 즉 첫 질문은 언제나 느리고, 그 뒤 "따라 들어오는" 유사 질문들이 이득을 봅니다. 캐시가 데워지는(warm-up) 시간이 필요하다는 뜻이죠.

4. 실전 — 60줄로 만드는 시맨틱 캐시

원리를 손으로 만져 보면 확실해집니다. 외부 라이브러리의 마법 없이, 임베딩과 코사인 유사도만으로 동작하는 가장 작은 시맨틱 캐시를 짜 봅시다. (설명용이라 저장소는 그냥 파이썬 리스트입니다.)

{{< highlight python >}} import numpy as np from openai import OpenAI client = OpenAI() def embed(text: str) -> np.ndarray: # 문장을 하나의 벡터로 바꾼다 (임베딩 모델 호출) v = client.embeddings.create( model="text-embedding-3-small", input=text ).data[0].embedding return np.array(v) def cosine(a: np.ndarray, b: np.ndarray) -> float: # 두 벡터가 이루는 각도의 코사인 = 방향이 얼마나 같은가 (1.0이 최대) return float(a @ b / (np.linalg.norm(a) * np.linalg.norm(b))) class SemanticCache: def __init__(self, threshold: float = 0.95): self.threshold = threshold # 이 값 이상이면 '같은 질문'으로 인정 self.store: list[tuple[np.ndarray, str, str]] = [] # (질문벡터, 질문, 답) def get(self, question: str): qv = embed(question) best_sim, best_ans = 0.0, None for vec, _q, ans in self.store: # 저장된 예전 질문들과 비교 sim = cosine(qv, vec) if sim > best_sim: best_sim, best_ans = sim, ans if best_sim >= self.threshold: # 임계값 통과 → 캐시 히트 return best_ans, best_sim return None, best_sim # 미스 def put(self, question: str, answer: str): self.store.append((embed(question), question, answer)) {{< /highlight >}}

이제 이 캐시를 LLM 호출 앞에 끼워 넣습니다. 히트면 모델을 건너뛰고, 미스면 모델을 부른 뒤 결과를 저장합니다.

{{< highlight python >}} cache = SemanticCache(threshold=0.95) def ask(question: str) -> str: answer, sim = cache.get(question) if answer is not None: print(f"[캐시 히트 sim={sim:.3f}] 모델 호출 안 함") return answer # 미스 → 진짜 LLM 호출 answer = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": question}], ).choices[0].message.content cache.put(question, answer) # 다음을 위해 반드시 저장 return answer ask("환불은 어떻게 하나요?") # 미스 → 모델 호출, 저장 ask("환불 절차가 어떻게 되죠?") # 히트! 뜻이 같아 저장된 답을 즉시 반환 {{< /highlight >}}

두 번째 질문은 글자가 완전히 다른데도 캐시 히트가 납니다. 프롬프트 캐시라면 절대 못 잡을 상황이죠. 이게 시맨틱 캐시의 힘입니다.

실무에서는 직접 짜지 마세요. 위 코드는 원리를 보여 주려는 것이고, 실제로는 검증된 도구를 씁니다. GPTCache는 임베딩·벡터 저장소(FAISS/Milvus)·스칼라 저장소(SQLite/MySQL)를 조립해 주고 LangChain·llama_index와 바로 붙습니다. LangChain의 RedisSemanticCache나 RedisVL도 흔한 선택지고, Bifrost·Gravitee·Portkey 같은 게이트웨이는 코드 한 줄 없이 프록시 단에서 시맨틱 캐시를 켜 줍니다. 선형 탐색 대신 벡터 인덱스(ANN)를 써서 수백만 건에서도 빠르게 검색한다는 점이 미니 구현과의 결정적 차이입니다.

5. 임계값이라는 손잡이 — 히트율 vs 오답률

시맨틱 캐시에는 딱 하나의 중요한 손잡이가 있습니다. 유사도 임계값입니다. 이 손잡이를 돌리면 두 지표가 서로 반대로 움직입니다.

  • 임계값을 낮추면(예: 0.85) — 웬만큼 비슷해도 히트로 쳐 줍니다. 히트율(hit rate)이 올라가 비용은 더 아끼지만, 사실은 다른 질문을 같다고 착각하는 거짓 양성(false positive)이 늘어 엉뚱한 답을 줄 위험이 커집니다.
  • 임계값을 높이면(예: 0.98) — 거의 판박이일 때만 히트로 인정합니다. 오답 위험은 뚝 떨어지지만, 히트율이 낮아져 절약 효과가 줄어듭니다.

그럼 얼마로 시작할까요? 실무 가이드들은 대체로 보수적으로 높게 잡으라고 합니다. 아무 데이터가 없는 첫 배포라면 0.95~0.97쯤에서 출발해, 캐시 히트의 1~5%를 샘플링해 "이 답이 정말 맞았나"를 사람이나 강한 LLM(LLM-as-judge)으로 채점하며 거짓 양성률을 지켜봅니다. 오답이 거의 없으면 0.01씩 내려 히트율을 넓혀 가는 식이죠. 임계값 0.97처럼 아주 높게 잡으면 히트율은 5~10%로 낮아도 거짓 양성률은 0.5% 밑으로 억누를 수 있습니다.

효과가 어느 정도인지 감을 주는 숫자도 있습니다. 시맨틱 캐시를 정면으로 연구한 GPT Semantic Cache(arXiv 2411.05276)는 실제 챗봇 트래픽에서 API 호출을 최대 68.8% 줄이고, 캐시 히트율 61.6~68.8%를 달성하면서도 히트가 맞은 비율(positive hit rate)을 97% 이상으로 유지했다고 보고합니다. 반복 질문이 많은 FAQ·고객지원 성격의 트래픽에서 특히 강하다는 걸 보여 주죠. 현업에서 흔히 보고되는 히트율은 트래픽 성격에 따라 대략 30~70% 사이입니다.

6. 언제 켜고, 언제 절대 켜지 말아야 하나

시맨틱 캐시는 만능이 아닙니다. 오히려 켜면 안 되는 곳에 켜면 조용히 사고를 냅니다. 판단 기준은 "이 질문의 답이 누가·언제·어떤 맥락에서 물어도 같은가?"입니다.

상황시맨틱 캐시이유
FAQ·제품 설명·정책 안내처럼 반복되는 질문✅ 아주 좋음답이 안정적이고 표현만 다른 같은 질문이 쏟아짐
문서 요약·번역·분류 등 입력이 겹치는 배치 작업✅ 좋음같은 입력이 자주 반복됨
"지금 잔액", "오늘 배송 상태" 같은 실시간·개인화 질의❌ 위험답이 사용자·시점마다 다름 → 남의 답이 새어 나감
정확한 수치·법률·의료처럼 한 단어 차이가 치명적인 질의⚠️ 매우 주의임베딩이 가까워도 정답은 완전히 다를 수 있음
창작·브레인스토밍처럼 매번 다른 답을 원하는 질의❌ 부적합같은 답 재사용이 오히려 나쁨

실무에서는 보통 전부 켜거나 전부 끄지 않습니다. 라우팅 앞단에서 "이 요청은 캐시 가능한 종류인가"를 먼저 판단해, FAQ성 트래픽에만 시맨틱 캐시를 태우고 개인화·실시간 질의는 통과시킵니다. 캐시 키에 사용자 ID나 언어·테넌트 같은 맥락을 함께 넣어 네임스페이스를 나누는 것도 기본기고요(남의 답이 새는 걸 막는 핵심 장치입니다).

7. 흔한 실수 · 팁

1) 거짓 양성 — 가장 무서운 실수. 임계값을 욕심내 낮추면, "서울 날씨"에 "부산 날씨" 답을, "2023년 매출"에 "2024년 매출" 답을 자신 있게 내놓습니다. 사용자는 캐시가 있는지도 모르니 그냥 틀린 답을 믿습니다. 임계값은 항상 보수적으로 높게 시작하고, 히트를 샘플링해 오답률을 계속 감시하세요.
2) 개인화·맥락 누출. 사용자 A의 "내 주문 어디쯤이야?" 답이 캐시에 남아, 비슷하게 물은 사용자 B에게 A의 개인정보가 그대로 나갈 수 있습니다. 개인화 질의는 캐시에서 빼거나, 캐시 키에 사용자·세션·테넌트를 반드시 포함해 격리하세요.
3) 신선도(staleness)와 TTL. "최신 요금제는?", "오늘 환율은?"의 답은 시간이 지나면 틀린 답이 됩니다. 시맨틱 캐시는 오래된 답을 영원히 돌려줄 수 있으니, 항목마다 만료 시간(TTL)을 두고, 시간에 민감한 질의는 아예 캐시하지 마세요.
4) 임베딩 모델을 바꾸면 캐시가 깨진다. 저장된 벡터는 그때 쓴 임베딩 모델의 좌표계입니다. 모델을 교체하면 새 질문 벡터와 옛 저장 벡터의 유사도가 무의미해집니다. 임베딩 모델을 바꾸면 캐시를 통째로 다시 채워야(재인덱싱) 합니다.
5) 히트율을 측정하지 않으면 켠 의미가 없다. 히트율·거짓 양성률·절감액을 대시보드로 보세요. 히트율이 몇 %밖에 안 되면 임계값이 너무 높거나 트래픽이 애초에 안 겹치는 것이고, 오답 신고가 늘면 임계값이 너무 낮은 겁니다. 측정 없이는 튜닝도 없습니다.
6) 프롬프트 캐시와 같이 쓰기. 시맨틱 캐시가 미스여서 모델을 부를 때, 그 호출은 여전히 프롬프트 캐시의 혜택을 받습니다. 둘은 층이 달라 충돌하지 않습니다. 시맨틱 캐시로 "부를지 말지"를 거르고, 부를 때는 프롬프트 캐시로 "얼마나 싸게 부를지"를 챙기는 조합이 이상적입니다.
7) 질문을 정규화(normalize)하면 히트율이 오른다. 앞뒤 공백·대소문자·불필요한 인사말("안녕하세요, 혹시…")을 임베딩 전에 다듬으면 같은 뜻의 질문이 더 잘 뭉칩니다. 다만 정규화가 뜻을 바꾸지 않도록 조심하세요(부정어 "안", "말고" 같은 건 절대 지우면 안 됩니다).

8. 마무리 (TL;DR)

  • 시맨틱 캐시는 질문을 임베딩으로 바꿔 의미가 비슷한 예전 질문의 답을 재사용합니다. 프롬프트 캐시가 못 잡는 "표현만 다른 같은 질문"을 잡아 LLM 호출 자체를 건너뜁니다.
  • LLM 캐시는 셋입니다. KV 캐시(한 생성 안), 프롬프트 캐시(같은 프리픽스 계산, 무손실), 시맨틱 캐시(같은 뜻이면 호출 생략, 근사 일치라 틀릴 수 있음). 서로 층이 달라 함께 씁니다.
  • 원리는 임베딩 → 코사인 유사도 → 임계값입니다. 유사도 0.95는 "95% 정답"이 아니라 "방향이 그만큼 비슷하다"는 기하학적 수치일 뿐입니다.
  • 핵심 손잡이는 임계값입니다. 낮추면 히트율↑·오답위험↑, 높이면 오답위험↓·히트율↓. 보수적으로 높게(≈0.95~0.97) 시작해 오답률을 보며 내리세요.
  • 효과는 큽니다 — GPT Semantic Cache는 API 호출 최대 68.8%↓, 히트 정확도 97%+를 보고합니다. 현업 히트율은 대략 30~70%.
  • 가장 무서운 건 거짓 양성·개인화 누출·신선도입니다. FAQ성 질의에만 켜고, 캐시 키에 맥락을 넣어 격리하고, TTL을 두고, 히트를 샘플링해 계속 감시하세요.
