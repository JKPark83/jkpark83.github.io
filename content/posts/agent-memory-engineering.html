모델은 똑똑한데, 어제 한 대화를 오늘은 통째로 잊어버립니다. 이름도, 말투도, 지난주에 겪은 사고도요. LLM은 기본적으로 상태가 없는(stateless) 함수라서 그렇습니다.

— 그래서 "어떻게 기억하게 만들 것인가"가, 2026년 에이전트 엔지니어링에서 모델 선택만큼 중요한 설계 문제가 됐습니다.

챗봇에게 "저는 매운 음식을 못 먹어요"라고 말한 적이 있을 겁니다. 그런데 다음 날 같은 챗봇이 떡볶이 맛집을 추천하면 어떤 기분이 드나요? 모델이 멍청해서가 아닙니다. 기억할 곳이 없어서 그렇습니다.

LLM은 본질적으로 "입력을 넣으면 출력을 뱉는" 함수입니다. 함수는 호출이 끝나면 아무것도 남기지 않습니다. 우리가 ChatGPT 같은 서비스에서 "대화가 이어진다"고 느끼는 건, 사실 매 요청마다 이전 대화 기록을 통째로 다시 입력에 붙여 넣기 때문입니다. 창을 닫으면, 또는 대화가 너무 길어져 기록을 잘라내면, 그 순간 모델은 당신을 처음 만나는 사람처럼 대합니다.

이 문제를 제대로 풀자는 게 바로 에이전트 메모리(agent memory)입니다. 이번 주에도 "메모리는 컨텍스트 엔지니어링(context engineering)의 다음 장"이라는 글들이 줄줄이 올라왔고, Mem0·Letta·Zep 같은 메모리 프레임워크 비교 글이 실무자들 사이에서 활발히 공유됐습니다. 2026년 들어 메모리는 "있으면 좋은 것"이 아니라, 벤치마크와 논문과 전용 생태계를 갖춘 독립된 엔지니어링 분야가 됐습니다.

이 글은 그 메모리를 기초부터 실전까지 한 바퀴 돕니다. 벡터 DB를 깊이 안 다뤄 본 주니어 개발자도 따라올 수 있게, 비유와 그림부터 시작하겠습니다.

1. 컨텍스트 윈도우는 '기억'이 아니다

"요즘 모델은 컨텍스트 윈도우(context window)가 100만 토큰이라던데, 그냥 대화 기록을 다 넣으면 되는 거 아닌가요?" — 가장 흔한 오해입니다.

컨텍스트 윈도우는 모델이 "이번 한 번의 호출에서" 한꺼번에 읽을 수 있는 토큰의 한도입니다. 사람으로 치면 책상 위에 펼쳐 둔 종이의 크기죠. 책상이 아무리 넓어도 세 가지 문제가 있습니다.

  • 비용: 입력 토큰은 돈입니다. 매 요청마다 수만 토큰짜리 대화 기록을 통째로 다시 넣으면, 대화가 길어질수록 호출 한 번의 가격이 계속 비싸집니다.
  • 지연(latency): 넣는 토큰이 많을수록 모델이 입력을 읽는 시간이 길어집니다. 첫 글자가 뜰 때까지 더 오래 기다리게 되죠.
  • 주의 분산: 토큰을 많이 넣는다고 모델이 그걸 다 잘 쓰는 건 아닙니다. 긴 입력의 한가운데 묻힌 정보는 자주 놓칩니다(흔히 'lost in the middle'이라 부릅니다). 책상이 종이로 뒤덮이면 정작 필요한 한 장을 못 찾는 것과 같습니다.

게다가 책상은 결국 치워집니다. 세션이 끝나거나, 대화가 한도를 넘겨 오래된 부분을 잘라내는 순간, 거기 있던 내용은 사라집니다. 즉 컨텍스트 윈도우는 작업용 책상이지, 서랍이나 캐비닛이 아닙니다. 진짜 '기억'은 호출이 끝나도 남아 있고, 다음에 필요할 때 다시 꺼내 쓸 수 있어야 합니다. 그걸 모델 바깥에 따로 만들어 주는 일이 바로 메모리 엔지니어링입니다.

핵심 한 줄: 컨텍스트 윈도우 = 매 호출마다 비워지는 작업 공간, 메모리 = 호출이 끝나도 남는 저장 공간. 메모리의 일은 "장기 저장소에서 지금 필요한 조각만 골라 작업 공간에 올려 주는 것"입니다.

2. 단기 기억과 장기 기억

사람의 기억을 빌려 오면 그림이 또렷해집니다. 우리에게는 방금 들은 전화번호를 잠깐 붙잡아 두는 단기 기억과, 몇 년 전 일도 떠올리는 장기 기억이 있죠. 에이전트도 똑같이 둘로 나눠 생각하면 편합니다.

  • 단기 기억(short-term memory) = 지금 진행 중인 대화. 곧 컨텍스트 윈도우 안에 들어 있는 내용입니다. 빠르고 정확하지만 좁고, 세션이 끝나면 사라집니다.
  • 장기 기억(long-term memory) = 세션을 넘어 계속 남는 저장소. 모델 바깥의 데이터베이스(주로 벡터 DB, 때로는 그래프 DB)에 보관하고, 필요할 때만 검색해서 단기 기억으로 끌어올립니다.

이 구조를 가장 먼저 깔끔하게 정리한 연구가 2023년의 MemGPT(현재 Letta 프로젝트로 이어짐)입니다. 핵심 비유가 운영체제(OS)예요. 컴퓨터에서 RAM은 빠르지만 작고, 디스크는 느리지만 거의 무한합니다. OS는 둘 사이에서 데이터를 부지런히 옮깁니다(paging). MemGPT는 LLM에게 똑같은 일을 시킵니다.

  • 메인 컨텍스트(main context) = RAM. 지금 컨텍스트 윈도우에 올라와 있는 내용.
  • 리콜 메모리(recall memory) = 검색 가능한 과거 대화 로그.
  • 아카이브 메모리(archival memory) = 사실을 쌓아 두는 거의 무한한 외부 저장소.

중요한 건 "무엇을 RAM에 올리고 무엇을 디스크로 내릴지"를 모델 스스로 함수 호출로 결정한다는 점입니다. 컨텍스트가 꽉 차 가면 오래된 내용을 아카이브로 내리고(page out), 옛날 약속이 필요해지면 검색해서 다시 올립니다(page in). 덕분에 작은 컨텍스트 윈도우로도 사실상 끝없는 대화를 이어 갈 수 있습니다.

3. 장기 기억의 세 종류 — 의미 · 일화 · 절차

장기 기억이라고 다 같지 않습니다. 인지심리학자 엔델 툴빙(Endel Tulving)이 1972년에 정리한 인간 기억의 분류를, 에이전트 메모리도 거의 그대로 빌려 씁니다. 세 종류로 나뉩니다.

  • 의미 기억(semantic memory) — "무엇이 사실인가". 사용자의 이름, 직업, 선호("매운 음식 못 먹음"), 정책·설정 같은 시간과 무관한 사실입니다. 사람으로 치면 "서울은 한국의 수도다"를 아는 것과 같죠.
  • 일화 기억(episodic memory) — "언제 무슨 일이 있었나". 특정 시점의 사건과 상호작용입니다. "지난주 화요일에 결제 오류로 환불을 요청했다" 같은, 타임스탬프가 붙은 경험입니다.
  • 절차 기억(procedural memory) — "어떻게 하는가". 학습된 행동 방식과 규칙입니다. "이 사용자에게는 코드 예시를 먼저 보여 준 뒤 설명한다" 같은, 반복으로 굳어진 일하는 방식이죠. 사람으로 치면 자전거 타는 법입니다.

왜 굳이 나눌까요? 저장하고 꺼내는 방식이 다르기 때문입니다. 의미 기억은 충돌하면 최신 사실로 덮어써야 합니다("이직했어요" → 직업 갱신). 일화 기억은 덮어쓰면 안 되고 시간순으로 쌓아야 합니다(과거 사건은 그대로 보존). 절차 기억은 보통 시스템 프롬프트나 규칙 형태로 녹여 넣습니다. 종류를 섞어 한 통에 담으면, 갱신해야 할 것과 보존해야 할 것이 뒤엉켜 엉뚱한 답이 나옵니다.

주니어 팁: 처음부터 셋을 다 만들 필요는 없습니다. 대부분의 서비스는 의미 기억(사용자 사실·선호) 하나만 잘 만들어도 체감 품질이 확 올라갑니다. 일화·절차는 필요해질 때 붙이세요.

4. 메모리는 어떻게 동작하나 — 쓰기 길과 읽기 길

이제 진짜 동작입니다. 메모리 시스템은 두 갈래 길로 이루어집니다. 대화를 기억으로 바꿔 쌓는 쓰기 길(write path), 그리고 쌓인 기억에서 지금 필요한 것만 꺼내는 읽기 길(read path)입니다.

쓰기 길은 보통 세 단계입니다. 대화 기록을 통째로 저장하는 게 아니라, 가공해서 핵심만 남깁니다.

  1. 추출(extract): LLM이 대화를 읽고 "기억할 만한 사실"만 짧은 문장으로 뽑습니다. "어제 점심에 떡볶이 먹고 속이 안 좋았는데, 사실 저 매운 거 잘 못 먹거든요" → 사용자는 매운 음식을 잘 못 먹는다. 잡담은 버립니다.
  2. 통합(consolidate): 새 기억을 기존 기억과 대조합니다. 똑같으면 버리고(중복 제거), 충돌하면 갱신하고("이제 매운 거 괜찮아졌어요" → 기존 사실 수정), 새로우면 추가합니다. 이 단계가 없으면 같은 사실이 수백 개씩 쌓입니다.
  3. 저장(store): 정리된 기억을 임베딩(embedding)으로 바꿔 벡터 DB에 넣습니다. 사람·주제 사이의 관계가 중요하면 그래프 DB를 같이 씁니다.

읽기 길은 두 단계입니다.

  1. 검색(retrieve): 지금 사용자의 입력을 임베딩으로 바꿔, 의미가 가까운 기억 몇 개를 벡터 DB에서 찾습니다. "이번 주말에 뭐 먹을까?"라는 질문에 사용자는 매운 음식을 잘 못 먹는다가 딸려 나오는 식입니다.
  2. 주입(inject): 찾아온 기억 몇 줄을 프롬프트(시스템 메시지 등)에 끼워 넣고 모델을 호출합니다. 전체 대화 기록이 아니라 관련 있는 몇 줄만 들어가는 게 핵심입니다.

흐름을 의사 코드로 보면 이렇습니다. 실제 엔진이 아니라 모양만 보여 주는 예시입니다.

{{< highlight python >}} # ── 쓰기 길: 대화 한 토막이 끝날 때마다 ── def remember(conversation, user_id): facts = llm_extract(conversation) # 1) 핵심 사실만 뽑기 for fact in facts: similar = db.search(embed(fact), user_id) # 2) 비슷한 기억 찾기 if is_duplicate(fact, similar): continue # 같은 사실이면 버림 elif conflicts(fact, similar): db.update(similar.id, fact) # 바뀐 사실이면 갱신 else: db.add(embed(fact), fact, user_id) # 3) 새 사실이면 저장 # ── 읽기 길: 사용자가 새 메시지를 보낼 때마다 ── def answer(user_msg, user_id): memories = db.search(embed(user_msg), user_id, top_k=5) # 가까운 기억 5개 prompt = f"[기억]\n{format(memories)}\n\n[질문]\n{user_msg}" # 몇 줄만 주입 return llm(prompt) # 전체 기록이 아니라 '관련 기억'만 {{< /highlight >}}

포인트는 두 가지입니다. 첫째, 대화를 그대로 쌓지 않고 사실로 가공합니다. 둘째, 매번 전부가 아니라 관련된 것 몇 개만 컨텍스트에 넣습니다. 이 두 가지가 비용·지연·정확도를 동시에 잡는 비결입니다.

5. RAG랑 뭐가 다른가요?

"검색해서 프롬프트에 넣는다"니까 RAG(retrieval-augmented generation, 검색 증강 생성)와 똑같아 보입니다. 실제로 기술 스택(임베딩 + 벡터 DB + 검색)은 많이 겹칩니다. 하지만 무엇을 기억하느냐가 다릅니다.

RAG에이전트 메모리
대상외부 지식 — 문서, 매뉴얼, 위키상호작용 — 사용자와 나눈 대화·사실
출처미리 준비된 정적 문서 더미대화하면서 그때그때 생기고 바뀜
갱신대체로 읽기 전용(문서를 다시 인덱싱)계속 추가·수정·삭제(살아 있음)
질문 예"환불 정책이 어떻게 되나요?""제가 저번에 요청한 환불 어떻게 됐죠?"

한 줄로 정리하면, RAG는 "세상에 대한 지식"을 검색하고, 메모리는 "이 사용자에 대한 기억"을 검색합니다. 그래서 둘은 경쟁 관계가 아니라 보완 관계입니다. 실전 에이전트는 보통 둘을 같이 씁니다. 회사 정책은 RAG로, 이 고객이 누구인지는 메모리로 가져와 한 프롬프트에 합치는 식이죠.

6. 숫자로 보는 효과

"그래서 대화 기록 다 넣는 것보다 정말 나아요?"를 숫자로 보겠습니다. 메모리 분야에서 가장 많이 인용되는 벤치마크가 LOCOMO입니다. 여러 세션에 걸친 긴 대화 10개(대화당 평균 약 600개 발화, 약 26,000토큰)에서 "이 사람에 대해 기억해야 답할 수 있는" 질문에 얼마나 잘 답하는지를 잽니다. LongMemEval, BEAM 같은 벤치마크도 함께 쓰입니다.

대표적인 메모리 프레임워크 Mem0의 논문(arXiv:2504.19413, ECAI 2025)이 보고한 결과가 인상적입니다. 대화 기록 전체를 그대로 욱여넣는 full-context 방식과 비교해, Mem0 같은 메모리 계층은 p95 지연을 약 91% 낮추고, 토큰 비용을 90% 넘게 줄였습니다. 정확도(LLM-as-Judge 기준)는 OpenAI의 내장 메모리보다 약 26% 더 높았고요. 즉 "더 싸고, 더 빠른데, 더 정확"할 수 있다는 뜻입니다.

주의: 벤치마크 숫자는 "메모리가 항상 정확도까지 이긴다"는 뜻이 아닙니다. 대화 전체를 다 넣으면 정확도 자체는 더 높을 때도 있습니다. 메모리의 진짜 가치는 "거의 비슷한 정확도를 1/10의 비용·지연으로" 얻는다는 데 있습니다. 대화가 길어질수록 이 차이는 더 벌어집니다.

7. 실전: Mem0로 다섯 줄 만에 기억 붙이기

이론은 충분합니다. 직접 붙여 봅시다. 오픈소스 메모리 라이브러리 Mem0를 쓰면, 앞서 본 추출·통합·저장·검색을 직접 구현하지 않고도 두 개의 메서드로 끝납니다. 기억을 쌓는 add()와 꺼내는 search()입니다.

{{< highlight python >}} # pip install mem0ai from mem0 import Memory m = Memory() # 기본은 로컬 벡터 DB로 동작 USER = "jihu" # 사용자별로 기억을 분리하는 키 # 1) 대화에서 기억 쌓기 — 추출·통합·저장을 Mem0가 알아서 처리 m.add( [ {"role": "user", "content": "저 매운 음식 잘 못 먹어요. 채식도 선호하고요."}, {"role": "assistant", "content": "알겠습니다, 순하고 채식 위주로 추천할게요."}, ], user_id=USER, ) # 2) 나중에, 완전히 다른 세션에서 — 관련 기억만 검색 hits = m.search("주말에 외식 뭐 할지 추천해줘", user_id=USER) for h in hits["results"]: print(h["memory"]) # → "매운 음식을 잘 못 먹는다" # → "채식을 선호한다" {{< /highlight >}}

핵심은 add()에 대화를 통째로 넘겨도, 저장되는 건 "매운 음식 못 먹음", "채식 선호" 같은 짧은 사실이라는 점입니다. 잡담은 알아서 걸러집니다. 그리고 며칠 뒤 다른 세션에서 "주말 외식"을 물으면, 단어가 하나도 안 겹쳐도 의미가 가까운 기억이 딸려 나옵니다.

실제 에이전트에 붙일 땐 이렇게 감쌉니다. 답하기 전에 search()로 기억을 끌어와 프롬프트에 넣고, 답한 뒤 add()로 이번 대화를 다시 기억에 쌓는 고리입니다.

{{< highlight python >}} def chat(user_msg, user_id): # ① 읽기: 관련 기억을 검색해 시스템 프롬프트에 주입 memories = m.search(user_msg, user_id=user_id, limit=5)["results"] facts = "\n".join(f"- {x['memory']}" for x in memories) system = f"다음은 사용자에 대해 기억하는 사실이다:\n{facts}" reply = llm(system=system, user=user_msg) # ② 모델 호출 # ③ 쓰기: 이번 대화도 기억으로 쌓기 (다음 세션을 위해) m.add( [{"role": "user", "content": user_msg}, {"role": "assistant", "content": reply}], user_id=user_id, ) return reply {{< /highlight >}}

이 짧은 고리만으로도 에이전트는 "어제 한 말을 오늘도 기억하는" 서비스가 됩니다. Mem0 외에 OS 비유를 끝까지 밀어붙인 Letta(MemGPT 후신), 시간 지식 그래프로 사실의 변화를 추적하는 Zep 같은 선택지도 있습니다. 한 줄 요약: 직접 구현하기 전에, 이미 잘 만들어진 메모리 계층부터 써 보세요.

8. 흔한 실수 · 팁

메모리를 처음 붙일 때 주니어가 자주 밟는 함정들입니다.

  • 대화를 통째로 저장한다. 가장 흔한 실수입니다. 원본 대화를 그대로 쌓으면 DB가 잡담으로 가득 차고, 검색 정확도가 떨어집니다. 반드시 짧은 사실로 추출해서 저장하세요.
  • 통합(중복·충돌 정리)을 건너뛴다. "사용자는 매운 걸 못 먹는다"가 50번 저장되거나, "이직했어요" 뒤에도 옛 직업이 그대로 남아 충돌합니다. 저장 전에 비슷한 기억과 대조하는 단계를 꼭 넣으세요. 대부분의 라이브러리는 이걸 기본 제공합니다.
  • 너무 많이 검색해 다시 컨텍스트를 채운다. 메모리를 쓰는 이유가 컨텍스트 절약인데, top_k를 50으로 잡으면 도로 아미타불입니다. 보통 3~7개면 충분합니다.
  • 잘못된 기억이 영구 박제된다. 사용자가 농담으로 한 말이나 일시적 사실이 "영구 진실"로 굳으면 두고두고 헛소리를 합니다. 기억에 삭제·수정·만료(TTL) 경로를 처음부터 만들어 두세요.
  • 프라이버시를 안 챙긴다. 메모리는 본질적으로 개인정보 저장소입니다. 반드시 user_id로 기억을 철저히 격리하고("A의 기억이 B에게 새지 않게"), 사용자가 자기 기억을 보고 지울 수 있게 하세요. GDPR의 '잊힐 권리'는 메모리 설계에서 선택이 아니라 필수입니다.
  • 메모리를 평가하지 않는다. "기억이 잘 되나"를 감으로 판단하면 조용히 망가집니다. 엉뚱한 기억을 불러오는지, 충돌을 제대로 갱신하는지 작은 회귀 테스트 세트로 점검하세요. LOCOMO 같은 공개 벤치마크의 발상을 빌려 와도 좋습니다.

9. 마무리 (TL;DR)

  • LLM은 상태가 없다. 대화가 이어지는 듯 보이는 건 매번 기록을 다시 넣기 때문이고, 그것만으로는 세션을 넘기면 다 잊는다.
  • 컨텍스트 윈도우 ≠ 메모리. 윈도우는 매번 비워지는 작업 책상, 메모리는 호출이 끝나도 남는 저장 공간이다.
  • 기억은 단기(컨텍스트)와 장기(외부 DB)로 나뉘고, 장기 기억은 다시 의미(사실)·일화(사건)·절차(방법) 세 종류로 갈린다.
  • 동작은 두 갈래 길. 쓰기 길(추출 → 통합 → 저장)로 대화를 사실로 가공해 쌓고, 읽기 길(검색 → 주입)로 관련 기억 몇 개만 꺼내 쓴다.
  • RAG는 "세상 지식", 메모리는 "이 사용자 기억". 경쟁이 아니라 보완. 실전에선 같이 쓴다.
  • 효과는 분명하다. 통째로 넣기 대비 토큰·지연을 90% 넘게 줄이면서 비슷하거나 더 나은 답을 낼 수 있다(Mem0/LOCOMO 기준).
  • 직접 만들기 전에 Mem0·Letta·Zep부터. add() / search() 두 메서드면 시작할 수 있다.
  • 함정은 정해져 있다. 통째 저장·통합 생략·과검색·잘못된 기억 박제·프라이버시·평가 누락 — 이 여섯 개만 피해도 절반은 성공이다.

모델은 계속 똑똑해지지만, "이 사용자가 누구인지"는 모델 안에 들어 있지 않습니다. 그건 우리가 바깥에 지어 줘야 하는 구조입니다. 좋은 에이전트와 그냥 그런 챗봇을 가르는 건, 점점 더 "무엇을 기억하고 무엇을 잊느냐"가 됩니다.
