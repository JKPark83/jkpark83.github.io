컨텍스트를 모델의 토큰(attention 윈도우)으로 다루지 말고, 프로그램이 다룰 수 있는 환경(변수)으로 다뤄라.

— Recursive Language Models의 한 줄 요약. 이 관점 전환 하나가 "1M 토큰인데도 답이 흔들리는" 문제를 정면으로 뒤집습니다.

요즘 모델의 스펙표를 보면 컨텍스트 윈도우가 100만(1M) 토큰까지 늘었다고 자랑합니다. 그런데 실제로 긴 문서를 잔뜩 넣고 질문해 본 적이 있다면 느꼈을 겁니다. 토큰을 많이 넣을수록 오히려 엉뚱한 답이 나오고, 중간에 파묻힌 핵심 문장을 놓친다는 걸요. 스펙은 1M인데, 현장에서는 5만 토큰만 넘어가도 품질이 흔들리기 시작합니다.

이 현상에는 이름이 붙었습니다. context rot(컨텍스트 로트), 즉 "컨텍스트가 길어질수록 그 안의 정보를 정확히 떠올리는 능력이 떨어지는" 현상입니다. 윈도우를 아무리 키워도 이 문제는 사라지지 않습니다. 오히려 커진 윈도우를 믿고 다 밀어넣을수록 더 심해지죠.

2026년 상반기 내내 실무자들의 화두가 된 해법이 하나 있습니다. 바로 재귀 언어 모델(RLM, Recursive Language Models)입니다. MIT CSAIL의 Alex Zhang이 2025년 10월 블로그로 처음 공개하고 연말에 논문(arXiv:2512.24601)으로 정리한 이 기법은, 최근 DSPy에 dspy.RLM 모듈로 정식 편입되고 Google의 ADK·Prime Intellect의 환경(Environments Hub)으로 번지면서 "2026년의 패러다임"이라는 별명까지 얻었습니다. 이번 글에서 왜 필요한지부터, 어떻게 도는지, 실제 코드로 어떻게 써 보는지, 그리고 어디서 발등을 찍는지까지 하나씩 풀어 보겠습니다.

1. 문제 — 윈도우는 1M인데 왜 답이 흔들릴까

먼저 왜 이게 문제인지부터 짚겠습니다. LLM은 프롬프트를 통째로 attention 연산에 넣습니다. 토큰 하나가 다른 모든 토큰을 "돌아보며" 관계를 계산하죠. 문제는 토큰이 늘수록 이 "돌아볼 대상"이 폭발적으로 많아진다는 점입니다. 신호(정답에 필요한 문장)는 그대로인데 잡음(상관없는 문장)만 계속 쌓이면, 모델이 핵심에 집중하기가 점점 어려워집니다.

결과가 context rot입니다. 유명한 "바늘 찾기(needle-in-a-haystack)" 실험을 넘어, 실제로 여러 정보를 종합해야 하는 어려운 질문에서는 컨텍스트가 길어질수록 정확도가 눈에 띄게 떨어집니다. 즉 "윈도우에 들어간다"와 "윈도우 안의 내용을 제대로 쓴다"는 전혀 다른 이야기입니다.

지금까지의 대응은 대체로 덜 넣기였습니다. RAG로 관련 청크만 검색해 넣거나, 오래된 대화를 요약(compaction)해 압축하거나, 컨텍스트를 정성껏 큐레이션하는 컨텍스트 엔지니어링이죠. 다 유효한 방법이지만 공통된 약점이 있습니다. 모델에게 넣기 전에, 사람이 만든 파이프라인이 "무엇이 중요한지"를 미리 정해 버린다는 점입니다. 정작 무엇이 중요한지 가장 잘 아는 건 질문을 받은 모델 자신인데 말이죠.

핵심 질문. "미리 잘라서 넣는" 대신, 모델이 스스로 방대한 원본을 뒤져 필요한 조각을 그때그때 꺼내 보게 하면 어떨까요? RLM은 바로 이 질문에서 출발합니다.

2. 배경 — 컨텍스트를 '토큰'으로 보는 관점의 한계

비유를 하나 들어 보죠. 1,000쪽짜리 보고서에서 "3분기 영업이익률과 그 원인"을 찾아야 한다고 합시다. 두 가지 방식이 있습니다.

  • 방식 A(밀어넣기): 1,000쪽을 처음부터 끝까지 눈으로 다 읽고 머릿속에 담은 뒤 답한다. 페이지가 많아질수록 앞부분을 잊고, 집중력이 흐려진다. — 지금의 롱컨텍스트 LLM.
  • 방식 B(탐색): 책은 책상 위에 그냥 둔다. 대신 검색창과 메모장을 손에 쥐고, "영업이익"이 나오는 쪽만 펴서 읽고, 필요하면 그 장(章)만 따로 정리해 메모한 뒤 답한다. — 이게 RLM의 발상입니다.

방식 B의 핵심은 원본 전체를 머리에 담지 않는다는 겁니다. 원본은 바깥(책상 = 변수)에 두고, 필요한 부분만 그때그때 꺼내 봅니다. 프로그래밍으로 옮기면 이렇습니다. 긴 프롬프트를 모델의 컨텍스트 윈도우에 넣는 대신, Python 변수에 담아 두고, 모델이 코드를 써서 그 변수를 들여다보게 하는 거죠. 마치 주피터 노트북에서 거대한 문자열이나 DataFrame을 변수에 올려 두고 len(), 슬라이싱, 검색으로 조금씩 살펴보듯이요.

3. 핵심 아이디어 — 컨텍스트를 '변수'로 두기

이제 용어를 정리하겠습니다. RLM에는 두 종류의 모델 호출이 등장합니다.

  • 루트 LM(root LM): 질문을 받고 REPL(Read-Eval-Print Loop, 코드를 한 줄씩 실행하고 결과를 돌려받는 대화형 실행 환경) 안에서 지휘하는 모델입니다. 긴 원본을 직접 읽지 않습니다. 대신 prompt라는 변수에 원본이 들어 있다는 걸 알고, 그 변수를 다룰 코드를 씁니다.
  • 서브 LM(sub-LM) 호출: 루트 LM이 원본에서 잘라 낸 조각과, 그 조각에 던질 질문을 넘겨 새 모델 인스턴스를 부르는 것입니다. 이 서브 호출은 백지 상태로 태어나 조각과 질문만 보고 답을 만든 뒤 사라집니다. 그 답은 루트의 REPL로 Python 값이 되어 돌아옵니다.

여기서 "재귀(recursive)"라는 이름의 뜻이 드러납니다. 서브 호출 역시 또 하나의 RLM일 수 있습니다. 조각이 여전히 너무 크면, 서브가 다시 그 조각을 변수로 두고 더 잘게 쪼개 자기 자신을 부릅니다. 큰 문제를 작은 문제로 쪼개고, 작은 문제를 또 쪼개는 분할 정복(divide and conquer)이 언어 모델 버전으로 구현된 셈이죠.

왜 이게 context rot을 피할까요? 어떤 단일 모델 호출도 짧은 컨텍스트만 본다는 게 핵심입니다. 루트는 원본 대신 "지금까지 REPL에서 관찰한 것"만 보고, 서브는 자기 조각만 봅니다. 전체가 1천만 토큰이어도, 개별 호출은 늘 모델이 편안하게 다루는 길이 안에 머뭅니다. 긴 입력은 코드가 훑고, 모델은 짧은 판단만 반복합니다.

비유. 루트 LM은 대용량 로그를 다루는 엔지니어와 같습니다. 로그 파일 전체를 눈으로 읽지 않죠. grep으로 걸러 보고, head로 앞부분만 엿보고, 수상한 구간만 따로 열어 봅니다. RLM은 이 습관을 언어 모델에게 그대로 가르친 것입니다.

4. 어떻게 도는가 — REPL 루프와 재귀 호출

루트 LM이 REPL에서 도는 과정을 단계로 풀면 이렇습니다.

  1. 엿보기(peek): len(prompt), prompt[:2000]처럼 원본의 크기와 앞머리, 구조를 먼저 파악합니다.
  2. 좁히기(slice/grep): 질문과 관련된 키워드를 찾아 그 주변만 잘라 냅니다. 예: prompt.find("영업이익") 후 인덱스 주변 슬라이싱.
  3. 위임(recurse): 잘라 낸 조각이 아직 크거나 정밀 분석이 필요하면, 그 조각을 서브 호출(재귀 RLM)에 넘겨 "이 조각에서 X를 뽑아 줘"라고 시킵니다.
  4. 모으기(aggregate): 여러 서브 호출이 돌려준 값들을 REPL 변수에 쌓아 두고, 마지막에 종합해 최종 답을 만듭니다.

중요한 점은 이 루프를 사람이 설계한 고정 파이프라인이 아니라 모델이 그때그때 코드로 만든다는 겁니다. 질문이 단순하면 한두 번의 슬라이싱으로 끝나고, 여러 곳을 종합해야 하면 서브 호출을 여러 개 병렬로 띄웁니다. 아래 의사코드로 흐름을 감 잡아 보세요.

{{< highlight python >}} # RLM이 내부적으로 하는 일 (개념을 보여 주는 의사코드) def rlm(prompt, question, depth=0): env = REPL(variables={"prompt": prompt}) # ① 긴 입력을 '변수'로 보관 (윈도우에 안 넣음) scratch = [] # 서브 호출 결과를 모아 둘 메모장 while not done: # ② 루트 LM이 '다음에 실행할 코드'를 직접 작성한다 code = root_lm.write_code(question, observation=env.tail(), notes=scratch) result = env.run(code) # ③ 코드 실행 — 슬라이싱/검색/집계 등 # ④ 조각이 여전히 크면 자기 자신을 재귀 호출 (분할 정복) # 실제로는 root_lm이 코드 안에서 아래 같은 호출을 직접 만들어 낸다: # answer = rlm(prompt[a:b], "이 구간에서 3분기 영업이익률을 찾아라", depth+1) if result.is_subcall: scratch.append(result.value) # 서브가 돌려준 '짧은 답'만 보관 done = root_lm.thinks_it_has_enough(scratch) # ⑤ 모아 둔 짧은 답들만으로 최종 응답 — 여기서도 원본 전체는 안 본다 return root_lm.final_answer(question, notes=scratch) {{< /highlight >}}

주목할 지점은 두 곳입니다. 하나, prompt는 처음부터 끝까지 변수로만 존재합니다. 둘, 루트가 실제로 "읽는" 것은 env.tail()(방금 실행한 코드의 결과)과 scratch(서브들이 돌려준 짧은 답)뿐입니다. 원본이 아무리 커도 루트의 컨텍스트는 짧게 유지됩니다.

5. 숫자로 보기 — OOLONG·BrowseComp-Plus

발상은 그럴듯한데, 실제로 효과가 있을까요? 논문은 두 가지 어려운 롱컨텍스트 벤치마크로 검증합니다. 숫자가 꽤 인상적입니다.

OOLONG은 가장 어려운 축에 드는 롱컨텍스트 벤치마크입니다. 여기서 작은 모델 GPT-5-mini에 RLM을 씌운 조합이, 큰 모델 GPT-5 단독보다 정답 개수를 2배 이상 맞혔습니다. 게다가 질문당 평균 비용은 오히려 더 쌌죠(최대 3배가량). 132k 토큰짜리 어려운 질의 구간에서도 이 우위가 유지됐습니다.

BrowseComp-Plus는 6~11백만(6~11M) 토큰짜리 입력을 다루는, 딥리서치 성격의 극단적 벤치마크입니다. 일반 모델은 입력이 윈도우를 한참 넘어 사실상 0%에 그친 반면, GPT-5 기반 RLM은 91.33%를 기록했습니다. 질문당 평균 비용은 약 $0.99였고요. 요컨대 "넣을 수조차 없던" 크기의 입력을, 코드로 훑으니 풀리기 시작한 겁니다.

왜 작은 모델이 큰 모델을 이길까? 큰 모델도 롱컨텍스트에서는 context rot을 겪습니다. RLM은 애초에 각 호출을 짧게 유지하니, "작지만 짧은 입력에서 또렷한" 모델이 "크지만 긴 입력에서 흐려지는" 모델을 앞지를 수 있습니다. 지능을 키우는 대신, 입력을 관리해서 이긴 셈이죠.

6. 실전 — DSPy의 dspy.RLM로 5줄 만에

다행히 이걸 직접 REPL 루프부터 짤 필요는 없습니다. DSPy가 3.1.2 버전대부터 dspy.RLM을 일급 모듈로 넣었습니다. 긴 컨텍스트를 프롬프트가 아니라 변수처럼 넘기면, 내부에서 샌드박스 REPL을 띄우고 재귀 탐색을 대신 돌려 줍니다. 최소 코드는 이렇게 짧습니다.

{{< highlight python >}} import dspy # 1) 평범한 LM 하나만 지정하면 됩니다. 작은 모델로도 충분합니다. dspy.configure(lm=dspy.LM("openai/gpt-5-mini")) # 2) RLM 모듈 생성. 시그니처는 "입력 -> 출력" 형태로 선언합니다. # context는 프롬프트가 아니라 '변수'로 취급됩니다. rlm = dspy.RLM("context, question -> answer") # 3) 30만 자짜리 문서를 통째로 넘겨도 됩니다. 자르지 않아도 돼요. huge_doc = open("annual_report_2025.txt").read() out = rlm(context=huge_doc, question="3분기 영업이익률과 그 원인은?") print(out.answer) # 내부에서 REPL·재귀 서브호출을 거쳐 나온 답 {{< /highlight >}}

여기서 dspy.RLM은 코드 실행을 샌드박스에서 합니다. 기본 인터프리터는 Deno + Pyodide(WASM) 런타임으로, 파일시스템·네트워크 접근이 막혀 있고 forward() 호출마다 새로 만들어졌다가 끝나면 폐기됩니다. 모델이 쓴 코드를 실행한다는 게 불안하게 들릴 수 있는데, 바로 이 격리 덕분에 임의 코드 실행의 위험을 상당 부분 눌러 둔 것입니다. (에이전트가 만든 코드를 안전하게 굴리는 샌드박스 격리 자체는 이 블로그의 코드 실행 샌드박스 글에서 더 깊게 다뤘습니다.)

보안·의존성 요구가 다르면 인터프리터를 바꿀 수 있습니다. 로컬에 신뢰된 패키지가 필요하면 LocalInterpreter를, 완전 격리가 필요하면 기본 WASM 샌드박스를 유지하는 식이죠. 핵심은 호출부 코드가 거의 그대로라는 점입니다. 컨텍스트를 넘기는 방식만 "프롬프트 문자열"에서 "변수"로 바뀝니다.

DSPy를 고른 이유. RLM은 프롬프트를 코드로 다루므로, "어떤 코드를 쓰게 할지"를 프롬프트로 일일이 지시하면 금세 지저분해집니다. DSPy는 시그니처("context, question -> answer")만 선언하면 프롬프트를 알아서 구성·최적화해 주는 프레임워크라, RLM 같은 코드-매개 방식과 궁합이 좋습니다. GEPA 같은 프롬프트 최적화와도 붙습니다.

7. 언제 쓰고, 무엇을 내주는가

RLM은 만능이 아닙니다. 공짜 점심도 아니고요. 언제 쓰면 이득이고 무엇을 대가로 내주는지 분명히 알아야 합니다.

상황RLM이 유리한가이유
입력이 모델 윈도우를 넘어선다 (수백만 토큰)✅ 매우 유리애초에 밀어넣을 수 없다. 변수로 두고 훑는 게 유일한 길에 가깝다.
윈도우엔 들어가지만 정보가 넓게 흩어져 종합이 필요✅ 유리여러 조각을 재귀로 종합. context rot을 피한다.
짧은 프롬프트, 단순 질의❌ 손해REPL·서브호출 오버헤드만 늘고 지연·비용이 커진다. 그냥 한 번 호출하라.
낮은 지연이 절대적으로 중요⚠️ 주의여러 왕복이 생겨 TTFT가 길어질 수 있다. 병렬화로 완화한다.

대가는 크게 셋입니다. 첫째, 지연(latency)입니다. 한 번의 forward에 코드 실행과 여러 서브 호출이 들어가니 왕복이 늘죠. 그래서 Google ADK 구현은 서브 호출을 대규모 병렬로 돌려 이 지연을 줄입니다. 조각들은 서로 독립적이라 병렬화가 잘 먹힙니다.

둘째, 비용의 성격입니다. 총비용 자체는 앞서 봤듯 오히려 낮을 수 있지만, 호출 횟수가 늘어 요금 청구·레이트리밋 관리가 복잡해집니다. 셋째, 디버깅입니다. 실행 경로가 "모델이 쓴 코드 + 재귀 트리"라 재현과 추적이 단발 호출보다 까다롭습니다. 그래서 관측성(트레이싱)이 특히 중요해집니다.

큰 그림. RLM은 요즘 흐름의 자연스러운 결론입니다. 컨텍스트 오프로딩(긴 내용을 파일·변수 같은 바깥에 두기), 서브에이전트(격리된 컨텍스트에 일 위임하기), 코드 실행(모델이 도구 대신 코드를 쓰기)이 하나로 합쳐진 형태거든요. "모델을 더 똑똑하게"가 아니라 "모델의 입력을 더 잘 관리해서" 성능을 얻는다는 2026년의 큰 방향과 정확히 맞닿아 있습니다.

8. 흔한 실수 · 팁

실수 ①: 짧은 입력에도 RLM을 쓴다. 몇천 토큰짜리 질문에 RLM을 씌우면 REPL·서브호출 오버헤드만 붙고 이득은 없습니다. RLM은 "밀어넣기가 아프기 시작하는 지점"부터 켜세요. 그 전까지는 평범한 단발 호출이 더 빠르고 쌉니다.
실수 ②: 샌드박스 없이 모델이 쓴 코드를 실행한다. RLM의 코어는 "모델이 만든 코드를 실행"하는 것입니다. 격리되지 않은 환경(로컬 셸, 프로덕션 서버)에서 그대로 돌리면 임의 코드 실행 취약점이 됩니다. 반드시 네트워크·파일시스템이 차단된 샌드박스(예: Pyodide WASM)에서 실행하고, 호출마다 새로 만들었다가 폐기하세요.
실수 ③: 재귀 깊이·서브호출 수에 상한을 안 둔다. 모델이 컨텍스트를 잘게 쪼개다 보면 서브 호출이 폭증하거나 재귀가 깊어져 비용·지연이 통제 불능이 될 수 있습니다. 최대 깊이, 서브호출 개수, 총 토큰 예산에 가드레일을 두세요. depth>1 재귀와 예산 제어는 실무 도입에서 반복적으로 등장하는 요구사항입니다.
실수 ④: 관측성 없이 프로덕션에 올린다. 실행 경로가 코드+재귀 트리라 "왜 이 답이 나왔는지"를 나중에 추적하기 어렵습니다. 각 서브 호출의 입력 조각·질문·반환값을 스팬으로 남겨 두지 않으면, 실패했을 때 손을 못 댑니다. 트레이싱을 처음부터 켜 두세요.
팁: RAG를 대체가 아니라 보완으로. RLM과 RAG는 경쟁 관계가 아닙니다. 후보 문서를 좁히는 데는 RAG(검색)를, 좁혀 온 방대한 원문을 정확히 종합하는 데는 RLM을 쓰는 식으로 겹쳐 쓸 수 있습니다. "검색으로 100개 문서 → RLM으로 통독·교차 검증"이 대표적인 결합 패턴입니다.

9. 마무리 — TL;DR

  • 문제 — 컨텍스트 윈도우는 1M까지 커졌지만, 길어질수록 정확도가 떨어지는 context rot 때문에 "들어간다"와 "제대로 쓴다"는 다릅니다.
  • 발상 — RLM은 긴 프롬프트를 윈도우에 밀어넣지 말고 변수로 두고, 모델이 코드로 필요한 부분만 잘라 보게 합니다.
  • 구조 — 루트 LM이 REPL에서 지휘하고, 조각마다 서브 LM을 재귀 호출합니다. 어떤 단일 호출도 짧은 컨텍스트만 보므로 rot을 피합니다.
  • 효과 — BrowseComp-Plus(6~11M 토큰)에서 일반 LLM 0% → RLM 91.3%. OOLONG에서 GPT-5-mini+RLM이 GPT-5 단독보다 정답 2배·비용은 더 낮음. 10M+ 토큰에서도 성능이 무너지지 않습니다.
  • 실전 — DSPy의 dspy.RLM(3.1.2+)로 5줄이면 시작합니다. 코드 실행은 Deno+Pyodide WASM 샌드박스에서 격리됩니다.
  • 주의 — 짧은 입력엔 오히려 손해. 샌드박스·재귀 깊이 가드레일·관측성은 필수. RAG와는 대체가 아니라 보완으로.
