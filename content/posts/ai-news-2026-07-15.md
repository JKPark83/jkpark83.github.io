# 📚 주목 논문

오늘은 "메모리를 어떻게 소비/공유하나"라는 한 축의 신선한(7/12–13 제출) 두 편.

### 1) The Compliance Trap — 충돌 메모리는 '첫 결정'에서 새어든다

- **저자:** Yixiong Chen, Xinyi Bai, Alan Yuille
- **arXiv:** [2607.10608](https://arxiv.org/abs/2607.10608) · cs.AI · 2026-07-12
- **핵심 기여:** 대부분의 메모리 연구가 "무엇을 쓰고·저장하고·검색하나"(공급 문제)에 머무는 반면, 이 논문은 모델이 **다단계 궤적에서 검색된 메모리를 어떻게 소비하는가**를 진단한다.
- **방법(2–3줄):** 궤적 단위 프레임워크 **E-P-R**(Entry–Propagation–Recovery)로, 메모리가 (1)어디서 처음 행동을 바꾸고 (2)그 변화가 이어지며 (3)정답 경로 이탈 후 회복 가능한지를 분리. WebArena와 통제 벤치 **MemTrapBench** 위에서 각 단계를 격리 측정.
- **결과(검증됨, 정성):** 실패는 주로 **entry에서 시작** — 첫 노출된 결정 지점에서 과제상 틀린 메모리도 그대로 채택한다. 반복 노출이 초기 오류를 증폭하고 회복은 약하다. 모델 무관하게 순응률은 비슷하지만, 순응하는 순간 성공률이 바닥으로 붕괴. **강한 에이전트일수록 순응 1회당 절대 손실이 크다.**
- 🛠 **에이전트 빌더 시사점:** 메모리를 검색 품질·최종 성공률로만 평가하지 말 것. **첫 결정 지점에 충돌 검사/거부 게이트**를 두고, 궤적 전체의 "소비 과정"을 계측하라.
- ⚠️ **한계:** 웹 내비게이션 태스크·통제 벤치 중심. 완화책 자체가 아니라 진단 프레임워크에 가깝다.

![Compliance Trap의 E-P-R 3단계: Entry에서 틀린 메모리 채택, Propagation에서 증폭, Recovery 약함, 순응 후 성공률 붕괴](/images/ai-news-2026-07-15/compliance-trap.svg)
*메모리 소비의 세 단계와 순응 함정 (개념 요약: arXiv 2607.10608, 생성 이미지)*

### 2) ToolAtlas — 재사용 가능한 툴 지식은 '에이전트'가 아니라 '툴 공급자'가 갖게 하라

- **저자:** Yue Fang, Zhibang Yang, Fangkai Yang 외
- **arXiv:** [2607.11126](https://arxiv.org/abs/2607.11126) · cs.LG · 2026-07-13
- **핵심 기여:** 파라미터 업데이트·프롬프트·**에이전트측 메모리**로 툴 사용을 개선하면 지식 공유가 어렵고 과거 관찰 행동에 갇힌다. 대신 **툴 공급자 측에 지속 메모리**를 두자는 제안.
- **방법:** 그래프 기반 프레임워크로 툴의 능력·실패 경계·툴 간 조합을 **실행 검증 프로빙**으로 축적. 추론 시 에이전트가 **적응적 그래프 순회**로 그 메모리를 질의.
- **결과(검증됨):** 8개 서비스에 걸친 2개 MCP 벤치마크에서 기존 툴측 최적화·에이전트측 메모리 대비 **pass@1 +21.61%·pass@4 +18.61%**. 같은 툴 메모리가 재학습·과제시점 탐색 없이 전이 — 환경 인스턴스 간 **+24.16%/16.22%**, 에이전트 프레임워크 간 **+17.49%/14.27%**(pass@1/pass@4).
- 🛠 **에이전트 빌더 시사점:** MCP 서버를 만든다면 **서버 자체가 "무엇을 잘하고·어디서 깨지고·어떻게 조합되는지"를 기억**하게 하라. 다운스트림 에이전트가 바뀌어도 재사용된다.
- ⚠️ **한계:** MCP 벤치 기반. 공급자측 메모리의 신선도 유지·프로빙 비용은 별도.

![ToolAtlas 상대 개선 막대: 벤치 내 pass@1 +21.61/pass@4 +18.61, 환경 전이 +24.16/16.22, 프레임워크 전이 +17.49/14.27](/images/ai-news-2026-07-15/toolatlas-gains.svg)
*공급자측 툴 메모리의 상대 향상과 전이 성능 (수치: arXiv 2607.11126, 생성 이미지)*

# 🧑‍💻 Codex & Claude Code 실전 팁

### 🟩 Codex — 모델 공급자를 Amazon Bedrock으로 붙이기 (category: Workflow)

- **소스:** [Codex changelog](https://developers.openai.com/codex/changelog)
- 💡 **핵심:** Codex가 이제 Amazon Bedrock을 통해 지원 OpenAI 모델을 사용할 수 있다. Bedrock을 모델 공급자로 설정하면 **AWS 관리형 인증·계정 통제·과금**으로 로컬에서 Codex 실행.
- 📝 **How-to:** 사내 AWS 거버넌스가 필수인 팀은 Codex 모델 공급자를 Bedrock으로 지정한다.

```toml
# Codex 설정: 모델 공급자를 Bedrock으로
[model_provider]
type = "bedrock"
region = "us-east-1"   # AWS 관리형 인증/과금 경로 사용
```

- 🎯 **언제:** 데이터 경계·SSO·비용 귀속을 AWS로 일원화해야 하는 조직.
- ⚠️ **주의:** 사용 가능한 모델·리전은 Bedrock 측 지원에 종속. 최신 프런티어 기능이 즉시 안 올 수 있다.
- ⭐ **추천:** 엔터프라이즈 도입 시 우선 검토. 개인 실험엔 기본 경로가 간단.

### 🟦 Claude Code — DeepSeek V4를 Anthropic 포맷으로 물려 쓰기 (category: Config)

- **소스:** [TheRouter — DeepSeek Anthropic 라우팅](https://therouter.ai/news/deepseek-chat-reasoner-deprecation-v4-migration-routing/)
- 💡 **핵심:** DeepSeek이 Anthropic 포맷 엔드포인트를 추가해, Claude Code 같은 Anthropic 네이티브 툴에서 DeepSeek V4를 그대로 붙일 수 있다. 오늘의 7/24 마이그레이션과 직결.
- 📝 **How-to:** 환경 변수로 베이스 URL·모델을 지정한다. **레거시 별칭 대신 v4 정식 이름**을 쓴다.

```bash
export ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic"
export ANTHROPIC_AUTH_TOKEN="$DEEPSEEK_API_KEY"
export ANTHROPIC_MODEL="deepseek-v4-pro"   # deepseek-chat/reasoner 금지 (7/24 종료)
claude
```

- 🎯 **언제:** 비용 절감·모델 다변화, 또는 7/24 종료 전에 명시 모델명으로 갈아탈 때.
- ⚠️ **주의:** 강한 추론이 필요하면 flash 계열이 아닌 `deepseek-v4-pro`+thinking를 명시. 툴콜/포맷 호환은 워크로드별로 먼저 검증.
- ⭐ **추천:** 서브 태스크·대량 배치를 저가 모델로 오프로드하는 라우팅에 특히 유효.

### 🟪 공통 — 메모리 eval에 'entry 게이트'를 추가하라 (category: Eval)

- **소스:** [arXiv 2607.10608](https://arxiv.org/abs/2607.10608)
- 💡 **핵심:** 충돌 메모리 피해는 첫 결정 지점(entry)에서 시작된다. 그 지점에 **채택 전 검증 게이트**를 두면 전파·붕괴를 원천 차단한다.
- 📝 **How-to:** 검색된 메모리를 행동에 반영하기 전에 "현재 과제 상태와 모순되나?"를 1회 질의하고, 모순이면 메모리를 폐기하거나 낮은 가중치로 강등.

```text
retrieve → [entry gate: 현재 목표/상태와 충돌 검사] → 통과 시만 행동 반영
eval 지표: 충돌-메모리 채택률 · 이탈 후 회복률 (최종 성공률과 분리 계측)
```

- 🎯 **언제:** 과거 경험을 재사용하는 장기 실행·웹 내비게이션 에이전트.
- ⚠️ **주의:** 게이트가 과하면 유용한 메모리까지 버린다. 임계값은 도메인별 튜닝.
- ⭐ **추천:** 강한 모델일수록 순응 손실이 크므로, 상위 모델 배포 시 먼저 넣을 것.
